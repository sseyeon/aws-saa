## SQS, SNS, Kinesis, Amazon MQ

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **동기식 통신으로 백엔드가 압도당해 다운되는 문제 해결** $\rightarrow$ `SQS로 애플리케이션 Decouple(분리)`
* **트래픽 폭주 시 데이터베이스 쓰기(Write) 유실 방지 및 버퍼링** $\rightarrow$ `SQS를 DB 앞단에 버퍼로 배치`
* **SQS 기반 오토 스케일링 트리거 지표** $\rightarrow$ `CloudWatch Alarm + ApproximateNumberOfMessages 지표`
* **소비자가 메시지를 처리하는 동안 다른 소비자가 못 보게 숨기는 시간** $\rightarrow$ `Visibility Timeout (기본 30초)`
* **API 호출 횟수를 줄여 SQS 비용을 획기적으로 아끼고 싶다** $\rightarrow$ `Long Polling (WaitTimeSeconds=20)`
* **은행 거래나 주문처럼 중복 절대 안 되고 무조건 순서가 중요함** $\rightarrow$ `SQS FIFO Queue (Exactly-once, 순서 보장)`
* **하나의 이벤트를 여러 독립 서비스로 동시 전송 (팬아웃)** $\rightarrow$ `SNS + SQS Fan-out 패턴`
* **동일한 S3 이벤트 알림을 여러 대기열/엔드포인트로 전송** $\rightarrow$ `S3 → SNS → 다수의 SQS 팬아웃`
* **금융 거래처럼 순서 + 중복 제거가 모두 필요한 팬아웃** $\rightarrow$ `SNS FIFO Topic + SQS FIFO Queue 조합`
* **특정 조건에 맞는 구독자에게만 메시지 선별 전달** $\rightarrow$ `SNS Message Filtering (JSON 필터 정책)`
* **앱 로그, IoT, 웹 클릭 스트림 대용량 실시간 수집 및 재생(Replay)** $\rightarrow$ `Kinesis Data Streams`
* **실시간 데이터를 S3, Redshift, OpenSearch로 자동 적재 (변환 없이)** $\rightarrow$ `Kinesis Data Firehose`
* **Firehose 통과 중 데이터 형식 변환(ETL)** $\rightarrow$ `Kinesis Data Firehose + Lambda`
* **빅데이터 스트림에서 장비/트럭별 데이터 순서 완벽 보장** $\rightarrow$ `Kinesis 파티션 키(Partition Key) 활용`
* **온프레미스 RabbitMQ / ActiveMQ를 AWS로 마이그레이션 (코드 수정 없이)** $\rightarrow$ `Amazon MQ`
* **MQTT, AMQP, STOMP 프로토콜을 그대로 유지** $\rightarrow$ `Amazon MQ`
* **Amazon MQ 고가용성 + 장애 시 데이터 유실 없이 전환** $\rightarrow$ `Amazon MQ + Amazon EFS 공유 마운트`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📌 0. 왜 애플리케이션을 분리(Decouple)해야 할까?

* **동기(Synchronous) 통신:** 프론트엔드가 백엔드에게 요청하고 끝날 때까지 기다리는 구조입니다. 평소에는 괜찮지만 갑자기 요청이 10개에서 1,000개로 늘어나면 백엔드가 과부하로 터집니다. 백엔드가 죽으면 프론트도 같이 죽는 운명공동체 구조입니다.
* **비동기(Asynchronous) 통신:** 중간에 안전한 '우체통'을 두는 것입니다. 프론트는 우체통에 메시지만 쏙 넣고 자기 할 일을 합니다. 백엔드는 체력(CPU)이 허락하는 만큼만 메시지를 꺼내 처리합니다. 이 우체통 역할을 하는 삼총사가 **SQS(대기열), SNS(발행/구독), Kinesis(실시간 대용량 스트리밍)**입니다.

### 📌 1. Amazon SQS Standard Queue (기본 우체통)

AWS에서 가장 역사가 깊고 강력한 완전 관리형 대기열 서비스입니다.

| 속성 | 내용 |
| --- | --- |
| 처리량 | 무제한 (메시지 수 제한 없음) |
| 메시지 크기 | 최대 256 KB |
| 보존 기간 | 기본 4일, 최대 14일 설정 가능 |
| 전송 지연 | 10ms 미만 |
| 전송 방식 | At-least-once (중복 발생 가능) |
| 순서 보장 | Best-effort ordering (순서 미보장) |

#### SQS Producer / Consumer 흐름

* **생성자(Producer):** `SendMessage` API로 메시지를 대기열에 밀어 넣습니다.
* **소비자(Consumer):** EC2 인스턴스 또는 Lambda가 폴링(Polling)하여 한 번에 최대 10개 메시지를 가져와 처리합니다.
* **★ 핵심 규칙:** 소비자는 처리 완료 후 반드시 `DeleteMessage` API로 메시지를 직접 삭제해야 합니다. 삭제하지 않으면 다른 소비자가 동일 메시지를 다시 가져가는 중복이 발생합니다.

#### SQS + ASG 오토 스케일링 조합

* **트리거 지표:** CPU 사용량이 아닌 `ApproximateNumberOfMessages` (대기열 메시지 수)를 기준으로 삼습니다.
* **흐름:** 주문 폭주로 대기열에 메시지가 대량 쌓임 → CloudWatch 경보 발동 → ASG가 EC2 인스턴스(소비자)를 Scale-out → 적체된 메시지 빠르게 처리.

#### SQS Write Buffer (DB 앞단 쓰기 방패)

* **문제:** 대규모 트래픽이 DB에 직접 쓰기를 시도하면 DB가 과부하로 다운되거나 트랜잭션이 유실됩니다.
* **해결:** 애플리케이션이 DB에 직접 쓰지 않고 SQS 대기열에 먼저 저장합니다. SQS 처리량은 무제한이라 유실이 없으며, 백엔드 일꾼이 DB가 소화 가능한 속도로 차근차근 밀어 넣어 유실률 0%를 달성합니다.

### 📌 3. Visibility Timeout (메시지 숨바꼭질 시간)

소비자 A가 메시지를 가져가는 순간 SQS가 그 메시지를 **숨겨서(Invisible)** 다른 소비자가 중복으로 가져가지 못하게 차단합니다. 기본값은 **30초**입니다.

* **정상 종료:** A가 30초 이내에 처리하고 `DeleteMessage`를 날리면 메시지 영구 소멸.
* **시간 초과 시:** 30초가 지나면 SQS가 메시지를 다시 노출시키고, 다른 소비자가 동일 메시지를 중복 처리하는 위험 발생.
* **해결책:** 처리 시간이 길어질 것 같으면 `ChangeMessageVisibility` API를 호출하여 숨김 시간을 연장합니다.
* **튜닝 포인트:** 너무 길면 실제 장애 시 복구가 늦어지고, 너무 짧으면 정상 처리 중에도 중복이 발생합니다.

### 📌 4. Long Polling (진득하게 기다리기)

* **짧은 폴링(Short Polling):** 메시지가 없어도 계속 "왔어? 왔어?" 하며 API를 두드려 요금 폭탄 유발.
* **긴 폴링(Long Polling):** 메시지가 없으면 최대 20초간 연결을 열어둔 채로 대기하다가 메시지가 오면 즉시 수신합니다.
* **장점:** API 호출 수 극감 → 비용 최소화, 효율성 향상, 지연 감소. `WaitTimeSeconds` 파라미터로 설정합니다. AWS는 20초 설정을 강력 권장합니다.

### 📌 5. SQS FIFO Queue (절대 순서 보장 우체통)

Standard Queue의 단점(순서 뒤바뀜, 중복 가능)을 완벽 해결한 고급 대기열입니다.

| 항목 | Standard | FIFO |
| --- | --- | --- |
| 전송 방식 | At-least-once (중복 가능) | Exactly-once (중복 제거) |
| 순서 보장 | Best-effort (미보장) | 완전 보장 |
| 기본 처리량 | 무제한 | 초당 300개 |
| 배치 처리량 | 무제한 | 초당 최대 3,000개 |

> **⚠️ 오답 함정:** FIFO는 처리량이 제한됩니다. "무제한 처리량이 필요하다"면 Standard가 정답입니다.

### 📌 6. Amazon SNS (1:N 브로드캐스트 방송국)

SNS는 하나의 메시지를 여러 구독자에게 동시에 뿌리는 **Pub/Sub 모델**입니다.

* **스케일 제한 (필수 암기):** 채널(주제)당 최대 **1,250만 명** 구독자, 생성 가능 채널 최대 **10만 개**.

#### SNS 구독 가능 대상 (Subscribers)

| 대상 유형 | 구체적 대상 |
| --- | --- |
| 인간/외부 | 이메일, SMS, 모바일 푸시, HTTP/HTTPS 엔드포인트 |
| AWS 내부 서비스 | SQS, Lambda, Kinesis Data Firehose |

#### SNS Security

SQS와 동일한 메커니즘입니다. HTTPS(전송 중) + KMS(저장 중) + IAM 정책 + SNS Access Policies(교차 계정 / 다른 AWS 서비스 허용).

### 📌 7. SNS + SQS Fan-out 패턴 (★시험 출제율 100%)

메시지를 여러 SQS에 동시 전달하려고 앱이 각 SQS에 직접 전송하면 중간 에러 시 데이터 유실 위험이 있습니다. 이때 **SNS 아래에 여러 SQS를 구독자로 연결하는 팬아웃 패턴**을 사용합니다.

* **구조:** `앱 → SNS 주제 → [SQS A, SQS B, SQS C, Lambda, ...]`
* **장점 1 (유실 방지):** SNS가 한 번 푸시하면 SQS가 버퍼로 메시지를 보존해 백엔드 장애 시에도 재시도 가능.
* **장점 2 (자유로운 확장):** 새 SQS를 만들어 SNS 주제에 구독만 추가하면 되므로 기존 코드 수정 불필요.
* **장점 3 (리전 돌파):** 다른 리전의 SQS도 구독 가능.
* **주의:** SQS Access Policy에서 해당 SNS의 쓰기를 허용해야 합니다.

#### S3 이벤트 팬아웃 (S3 구조적 한계 극복)

* **문제:** 동일한 S3 이벤트 조건(이벤트 유형 + 접두사 조합)의 목적지는 딱 1개만 지정 가능.
* **해결:** S3 이벤트 목적지를 SNS로 설정 → SNS 아래 다수의 SQS/Lambda를 구독자로 연결.

### 📌 8. SNS FIFO Topic & Message Filtering

**SNS FIFO Topic:**
* 순서 보장 + 중복 제거 기능을 제공하는 고급 채널.
* **★ 철칙:** SNS FIFO의 구독자는 오직 **SQS FIFO 대기열만** 가능합니다. 일반 SQS, 이메일 등은 구독 불가.

**SNS Message Filtering:**
* 구독자가 JSON 필터 정책(Filter Policy)을 등록하면, SNS가 메시지 속성을 확인하여 조건에 맞는 구독자에게만 선별 전달합니다.
* 필터 정책이 없는 구독자는 모든 메시지를 그대로 수신합니다.

### 📌 9. Amazon Kinesis Data Streams (불변의 실시간 데이터 파이프)

IoT 장비, 클릭 로그, 시스템 메트릭 등 쉴 새 없이 쏟아지는 스트리밍 데이터를 실시간 수집/처리/보존하는 파이프라인입니다.

| 속성 | 내용 |
| --- | --- |
| 데이터 보존 | 1일 ~ 365일 |
| 불변성 | 삽입 후 수정/삭제 불가 |
| 순서 보장 | 동일 파티션 키 → 동일 샤드 → 순서 보장 |

#### 용량 모드 비교 (Capacity Modes)

| 모드 | 관리 방식 | 입력 | 출력 |
| --- | --- | --- | --- |
| Provisioned | 샤드 수 수동 설정, 샤드당 시간 요금 | 샤드당 1MB/s 또는 1,000건/s | 샤드당 2MB/s |
| On-demand | 자동 스케일링, 데이터량 기준 요금 | 기본 4MB/s 또는 4,000건/s | 자동 조절 |

#### Kinesis Data Streams vs SQS FIFO 순서 보장 비교

* **Kinesis:** 파티션 키로 동일 샤드에 정렬. 샤드 5개면 병렬 소비자 최대 5개로 제한. 빅데이터/대용량/샤드 기반 정렬에 적합.
* **SQS FIFO:** 그룹 ID(Message Group ID)로 동적 그룹 생성. 트럭 100대 → 그룹 100개 → 소비자 최대 100개까지 병렬 확장 가능. 중소량이지만 소비자를 유연하게 늘려야 할 때 적합.

### 📌 10. Kinesis Data Firehose (자동 스케일링 데이터 배달부)

스트리밍 데이터를 받아 AWS 저장소로 자동 적재하는 완전 관리형 서버리스 서비스입니다. 샤드 개념 없이 통과 데이터양만큼 요금을 냅니다.

| 항목 | 내용 |
| --- | --- |
| 연결 가능 목적지 | S3, Redshift, OpenSearch, Splunk, MongoDB, DataDog, HTTP 엔드포인트 |
| 지연 시간 | 최소 60초 지연 또는 최소 1MB 버퍼 누적 후 전송 (Near Real-Time) |
| 변환 | Lambda 연동으로 커스텀 데이터 변환 가능 |
| 백업 | 실패하거나 원본이 필요하면 S3 버킷으로 흘려보낼 수 있음 |

### 📌 11. Amazon MQ (레거시 구원자)

온프레미스에서 RabbitMQ, ActiveMQ를 쓰던 기업이 AWS로 마이그레이션할 때 코드를 단 한 줄도 바꾸지 않고 이사할 수 있게 해주는 관리형 서비스입니다.

| 항목 | SQS / SNS | Amazon MQ |
| --- | --- | --- |
| 프로토콜 | AWS 독점 API | AMQP, MQTT, STOMP, OpenWire, WSS |
| 확장성 | 무제한 서버리스 | 서버 기반 (무제한 확장 불가) |
| 사용 시나리오 | 클라우드 네이티브 신규 개발 | 레거시 마이그레이션 |
| 기능 | Queue(SQS 유사) 또는 Topic(SNS 유사) 별도 | Queue + Topic 동시 제공 |

#### Amazon MQ 고가용성 아키텍처 (Active / Standby)

서버 기반이므로 단일 서버 장애 시 전체 서비스가 마비됩니다. **EFS를 활용한 Multi-AZ HA 아키텍처**가 시험에 자주 출제됩니다.

* **구성:** Active 브로커(AZ-A) + Standby 브로커(AZ-B) + 공유 Amazon EFS 마운트.
* **평소:** Active 브로커가 클라이언트와 통신하며 모든 데이터를 EFS에 실시간 저장.
* **장애 시:** AZ-A가 다운되면 AZ-B의 Standby 브로커가 Active로 전환. EFS에 데이터가 공유되어 있으므로 **데이터 유실 없이** 서비스 계속.
