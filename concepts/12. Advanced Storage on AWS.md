# 🗄️ Advanced Storage on AWS 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **네트워크 전송 시 일주일 이상 걸리는 대용량 이사** $\rightarrow$ `Snowball Edge (물리 장치 사용이 정답)`
* **2.1kg 초소형, 공간이 매우 협소한 환경, 8TB HDD 또는 14TB SSD** $\rightarrow$ `Snowcone`
* **Snowcone에서 배터리·케이블 준비** $\rightarrow$ `직접 준비 필요 (AWS가 제공 안 함)`
* **80TB HDD, 데이터 이사 주목적** $\rightarrow$ `Snowball Edge Storage Optimized`
* **42TB HDD 또는 28TB NVMe, GPU 옵션, 현장 ML·렌더링** $\rightarrow$ `Snowball Edge Compute Optimized`
* **엑사바이트급, 트럭 배송, 100PB/대, 10PB 이상** $\rightarrow$ `Snowmobile`
* **인터넷 불가 환경에서 EC2/Lambda 구동, IoT Greengrass** $\rightarrow$ `Snow Family 엣지 컴퓨팅`
* **Snowball 데이터를 Glacier에 저장하는 방법** $\rightarrow$ `S3로 먼저 업로드 → S3 수명주기 정책으로 Glacier 전환 (Snowball에서 Glacier 직행 불가)`
* **Snow 장치 GUI 관리 도구, CLI 노가다 없이 마우스 클릭으로 관리** $\rightarrow$ `OpsHub`
* **Windows AD 통합, SMB+NTFS 프로토콜, 사내 Z 드라이브 클라우드 이사, Multi-AZ** $\rightarrow$ `FSx for Windows File Server`
* **Linux+Cluster, HPC·ML·렌더링, S3 완벽 통합(읽기/쓰기), sub-ms 지연** $\rightarrow$ `FSx for Lustre`
* **Lustre 임시(빠름, 복제 없음) vs 장기(복제, 데이터 보존)** $\rightarrow$ `Scratch vs Persistent 파일 시스템`
* **NFS+SMB+iSCSI 멀티OS, 데이터 중복 제거(Deduplication), 온프레미스 NetApp 이전** $\rightarrow$ `FSx for NetApp ONTAP`
* **ZFS 마이그레이션, 0.5ms 미만, 100만 IOPS, 중복 제거 없음** $\rightarrow$ `FSx for OpenZFS`
* **중복 제거(Deduplication)로 비용 절감** $\rightarrow$ `FSx for NetApp ONTAP (OpenZFS는 중복 제거 없음!)`
* **온프레미스 서버의 NFS/SMB 파일 → S3에 저장, 로컬 캐시** $\rightarrow$ `S3 File Gateway`
* **SMB/NTFS/AD 온프레미스 → FSx for Windows로 연결** $\rightarrow$ `FSx File Gateway`
* **iSCSI 블록 스토리지, 전체 데이터 S3 + 최근 데이터만 로컬 캐시** $\rightarrow$ `Volume Gateway Cached`
* **iSCSI 블록 스토리지, 전체 데이터 온프레미스 + S3 비동기 백업 (인터넷 끊겨도 접근 가능)** $\rightarrow$ `Volume Gateway Stored`
* **레거시 백업 소프트웨어(Veritas/Commvault) + iSCSI + 물리 테이프 대체** $\rightarrow$ `Tape Gateway`
* **VMware 없는 환경, 가상화 하이퍼바이저 없어서 소프트웨어 못 깐다** $\rightarrow$ `Hardware Appliance (amazon.com에서 구매)`
* **FTP/FTPS/SFTP 프로토콜 그대로 유지, 뒷단은 S3/EFS에 저장** $\rightarrow$ `AWS Transfer Family`
* **스케줄 기반(매시간/매일/매주), 온프레미스→AWS 대량 이동, 파일 권한·메타데이터 보존** $\rightarrow$ `AWS DataSync`
* **DataSync 온프레미스 사용 시** $\rightarrow$ `에이전트(Agent) 설치 필요`
* **DataSync AWS 서비스 간(S3↔EFS↔FSx) 이동 시** $\rightarrow$ `에이전트 불필요`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🚚 1. AWS Snow Family

테라바이트(TB), 페타바이트(PB) 단위를 넘어서면 인터넷 랜선으로 보내는 것보다 하드디스크를 트럭에 싣고 달리는 게 훨씬 빠릅니다. 이 패러다임을 이해하면 Snow Family 전체가 명확해집니다.

#### 네트워크 전송의 한계와 Snow Family가 필요한 이유

* **시간이 너무 오래 걸림:** 100TB를 일반 네트워크로 올리려면 몇 달이 걸릴 수 있습니다.
* **대역폭 제한 및 비용:** 네트워크 라인 전체를 풀 가동하면 비용이 폭발하고 회사 내 다른 대역폭을 다 잡아먹습니다.
* **불안정한 연결성:** 기껏 올리다가 네트워크가 끊기면 대참사가 발생합니다.

> **SAA 시험 합격 기준점:** 만약 네트워크를 통한 데이터 전송 계산 결과가 **"일주일(1주일) 이상"** 걸린다고 나오면, 무조건 네트워크 업로드를 포기하고 **AWS Snowball 장치를 요청하는 것이 정답**입니다.

#### 데이터 마이그레이션 삼총사 (용량별 비교)

**① AWS Snowcone & Snowcone SSD (초소형)**

* 무게가 단 **2.1kg(4.5파운드)** 밖에 안 되는 작고 가벼운 휴대용 장치입니다.
* 거친 환경(진동, 먼지 등)도 잘 견디도록 견고하게 설계되었습니다.
* 용량: 기본 Snowcone **8 TB HDD** / Snowcone SSD **14 TB SSD**
* 주의점: 배터리와 연결 케이블은 AWS가 제공하지 않습니다. **직접 준비**해야 합니다.
* 데이터 전송 방식: 작업 후 AWS로 택배를 보내거나, 현장에서 네트워크에 연결해 **AWS DataSync**로 데이터를 전송할 수도 있습니다.

**② AWS Snowball Edge (중형 / 만능 일꾼)**

물리적으로 TB~PB 단위의 데이터를 AWS 안팎으로 이동시키는 핵심 장치입니다. 용도에 따라 두 가지 모델로 나뉩니다.

| 모델 | 용량 | 주 목적 |
| --- | --- | --- |
| Storage Optimized (스토리지 최적화) | 80 TB HDD | 데이터 이사 주목적, S3 호환 객체 스토리지·블록 볼륨 |
| Compute Optimized (컴퓨팅 최적화) | 42 TB HDD 또는 28 TB NVMe | 이사하면서 고성능 연산, 선택적 GPU (ML/비디오 처리) |

**③ AWS Snowmobile (대형 / 초대형 트럭)**

* 용량: **엑사바이트(EB)급** — 트럭 한 대당 **100 PB**를 담을 수 있고, 여러 대를 병렬로 세울 수 있습니다.
* 특징: 장치가 아니라 **거대한 컨테이너 트럭**이 회사 앞에 주차합니다. GPS 추적, 24시간 영상 감시, 온도 조절 등 첩보 영화급 보안 시설이 가동됩니다.
* 기준: 최소 **10 PB 이상의 데이터** 마이그레이션 시 Snowball 여러 대 쓰는 것보다 훨씬 탁월합니다.

#### Snow Family 이용 프로세스 (A to Z)

1. **배송 요청:** AWS 콘솔에서 Snowball을 신청합니다.
2. **연결 및 복사:** 장비가 사무실에 도착하면 서버에 연결합니다. Snowball 클라이언트나 AWS OpsHub 소프트웨어를 설치하고 파일을 장비 안으로 복사합니다.
3. **반송:** 복사가 끝나면 장비 겉면에 달린 전자잉크 디스플레이 주소가 AWS 센터로 자동으로 바뀝니다. 그대로 택배(반송)로 보냅니다.
4. **S3 로드 및 파기:** AWS 직원이 장비를 받아 **S3 버킷에 데이터를 안전하게 로드**합니다. 업로드가 끝나면 장비 내 데이터는 완전하게 삭제(Wipe) 처리됩니다.

#### 엣지 컴퓨팅(Edge Computing): "클라우드랑 연락이 안 돼요"

인터넷이 아예 안 터지거나 클라우드와 너무 먼 곳(예: 사막 위의 트럭, 태평양 한가운데의 군함, 지하 깊은 탄광)에서 현장 데이터를 즉시 처리해야 할 때, Snow 장비를 가져다 놓고 서버처럼 씁니다.

* **Snowcone / Snowcone SSD:** 2 vCPU, 4GB 메모리. USB-C 전원이나 배터리로 구동됩니다.
* **Snowball Edge Compute Optimized:** 무려 **104개 vCPU, 416 GiB RAM**, 머신러닝/비디오 처리를 위한 선택적 GPU까지 달 수 있습니다.
* **Snowball Edge Storage Optimized:** **최대 40 vCPU, 80GB RAM**, 여러 대를 묶어 객체 스토리지 클러스터링을 할 수 있습니다.
* 공통 사양: 모든 Snow 장비는 **AWS IoT Greengrass**를 사용하여 장비 안에서 직접 **EC2 인스턴스**나 **AWS Lambda 함수**를 실행할 수 있습니다.

#### AWS OpsHub: CLI 노가다는 끝났다

과거에는 Snow 장비를 조작하려면 CLI로 명령어를 쳐야 했습니다. 현재는 노트북에 **AWS OpsHub**라는 편리한 GUI 프로그램을 설치해서 마우스 클릭 몇 번으로 편하게 관리합니다.

* 단일 장치 또는 클러스터 장치의 잠금 해제 및 초기 구성
* 직관적인 파일 전송 (드래그 앤 드롭)
* Snow 장치 안에서 EC2 인스턴스 켜고 끄기
* 저장 용량, 활성화된 인스턴스 개수 등 장치 메트릭 모니터링
* AWS DataSync, NFS 등 AWS 호환 서비스 실행

#### ⚠️ 단골 아키텍처 문제: Snowball에서 Glacier로 직행하기? (X)

* **시험 함정:** "대용량 아카이브 데이터를 비용을 아끼기 위해 Snowball을 이용해 **직접 Glacier 스토리지로 업로드**한다." $\rightarrow$ **틀린 보기**입니다!
* **정답 메커니즘:** Snowball은 오직 **Amazon S3로만** 데이터를 넣을 수 있습니다. 일단 Snowball을 통해 데이터를 **Amazon S3 버킷으로 먼저 가져온 뒤**, S3의 수명 주기 정책(Lifecycle Policy)을 세팅하여 객체가 자동으로 Glacier로 전환(Transition)되도록 설계해야 합니다.

### 💾 2. Amazon FSx

타사의 유명한 고성능 파일 시스템 엔진을 AWS가 완전 관리형으로 운영해 주는 서비스입니다. RDS가 MySQL이나 PostgreSQL 엔진을 관리해 주듯, FSx는 기존 특수 파일 시스템 제품들을 수정 없이 클라우드로 가져와 쓸 수 있게 해줍니다.

#### FSx for Windows File Server (윈도우 전용 공유 드라이브)

회사 사무실에서 흔히 쓰던 "Z 드라이브(사내 공용 폴더)"를 AWS로 그대로 옮겨온 것입니다.

* **핵심 프로토콜:** Windows 표준인 **SMB**와 **NTFS** 파일 시스템을 완벽하게 지원합니다.
* **보안 및 권한 관리:** Microsoft Active Directory(AD)와 통합되므로, ACL(액세스 제어 목록)을 사용한 "인사팀만 이 폴더 접근 가능", "사용자별 용량 제한(할당량)" 같은 Windows식 보안 제어가 그대로 작동합니다.
* 이름은 Windows지만 Linux EC2 인스턴스에도 마운트(연결)할 수 있습니다.
* Windows의 **DFS(분산 파일 시스템) 네임스페이스**를 지원해서 여러 파일 시스템을 하나의 거대한 드라이브명으로 묶어 관리할 수 있습니다.
* **스토리지 옵션:** 속도가 생명인 DB나 분석에는 **SSD**, 홈 디렉토리나 간단한 콘텐츠 관리에는 가성비의 **HDD**를 선택합니다.
* **고가용성:** 온프레미스(VPN/Direct Connect)에서도 접속 가능, **다중 AZ(Multi-AZ)** 구성 지원, 데이터는 매일 **S3로 자동 백업**됩니다.

#### FSx for Lustre (슈퍼컴퓨터용 광속 엔진)

**Linux + Cluster**를 합친 이름. 리눅스 컴퓨터 여러 대를 묶어서 초고속 연산을 하기 위해 태어났습니다.

* 머신러닝(ML), 고성능 연산(HPC), 동영상 렌더링, 금융 모델링, 전자 설계 자동화(EDA)처럼 "압도적인 속도로 대량의 데이터를 씹어 먹어야 할 때" 무조건 정답입니다.
* **S3와의 완벽한 통합 (Seamless Integration):** S3에 있는 대용량 데이터를 FSx로 가져와서 로컬 파일 시스템처럼 읽어 연산합니다. 연산이 끝난 출력값(결과물)은 다시 **S3로 직접 기록**해 저장합니다.
* 초당 수백 GB의 처리량과 **sub-ms(1밀리초 미만)** 의 지연 시간을 자랑합니다.

**⚠️ Lustre의 두 가지 배포 옵션 (시험 단골)**

| 옵션 | 특징 | 적합한 용도 |
| --- | --- | --- |
| Scratch (스크래치) | 복제 없음, 서버 고장 시 데이터 유실, 버스트 성능 6배 빠름, 가격 저렴 | 단기 처리용, 비용 최적화 워크로드 |
| Persistent (퍼시스턴트) | 동일 AZ 내 데이터 복제, 서버 고장 시 몇 분 안에 복구 | 장기 보관, 민감한 데이터 |

#### FSx for NetApp ONTAP (기업용 스토리지의 끝판왕 가상화)

대기업 전산실에서 가장 많이 쓰던 명품 스토리지 장비 브랜드인 'NetApp NAS'를 통째로 AWS 내부에 소프트웨어로 구현해 놓은 것입니다.

* **프로토콜 깡패:** **NFS, SMB, iSCSI**를 전부 지원하므로 Linux, Windows, MacOS, VMware Cloud, Amazon WorkSpaces 등 거의 모든 OS와 서비스에 한 번에 연결할 수 있습니다.
* 스토리지가 알아서 줄어들고 늘어나는 **오토 스케일링**을 지원합니다.
* **데이터 압축 및 중복 제거(Deduplication)** 기능이 기본 탑재되어 있어, 똑같은 데이터가 여러 개 있으면 하나만 남기고 용량을 엄청나게 아껴줍니다.
* **지정 시간 복제(Point-in-time Clone):** 용량을 거의 차지하지 않고 즉시 똑같은 테스트용 복사본 드라이브를 만들어냅니다.

#### FSx for OpenZFS (오픈소스 고성능 ZFS의 클라우드화)

오픈소스 기반의 강력한 파일 시스템인 **ZFS 워크로드**를 AWS로 그대로 이전할 때 사용합니다.

* 여러 버전의 NFS 프로토콜(v3, v4, v4.1, v4.2)과 호환됩니다.
* **성능:** **0.5밀리초(ms) 미만**의 초저지연 성능으로 최대 **1,000,000 IOPS**를 제공합니다.
* 지정 시간 복제(즉각적인 복제), 스냅샷, 데이터 압축을 모두 지원합니다.
* **NetApp ONTAP과의 결정적 차이점 (암기 필수):** **'데이터 중복 제거(Deduplication)' 기능이 없습니다.** 시험에서 중복 제거로 비용 절감이 언급되면 무조건 NetApp ONTAP이 정답입니다!

#### FSx 4형제 비교표

| 서비스 | 프로토콜 | 핵심 특징 | 시험 키워드 |
| --- | --- | --- | --- |
| FSx for Windows | SMB, NTFS | AD 통합, DFS 네임스페이스, Multi-AZ, S3 자동 백업 | Windows AD 통합, 사내 Z 드라이브 이사 |
| FSx for Lustre | Linux/Cluster | S3 완벽 통합, sub-ms 지연, Scratch/Persistent 옵션 | HPC, ML, 렌더링, S3 직접 읽기/쓰기 |
| FSx for NetApp ONTAP | NFS, SMB, iSCSI | 멀티OS, 오토스케일링, 중복 제거, 지정 시간 복제 | 온프레미스 NetApp 이전, 데이터 중복 제거 |
| FSx for OpenZFS | NFS (v3~v4.2) | 0.5ms 미만, 100만 IOPS, 중복 제거 없음 | ZFS 워크로드 마이그레이션, 초저지연 |

### 🌉 3. AWS Storage Gateway

온프레미스 서버들이 사용하는 NFS나 SMB 같은 파일 시스템 인터페이스와, AWS S3(HTTP API 기반)를 연결하는 **번역기 겸 캐시 서버** 역할을 하는 가상 머신(혹은 물리 장비)입니다.

* **최대 장점 (로컬 캐시):** 자주 쓰는 데이터는 온프레미스 게이트웨이 내부의 SSD에 임시 보관(캐시)합니다. 클라우드에 있는 데이터인데도 사무실에서는 랜선 속도(초저지연)로 빠르게 열어볼 수 있습니다.
* **주요 용도:** 백업 및 복원, 재해 복구(DR), 온프레미스 저장 공간 부족 해결(계층화).

#### 4가지 핵심 게이트웨이 유형

시험 문제에 나오는 '프로토콜'과 '최종 목적지'를 매칭하면 10초 만에 정답을 고를 수 있습니다.

**① Amazon S3 File Gateway (파일 형태 그대로 S3 연동)**

* 연결 프로토콜: **NFS** (리눅스용) 또는 **SMB** (윈도우용)
* 온프레미스 서버가 게이트웨이에 파일을 저장하면, 게이트웨이가 이 파일을 Amazon S3 버킷에 1:1 매칭 파일 객체로 업로드합니다.
* 전체 S3 버킷 내용이 다 사무실에 내려와 있는 게 아니라, **최근에 쓴 파일만 로컬에 캐시**로 보관합니다.
* S3 Standard, IA, Intelligent-Tiering 등 다양한 S3 클래스를 지원하며, S3 수명 주기 정책을 걸어두면 나중에 자동으로 Glacier로 넘길 수 있습니다.
* SMB 프로토콜을 쓸 때는 사용자 인증을 위해 **Active Directory(AD)와 연동**됩니다.

**② Amazon FSx File Gateway (윈도우 서버 전용 확장)**

* 연결 프로토콜: **SMB, NTFS, Active Directory 통합** (완전한 Windows 네이티브)
* 최종 목적지: S3가 아니라 **Amazon FSx for Windows File Server**입니다.
* 회사 사내 공용 폴더(홈 디렉토리)나 그룹 파일 공유를 클라우드로 마이그레이션할 때 씁니다. 자주 쓰는 데이터는 로컬 캐시로 보관하여 지연 시간을 줄입니다.

**③ Volume Gateway (S3를 백업으로 두는 가상 하드디스크)**

* 연결 프로토콜: **iSCSI** (블록 스토리지 전용 프로토콜)
* 온프레미스 서버에 가상 하드디스크(블록 볼륨)를 제공합니다. 이 볼륨의 데이터는 클라우드에 백업되며, **EBS 스냅샷** 형태로 구워져서 온프레미스로 복원하거나 EC2에 바로 붙일 수 있습니다.

**두 가지 세부 옵션 (★시험 단골):**

| 옵션 | 실제 데이터 위치 | 로컬 보관 데이터 | 적합한 경우 |
| --- | --- | --- | --- |
| Cached Volumes (캐시 볼륨) | S3 클라우드 | 최근에 쓴 데이터만 캐시 | 사무실 하드 용량이 부족할 때 |
| Stored Volumes (저장 볼륨) | 온프레미스 하드디스크 (100%) | 전체 데이터셋 상주 | 인터넷 끊겨도 로컬에서 100% 데이터 접근 필요 시 |

**④ Tape Gateway (물리 테이프 백업실 탈출)**

* 기존에 회사에서 쓰던 레거시 백업 소프트웨어(Veritas, Commvault 등)와 **iSCSI**로 연결합니다. 소프트웨어는 자기가 진짜 물리 테이프 장비에 백업하는 줄 알지만, 실제로는 S3 및 S3 Glacier 가상 테이프 라이브러리(VTL)로 안전하게 저장됩니다.
* 기존 백업 프로세스와 소프트웨어는 단 한 줄도 고치기 싫고, 무겁고 관리하기 힘든 물리 테이프만 없애고 싶을 때 씁니다.

#### Storage Gateway – Hardware Appliance (가상화 서버가 없을 때)

* 보통 Storage Gateway는 온프레미스의 VMware나 Hyper-V 같은 가상화 환경 위에 소프트웨어(VM)로 설치합니다.
* 소규모 지사나 가상화 인프라 자체가 없는 현장(소규모 데이터 센터의 일일 NFS 백업 등)에서는 설치할 곳이 없습니다.
* **해결책:** **amazon.com에서 진짜 하드웨어 미니 서버(실물 장비)를 주문**할 수 있습니다. 배송받아 사무실 랙에 꽂은 뒤, 원하는 유형(파일, 볼륨, 테이프 게이트웨이)으로 세팅하면 됩니다.

### 📤 4. AWS Transfer Family (레거시 FTP 유저를 위한 클라우드 대문)

많은 기업이나 협력업체들이 여전히 FTP, SFTP 같은 고전적인 파일 전송 방식을 씁니다. "문은 평소에 쓰던 FTP 문을 그대로 열어둘 테니, 들어오는 데이터는 자동으로 S3나 EFS에 쌓이게 해줄게" 하는 완전 관리형 서비스입니다.

* **지원 프로토콜:** **FTP**, 보안이 강화된 **FTPS**(SSL 기반), **SFTP**(SSH 기반)를 완벽 지원합니다.
* **인프라 걱정 제로:** AWS가 다중 가용 영역(Multi-AZ) 기반으로 고가용성과 확장성을 관리합니다. 비용은 프로비저닝된 엔드포인트 시간당 비용과 데이터 전송량(GB)만큼 냅니다.
* **기존 인증 시스템 재활용:** 로그인 자격 증명을 서비스 내에 저장하거나, 기업들이 원래 쓰던 **Microsoft Active Directory, LDAP, Okta, Amazon Cognito** 등과 그대로 연동할 수 있습니다.
* **실무 예시:** 대외 협력사들과 매일 대용량 정산 파일이나 CRM/ERP 데이터를 주고받는 파일 공유 시스템 구축 시 최고입니다.

### 🔄 5. AWS DataSync (스케줄 기반 데이터 대이동 셔틀)

Snow Family가 '물리적인 트럭'이었다면, DataSync는 '인터넷 랜선을 타고 달리는 초고속 자동 데이터 셔틀'입니다. 온프레미스, 타사 클라우드, 혹은 AWS 내부 스토리지끼리 엄청난 양의 데이터를 동기화할 때 씁니다.

* **온프레미스/타사 클라우드 → AWS 이동 시:** 온프레미스 환경에 DataSync **에이전트(Agent)** 라는 전용 소프트웨어를 설치해야 합니다. NFS, SMB, HDFS, S3 API 등을 지원합니다.
* **AWS 서비스 간 이동 시:** S3에서 EFS로, 혹은 EFS에서 FSx로 옮길 때는 **에이전트가 필요 없습니다.** AWS 내부에서 알아서 통신하기 때문입니다.
* **동기화 가능한 목적지:** Amazon S3(Glacier 포함 전 클래스), EFS, FSx 제품군 전체.
* **★ 시험 단골 포인트 (작동 방식):** 복제 작업이 실시간(Continuous)이 아닙니다! **매시간, 매일, 매주 등 정해진 일정(Schedule)에 따라 예약 실행**됩니다.
* **똑똑한 기능:** 파일 권한(NFS POSIX, SMB 등)과 메타데이터를 깨지지 않게 그대로 보존하며, 에이전트 1개당 **초당 최대 10 Gbps**의 대역폭을 쓸 수 있습니다. (회사 랜선이 먹통 되지 않게 대역폭 제한 기능도 제공)

| Transfer Family | 레거시 프로토콜 대문 | 사용자는 FTP, SFTP, FTPS로 접속하는데, 뒷단은 S3/EFS에 자동 저장 |
| DataSync | 데이터 자동 동기화 셔틀 | 온프레미스 → AWS 대량 이사, 스케줄링 기반 자동 예약 실행, 파일 메타데이터 보존 |
| Snow Family | 물리적 데이터 이사 장비 | 네트워크로 올리면 일주일 넘게 걸리는 대용량(TB, PB, EB) 데이터 트럭 배송 |
| Database | 특화 데이터 구조 | 단순 파일 저장이 아닌, 빠른 검색(인덱스) 및 정교한 쿼리 기능이 필요한 워크로드 |

### 🎯 마지막 핵심 퀴즈 세팅으로 개념 다지기!

1. **"협력업체가 SFTP로 정산 파일을 보냅니다. 개발 공수를 최소화하여 S3에 저장하려면?"** $\rightarrow$ **AWS Transfer Family**
2. **"매일 밤 12시에 온프레미스 NAS의 변경 사항을 S3로 자동 동기화하고 싶다면?"** $\rightarrow$ **AWS DataSync**
