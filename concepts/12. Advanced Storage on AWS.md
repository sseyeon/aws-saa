# 1. AWS Snow Family
이번에는 물리적인 장치로 데이터를 이동시키는 **AWS Snow Family**와 편리한 관리 도구인 **AWS OpsHub**, 그리고 단골 아키텍처 문제까지 싹 정리해 드릴게요.

인터넷 속도가 아무리 빨라졌어도, **테라바이트(TB), 페타바이트(PB)** 단위를 넘어서면 랜선으로 보내는 것보다 **하드디스크를 트럭에 싣고 달리는 게 훨씬 빠른 법**입니다. 이 패러다임을 이해하면 아주 쉽습니다.

---

## 1-1. 네트워크 전송의 한계와 Snow Family가 필요한 이유

인터넷망(네트워크)을 통해 대용량 데이터를 AWS S3로 직접 업로드할 때는 여러 장벽을 만납니다.

* **시간이 너무 오래 걸림:** 100TB를 일반 네트워크로 올리려면 몇 달이 걸릴 수 있습니다.
* **대역폭 제한 및 비용:** 네트워크 라인 전체를 풀 가동하면 비용이 폭발하고 회사 내 다른 대역폭을 다 잡아먹습니다.
* **불안정한 연결성:** 기껏 올리다가 네트워크가 툭 끊기면 대참사가 일어납니다.

> 💡 **SAA 시험 합격 기준점:** > 만약 네트워크를 통한 데이터 전송 계산 결과가 **"일주일(1주일) 이상"** 걸린다고 나오면, 무조건 네트워크 업로드를 포기하고 **AWS Snowball 장치를 요청하는 것이 정답**입니다.

---

## 1-2. 🚚 데이터 마이그레이션 삼총사 (용량별 비교)

"우리 회사 데이터가 몇 톤(?)이냐"에 따라 장비를 골라 배송받습니다.

### ① AWS Snowcone & Snowcone SSD (초소형 / 귀요미)

* **특징:** 무게가 단 2.1kg(4.5파운드)밖에 안 되는 작고 가벼운 휴대용 장치입니다. 거친 환경(진동, 먼지 등)도 잘 견디도록 견고하게 설계되었습니다.
* **용량:** * 기본 Snowcone: **8 TB HDD**
* Snowcone SSD: **14 TB SSD**


* **용도:** 공간이 너무 좁아서 큰 장비를 못 두는 환경에 딱입니다.
* **주의점:** 배터리와 연결 케이블은 AWS가 안 줍니다. **직접 준비**해야 합니다.
* **데이터 전송 방식:** 작업을 마친 뒤 AWS로 다시 택배를 보내거나, 현장에서 네트워크에 연결해 **AWS DataSync** 서비스를 이용해 데이터를 쏘아 보낼 수도 있습니다.

### ② AWS Snowball Edge (중형 / 만능 일꾼)

* 물리적으로 TB에서 PB 단위의 데이터를 AWS 안팎으로 이동시키는 핵심 장치입니다. 데이터 전송 작업 건당 비용이 청구되며, 블록 스토리지와 S3 호환 스토리지를 제공합니다. 용도에 따라 두 가지 모델로 나뉩니다.
* **Storage Optimized (스토리지 최적화):** 데이터 이사가 주 목적입니다. **80 TB HDD** 용량을 제공합니다. (S3 호환 객체 스토리지나 블록 볼륨으로 사용)
* **Compute Optimized (컴퓨팅 최적화):** 이사하면서 계산도 세게 해야 할 때 씁니다. **42 TB HDD 또는 28 TB NVMe** 용량을 제공합니다.

### ③ AWS Snowmobile (대형 / 초대형 트럭)

* **용량:** **엑사바이트(EB)** 급 데이터를 옮기는 괴물입니다. (1 EB = 1,000 PB = 1,000,000 TB)
* **특징:** 장치가 아니라 **거대한 컨테이너 트럭**이 회사 앞에 주차합니다. 트럭 한 대당 **100 PB**를 담을 수 있고, 여러 대를 병렬로 세워둘 수도 있습니다.
* **보안:** 온도 조절은 기본, GPS 추적과 24시간 연중무휴 영상 감시 등 첩보 영화급 고급 보안 시설이 가동됩니다.
* **기준:** 최소 **10 PB 이상의 데이터**를 마이그레이션 해야 할 때 Snowball 여러 대 쓰는 것보다 훨씬 탁월한 성능을 냅니다.

---

## 1-3. 📦 Snow Family 이용 프로세스 (A to Z)

1. **배송 요청:** AWS 콘솔에서 "저희 이사할게요. 장비 보내주세요" 하고 Snowball을 신청합니다.
2. **연결 및 복사:** 장비가 사무실에 도착하면 서버에 연결합니다. 내 컴퓨터에 **Snowball 클라이언트**나 **AWS OpsHub** 소프트웨어를 깔고 파일을 장비 안으로 복사합니다.
3. **반송:** 복사가 끝나면 장비 겉면에 달린 전자잉크 디스플레이 주소가 AWS 센터로 자동 바뀝니다. 그대로 택배(반송) 보냅니다.
4. **S3 로드 및 파기:** AWS 직원이 장비를 받아 **S3 버킷에 데이터를 안전하게 로드**해 줍니다. 업로드가 끝나면 장비 내 데이터는 보안을 위해 **완전하게 삭제(Wipe)** 처리됩니다.

---

## 1-4. 💻 엣지 컴퓨팅(Edge Computing): "클라우드랑 연락이 안 돼요"

데이터를 단순히 옮기는 게 아니라, 인터넷이 아예 안 터지거나 클라우드와 너무 먼 곳(엣지 로케이션)에서 현장 데이터를 즉시 처리(컴퓨팅)해야 할 때가 있습니다. (예: 사막 위의 트럭, 태평양 한가운데의 군함, 지하 깊은 탄광소)

이곳에서 Snow 장비를 가져다 놓고 서버처럼 쓰면서 데이터 전처리, 현장 머신러닝(ML), 미디어 스트림 트랜스코딩 등을 수행합니다. 필요하다면 이렇게 현장에서 처리한 장비를 나중에 AWS로 보내 전송할 수도 있습니다.

### 장비별 엣지 컴퓨팅 스펙

* **Snowcone / Snowcone SSD:** 가벼운 만큼 2 vCPU, 4GB 메모리를 탑재했습니다. USB-C 전원이나 배터리로 구동됩니다.
* **Snowball Edge - Compute Optimized:** 대형 연산용입니다. 무려 **104개 vCPU, 416 GiB RAM**에 머신러닝/비디오 처리를 위한 **선택적 GPU**까지 달 수 있습니다.
* **Snowball Edge - Storage Optimized:** **최대 40 vCPU, 80GB RAM**을 제공하며, 여러 대를 묶어 객체 스토리지 클러스터링을 할 수 있습니다.

> 🛠️ **공통 사양:** 모든 Snow 장비는 **AWS IoT Greengrass**를 사용하여 장비 안에서 직접 **EC2 인스턴스**나 **AWS Lambda 함수**를 돌릴 수 있습니다. 장기로 쓸 사람들을 위해 1년 및 3년 약정 할인 옵션도 있습니다.

---

## 1-5. 🛠️ AWS OpsHub: CLI 노가다는 끝났다

과거에는 Snow 장비를 조작하려면 검은 화면의 CLI(명령줄 창)로 어렵게 명령어를 쳐야 했습니다.
현재는 노트북에 **AWS OpsHub**라는 편리한 GUI(그래픽 프로그램)를 설치해서 마우스 클릭 몇 번으로 편하게 관리합니다.

* 단일 장치 또는 클러스터(묶음) 장치의 잠금 해제 및 초기 구성
* 직관적인 파일 전송 (드래그 앤 드롭)
* Snow 장치 안에서 EC2 인스턴스 켜고 끄기
* 저장 용량, 활성화된 인스턴스 개수 등 장치 메트릭 모니터링
* AWS DataSync, NFS(네트워크 파일 시스템) 등 AWS 호환 서비스 실행

* **시험 함정:** "대용량 아카이브 데이터를 비용을 아끼기 위해 Snowball을 이용해 **직접 Glacier 스토리지로 업로드**한다." ➔ **틀린 보기**입니다!
* **정답 메커니즘:** Snowball은 오직 **Amazon S3로만 데이터를 넣을 수 있습니다.** * **해결책:** 일단 Snowball을 통해 데이터를 **Amazon S3 버킷으로 먼저 가져온 뒤**, S3의 수명 주기 정책(Lifecycle Policy)을 세팅하여 객체가 자동으로 Glacier로 전환(Transition)되도록 설계해야 합니다. (시험에 정말 단골로 출제되는 정답 시나리오입니다!)

# 2. Amazon FSx
이번에는 스토리지 섹션의 고난도 단골 문제이자, 다양한 운영체제와 고성능 컴퓨팅을 위한 **Amazon FSx 제품군**을 싹 정리해 드릴게요.

AWS에는 이미 S3나 EFS 같은 훌륭한 스토리지가 있지만, 기업들이 온프레미스(자체 서버실)에서 쓰던 **기존의 특수한 파일 시스템 제품들**을 수정 없이 그대로 클라우드로 가져와 쓰고 싶을 때가 있습니다. RDS가 MySQL이나 PostgreSQL 엔진을 그대로 관리해 주듯, **타사의 유명한 고성능 파일 시스템 엔진을 AWS가 똑같이 완전 관리형으로 운영해 주는 서비스**가 바로 FSx입니다.

SAA 시험에서는 "어떤 워크로드에 어떤 FSx 유형을 매칭할 것인가"가 핵심입니다. 네 가지 형제를 아주 직관적인 비유로 풀어드릴게요!

---

## 2-1. Amazon FSx for Windows File Server (윈도우 전용 공유 드라이브)

* **비유:** 회사 사무실에서 흔히 쓰던 "Z 드라이브(사내 공용 폴더)"를 AWS로 그대로 옮겨온 것입니다.
* **핵심 프로토콜:** Windows 표준인 **SMB**와 **NTFS** 파일 시스템을 완벽하게 지원합니다.
* **보안 및 권한 관리:** Microsoft Active Directory(AD)와 깔끔하게 통합되므로, ACL(액세스 제어 목록)을 사용해 "인사팀만 이 폴더 접근 가능", "사용자별 용량 제한(할당량)" 같은 윈도우식 보안 제어가 그대로 작동합니다.
* **실무 특징:** * 이름은 Windows지만 Linux EC2 인스턴스에도 마운트(연결)할 수 있습니다.
* Windows의 **DFS(분산 파일 시스템) 네임스페이스**를 지원해서 여러 파일 시스템을 하나의 거대한 드라이브명으로 묶어 관리할 수 있습니다.
* 초당 수십 GB의 처리량, 수백만 IOPS, 수백 PB까지 확장이 가능합니다.
* **스토리지 옵션:** 속도가 생명인 DB나 분석에는 **SSD**, 홈 디렉토리나 간단한 컨텐츠 관리에는 가성비의 **HDD**를 고릅니다.
* **고가용성:** 온프레미스(VPN/Direct Connect)에서도 접속할 수 있고, **다중 AZ(Multi-AZ)** 구성이 가능해 한쪽 서버실이 터져도 끄떡없으며, 데이터는 매일 **S3로 자동 백업**됩니다.



---

## 2-2. Amazon FSx for Lustre (슈퍼컴퓨터용 광속 엔진)

* **이름의 유래:** **Linux** + **Cluster**를 합친 단어입니다. 말 그대로 리눅스 컴퓨터 여러 대를 묶어서 초고속 연산을 하기 위해 태어났습니다.
* **비유:** F1 레이싱카 같은 스토리지입니다. 머신러닝(ML), 고성능 연산(HPC), 동영상 렌더링, 금융 모델링, 전자 설계 자동화(EDA)처럼 **"압도적인 속도로 대량의 데이터를 씹어 먹어야 할 때"** 무조건 정답으로 튀어나옵니다.
* **S3와의 완벽한 콤비 (Seamless Integration):** * S3에 있는 무지막지하게 큰 데이터를 FSx로 가져와서 마치 로컬 파일 시스템처럼 슥 읽어 연산합니다.
* 연산이 끝난 웅장한 출력값(결과물)은 **다시 S3로 직접 기록**해 저장합니다.
* 초당 수백 GB의 처리량과 sub-ms(1밀리초 미만)의 미친 지연 시간을 자랑합니다. (SSD/HDD 옵션 선택 가능)



### ⚠️ 중요: Lustre의 두 가지 배포 옵션 (시험 단골)

* **Scratch(스크래치) 파일 시스템:** **임시 보관소**입니다. 복제를 안 하기 때문에 기저 서버가 고장 나면 데이터가 다 날아갑니다. 대신 버스트 성능이 6배나 더 빠르고 가격이 저렴합니다. 단기 처리용, 비용 최적화용 워크로드에 씁니다.
* **Persistent(퍼시스턴트) 파일 시스템:** **장기 보관소**입니다. 민감한 데이터를 위해 **동일한 가용 영역(AZ) 내에서 데이터를 복제**해 둡니다. 서버가 뻗어도 몇 분 안에 새 장비로 교체되어 데이터가 유지됩니다.

## 2-3. Amazon FSx for NetApp ONTAP (기업용 스토리지의 끝판왕 가상화)

* **비유:** 대기업 전산실에서 가장 많이 쓰던 명품 스토리지 장비 브랜드인 **'NetApp NAS'를 통째로 AWS 내부에 소프트웨어로 구현**해 놓은 것입니다. 온프레미스에서 NetApp ONTAP을 쓰던 회사들이 인프라를 바꿀 필요 없이 그대로 클라우드로 마이그레이션할 때 씁니다.
* **프로토콜 깡패:** **NFS, SMB, iSCSI**를 전부 지원하므로 Linux, Windows, MacOS, VMware Cloud, Amazon WorkSpaces 등 **거의 모든 OS와 서비스에 한 번에 다 연결**할 수 있습니다.
* **지갑을 지켜주는 마법 기능:** * 스토리지가 알아서 줄어들고 늘어나는 **오토 스케일링**을 지원합니다.
* **데이터 압축 및 중복 제거(Deduplication)** 기능이 기본 탑재되어 있어, 똑같은 데이터가 여러 개 있으면 하나만 남기고 숨을 죽여 용량을 엄청나게 아껴줍니다. 비용 효율성이 극도로 높습니다.
* **지정 시간 복제(Point-in-time Clone):** 용량을 거의 차지하지 않고 눈 깜짝할 새(즉시) 똑같은 테스트용 복사본 드라이브를 만들어냅니다. 새로운 코드를 마음껏 테스트해 보기에 완벽합니다.



---

## 2-4. Amazon FSx for OpenZFS (오픈소스 고성능 ZFS의 클라우드화)

* **정의:** 오픈소스 기반의 강력한 파일 시스템인 **ZFS 워크로드**를 AWS로 그대로 이전할 때 사용합니다.
* **호환성:** 여러 버전의 NFS 프로토콜(v3, v4, v4.1, v4.2)과 호환됩니다.
* **대상 OS:** NetApp과 마찬가지로 Linux, Windows, MacOS, WorkSpaces, EC2, ECS, EKS 등 다양한 환경을 다 커버합니다.
* **성능:** 0.5밀리초(ms) 미만의 초저지연 성능으로 최대 1,000,000 IOPS를 뿜어냅니다.
* **NetApp ONTAP과의 차이점 (암기 팁):**
* 얘도 지정 시간 복제(즉각적인 복제)와 스냅샷, 데이터 압축을 다 지원해서 테스트 환경 구축에 유용하고 비용이 적게 듭니다.
* **하지만 결정적으로 '데이터 중복 제거(Deduplication)' 기능은 없습니다.** 시험에서 중복 제거로 비용 아낀다는 말이 나오면 무조건 NetApp이 정답입니다!

2. **"머신러닝(ML), 초고속 HPC 연산, S3 데이터 직접 읽기/쓰기"** ➔ **FSx for Lustre**

4. **"NFS 프로토콜 기반, ZFS 워크로드를 그대로 마이그레이션, 초저지연 연산"** ➔ **FSx for OpenZFS**

# 3. AWS Storage Gateway
이번에는 하이브리드 클라우드(클라우드 + 온프레미스) 아키텍처의 핵심 가교 역할을 하는 **AWS Storage Gateway**입니다!

회사들이 "우리 서버실에 있는 장비나 프로그램들을 그대로 쓰면서, 데이터 저장소만 무제한 공간인 AWS S3나 클라우드로 바꾸고 싶어!"라고 할 때 사용하는 마법 같은 연결 통로입니다.

기존 온프레미스 시스템이 사용하던 익숙한 프로토콜(NFS, SMB, iSCSI)을 그대로 지원하기 때문에, 시스템 수정 없이 뒷단만 클라우드로 교체할 수 있습니다. 네 가지 핵심 유형을 비유와 함께 완벽하게 정리해 드릴게요.

---

## 3-1. 하이브리드 클라우드의 구원자: Storage Gateway 개요

온프레미스 서버들은 주로 NFS나 SMB 같은 파일 시스템 인터페이스를 씁니다. 반면 AWS S3는 HTTP API 기반의 독점 기술이라 서버실 장비들이 직접 알아듣지 못합니다.

이 둘 사이에 **번역기 겸 캐시 서버** 역할을 하는 가상 머신(혹은 물리 장비)을 사무실에 설치하는 것이 바로 Storage Gateway입니다.

* **최대 장점 (로컬 캐시):** 자주 쓰는 데이터는 온프레미스 게이트웨이 내부의 SSD에 임시 보관(**캐시**)해 둡니다. 덕분에 클라우드에 있는 데이터인데도 사무실에서는 랜선 속도(초저지연)로 빠르게 열어볼 수 있습니다.
* **주요 용도:** 백업 및 복원, 재해 복구(DR), 온프레미스 저장 공간 부족 해결(계층화).

---

## 3-2. 4가지 핵심 게이트웨이 유형 (시험 정답 고르는 기준)

시험 문제에 나오는 '프로토콜'과 '최종 목적지'를 매칭하면 10초 만에 정답을 고를 수 있습니다.

### ① Amazon S3 File Gateway (파일 형태 그대로 S3 연동)

* **연결 프로토콜:** **NFS** (리눅스용) 또는 **SMB** (윈도우용)
* **작동 방식:** 온프레미스 서버가 게이트웨이에 파일을 저장하면, 게이트웨이가 이 파일을 Amazon S3 버킷에 '1:1 매칭 파일 객체'로 고스란히 업로드합니다.
* **특징:** * 전체 S3 버킷 내용이 다 사무실에 내려와 있는 게 아니라, **최근에 쓴 파일만 로컬에 캐시**로 들고 있습니다.
* S3 Standard, IA, Intelligent-Tiering 등 다양한 S3 클래스를 다 지원하며, S3 수명 주기 정책(Lifecycle)을 걸어두면 나중에 자동으로 Glacier로 넘길 수도 있습니다.
* **보안:** IAM Role을 통해 S3 접근 권한을 관리하며, SMB 프로토콜을 쓸 때는 윈도우 사용자 인증을 위해 Active Directory(AD)와 깔끔하게 연동됩니다.



### ② Amazon FSx File Gateway (윈도우 서버 전용 확장)

* **연결 프로토콜:** **SMB, NTFS, Active Directory 통합** (완전한 Windows 네이티브)
* **최종 목적지:** S3가 아니라, 앞서 배운 **Amazon FSx for Windows File Server**입니다.
* **용도:** 회사 사내 공용 폴더(홈 디렉토리)나 그룹 파일 공유를 클라우드로 깔끔하게 마이그레이션할 때 씁니다. 자주 쓰는 데이터는 당연히 로컬 캐시로 사무실에 보관하여 지연 시간을 줄입니다.

### ③ Volume Gateway (S3를 백업으로 두는 가상 하드디스크)

* **연결 프로토콜:** **iSCSI** (블록 스토리지 전용 프로토콜)
* **작동 방식:** 온프레미스 서버에 가상 하드디스크(블록 볼륨)를 제공합니다. 이 볼륨의 데이터는 클라우드에 백업되며, 필요할 때 **EBS 스냅샷** 형태로 구워져서 나중에 온프레미스로 복원하거나 EC2에 바로 붙일 수 있습니다.
* **두 가지 세부 옵션 (★시험 단골):**
* **Cached Volumes (캐시 볼륨):** 진짜 전체 데이터는 **S3 클라우드**에 저장합니다. 사무실 게이트웨이에는 **최근에 쓴 데이터만 캐시**로 남겨둡니다. (사무실 하드 용량이 부족할 때 유용!)
* **Stored Volumes (저장 볼륨):** 진짜 전체 데이터셋이 **사무실 온프레미스 하드디스크**에 다 상주합니다. 게이트웨이는 그저 이 데이터를 주기적으로 **S3에 비동기 백업**만 해주는 안전장치 역할을 합니다. (인터넷이 끊겨도 로컬에서 100% 데이터 접근 가능해야 할 때 유용!)



### ④ Tape Gateway (물리 테이프 백업실 탈출)

* **비유:** 옛날 영화나 대기업 전산실 보면 테이프 릴이 돌아가는 거대한 백업 장비(VTL)가 있죠? 그걸 클라우드로 가상화한 것입니다.
* **작동 방식:** 기존에 회사에서 쓰던 레거시 백업 소프트웨어(Veritas, Commvault 등)와 **iSCSI**로 연결합니다. 소프트웨어는 자기가 진짜 물리 테이프 장비에 백업하는 줄 알지만, 실제로는 S3 및 S3 Glacier 가상 테이프 라이브러리(VTL)로 안전하게 들어갑니다.
* **용도:** 기존 백업 프로세스와 소프트웨어는 단 한 줄도 고치기 싫고, 무겁고 관리하기 힘든 물리 테이프만 없애고 싶을 때 씁니다.

---

## 3-3. Storage Gateway – Hardware Appliance (가상화 서버가 없을 때)

* **개념:** 보통 Storage Gateway는 온프레미스의 VMware나 Hyper-V 같은 가상화 환경 위에 소프트웨어(VM)로 설치합니다.
* **문제 상황:** 하지만 소규모 지사나 가상화 인프라 자체가 아예 없는 현장(소규모 데이터 센터의 일일 NFS 백업 등)에서는 설치할 곳이 없습니다.
* **해결책:** 이럴 땐 **amazon.com에서 진짜 하드웨어 미니 서버(실물 장비)를 주문**할 수 있습니다.
* 장비를 배송받아 사무실 랙에 꽂은 뒤, 내가 원하느 유형(파일, 볼륨, 테이프 게이트웨이)으로 세팅하면 끝납니다. 원활한 동작을 위해 충분한 CPU, 메모리, 네트워크, 그리고 SSD 캐시 자원이 탑재되어 있습니다.

---

### 🎯 한눈에 외우는 SAA 시험 정답 매칭 공식

1. **"NFS/SMB 프로토콜 사용 + 온프레미스 가속 캐시 + 최종 파일은 S3에 저장"** ➔ **S3 File Gateway**
2. **"기존 온프레미스 앱이 iSCSI 블록 스토리지를 요구함 + EBS 스냅샷 백업"** ➔ **Volume Gateway**
3. **"용량이 부족해 최근 데이터만 로컬에 두고 전체 데이터는 S3에 두고 싶다"** ➔ **Cached Volume Gateway**
4. **"로컬에 100% 데이터가 다 있어야 하고, S3에는 백업만 하겠다"** ➔ **Stored Volume Gateway**
5. **"기존 레거시 백업 소프트웨어 수정 없이 물리 테이프 장치를 클라우드로 대체"** ➔ **Tape Gateway**
6. **"사무실에 가상화 하이퍼바이저(VMware 등)가 없어서 소프트웨어를 못 깐다"** ➔ **Hardware Appliance 구매**



이어서 파일 전송 전문 서비스인 **AWS Transfer Family**와 데이터 동기화 치트키인 **AWS DataSync**, 그리고 스토리지 섹션을 완벽하게 마무리하는 **최종 비교 요약**까지 싹 정리해 드릴게요!

이번 파트는 SAA 시험에서 **"FTP 프로토콜을 고집하는가?"**, "스케줄링 기반으로 대량의 데이터를 마이그레이션하는가?"를 구별하는 능력이 핵심입니다. 역시 생생한 비유와 실무 맥락을 얹어 한눈에 들어오게 풀어드릴게요.

# 4. AWS Transfer Family (레거시 FTP 유저를 위한 클라우드 대문)

많은 기업이나 협력업체들이 여전히 **FTP, SFTP** 같은 고전적인 파일 전송 방식으로 데이터를 주고받습니다. AWS S3나 EFS가 좋다고 해서 협력업체 보고 "이제 FTP 쓰지 말고 AWS API나 CLI 공부해서 업로드하세요!"라고 할 수는 없겠죠?

그럴 때 **"문은 평소에 쓰던 FTP 문을 그대로 열어둘 테니, 들어오는 데이터는 자동으로 S3나 EFS에 쌓이게 해줄게"** 하는 완전 관리형 서비스가 바로 Transfer Family입니다.

* **지원 프로토콜:** **FTP**, 보안이 강화된 **FTPS**(SSL 기반), **SFTP**(SSH 기반)를 완벽 지원합니다.
* **인프라 걱정 제로:** AWS가 알아서 다중 가용 영역(Multi-AZ) 기반으로 고가용성과 확장성을 관리해 줍니다. 비용은 프로비저닝된 엔드포인트 시간당 비용과 데이터 전송량(GB)만큼 냅니다.
* **기존 인증 시스템 재활용:** 로그인 자격 증명을 서비스 내에 저장할 수도 있고, 기업들이 원래 쓰던 **Microsoft Active Directory, LDAP, Okta, Amazon Cognito** 등과 그대로 연동할 수 있습니다.
* **실무 예시:** 대외 협력사들과 매일 대용량 정산 파일이나 CRM/ERP 데이터를 주고받는 파일 공유 시스템을 구축할 때 최고입니다.

# 5. AWS DataSync (스케줄 기반 데이터 대이동 셔틀)

앞서 배운 Snow Family가 '물리적인 트럭'이었다면, DataSync는 '인터넷 랜선을 타고 달리는 초고속 자동 데이터 셔틀'입니다. 온프레미스, 타사 클라우드, 혹은 AWS 내부 스토리지끼리 엄청난 양의 데이터를 동기화할 때 씁니다.

* **온프레미스/타사 클라우드 ➔ AWS 이동 시:** 온프레미스 환경에 DataSync 에이전트(Agent)라는 전용 소프트웨어를 설치해야 합니다. NFS, SMB, HDFS, S3 API 등을 지원합니다.
* **AWS 서비스 간 이동 시:** S3에서 EFS로, 혹은 EFS에서 FSx로 옮길 때는 **에이전트가 필요 없습니다.** AWS 내부에서 알아서 통신하기 때문입니다.
* **동기화 가능한 목적지:** Amazon S3(Glacier 포함 전 클래스), EFS, FSx 제품군 전체.
* **★ 시험 단골 포인트 (작동 방식):** 복제 작업이 실시간(Continuous)이 아닙니다! **매시간, 매일, 매주 등 정해진 일정(Schedule)에 따라 예약 실행**됩니다.
* **똑똑한 기능:** 파일 권한(NFS POSIX, SMB 등)과 메타데이터를 깨지지 않게 그대로 보존해 주며, 에이전트 1개당 **초당 최대 10 Gbps**의 무지막지한 대역폭을 쓸 수 있습니다. (회사 랜선이 먹통 되지 않게 대역폭 제한 기능도 제공)

| **FSx for NetApp ONTAP** | 멀티 OS 명품 NAS | 온프레미스 NetApp 인프라 그대로 이전, **데이터 중복 제거(비용 절감)** |
| **FSx for OpenZFS** | 리눅스 고성능 파일 | ZFS 워크로드 마이그레이션, NFS 호환, 0.5ms 미만 초저지연 연산 |
| **Storage Gateway** | 하이브리드 연결 게이트웨이 | 온프레미스 서버실에 **로컬 캐시** 두고, 실제 데이터는 S3/FSx/가상테이프(VTL)로 연동 |
| **Transfer Family** | 레거시 프로토콜 대문 | 사용자는 **FTP, SFTP, FTPS**로 접속하는데, 뒷단은 S3/EFS에 자동 저장 |
| **DataSync** | 데이터 자동 동기화 셔틀 | 온프레미스 ➔ AWS 대량 이사, **스케줄링 기반 자동 예약 실행**, 파일 메타데이터 보존 |
| **Snow Family** | 물리적 데이터 이사 장비 | 네트워크로 올리면 **일주일 넘게 걸리는 대용량(TB, PB, EB) 데이터 트럭 배송** |
| **Database** | 특화 데이터 구조 | 단순 파일 저장이 아닌, 빠른 검색(인덱스) 및 정교한 쿼리 기능이 필요한 워크로드 |
