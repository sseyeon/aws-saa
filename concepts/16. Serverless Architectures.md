# 🔥 Serverless Architectures 마스터 치트키
## MSA 패턴, 통신 방식, 소프트웨어 업데이트 부하 분산

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **기능별 독립 배포, 개별 수명주기, 독립 코드 리포지토리로 구성된 아키텍처** $\rightarrow$ `마이크로서비스 아키텍처 (MSA)`
* **요청을 보내고 응답이 올 때까지 기다리는 서비스 간 통신** $\rightarrow$ `동기적 패턴: API Gateway, 로드 밸런서`
* **요청을 던지고 응답 타이밍에 개의치 않는 서비스 간 통신** $\rightarrow$ `비동기적 패턴: SQS, Kinesis, SNS, S3 Event → Lambda`
* **새 마이크로서비스 생성마다 인프라 설정과 배포 파이프라인 구축 부담** $\rightarrow$ `MSA 초기 오버헤드 문제 → API Gateway + Lambda로 해결`
* **클라이언트가 수많은 개별 마이크로서비스 엔드포인트와 일일이 통합해야 하는 부담** $\rightarrow$ `API Gateway + Lambda + Swagger(OpenAPI) 통합 → 클라이언트 SDK 자동 생성`
* **컨테이너 기반 MSA 서비스 (DNS → LB → 컨테이너 → NoSQL)** $\rightarrow$ `Route53 → ELB → ECS → DynamoDB`
* **서버리스 기반 MSA 서비스 (함수 + 인메모리 캐시)** $\rightarrow$ `Lambda + ElastiCache`
* **전통적인 3-Tier MSA 서비스 (가상 서버 + 관계형 DB)** $\rightarrow$ `EC2 ASG + RDS`
* **대규모 소프트웨어 업데이트 배포 시 EC2/EFS/네트워크 비용 폭증 문제** $\rightarrow$ `CloudFront 추가 (코드 수정 없음) → 엣지 캐싱으로 비용 극적 감소`
* **코드 수정 없이 정적 파일 배포 비용과 EC2 부하를 동시에 줄이는 방법** $\rightarrow$ `기존 아키텍처 앞단에 CloudFront 배치`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📌 1. 마이크로서비스 아키텍처 (MSA) 개요

마이크로서비스는 거대한 단일 애플리케이션을 기능별로 작게 쪼개어 개발하는 방식입니다.

* **도입 목적:** 서비스별 개발 수명 주기(Lifecycle) 단축, 독립적인 확장(Scaling), 서비스별 개별 코드 리포지토리 운영.
* **핵심 특징:** 대다수 서비스가 **REST API**를 통해 직접 상호 통신합니다.

---

### 📌 2. 서비스별 아키텍처 예시 (Service 1 ~ 3)

각 마이크로서비스는 독립된 DNS 이름(URL)을 가지며, 서비스 특성에 맞는 전혀 다른 기술 스택을 자유롭게 조합할 수 있습니다.

#### Service 1: 컨테이너 기반 아키텍처

* **도메인:** `service1.example.com`
* **흐름:** 사용자 → Route 53(DNS, Alias Record) → HTTPS → ELB(로드 밸런서) → ECS(컨테이너) → DynamoDB(NoSQL DB)

#### Service 2: 서버리스(Serverless) 기반 아키텍처

* **구성:** Lambda(서버리스 함수) + ElastiCache(인메모리 캐시, DynamoDB 대신 조합)
* **서비스 간 상호작용:** Service 2의 Lambda 함수가 Service 1의 ELB를 호출하여 데이터를 가져와 최종 응답 생성.

#### Service 3: 전통적인 가상 서버(EC2) 기반 아키텍처

* **도메인:** `service3.example.com`
* **구성:** EC2 Auto Scaling(ASG) + RDS(관계형 DB) — 전형적인 3-Tier 아키텍처.
* **서비스 간 상호작용:** EC2 인스턴스가 비즈니스 로직 처리 전 Service 2를 호출하여 필요한 데이터를 협력해서 가져옵니다.

---

### 📌 3. MSA 통신 패턴 구분

| 패턴 | 방식 | 대표 서비스 |
| --- | --- | --- |
| 동기적(Synchronous) | 요청 후 응답이 올 때까지 대기 | API Gateway, 로드 밸런서 |
| 비동기적(Asynchronous) | 요청을 던져두고 응답 타이밍에 무관 | SQS, Kinesis, SNS, S3 Event → Lambda |

---

### 📌 4. MSA 도입 시 직면하는 4가지 문제점

| 문제 | 설명 |
| --- | --- |
| 초기 오버헤드 | 새 마이크로서비스마다 인프라 설정, 배포 파이프라인 구축 번거로움 발생 |
| 서버 밀도/사용률 최적화 어려움 | 서비스마다 서버를 띄우다 보니 유휴 서버 발생, 자원 낭비 |
| 버전 관리 복잡성 | 여러 버전의 여러 마이크로서비스가 동시 운용되어 관리 복잡도 증가 |
| 클라이언트 구현 부담 가중 | 클라이언트가 수많은 개별 엔드포인트와 일일이 통합해야 해서 프론트 코드 비대화 |

---

### 📌 5. 서버리스 패턴으로 MSA 문제 해결

**API Gateway + Lambda 조합**이 위 도전 과제 중 일부를 해결합니다.

| 문제 | 해결 방식 |
| --- | --- |
| 서버 사용률 최적화 어려움 | Lambda + API Gateway는 자가 확장 + 실제 사용량만큼만 과금 → 사용률 걱정 불필요 |
| 초기 오버헤드 (환경 재구성) | API Gateway에서 기존 API를 복제하여 스테이징/테스트 환경 빠르게 구축 |
| 클라이언트 구현 부담 | API Gateway + Swagger(OpenAPI) 통합 → 클라이언트용 SDK 자동 생성 후 배포 |

---

### 📌 6. 소프트웨어 업데이트 부하 분산 (Offloading)

#### Before: 기존 문제 상황

* **구조:** 다중 AZ에 걸쳐 ELB + ASG(M5 EC2) 운용, 업데이트 파일은 EFS에 저장.
* **이슈:** 대규모 업데이트 배포 시 전 세계 사용자가 동시에 다운로드 → 네트워크 대역폭 비용 폭증 + EC2 CPU 사용량 급증 + EFS 읽기 부하 급증 → 비용과 성능 모두 손해.

#### After: CloudFront 도입으로 해결

기존 EC2 애플리케이션 코드를 **단 한 줄도 수정하지 않고** 최상단에 Amazon CloudFront(CDN)를 배치합니다.

* **동작 원리:** 소프트웨어 업데이트 파일은 정적 콘텐츠입니다. 첫 사용자 요청 시 CloudFront 엣지 로케이션에 파일이 캐싱됩니다. 이후 사용자들은 EC2/EFS에 도달하지 않고 가장 가까운 엣지에서 파일을 수신합니다.

| 도입 효과 | 설명 |
| --- | --- |
| 비용 절감 | EC2, 네트워크, EFS 비용 극적 감소 |
| 인프라 안정성 | CloudFront가 자체적으로 무한 확장되어 ASG의 무리한 스케일아웃 불필요 |
| 글로벌 확장성 | 아키텍처 변경 없이 정적 캐싱으로 글로벌 다운로드 속도 향상, 원본 서버 완벽 보호 |

### 📌 7. 실전 시나리오: 게임 패치 업데이트

* **상황:** 전 세계 1,000만 명이 이용하는 모바일 게임의 시즌 2 대규모 패치 파일(정적 파일) 배포.

| 방식 | 결과 |
| --- | --- |
| 나쁜 예 (EC2+EFS 직접 다운로드) | 오픈 직후 EC2가 부하를 못 견디고 다운, 수천만 원 트래픽 청구 |
| 좋은 예 (CloudFront 추가) | 한국/미국/유럽 엣지에 최초 1회 캐시 → 나머지 9,999,999명은 엣지에서 초고속 수신 → EC2 평온 유지 |
