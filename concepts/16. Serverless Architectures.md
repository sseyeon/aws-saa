# 1. 마이크로서비스 아키텍처 (MSA) 개요

마이크로서비스는 거대한 하나의 애플리케이션을 기능별로 작게 쪼개어 개발하는 방식입니다. 정형화된 틀이 없기 때문에 팀이 원하는 형태대로 자유롭게 설계할 수 있습니다.

* **도입 목적:** 서비스별 개발 수명 주기(Lifecycle) 단축, 독립적인 확장(Scaling), 서비스별 개별 코드 리포지토리 운영.
* **핵심 특징:** 대다수의 서비스가 **REST API**를 통해 직접적으로 상호작용(통신)합니다.

---

# 2. 서비스별 아키텍처 예시 (Service 1 ~ 3)

각 마이크로서비스는 독립된 DNS 이름(URL)을 가지며, 서비스 특성에 맞춰 완전히 다른 기술 스택을 조합할 수 있습니다.

## 🏢 Service 1: 컨테이너 기반 아키텍처

* **도메인:** `service1.example.com`
* **아키텍처 흐름:** 1. 사용자가 Route 53(DNS)에 쿼리를 보내 별칭 레코드(Alias Record)를 받아옵니다.
2. 사용자는 HTTPS를 통해 ELB(로드 밸런서)에 접근합니다.
3. ELB가 ECS(컨테이너 서비스)로 트래픽을 분산하고, ECS는 DynamoDB(NoSQL DB)와 통신하여 데이터를 처리합니다.

## ⚡ Service 2: 서버리스(Serverless) 기반 아키텍처

* **아키텍처 흐름:** * 기본적으로 Lambda를 활용한 서버리스 형태입니다. 다만 DB 자리에 DynamoDB 대신 ElastiCache(인메모리 캐시)를 백엔드로 조합하여 자유롭게 구성했습니다.
* **서비스 간 상호작용 예시:** * Service 2가 작동 중에 Service 1의 데이터가 필요하다면, Service 2의 **Lambda 함수가 Service 1의 ELB 호출**을 보냅니다. Service 1로부터 정보를 받아와 최종 응답을 생성합니다.

## 🖥️ Service 3: 전통적인 가상 서버(EC2) 기반 아키텍처

* **도메인:** `service3.example.com`
* **아키텍처 흐름:** * 서버리스가 아닌 Amazon EC2 Auto Scaling(ASG)과 RDS(관계형 DB)를 사용하는 전형적인 3-Tier 아키텍처입니다.
* **서비스 간 상호작용 예시:** * EC2 인스턴스가 최종 비즈니스 로직(결정)을 내리기 전에, 필요한 데이터를 얻기 위해 **Service 2를 호출**하여 협력합니다.

# 3. 마이크로서비스 설계 패턴과 도전 과제

## 🔄 통신 패턴 구분

* **동기적 패턴 (Synchronous):** 요청을 보내고 응답이 올 때까지 기다리는 방식입니다. (예: API Gateway, 로드 밸런서 활용)
* **비동기적 패턴 (Asynchronous):** 요청을 일단 던져두고 응답 타이밍이나 결과에 개의치 않는 방식입니다. (예: SQS 메시지 큐에 메시지 남기기, Kinesis, SNS, S3 Event를 통한 Lambda 트리거)

## ⚠️ 마이크로서비스 도입 시 직면하는 문제점

1. **초기 오버헤드:** 새로운 마이크로서비스를 생성할 때마다 인프라 설정, 배포 파이프라인 구축 등의 번거로움이 발생합니다.
2. **서버 밀도 및 사용률 최적화의 어려움:** 각 서비스마다 서버를 띄우다 보니, 노는 서버가 생겨 자원 낭비가 발생할 수 있습니다.
3. **버전 관리의 복잡성:** 동시에 여러 버전의 여러 마이크로서비스가 띄워져 관리하기 복잡해집니다.
4. **클라이언트 구현 부담 가중:** 클라이언트(웹/앱) 측에서 수많은 개별 마이크로서비스 엔드포인트와 일일이 통합해야 하므로 프론트엔드 코드가 무거워집니다.

## ✨ 서버리스(Serverless) 패턴을 통한 문제 해결

위의 도전 과제 중 일부는 **API Gateway**와 **AWS Lambda** 조합을 통해 해결할 수 있습니다.

* **자동 확장 및 비용 최적화:** 자가 확장이 지원되며 실제 사용량만큼만 비용이 청구되므로 서버 사용률을 걱정할 필요가 없습니다.
* **환경 재생성 용이:** API Gateway에서 기존 API를 쉽게 복제하여 스테이징/테스트 환경을 빠르게 구축할 수 있습니다.
* **클라이언트 코드 감소:** API Gateway와 Swagger(OpenAPI)를 통합하여, 클라이언트용 SDK를 자동으로 생성해 배포할 수 있습니다.

# 4. 소프트웨어 업데이트 다운로드 부하 분산 (Offloading)

## ❌ 기존 문제 상황 (Before)

* **구조:** 다중 AZ(가용 영역)에 걸쳐 ELB + ASG(M5 EC2 인스턴스)가 구동 중이며, 업데이트 대상 소프트웨어 파일은 공용 스토리지인 **EFS**에 저장되어 있습니다.
* **이슈:** 대규모 소프트웨어 업데이트가 배포되면, 전 세계 사용자가 동시에 파일을 다운로드하기 위해 트래픽이 몰립니다. 이로 인해 **네트워크 대역폭 비용 폭증, EC2 CPU 사용량 급증, EFS 읽기 부하**가 발생하며 비용과 성능 모두 손해를 보게 됩니다.

## ⭕ 해결책: CloudFront 도입 (After)

기존 EC2 애플리케이션 코드를 단 한 줄도 수정하지 않고, 최상단에 Amazon CloudFront(CDN)를 배치합니다.

* **동작 원리:** 소프트웨어 업데이트 파일은 한 번 출시되면 내용이 바뀌지 않는 '정적 콘텐츠(Static Content)'입니다. 따라서 사용자가 다운로드를 요청하면 전 세계에 퍼져 있는 CloudFront의 엣지 로케이션(Edge Location) 캐시에 파일이 최초 1회 저장됩니다.
* **도입 후 효과:**
1. **비용 및 자원 절감:** 후속 사용자들은 EC2나 EFS까지 도달하지 않고 가장 가까운 엣지에서 파일을 받아 가므로, **EC2, 네트워크, EFS 비용이 극적으로 감소**합니다.
2. **인프라 안정성:** CloudFront는 자체적으로 무한히 확장되는 서버리스 형태이므로, 배포 당일 트래픽이 몰려도 오토 스케일링 그룹(ASG)이 무리하게 확장할 필요가 없어집니다.
3. **글로벌 확장성:** 기존 아키텍처의 변경 없이도 정적 캐싱을 활용해 글로벌 사용자의 다운로드 속도를 높이고 원본 서버를 완벽하게 보호할 수 있습니다.

## 💡 실사용 시나리오로 이해하기 (예시)

> **[게임 패치 업데이트 시나리오]**
> * **상황:** 전 세계 1,000만 명이 이용하는 모바일 게임의 '시즌 2 대규모 패치 파일(정적 파일)'을 배포해야 합니다.
> * **나쁜 예시:** 패치 파일을 EC2 서버와 연결된 EFS에 두고 유저들이 직접 다운로드하게 하면, 오픈 직후 EC2 서버들이 부하를 견디지 못하고 터지거나, 트래픽 비용으로 수천만 원이 청구됩니다.
> * **좋은 예시:** EC2 앞단에 **CloudFront**를 둡니다. 한국, 미국, 유럽의 유저들이 다운로드를 누르면, 각 나라에 있는 AWS 엣지 서버가 EC2에서 파일을 딱 한 번만 가져와 복사본(캐시)을 떠둡니다. 이후 나머지 9,999,999명의 유저들은 EC2 근처에도 가지 않고 각자 나라의 엣지 서버에서 초고속으로 패치를 받으므로 메인 서버(EC2)는 아주 평온한 상태를 유지합니다.
