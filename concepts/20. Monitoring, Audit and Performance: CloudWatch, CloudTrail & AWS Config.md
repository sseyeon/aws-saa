# 🛠️ AWS 관리 및 모니터링 서비스 마스터 치트키

## 🎯 1단계: 한 눈에 보는 모니터링 3대장 핵심 키워드 매칭

시험 문제에 아래 핵심 단어가 나오면 고민하지 말고 바로 해당 서비스를 고르세요!

* **성능 모니터링 + CPU/네트워크 지표 + 로그 집계/분석 + 인프라 경보** $\rightarrow$ `Amazon CloudWatch`
* **누가 어떤 API를 호출했는가? + 리소스 삭제 범인 찾기 + 감사/규정 준수** $\rightarrow$ `AWS CloudTrail`
* **시간에 따른 구성 변경 기록 + 규정 준수 여부(Compliance) 평가 + 자원 거부 기능 없음** $\rightarrow$ `AWS Config`

### 💡 특정 리소스(예: Elastic Load Balancer) 기준 비교 치트키

* **ELB 오류 코드 비율, 연결 수, 시각화 대시보드** $\rightarrow$ `CloudWatch`
* **누가 ELB 구성을 변경했는지 API 호출 추적** $\rightarrow$ `CloudTrail`
* **ELB 보안 그룹 변경 추적, SSL 인증서 상시 할당 규칙 검사** $\rightarrow$ `AWS Config`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📈 1. Amazon CloudWatch (성능 모니터링 및 로깅)

#### 📊 CloudWatch Metrics (지표)

* **기본 개념:** AWS 모든 서비스에 대한 모니터링 변수(CPUUtilization, NetworkIn 등)를 제공합니다. 지표는 시간을 기반으로 하므로 **타임스탬프가 필수**입니다.
* **네임스페이스 & 측정 기준:** 모든 지표는 고유한 네임스페이스(Namespace)에 속하며, 지표의 속성으로 **Dimension(측정 기준)**(인스턴스 ID, 환경 등)을 가집니다. 지표 하나당 최대 **10개의 Dimension**을 지정할 수 있습니다.
* **시각화:** 지표가 많아지면 CloudWatch 대시보드에 추가해 한눈에 볼 수 있으며, RAM 지표 같은 사용자 지정 지표(Custom Metrics)도 생성 가능합니다.

#### 🌊 CloudWatch Metric Streams

* **개념:** CloudWatch 지표를 외부로 지속적이고 짧은 지연 시간으로 스트리밍하여 거의 실시간 전송을 구현합니다.
* **대상(Destination):** **Amazon Kinesis Data Firehose**를 지정할 수 있으며, 이를 통해 다른 목적지로 이동하거나 제3자 솔루션 파트너(Datadog, Dynatrace, New Relic, Splunk, Sumo Logic)로 전송할 수 있습니다. 지표를 필터링하여 일부 지표만 골라 스트리밍하는 옵션도 제공합니다.

#### 📝 CloudWatch Logs

* **로그 그룹 (Log groups):** 동일한 보존 기간, 모니터링, 액세스 제어 설정을 공유하는 로그 스트림의 집합입니다. 각 로그 스트림은 무조건 하나의 로그 그룹에 속해야 하며 보통 애플리케이션을 나타내는 임의의 이름입니다.
* **로그 스트림 (Log stream):** 동일한 소스(인스턴스, 로그 파일, 컨테이너 등)를 공유하는 로그 이벤트 시퀀스입니다.
* **보존 기간:** 유효 기간을 지정할 수 있으며 영원히 유지하거나 30일 등으로 커스텀 설정이 가능합니다.
* **전송 대상:** `Amazon S3(내보내기)`, `Kinesis Data Streams`, `Kinesis Data Firehose`, `AWS Lambda`, `OpenSearch` 등으로 내보낼 수 있습니다.
* **주요 로그 소스(Sources):** * `SDK`(직접 전송), `CloudWatch Logs Agent`(구버전 EC2 수집기)
* `Elastic Beanstalk`(앱 로그), `ECS`(컨테이너 로그), `AWS Lambda`(함수 로그)
* `VPC Flow Logs`(네트워크 흐름), `API Gateway`, `CloudTrail based on filter`(필터 기반 수집), `Route 53`(DNS 쿼리 로그)



#### 🔍 Metric Filter & Insights

* **Metric Filter (지표 필터):** 필터 표현식을 사용해 로그 데이터에서 특정 IP를 찾거나 "ERROR" 같은 단어의 발생 횟수를 세어 지표로 만듭니다. 이 필터 지표를 기반으로 **CloudWatch 경보(Alarm)를 트리거**할 수 있습니다.
* **Insights (인사이트):** 로그 데이터에 쿼리를 수행하고 결과를 실시간 검색/분석하는 기능입니다. 쿼리 결과를 CloudWatch 대시보드에 추가하여 시각화할 수 있어 빠르고 효율적인 로그 분석을 돕습니다.

#### 📥 CloudWatch Logs – S3 Export vs 구독 필터

* **S3 Export (내보내기):** API 호출은 `CreateExportTask`입니다. 로그 데이터가 내보내기 가능해질 때까지 **최대 12시간**이 소요될 수 있으므로 **실시간 또는 거의 실시간으로 작동하지 않습니다.**
* **구독 필터 (Logs Subscriptions):** 실시간으로 로그를 스트리밍하고 싶다면 반드시 구독 필터를 사용하여 목적지(Lambda, Kinesis 등)로 쏴주어야 합니다. 교차 계정 구독(Cross-Account Subscriptions)을 통해 Multi-Account 및 Multi Region 로그 어그리게이션(집계)이 가능합니다.

#### 🤖 CloudWatch Logs Agent vs Unified Agent (통합 에코시스템)

* **EC2 기본 주의점:** 기본적으로 EC2 인스턴스의 내부 로그(OS/앱 로그)나 상세 메트릭은 CloudWatch로 전송되지 않습니다. 전송을 위해서는 EC2에 IAM Role(권한)을 부여하고 에코시스템 에인전트를 실행해야 합니다. 온프레미스 환경에서도 설정 가능합니다.
* **CloudWatch Logs Agent:** 오래된 구버전 에인전트이며, 오직 **CloudWatch Logs로 '로그'만 전송**할 수 있습니다.
* **CloudWatch Unified Agent (통합 에코시스템):** 로그 파일뿐만 아니라 RAM, 프로세스 등 시스템 레벨의 상세 지표를 함께 수집합니다. 중앙 집중식 환경 구성을 위해 **SSM Parameter Store**와 통합되어 설정 파일을 관리할 수 있습니다. (구버전 에이전트는 사장되는 추세입니다.)
* **통합 에이전트 수집 상세 지표:** * *CPU* (active, guest, idle, system, user, steal)
* *디스크 지표* (free, used, total), *디스크 IO* (writes, reads, bytes, iops)
* *RAM* (free, inactive, used, total, cached)
* *넷 상태(Netstat)* (TCP/UDP 연결 수, net packets, bytes)
* *프로세스* (total, dead, blocked, idle, running, sleep)
* *스와프 공간(Swap Space)* (free, used, used %)
* ※ 기본 EC2 모니터링보다 훨씬 더 세부적인 시스템 지표를 긁어옵니다.



#### 🚨 CloudWatch Alarms (경보)

* **경보 상태 3가지:** 1. `OK`: 임계값을 위반하지 않은 정상 상태
2. `INSUFFICIENT_DATA`: 상태를 결정할 데이터가 부족한 상태
3. `ALARM`: 임계값이 위반되어 알림이 발송되는 상태
* **평가 기간:** 지표를 평가하는 시간 길이(초 단위)입니다. **고해상도 사용자 지정 지표**의 경우 10초, 30초, 또는 60초의 배수로 설정 가능합니다.
* **경보 타겟(주 타겟 3가지):** 1. `EC2 인스턴스`: 인스턴스 중지(Stop), 종료(Terminate), 재부팅(Reboot), 또는 복구(Recover) 실행
2. `EC2 Auto Scaling`: 스케일 아웃 및 스케일 인 트리거
3. `Amazon SNS`: 알림 메시지 전송
* **✨ EC2 인스턴스 복구 (EC2 Instance Recovery):** 상태 확인(Status Check)을 통해 인스턴스 상태(VM 점검)와 시스템 상태(기반 하드웨어 점검)를 모니터링하다가 경보 위반 시 복구를 실행합니다. 복구된 인스턴스는 기존 호스트에서 **다른 호스트로 이동**하며, 이때 **동일한 사설/공인/탄력적(EIP) 주소, 동일한 메타데이터, 동일한 인스턴스 배치 그룹**을 그대로 유지합니다.
* **테스트 CLI 팁:** 특정 임계값에 도달하지 않아도 경보 강제 트리거 및 알림 테스트를 하고 싶다면 `set-alarm-state` API를 사용합니다.
* `aws cloudwatch set-alarm-state --alarm-name "myalarm" --state-value ALARM --state-reason "testing purposes"`
* 경보는 앞서 언급한 `CloudWatch Logs Metrics Filter`를 기반으로도 생성할 수 있습니다.

### 🔀 2. Amazon EventBridge (구 CloudWatch Events)

이벤트 기반으로 리소스를 연결하는 중앙 버스 아키텍처입니다.

* **동작 방식 2가지:** 1. `Schedule`: 크론(CRON) 탭 작업 방식을 통한 스크립트 예약 실행
2. `Event Pattern`: 특정 작업을 수행하는 서비스의 상태 변화에 반응하는 규칙 규칙
* **라우팅:** 대상(Destination)이 다양하다면 EventBridge 규칙으로 직접 쏘거나, 중간에 Lambda 함수를 트리거하여 SQS, SNS 등으로 메시지를 분산 전송할 수 있습니다.

#### 🚌 EventBridge Event Bus & Advanced 기능

* **교차 계정 공유:** 리소스 기반 정책(Resource-based Policy)을 사용하여 다른 AWS 계정이나 다른 리전의 이벤트 버스로부터 이벤트를 수신하거나 액세스를 허용/거부할 수 있습니다. (예: AWS Organizations 내 모든 조직 계정의 이벤트를 단일 계정/리전 버스로 집계)
* **아카이브 및 재생:** 이벤트 버스로 전송된 모든 이벤트 또는 필터링된 일부 서브셋을 아카이브(보존 기간: 무기한 또는 일정 기간)할 수 있으며, 아카이브된 이벤트를 다시 재생(Replay)하는 기능도 제공합니다.
* **Schema Registry (스키마 레지스트리):** 이벤트 버스의 이벤트를 분석하여 데이터 구조(스키마)를 자동으로 추론합니다. 개발자는 스키마 레지스트리를 통해 데이터 구조를 미리 파악하고 대응하는 애플리케이션 코드를 생성할 수 있으며 스키마는 버전 관리가 가능합니다.
* **API 호출 가로채기:** CloudTrail과 연동하여 AWS 내에서 발생하는 모든 API Call을 EventBridge가 가로채서(Intercept) 실시간 이벤트를 트리거할 수 있습니다.

### 👁️‍🗨️ 3. CloudWatch Insights & Operational Visibility (운영 가시성 모음)

* **🐳 CloudWatch Container Insights:** * `ECS`, `EKS`, `EC2 환경의 Kubernetes`, `Fargate`에서 실행되는 컨테이너의 지표와 로그를 수집, 집계, 요약합니다.
* ⚠️ **주의:** Amazon EKS 및 Kubernetes 환경의 경우, 컨테이너를 식별하기 위해 반드시 **컨테이너화된 CloudWatch 에이전트**를 별도로 실행해야 합니다.


* **⚡ CloudWatch Lambda Insights:** * 서버리스 환경의 모니터링 및 트러블 슈팅을 위한 솔루션입니다. CPU 시간, 메모리, 디스크, 네트워크 등 시스템 레벨 지표를 수집합니다.
* Lambda의 고질적인 문제인 **콜드 스타트(Cold Start)** 발생 여부나 Lambda 작업자(Worker) 종료 같은 진단 정보도 집계합니다. **Lambda 계층(Layer)** 형태로 간편하게 제공됩니다.


* **👥 CloudWatch Contributor Insights (상위 N개 찾기):** * 로그 데이터를 분석하여 시스템 성능에 가장 많은 영향을 미치는 기고자(Contributor) 데이터를 시계열로 표시합니다.
* **상위 N개(Top-N) 기고자 지표** 및 총 고유 기고자 수/사용량을 확인하여 네트워크 상위 대화자, 불량 호스트 식별, 사용량이 가장 많은 사용자, 가장 많은 오류를 뿜어내는 URL 등을 골라냅니다.
* `VPC logs`, `DNS logs` 등 AWS 전반의 로그에 작동하며 백그라운드에서 CloudWatch Logs를 활용합니다. 규칙은 커스텀 생성하거나 내장 샘플 규칙을 씁니다.


* **🌐 CloudWatch Application Insights (자동화 대시보드):** * 모니터링 대상 애플리케이션의 잠재적 문제와 진행 중인 장애 문제를 고립/분리할 수 있는 자동화 대시보드를 제공합니다.
* **대상 기술:** Java, .NET, Microsoft IIS 웹 서버, 데이터베이스 등이 구동되는 EC2 인스턴스뿐만 아니라 `EBS`, `RDS`, `ELB`, `ASG`, `Lambda`, `SQS`, `DynamoDB`, `S3`, `ECS`, `EKS`, `SNS`, `API Gateway` 등 연관 리소스 전반을 감시합니다.
* 내부적으로 **SageMaker 머신러닝 서비스**가 동작하여 정상 패턴 파악 및 복구 시간을 단축하며, 발견된 문제와 운영 알림은 **Amazon EventBridge** 및 **SSM OpsCenter**로 자동 전송됩니다.

### 🕵️ 4. AWS CloudTrail (사용자 행위 및 API 감사)

AWS 계정 내의 모든 행위와 API 호출 이력(콘솔, SDK, CLI, 타 AWS 서비스 백엔드 호출 포함)을 투명하게 기록하여 거버넌스, 규정 준수 및 감사를 제공하는 서비스입니다.

* **기본 활성화:** AWS 계정 생성 시 기본적으로 활성화되어 있습니다.
* **저장 및 집계:** 로그를 `CloudWatch Logs` 또는 `S3`에 안전하게 저장 가능하며, 전체 또는 단일 리전에 적용되는 Trail(트레일)을 생성해 전 세계 모든 리전의 이벤트를 한곳으로 중앙 집계할 수 있습니다.
* **시험 팁:** "어떤 리소스가 갑자기 삭제되었는데 범인이 누군지 모르겠다" 하면 무조건 CloudTrail 로그부터 뒤져야 합니다.

#### 📊 CloudTrail 이벤트를 구성하는 3가지 유형

1. **Management Events (관리 이벤트 - 기본 활성화):** * AWS 계정 내의 리소스 자체에서 수행되는 제어 평면(Control Plane) 작업들입니다. 트레일 생성 시 기본 로깅됩니다.
* *예시:* IAM 권한 변경(`AttachRolePolicy`), 네트워크 라우팅 설정(`CreateSubnet`), 로깅 설정(`CreateTrail`)
* 리소스를 수정하지 않는 **Read 이벤트**와 리소스를 수정하는 **Write 이벤트**를 필터에서 분리할 수 있습니다.


2. **Data Events (데이터 이벤트 - 기본 비활성화):** * 리소스 내부에서 일어나는 고용량(High Volume) 데이터 작업들입니다. 워낙 양이 많아 기본값은 꺼져 있습니다.
* *S3 데이터 이벤트:* S3 객체 수준 활동(`GetObject`, `DeleteObject`, `PutObject` 등 - Read/Write 분리 가능)
* *Lambda 데이터 이벤트:* Lambda 함수 실행 활동 (`Invoke API`)


3. **CloudTrail Insights Event (이상 행동 감지):** * 계정 내 비정상적인 쓰기(Write) 활동 패턴을 인공지능으로 감지합니다. (예: 부정확한 리소스 대량 프로비저닝, 서비스 한도 도달, IAM 권한 작업의 급격한 증가, 주기적인 유지보수 활동 중단 등)
* 정상적인 관리 이벤트를 분석해 기준선(Baseline)을 세운 후, 비정상 패턴이 감지되면 CloudTrail 콘솔에 표시하고 **Amazon S3 전송** 및 **EventBridge 이벤트를 생성**하여 자동화 대응을 지원합니다.



#### ⏳ CloudTrail 로그 보존 기한

* 이벤트는 CloudTrail 이벤트 기록에 **기본 90일 동안만 저장**되고 자동 삭제됩니다. 90일 이상 보존하며 과거 이벤트를 분석하려면 S3 버킷으로 로그를 상시 전송하도록 설정하고, 추후 **Amazon Athena**를 연결해 SQL로 조회하는 아키텍처를 구성해야 합니다.

### 📐 5. AWS Config (리소스 구성 및 규정 준수 감사)

AWS 리소스의 상세 스펙과 구성 요소를 지속적으로 감사하고, 사전에 정의된 규정에 준수하는지 기록·평가하는 서비스입니다.

* **해결 가능 문제 예시:** 특정 보안 그룹에 `0.0.0.0/0(무제한) SSH`가 열려 있는지, S3 버킷이 Public(공개) 상태로 풀려 있는지, 시간이 흐르며 ALB 구성이 어떻게 변해왔는지 추적합니다.
* **인프라 특징:** **리전별 서비스**이므로 모든 리전별로 각각 구성해야 합니다. (단, 리전과 계정 간 데이터를 하나로 모으는 데이터 통합(Aggregator)은 지원합니다.)
* **저장 및 분석:** 구성 변경 타임라인 히스토리 데이터를 S3에 저장하여 **Athena**로 스캔할 수 있으며 변경 사항 발생 시 **SNS 알림**을 보낼 수 있습니다.

#### 📜 Config Rules (규칙 규칙) 및 강제성 여부 (★최다 오답 감점 포인트)

* **제공 규칙:** AWS가 자체 제공하는 75개 이상의 사전 정의된 관리형 규칙이 있으며, **AWS Lambda**를 활용해 사용자 지정 규칙(Custom Rules)을 선언할 수 있습니다. (예: 생성되는 모든 EBS 디스크 포맷이 `gp2`인지 확인, 모든 EC2 타입이 `t2.micro`인지 감시)
* **평가 트리거 시점:** 구성 항목에 변경 변경 이벤트가 발생할 때마다 실시간 평가하거나, 정기적인 특정 시간 간격(주기적 평가)으로 트리거할 수 있습니다.
* **⚠️ 주의 (거부 기능 없음):** AWS Config Rules는 사용자가 잘못된 리소스를 생성하려 할 때 이를 **중간에 차단하거나 예방(Deny)하지 못합니다.** 오직 규정을 위반했는지 '준수/미준수 여부를 평가하고 보고'하는 사후 감사 역할만 수행합니다.
* **비용 구조 (프리 티어 없음):** 리전당 기록된 구성 항목(CI)당 $0.003, Config 규칙 평가당 $0.001가 청구됩니다.

#### 🔧 Remediations (자동 보완 메커니즘) & 알림

* **SSM 자동화 연동:** 리소스가 규칙을 위반하여 '미준수(Non-compliant)' 상태가 되면 **AWS Systems Manager(SSM) 자동화 문서**를 트리거하여 리소스를 정상 규격으로 강제 수정(Remediation)할 수 있습니다. 관리형 문서를 쓰거나 Lambda 함수를 호출하는 커스텀 자동화 문서를 씁니다.
* **보완 재시도 (Remediation Retries):** 자동 보완 조치를 실행했음에도 불구하고 리소스가 여전히 미준수 상태로 남아있다면, 실패로 끝내지 않고 지정을 통해 자동으로 재시도하도록 보완 재시도를 설정할 수 있습니다.
* **EventBridge 연동:** 리소스 준수 상태가 변할 때마다 **EventBridge** 규칙을 통해 관리자에게 알림을 트리거하거나, 모든 구성 변경 및 미준수 상태 변경 이벤트를 **SNS 토픽**으로 발송하여 클라이언트 측 필터링을 거쳐 소비할 수 있습니다.

## 🏗️ 3단계: 복합 운영 인프라 관제 아키텍처 시나리오

위의 모니터링/감사 조각들을 유기적으로 결합하여 엔터프라이즈 환경을 철통 보안 관제하는 핵심 시나리오 구조입니다.

### 시나리오: EC2 인스턴스의 하드웨어 장애 복구 및 비인가 보안 그룹 변경 자동 보완 파이프라인

```
[인프라 이벤트 발생]
  ├── (A) EC2 하드웨어 물리 장애 발생 
  │     └── [CloudWatch Status Check 경보 위반 (ALARM)] ──> [EC2 Instance Recovery 작동 (호스트 자동 이동, IP/메타데이터 유지)]
  │
  └── (B) 악의적인 사용자가 EC2 보안 그룹에 '0.0.0.0/0 (전체 오픈)' SSH 규칙 추가 API 호출
        ├── [CloudTrail API 가로채기] ──> [Amazon EventBridge 트리거] ──> [보안 팀에 SNS 경고 알림 전송]
        └── [AWS Config Rules 실시간 감지] ──> [미준수(Non-compliant) 판단] ──> [SSM Automation / Lambda 호출] ──> [비인가 포트 강제 폐쇄 (자동 보완)]

```

* **아키텍처 가치:** 물리 서버가 터지면 `CloudWatch`가 감지해 똑같은 메타데이터와 IP를 들고 다른 호스트 서버로 인스턴스를 즉시 자동 복구(`Recovery`) 시킵니다. 동시에 누군가 보안 구멍을 뚫는 나쁜 API 호출을 하면 `CloudTrail`이 범인 이력을 90일 넘게 S3+Athena 장부에 박제하고, `AWS Config`가 이를 실시간으로 잡아내어 `SSM 자동화`를 통해 소스 보안 그룹을 원래 안전한 상태로 즉시 강제 원복(`Remediation`) 시키는 완벽하게 자동화된 지능형 인프라 관제 탑이 완성됩니다.
