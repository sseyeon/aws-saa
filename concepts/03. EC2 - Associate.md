# 🌐 AWS EC2 네트워킹 토폴로지 및 고급 배치 전략 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **단일 AZ 배치 + 10Gbps 초저지연 내부 네트워킹 + HPC/분산 인공지능/빅데이터 연산** $\rightarrow$ `Cluster Placement Group (클러스터 배치 그룹)`
* **인프라 물리적 장애 격리 + 인스턴스 간 랙(Rack) 공유 전면 차단 + 고가용성이 필수인 소규모 핵심 자원(AZ당 최대 7대)** $\rightarrow$ `Spread Placement Group (분산 배치 그룹)`
* **빅데이터 에코시스템(Hadoop, Kafka, Cassandra) + 수백 대의 EC2 확장 + 파티션 단위 랙 격리 및 메타데이터 조회** $\rightarrow$ `Partition Placement Group (파티션 배치 그룹)`
* **인스턴스 재시작 시 퍼블릭 IP 변경 방지 + 장애 시 다른 인스턴스로 즉시 IP 주소 재매핑(인계)** $\rightarrow$ `Elastic IP (탄력적 IP / EIP)`
* **VPC 내 논리적 네트워크 카드 + 기본/보조 프라이빗 IP 보유 + 인스턴스 분리 후 타 인스턴스로 장애 조치(Failover) 이동** $\rightarrow$ `ENI (Elastic Network Interface)`
* **초기화 및 캐시 로딩 시간이 오래 걸리는 앱 + 인메모리(RAM) 상태를 루트 EBS 볼륨에 저장 후 고속 부팅** $\rightarrow$ `EC2 Hibernate (최대 절전 모드)`
* **EC2 Hibernate 활성화를 위한 필수 전제 조건** $\rightarrow$ `루트 볼륨이 반드시 'EBS'여야 하며, '암호화(Encrypted)'가 필수 적용되어야 함`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🌐 1. IP 주소 체계 (Public vs Private) 및 탄력적 IP (EIP)

#### 🌐 IPv4 주소의 기본 특징

* **Public IP (공용 IP):** 전 세계 인터넷(WWW) 공간에서 단 하나만 존재하는 고유 주소입니다. 외부에 노출되므로 지리적 위치 추적이 가능하며, 외부에서 내 서버로 접속할 때(예: SSH, HTTP) 반드시 이 퍼블릭 IP를 타겟으로 삼아야 합니다.
* **Private IP (사설 IP):** 사설 격리 네트워크(VPC) 내부에서만 장치들을 식별하는 주소입니다. 프라이빗 주소 대역은 정해져 있으며, 다른 회사/VPC라면 동일한 사설 IP를 중복 사용할 수 있습니다. 사설 IP를 가진 자원이 인터넷과 통신하려면 **NAT 게이트웨이** 또는 **인터넷 게이트웨이**를 거쳐야 합니다.

#### 💡 Elastic IP (탄력적 IP / EIP)

* **메커니즘:** 일반적인 EC2 퍼블릭 IP는 인스턴스를 중지(`Stop`) 후 다시 시작(`Start`)하면 주소가 매번 바뀝니다. 이를 막기 위해 사용자가 명시적으로 해제(삭제)하기 전까지 고정적으로 소유할 수 있게 해주는 **영구적 퍼블릭 IPv4 주소**입니다.
* **장애 조치(Failover):** 운영 중인 EC2 인스턴스에 에러가 발생하면, 해당 EIP를 계정 내의 다른 대기 중인 정상 EC2 인스턴스로 즉시 재매핑(네트워크 연결 인계)하여 다운타임을 최소화할 수 있습니다.
* **⚠️ 아키텍처 주의점 (시험 단골 감점 요인):** * EIP는 기본적으로 **계정당 리전별 5개로 제한**됩니다.
* 무분별한 EIP 매핑은 하드코딩 아키텍처를 유발하여 안 좋은 구조적 결정으로 평가받습니다. 퍼블릭 고정 IP가 필요하다면 EIP 대신 **임의의 퍼블릭 IP 위에 DNS 레코드 이름(Route 53)을 바인딩**하거나, 전면에 로드 밸런서(ALB/NLB)를 배치하여 퍼블릭 IP 노출 자체를 완전히 지우는 것이 AWS 표준 베스트 프랙티스입니다.

### 📦 2. EC2 배치 그룹 (Placement Groups) 3대 전략

AWS 가상 서버가 실제 물리 데이터 센터 하드웨어 하이퍼바이저 랙(Rack)에 배치되는 물리적 위치 위상을 개발자가 직접 통제하는 기술입니다.

#### 🪢 1. Cluster Placement Group (클러스터 배치 그룹)

* **구조:** **단일 가용 영역(Single AZ)** 내에서 모든 EC2 인스턴스들을 **동일한 물리적 하드웨어 랙(Rack) 시스템에 밀집 배치**합니다.
* **장점:** 인스턴스 간 네트워크 통신 지연 시간이 극도로 낮아지며, **최대 10 Gbps의 폭발적인 대역폭**을 보장받습니다.
* **단점:** 하드웨어 랙 자체에 전원 차단 등 물리 장애가 발생하면 배치 그룹 내의 **모든 EC2 인스턴스가 동시에 다운**되는 단일 장애점(SPOF) 리스크가 있습니다.
* **시험용 매칭 용도:** 초고속 대용량 빅데이터 연산, 고성능 컴퓨팅(HPC), 로우 레이턴시 전용 과학적 모델링 및 인공지능 분산 학습.

#### 🍞 2. Spread Placement Group (분산 배치 그룹)

* **구조:** 상호 장애 격리를 극대화하기 위해, 배치되는 모든 EC2 인스턴스를 **서로 다른 물리 하드웨어 랙(Rack)에 완전히 찢어서 분산 배치**합니다.
* **장점:** 여러 가용 영역(Multi-AZ)에 걸쳐 구성할 수 있으며, 특정 물리 랙이 무너져도 다른 인스턴스들은 완벽히 보호되므로 동시 실패 위험이 극도로 낮아집니다.
* **단점:** 엄격한 물리 격리를 보장해야 하므로, **배치 그룹당 가용 영역(AZ)별로 최대 7개의 인스턴스만 생성할 수 있다는 규모적 확장성 제한**이 걸려 있습니다.
* **시험용 매칭 용도:** 절대 꺼지면 안 되는 핵심 프로덕션 인스턴스, 상호 간의 하드웨어 장애 영향도가 완벽히 격리되어야 하는 고가용성 애플리케이션.

#### 🗄️ 3. Partition Placement Group (파티션 배치 그룹)

* **구조:** 분산(Spread)의 장점과 대규모 확장성을 결합한 형태입니다. 배치 그룹 내에 여러 개의 '파티션(물리 랙 세트)'을 정의하고 인스턴스들을 분할 배치합니다. **서로 다른 파티션끼리는 하드웨어 랙을 절대 공유하지 않습니다.**
* **장점:** 하나의 파티션(랙)이 통째로 고장 나도 다른 파티션에 속한 인스턴스들은 안전합니다. AZ당 최대 7개의 파티션을 둘 수 있으며, **그룹당 수백 대의 EC2 인스턴스로 거대하게 스케일 아웃**할 수 있습니다. EC2 인스턴스는 자체 메타데이터를 호출하여 자신이 현재 몇 번 파티션 랙에 있는지 인지할 수 있습니다.
* **시험용 매칭 용도:** 대규모 분산 파티션 데이터 스토어 에코시스템인 **Hadoop(HDFS), Apache HBase, Apache Kafka, Apache Cassandra** 워크로드용 인프라.

### 🎴 3. 탄력적 네트워크 인터페이스 (ENI)

ENI는 VPC 내부에서 가상 네트워크 카드 역할을 수행하는 논리적 네트워킹 구성 요소입니다. EC2 인스턴스에 탈부착이 가능합니다.

```
[ ENI 속성 풀 ]
├── 기본 프라이빗 IPv4 주소 (VPC 대역 내 1개 필수)
├── 보조(Secondary) 프라이빗 IPv4 주소 (여러 개 할당 가능)
├── 프라이빗 IP별 1:1 매핑된 탄력적 IP (EIP)
├── 퍼블릭 IPv4 주소 & IPv6 주소
└── 연결된 보안 그룹(Security Groups) 리스트, MAC 주소

```

* **장애 조치 유연성:** 구동 중인 EC2 인스턴스에 네트워크 장애가 발생하거나 장비 교체가 필요할 때, **ENI 가상 카드를 인스턴스로부터 독립적으로 떼어내어(Detach) 동일 AZ 내의 다른 대기중인 EC2 인스턴스에 즉시 핫 플러그인(Attach) 형태로 재연결**할 수 있습니다. 이 방식을 쓰면 인스턴스가 바뀌어도 사설 IP와 네트워크 속성을 그대로 유지하므로 고속 장애 조치가 가능합니다.
* **⚠️ 핵심 제약 조건:** ENI는 생성 시 특정 가용 영역(AZ)에 완전히 귀속(바인딩)됩니다. 따라서 **다른 AZ에 서 가동 중인 EC2 인스턴스로는 ENI를 이동시켜 연결할 수 없습니다.** (반드시 동일 AZ 내에서만 이동 가능)

### ❄️ 4. EC2 최대 절전 모드 (EC2 Hibernate)

EC2 인스턴스를 일반적인 중지(`Stop`) 상태로 돌리면 램(RAM) 안의 데이터는 증발하고 다음 부팅 시 OS 커널부터 새로 로드하므로 애플리케이션 가동 및 내부 캐시 메모리가 빌드될 때까지 긴 다운타임이 유발됩니다. 최대 절전 모드는 이를 해결하기 위한 고급 인프라 관리 기술입니다.

#### ⚙️ 동작 메커니즘 및 특징

* **상태 동결:** 절전 명령을 내리면 인스턴스가 완전히 꺼지기 직전, **현재 RAM(인메모리)에 적재되어 있던 모든 프로세스 데이터 상태를 있는 그대로 캡처하여 루트(Root) EBS 볼륨에 통째로 파일 형태로 동결 저장**합니다.
* **고속 부팅:** 재시작 시 OS 초기화 및 사용자 데이터(User Data) 스크립트를 처음부터 구동하지 않고, 루트 EBS에 박혀있던 RAM 상태 스냅샷을 메모리로 다이렉트 복구하므로 **이전 작업을 멈춘 시점부터 즉시 애플리케이션 가동이 재개**됩니다.

#### 🔒 시험 합격을 위한 필수 사양 셋 (100% 암기)

* **루트 볼륨 제약:** 인스턴스 임시 스토어 볼륨은 절대 불가능하며, **반드시 네트워크 기반인 EBS 볼륨**이어야 합니다.
* **★보안 필수 조건:** RAM에 저장되어 있던 사용자 세션, 커널 민감 정보가 디스크에 평문으로 기록되면 보안 사고로 이어지므로, **루트 EBS 볼륨은 반드시 '암호화(Encrypted)' 처리가 완료되어 있어야만 최대 절전 모드를 활성화할 수 있습니다.**
* **하드웨어 한계:** 인스턴스의 **최대 RAM 사이즈는 150GB 이하**여야 하며, 가상화 레이어가 없는 베어 메탈(Bare Metal) 인스턴스 기종은 최대 절전 모드를 지원하지 않습니다.
* **기타 규칙:** 주요 인스턴스 패밀리(C시리즈, M시리즈, R시리즈, T시리즈 등) 및 대다수 메이저 OS(Amazon Linux 2, Ubuntu, RHEL, CentOS, Windows)를 커버합니다. On-Demand, Reserved는 물론 Spot 인스턴스에서도 작동하며, 한 번 절전 모드로 동결된 상태는 **최대 60일까지 보존** 가능합니다.
