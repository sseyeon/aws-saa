# 📚 AWS 빅데이터 및 분석 서비스 마스터 치트키

## 🎯 1단계: 한 눈에 보는 빅데이터 서비스 핵심 키워드 매칭

시험 문제에 아래 핵심 단어가 나오면 고민하지 말고 바로 해당 서비스를 고르세요!

* **S3에 저장된 로그/파일 + 서버리스 SQL** $\rightarrow$ `Amazon Athena`
* **PB 단위 + 대규모 엔터프라이즈 DW + 컬럼 기반 + OLAP** $\rightarrow$ `Amazon Redshift`
* **Redshift로 로드하지 않고 S3 데이터 바로 쿼리** $\rightarrow$ `Redshift Spectrum`
* **자유로운 텍스트 검색 + 부분 일치 + SQL 미지원 + 비정형** $\rightarrow$ `Amazon OpenSearch`
* **하둡(Hadoop) / 스파크(Spark) 클러스터 + 수백 대 EC2** $\rightarrow$ `Amazon EMR`
* **서버리스 BI + 시각화 대시보드 + 인메모리 계산** $\rightarrow$ `Amazon QuickSight`
* **서버리스 ETL + 메타데이터 장부 + 파일 포맷 변환** $\rightarrow$ `AWS Glue`
* **데이터 레이크 구축 자동화 + 행/열 수준 정밀 보안 통제** $\rightarrow$ `AWS Lake Formation`
* **스트리밍 데이터 실시간 분석 + SQL / Apache Flink** $\rightarrow$ `Kinesis Data Analytics`
* **관리형 Apache Kafka** $\rightarrow$ `Amazon MSK`

## 📑 2단계: 서비스별 100% 디테일 분석 및 최적화 기법

### 🔍 1. Amazon Athena

S3에 저장된 데이터를 분석하기 위한 **서버리스** 쿼리 서비스입니다.

* **상세 개념:** Presto 엔진 기반으로 빌드되었으며, 표준 SQL 언어를 사용하여 파일을 쿼리합니다.
* **지원 포맷:** CSV, JSON, ORC, Avro, Parquet 등 다양한 형식을 지원합니다.
* **비용 구조:** 검색(스캔)된 데이터 **1 TB당 $5.00**를 지불합니다.
* **주요 연동:** 보고서 및 대시보드 시각화를 위해 주로 **Amazon QuickSight**와 함께 사용됩니다.
* **사용 사례:** 비즈니스 인텔리전스(BI), 분석, 보고서 작성뿐만 아니라 `VPC Flow Logs`, `ELB Logs`, `CloudTrail` 추적 등 AWS 서비스에서 발생하는 모든 로그를 쿼리하고 분석할 때 사용합니다.

#### ⚡ Athena 성능 및 비용 최적화 (★시험 출제 1순위)

* **컬럼(열) 기반 포맷 변환:** `Apache Parquet` 또는 `ORC` 사용을 강력히 권장합니다. 필요한 열만 골라서 스캔하므로 데이터 스캔량이 대폭 줄어들어 비용이 절감되고 성능이 크게 향상됩니다. (기존 데이터를 변환할 때는 **AWS Glue**를 활용합니다.)
* **데이터 압측:** `gzip`, `bzip2`, `lz4`, `snappy`, `zlib`, `zstd` 등으로 데이터를 압축하여 스캔 오버헤드를 줄입니다.
* **데이터 분할(파티셔닝):** 특정 열을 항상 쿼리한다면 S3 버킷의 전체 경로를 슬래시(`/`)로 분할하여 열별로 특정 값을 저장합니다. (예시: `s3://athena-examples/flight/parquet/year=1991/month=1/day=1/`)
* **파일 크기 제어:** 자잘한 파일 여러 개보다 **128MB보다 큰 파일**들을 사용하여 파일 오픈 오버헤드를 최소화합니다.

#### 🌐 Athena Federated Query (연합 쿼리)

* **개념:** 관계형, 비관계형, 객체, 사용자 지정 데이터 소스(AWS 또는 온프레미스 상관없이)에 저장된 모든 데이터를 대상으로 SQL 쿼리를 실행할 수 있게 합니다.
* **구동 방식:** **AWS Lambda**에서 실행되는 데이터 소스 커넥터를 사용하여 `CloudWatch Logs`, `DynamoDB`, `RDS` 등과 통신합니다.
* **결과 처리:** 연합 쿼리의 실행 결과는 다시 **Amazon S3**에 저장됩니다.

### 🛢️ 2. Amazon Redshift

PostgreSQL을 기반으로 하지만 OLTP(트랜잭션)가 아닌 **OLAP(온라인 분석 처리)** 유형의 대규모 데이터 웨어하우징 서비스입니다.

* **상세 개념:** 다른 데이터 웨어하우스에 비해 10배 빠른 성능을 제공하며 **PB(페타바이트) 단위**의 데이터까지 확장 가능합니다.
* **저장 아키텍처:** 데이터를 행(Row) 기반이 아닌 **컬럼(열) 기반으로 저장**하며, 대규모 대형 쿼리를 위해 대규모 병렬 처리(MPP) 엔진을 사용합니다.
* **비용 및 인터페이스:** 프로비저닝된 인스턴스 스펙과 시간에 따라 요금을 지불하며, 쿼리를 위해 표준 SQL 인터페이스를 제공하여 `QuickSight`나 `Tableau` 같은 BI 도구와 쉽게 통합됩니다.
* **Athena와의 차이점:** Athena와 달리 물리적인 **인덱스**를 활용하므로 대규모 데이터 간의 복잡한 쿼리, 조인(Join), 집계 연산 속도가 압도적으로 빠릅니다.

#### 🏗️ Redshift Cluster 아키텍처

* **리더 노드(Leader Node):** 클라이언트 연결을 받고, SQL 쿼리를 받아 최적의 쿼리 실행 계획을 수립하며, 하위 노드들이 넘겨준 결과를 최종 집계하여 클라이언트에 돌려주는 노드입니다.
* **컴퓨트 노드(Compute Node):** 리더 노드의 명령을 받아 실제 쿼리를 실행하고 데이터를 로컬에 저장하며, 결과를 다시 리더 노드로 전송하는 실무 노드입니다. (노드 크기는 미리 프로비저닝해야 합니다.)
* **비용 절감:** 상시 구동되므로 예약 인스턴스(Reserved Instances, RI)를 구매하여 비용을 절감하는 것이 좋습니다.

#### 🛡️ Snapshots & DR (재해 복구)

* **⚠️ 주의 (다중 AZ 불가):** Redshift는 기본적으로 **다중 AZ(Multi-AZ) 모드를 지원하지 않습니다.** 대신 고도의 스냅샷 전략을 씁니다.
* **스냅샷 특징:** 클러스터의 지정 시간 백업본이며 **Amazon S3** 내부에 저장됩니다. 변경 사항만 저장하는 **증분(Incremental) 방식**이며, 이를 통해 새로운 클러스터로 그대로 복원할 수 있습니다.
* **자동 백업:** 매 8시간마다, 혹은 5GB 데이터가 바뀔 때마다, 또는 유저가 설정한 일정에 따라 자동 생성됩니다. (보존 기간은 1일~35일 설정 가능)
* **수동 백업:** 유저가 직접 삭제할 때까지 영구 보존됩니다.
* **교차 리전 복제 (DR):** 클러스터의 자동/수동 스냅샷을 **다른 AWS 리전으로 자동으로 복사**하도록 구성하여 리전 장애에 대비하는 재해 복구 전략을 사용할 수 있습니다.

#### 📥 데이터 로딩 및 Spectrum

* **데이터 로드 규칙:** 자잘하게 Insert 구문을 수만 번 날리는 것보다, 대량의 데이터를 한 번에 밀어 넣는 대규모 인서트(Large inserts)가 성능상 훨씬 유리합니다.
* **✨ Redshift Spectrum:** Amazon S3에 저장된 대용량 데이터를 Redshift 클러스터 내부로 **로드(이동)하지 않고도 즉시 분석**할 수 있게 하는 기능입니다. 쿼리를 시작하려면 활성화된 Redshift 클러스터가 있어야 하며, 쿼리 실행 시 수천 개의 독립된 Redshift Spectrum 노드가 할당되어 연산을 처리합니다.

### 🔍 3. Amazon OpenSearch Service

Amazon Elasticsearch의 라이선스 문제로 인해 이름이 변경된 후속작으로, 전문 검색 및 로그 분석 엔진입니다.

* **상세 개념:** DynamoDB 같은 일반 NoSQL은 기본 키나 인덱스로만 조회가 가능하지만, OpenSearch는 **부분적으로 일치하는 필드를 포함하여 모든 필드를 자유롭게 검색**할 수 있습니다. (보통 단독 메인 DB보다는 다른 DB의 검색 기능을 보완하기 위해 조합하여 사용합니다.)
* **인프라 구조:** 서버리스가 아닙니다. 사용하려면 인스턴스 클러스터를 직접 생성하고 크기를 프로비저닝해야 합니다.
* **쿼리 언어:** 자체 쿼리 언어(DSL)를 사용하며 기본적으로 **SQL을 지원하지 않습니다.**
* **데이터 주입(Ingestion) 소스:** 주로 `Kinesis Data Firehose`, `Kinesis Data Streams`, `CloudWatch Logs`, `AWS IoT`, 사용자 지정 앱의 로그 데이터를 주입받아 저장합니다.
* **보안 및 시각화:** Amazon Cognito 및 IAM과 통합되어 완벽한 인증 및 저장/전송 중 암호화를 지원하며, 강력한 전용 시각화 도구인 OpenSearch 대시보드(구 Kibana)가 함께 제공됩니다.

### 🐘 4. Amazon EMR (Elastic MapReduce)

AWS에서 대규모 빅데이터 처리를 위해 **하둡(Hadoop) 및 스파크(Spark) 클러스터**를 손쉽게 생성하고 관리해 주는 서비스입니다.

* **에코시스템 통합:** `Apache Spark`, `HBase`, `Presto`, `Flink` 등 오픈소스 빅데이터 도구들의 복잡한 프로비저닝과 인프라 구성을 AWS가 대신 처리해 줍니다.
* **확장성:** 클러스터는 수백 개의 EC2 인스턴스로 구성될 수 있으며 오토스케일링을 지원합니다.
* **사용 사례:** 대용량 데이터 처리, 머신러닝 모델링, 대규모 웹 인덱싱, 빅데이터 로그 분석 등.

#### 🖥️ EMR 노드 유형 및 구매 옵션 (★시험 단골 비용 최적화)

* **노드 유형:**
1. **마스터 노드:** 클러스터 전체를 총괄 관리하고 하위 노드들의 상태와 작업 일정을 조율/모니터링합니다. (지속 실행 필수)
2. **코어 노드:** 마스터의 명령을 받아 실제 태스크를 실행함과 동시에 클러스터 내부의 데이터를 물리적으로 저장(HDFS)하는 역할을 합니다. (지속 실행 필수)
3. **태스크 노드 (선택 사항):** 데이터를 저장하지 않고, 오직 **작업 실행(연산)에만** 세를 보태는 노드입니다.


* **인스턴스 구매 및 배포 전략:**
* **On-Demand / Reserved(예약 인스턴스):** 작업 도중 절대 죽으면 안 되고 장기 실행되어야 하는 **마스터 노드와 코어 노드**에 적합합니다. (최소 1년 이상 장기 실행 시 EMR이 RI를 자동 적용하여 비용을 아낍니다.)
* **Spot 인스턴스(스팟):** 언제든 종료될 수 있어 신뢰성은 낮지만 비용이 최대 90% 저렴하므로, 데이터 유실 위험이 없고 연산만 돕는 **태스크 노드**에 배치하기 가장 좋습니다.


* **운영 패턴:** 24시간 내내 켜두는 **장기 실행 클러스터**에 RI를 적용해 쓰거나, 특정 빅데이터 배치 작업이 있을 때만 **임시 클러스터**를 생성해 분석을 끝내고 바로 클러스터를 통째로 삭제하여 비용을 극단적으로 아끼는 방식을 씁니다.

### 📊 5. Amazon QuickSight

서버리스 환경에서 구동되는 머신러닝 기반의 비즈니스 인텔리전스(BI) 대시보드 시각화 서비스입니다.

* **상세 개념:** 별도의 인프라 관리 없이 대화형 대시보드를 생성할 수 있으며, 오토스케일링이 탑재되어 있고 세션별로 비용이 책정됩니다.
* **데이터 소스 연결:** `RDS`, `Aurora`, `Athena`, `Redshift`, `S3` 등 다양한 AWS 내 데이터 소스들과 네이티브하게 연결됩니다.
* **🚀 SPICE 엔진:** QuickSight 내부로 소스 데이터를 복사해 들여왔을 때 활성화되는 **초고속 인메모리 계산 엔진**입니다. 대규모 데이터의 다차원 계산과 시각화 반응 속도를 극대화해 줍니다.
* **보안 기능:** 엔터프라이즈 에디션(Enterprise Edition)을 사용하면 컬럼 수준 보안(`Column-Level security, CLS`) 설정이 가능하여, 특정 유저 그룹에게 특정 열의 노출을 완벽히 차단할 수 있습니다.

#### 👥 QuickSight 권한 및 구성 요소 (대시보드 vs 분석)

* **사용자 및 그룹 관리:** * **Standard 에디션:** 개별 사용자 정의만 가능합니다.
* **Enterprise 에디션:** 사용자들을 묶는 '그룹(Group)' 정의가 가능합니다.
* ⚠️ **주의:** QuickSight 내부의 사용자와 그룹은 **QuickSight 서비스 전용 독립 멤버십**이며, AWS **IAM 사용자와는 완전히 별개**입니다!


* **분석(Analysis)과 대시보드(Dashboard)의 메커니즘:**
* **분석(Analysis):** 개발자가 필터링, 매개변수 설정, 정렬 옵션 등을 조절해가며 시각화 화면을 직접 구성하고 편집하는 실시간 작업 공간입니다.
* **대시보드(Dashboard):** 분석 결과를 다른 사람들과 공유하기 위해 만드는 **읽기 전용 스냅샷**입니다.
* **공유 규칙:** 분석 화면을 공유하려면 먼저 반드시 **대시보드로 게시(Publish)** 단계를 거쳐야 유저나 그룹에게 공유할 수 있습니다. 대시보드 액세스 권한을 부여받은 일반 사용자들은 원본 수정은 못 하지만 내부에 맵핑된 기본 데이터들을 시각적으로 자유롭게 소비할 수 있습니다.

### 🧪 6. AWS Glue & Lake Formation

#### 🛠️ AWS Glue (완전 관리형 서버리스 ETL)

분석을 위해 원천 데이터를 추출(Extract), 변환(Transform), 로드(Load)하는 데 특화된 **완전 서버리스 ETL 서비스**입니다.

* **Glue Data Catalog:** 사방에 흩어진 데이터 세트들을 크롤러가 훑고 지나가며 스키마 정보를 수집해 두는 **중앙 메타데이터 카탈로그 장부**입니다. Athena나 Redshift가 S3 데이터를 읽을 때 이 장부를 참조합니다.
* **작업 북마크 (Job Bookmarks):** ETL 작업을 반복 실행할 때, 이미 이전에 성공적으로 처리한 과거 데이터를 기억하여 **중복 처리 및 재처리를 완벽히 방지**합니다.
* **Glue Elastic Views:** SQL을 활용하여 서로 다른 여러 데이터 저장소의 데이터를 결합하고 복제할 수 있는 기능입니다. 커스텀 코딩이 전혀 필요 없으며, 원본 데이터가 변경되면 Glue가 이를 감지하여 가상 테이블(`Materialized View`)에 실시간으로 반영해 주는 서버리스 기능입니다.
* **Glue DataBrew:** 코딩을 전혀 할 줄 몰라도, 이미 빌드된 수백 가지의 데이터 변환 아이콘들을 클릭하여 데이터를 깨끗하게 정제하고 정규화할 수 있는 시각적 툴입니다.
* **Glue Studio:** GUI(시각적 도구) 화면을 통해 복잡한 ETL 파이프라인 작업을 직관적으로 디자인하고 생성, 실행, 모니터링할 수 있는 콘솔입니다.
* **Glue 스트리밍 ETL:** Apache Spark 구조적 스트리밍을 기반으로 작동하며, `Kinesis Data Streams`, `Apache Kafka`, `Amazon MSK` 등 끊임없이 들어오는 실시간 스트림 데이터를 중간에 가로채어 실시간으로 데이터 포맷을 변환하고 정제해 줍니다.

#### 🏰 AWS Lake Formation

분석용 데이터 레이크(모든 정형/비정형 데이터를 모아두는 중앙 기지)를 단 며칠 만에 수월하게 안전 구축할 수 있도록 돕는 완전 관리형 데이터 보안 및 거버넌스 레이어입니다.

* **작업 자동화:** 복잡하고 수동으로 해야 했던 데이터 수집, 정제, 이동, 카탈로깅, 복제 단계를 자동화하며, 내장된 머신러닝(ML) 변환을 사용해 중복 데이터를 자동으로 걸러내 줍니다.
* **소스 블루프린트:** `S3`, `RDS`, 관계형 DB 및 `NoSQL DB` 등 다양한 소스로부터 데이터를 쉽게 퍼 올 수 있는 사전 구성된 블루프린트(템플릿)를 제공합니다.
* **세분화된 보안 제어 (★최다 출제):** 데이터 레이크에 접근하는 각 유저나 애플리케이션별로 **행(Row) 수준 및 열(Column) 수준의 초정밀 액세스 권한 제어**가 가능합니다.
* **⚠️ 아키텍처 주의 점:** 기술적으로 **AWS Glue 위에 빌드되는 상위 보안/관리 계층**입니다. 따라서 Lake Formation을 쓰면 사용자가 Glue API와 복잡하게 직접 상호 작용하지 않고 한 곳에서 통제가 가능해집니다.

### 🌊 7. 실시간 스트리밍 데이터 분석 서비스

#### 🔢 Kinesis Data Analytics

스트리밍 데이터가 대상을 향해 흘러가는 파이프라인 중간에서 서버 프로비저닝 없이 **실시간으로 데이터를 가공하고 분석 쿼리를 날리는** 완전 관리형 서버리스 서비스입니다.

* **Kinesis Data Analytics for SQL application:**
* `Kinesis Data Streams` 및 `Kinesis Data Firehose`를 통과하는 실시간 데이터에 대고 표준 SQL 구문을 사용해 실시간 분석을 수행합니다.
* 분석 도중 **Amazon S3**에 저장된 정적 참조 데이터(예: 사용자 프로필 장부)를 쿼리에 결합하여 스트리밍 데이터를 더 풍부하게 보강(Enrichment)할 수 있습니다.
* 인프라를 수동 관리하지 않고 처리량에 따라 알아서 오토스케일링되며, 사용한 데이터양만큼만 요금을 냅니다.
* **출력(Output) 방향:** 실시간 쿼리로 걸러낸 결과 스트림을 다시 `Kinesis Data Streams`로 보내 후속 처리를 하거나, `Kinesis Data Firehose`로 넘겨 타깃 저장소로 안전하게 밀어냅니다.
* **사용 사례:** 실시간 시계열 통계 분석, 실시간 관제 대시보드 메트릭 산출 등.


* **Kinesis Data Analytics for Apache Flink:**
* 단순 SQL을 넘어 Java, Scala, SQL 언어로 복잡한 스트리밍 연산이 가능한 **Apache Flink 애플리케이션**을 AWS가 관리하는 클러스터 위에서 구동시키는 기능입니다.
* 연산 리소스를 자동으로 프로비저닝하고 병렬 연산 및 오토스케일링을 수행합니다.
* 애플리케이션의 유실 없는 백업을 위해 내부적인 체크포인트(Checkpoint)와 스냅샷(Snapshot) 기능을 기본 제공합니다.
* ⚠️ **주의 (스트리밍 호환성 차이):** Apache Flink 엔진은 **Kinesis Data Firehose로부터 데이터를 직접 읽어 들이지 못합니다.** (Firehose를 직독직해하여 실시간 분석하려면 반드시 위의 SQL 애플리케이션용 버전을 선택해야 시험에서 감점이 안 됩니다.)



#### 🎼 Amazon MSK (Managed Streaming for Apache Kafka)

대규모 실시간 데이터 스트리밍 플랫폼인 오픈소스 `Apache Kafka` 클러스터를 AWS 환경에서 쉽게 다룰 수 있게 해주는 완전 관리형 서비스입니다. (Kinesis의 강력한 아키텍처 대안입니다.)

* **인프라 자동화:** 클러스터 생성, 업데이트, 삭제가 자유로우며 Kafka의 핵심인 브로커(Broker) 노드와 사육사 역할을 하는 주키퍼(Zookeeper) 노드를 AWS가 자동으로 백엔드에서 생성하고 관리합니다.
* **고가용성:** 기본적으로 사용자의 VPC 내부에서 최대 3개의 다중 가용 영역(Multi-AZ) 전체에 브로커들을 고르게 분산 배포하여 가용성을 보장하며, 일반적인 Kafka 장애 시 알아서 자동 복구합니다.
* **안전한 스토리지:** 유실되면 안 되는 스트림 데이터를 보존 기간 제한 없이 원하는 만큼 안정적인 물리 **EBS 볼륨**에 영구 저장할 수 있습니다.
* **MSK Serverless:** 브로커 개수나 스토리지 용량 산정(용량 계획)을 할 필요 없이, MSK가 트래픽을 모니터링하여 컴퓨팅 자원과 스토리지를 알아서 자동으로 늘리고 줄여주는 서버리스 모드입니다.
* **소비 아키텍처 (Consumers):** MSK에 쌓인 실시간 스트림 데이터는 주로 `AWS Glue 스트리밍 ETL 작업`, `대규모 데이터 주입 파이프라인`, 또는 커스텀하게 코딩된 카프카 컨슈머 앱들이 실시간으로 가져가서 소비하게 됩니다.

## 🏗️ 3단계: 종합 대규모 데이터 수집 파이프라인 (Big Data Ingestion Pipeline)

위의 모든 조각이 하나로 유기적으로 연결된 **완전 서버리스 대규모 실시간 수집 및 데이터 웨어하우징 파이프라인**의 전체 동작 메커니즘입니다.

1. **실시간 수집 단계 (IoT Core $\rightarrow$ Kinesis):** 수만 대의 온프레미스 IoT 디바이스에서 발생하는 데이터가 **IoT Core**를 거쳐 진입하며, 대규모 트래픽을 유실 없이 받아내기 위해 실시간 수집기인 **Kinesis Data Streams** 스트림 파이프에 즉시 적재됩니다.
2. **실시간 가공 및 수송 단계 (Firehose + Lambda):** 수집된 날것의 데이터 스트림을 **Kinesis Data Firehose**가 이어받습니다. Firehose는 데이터를 영구 저장소로 전달하기 전, 중간에 **AWS Lambda**를 결합하여 행(Row) 기반의 무거운 데이터를 정제하고 변환하는 작업을 거칩니다. (Firehose의 최소 전송 파일 생성 주기 설정은 1분 단위입니다.)
3. **중앙 데이터 레이크 안착 (Amazon S3):** 변환이 완료된 깨끗한 데이터 파일들은 무한 확장이 가능한 중앙 데이터 저장소인 **Amazon S3** 버킷(데이터 레이크)에 덤프 트럭처럼 적재됩니다.
4. **이벤트 비동기 트리거 (S3 $\rightarrow$ SQS $\rightarrow$ Lambda):** S3 버킷에 새로운 데이터 파일이 정상 업로드되면 **S3 Event Notification**이 비동기적으로 발동하여 Amazon SQS(메시지 대기열)로 알림 메시지를 던집니다. SQS를 리스닝하며 대기하고 있던 다음 단계의 **AWS Lambda**가 메시지를 받아 깨어나며 후속 백엔드 가공 및 자동화 인프라 프로세스를 안전하게 처리합니다.
5. **임시 서버리스 쿼리 분석 (Athena):** 분석가들은 인프라 서버 구동이나 로드 작업 없이, **Amazon Athena**를 활용해 S3에 적재된 대용량 로그나 변환 파일에 직접 표준 SQL 쿼리를 날려 필요한 인사이트를 즉석 쿼리합니다. Athena가 실행한 쿼리의 분석 결과 파일은 S3의 별도 '보고 버킷'에 안전하게 격리 저장됩니다.
6. **고성능 데이터 웨어하우징 및 대시보드 시각화 (Redshift $\rightarrow$ QuickSight):** 장기 보존 및 복잡한 다차원 집계 분석이 필요한 핵심 정제 데이터 세트는 PB 단위 고성능 인덱싱 처리가 가능한 **Amazon Redshift** 데이터 웨어하우스로 대량 인서트(Large inserts)되어 정교하게 구조화됩니다. 최종적으로 C-Level 임원들과 비즈니스 분석가들은 **AWS QuickSight(SPICE 고속 인메모리 엔진 구동)** 대시보드를 열어 실시간 비즈니스 인사이트 리포트 및 시각화 화면을 지연 없이 깔끔하게 소비하게 됩니다.
