# 📊 AWS 데이터베이스 아키텍처 및 인메모리 캐싱 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **관계형 DB 자동 패치 및 특정 시점 복원(PITR) + 수평 읽기 확장(최대 15대) + SSH 접속 절대 불가** $\rightarrow$ `Amazon RDS`
* **남은 디스크 공간 10% 미만 + 5분 이상 지속 시 자동으로 용량을 증설하여 수동 프로비저닝 오버헤드 제거** $\rightarrow$ `RDS Storage Auto Scaling`
* **메인 프로비저닝 DB에 영향을 주지 않고 무중단 대용량 데이터 보고서/분석 워크로드 분산(비동기 복제)** $\rightarrow$ `RDS Read Replicas (읽기 전용 복제본)`
* **동기식 복제 + 단일 DNS 엔드포인트 제공 + AZ/스토리지 전면 장애 시 무중단 자동 페일오버(재해 복구)** $\rightarrow$ `RDS Multi-AZ`
* **기저 OS 가상 가상 서버(EC2) 관리자 권한 필수 + Oracle/SQL Server 네이티브 기능 패치 및 SSH/SSM 직접 통제** $\rightarrow$ `RDS Custom`
* **3개 AZ에 6개 데이터 사본 분산 + 10ms 미만 지연 + 쿼터(Writer 1대/Reader 최대 15대) 엔드포인트 완벽 분리** $\rightarrow$ `Amazon Aurora`
* **예측 불가능하거나 간헐적인 트래픽 피크 + 인프라 용량 산정 불필요 + 초당 결제 기반 자동 DB 온앤오프** $\rightarrow$ `Aurora Serverless`
* **비즈니스 결함으로 데이터 오염 시 백업본 복원 없이 특정 과거 시점으로 DB 볼륨 전체를 즉시 되감기(Rollback)** $\rightarrow$ `Aurora Backtrack (백트랙)`
* **기존 라이브 프로데 DB에 부하를 주지 않고 테스트용 스테이징 환경을 초고속 및 비용 효율적으로 즉시 복제하는 기술** $\rightarrow$ `Aurora Database Cloning`
* **DB 자격 증명을 Secrets Manager와 연동 + 대규모 커넥션 풀 공유로 CPU 부하 최소화 + 완벽한 VPC 전용 가상 프록시** $\rightarrow$ `Amazon RDS Proxy`
* **읽기 집약 워크로드 DB 부하 완화 + 애플리케이션의 상태 없음(Stateless) 아키텍처 지원 + 고속 인메모리 스토어** $\rightarrow$ `Amazon ElastiCache`
* **캐시 데이터 무결성 영구 유지(AOF) + Multi-AZ 자동 페일오버 지원 + 게임 실시간 순위 산정을 위한 Sorted Sets 활용** $\rightarrow$ `ElastiCache for Redis`
* **데이터 조회 시 캐시에 없으면(Cache Miss) DB에서 읽어와 캐시를 나중에 채우는 가장 대중적인 지연 로딩 패턴** $\rightarrow$ `Lazy Loading / Cache-Aside`
* **인메모리 초고속 속도(Microsecond)와 복제 트랜잭션 로그 기반의 완벽한 데이터 내구성(영구 보존)을 동시 만족** $\rightarrow$ `Amazon MemoryDB for Redis`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 💾 1. Amazon RDS (Relational Database Service)

RDS는 클라우드 가상 공간에 관계형 데이터베이스(PostgreSQL, MySQL, MariaDB, Oracle, SQL Server, Aurora)를 빠르게 배포하고 운영 체제 패치, 백업 등의 반복 작업을 AWS가 전담하는 관리형 DB 서비스입니다. **가장 핵심적인 제약은 기저 인프라 가상 서버에 SSH로 직접 접속하는 것이 전면 차단된다는 점입니다.**

#### ⏫ RDS Storage Auto Scaling

* **작동 원리:** DB 디스크 용량이 한계에 도달하면 다운타임 없이 스토리지를 동적으로 자동 스케일 아웃합니다.
* **임계값 트리거 조건 (아래 조건 동시 충족 시 자동 스케일링 수행):**
1. 남은 데이터 공간이 **전체 스토리지의 10% 미만**으로 떨어질 때.
2. 이 스토리지 부족 상태가 최소 **5분 이상 지속**될 때.
3. 이전 용량 수정 작업이 일어난 지 최소 **6시간이 경과**했을 때.



#### 🗂️ 읽기 복제본(Read Replicas) vs 다중 가용 영역(Multi-AZ) 집중 비교

| 비교 아키텍처 축 | 🗂️ 읽기 전용 복제본 (Read Replicas) | 📍 다중 가용 영역 (Multi-AZ) |
| --- | --- | --- |
| **주 목적** | **읽기 스케일링 성능 확장 (Scalability)** | **고가용성 및 인프라 재해 복구 (Disaster Recovery)** |
| **복제 메커니즘** | **비동기식 복제 (Asynchronous)** | **동기식 복제 (Synchronous)** |
| **가동 스펙** | 각 복제본마다 고유의 독립 DNS 엔드포인트 할당 | **단 하나의 대표 DNS 이름 공유** (장애 시 스탠바이로 자동 승격) |
| **최대 확장 수** | 클러스터당 **최대 15개**까지 스케일 아웃 가능 | 원본 마스터 1대 + 타 AZ 대기 조(Standby) 1대 이중화 고정 |
| **쓰기/조회 권한** | 오직 `SELECT` 조회 전용 (필요시 독자 DB로 승격 가능) | 스탠바이 DB는 평소에 접속 불가 (트래픽 처리 X) |
| **리전 범위** | 동일 AZ, 타 AZ는 물론 **타 리전(Cross-Region) 구성 가능** | 동일 리전 내의 서로 다른 가용 영역(AZ) 간 복제 구조 |
| **네트워크 비용** | **동일 리전 내 타 AZ 간 복제는 네트워크 비용 면제(Free)** | 기본 가용성 동기화 트래픽 비용은 서비스 요금에 포함됨 |

#### 🪄 Single-AZ에서 Multi-AZ로의 무중단 전환 공정

서비스 가동 중에 단일 DB를 다중 AZ 고가용성 구조로 가동 중단 없이 원클릭 전환할 수 있습니다.

1. 관리자가 콘솔에서 Multi-AZ 활성화를 클릭하면, RDS가 가동 중인 원본 DB의 **자동 스냅샷을 즉시 백그라운드에서 촬영**합니다.
2. 해당 스냅샷을 기반으로 **새로운 가용 영역(AZ)에 똑같은 스탠바이 DB를 무중단 복원**합니다.
3. 두 DB 간의 **동기식(Synchronous) 데이터 복제 채널을 최종 연결**하여 이중화를 완료합니다.

#### 🏷️ OS 통제권 융합 솔루션: RDS Custom

* **정의:** 완전 관리형 RDS의 이점과 EC2의 자유도를 결합한 특수 모델입니다. (**Oracle 및 Microsoft SQL Server 엔진 한정 지원**)
* **차별점:** 기저 가상 서버에 관리자 권한으로 직접 잠입할 수 있으며, **SSH 또는 SSM 세션 매니저를 통한 직접 OS 접속이 허용**됩니다. 내부 커널 구성 변경, 레거시 서드파티 네이티브 패치 설치가 가능합니다.
* **주의점:** 임의 변경으로 인한 시스템 붕괴 리스크가 있으므로, 커스텀 작업 중에는 AWS의 자동 유지 관리(유지보수 놉) 기능을 잠시 꺼두어야 하며, 상시 스냅샷 촬영이 권장됩니다.

### 👑 2. Amazon Aurora 엔터프라이즈 아키텍처

Aurora는 AWS의 클라우드 인프라에 완전 최적화되어 빌드된 독자적인 고성능 관계형 데이터베이스 엔진입니다. (MySQL 및 PostgreSQL 호환)

#### 🔖 고가용성 및 스토리지 가상화 분산 기술

* **6-Way 복제 스펙:** Aurora는 데이터를 단일 드라이브에 넣지 않고, **3개의 가용 영역(AZ)에 걸쳐 총 6개의 데이터 사본을 분산 저장**하는 독보적인 스토리지 가상화 구조를 취합니다.
* **합의 쿼럼(Quorum) 규칙:**
* **쓰기(Write) 검증:** 6개의 사본 중 **최소 4개의 승인**이 떨어지면 쓰기 성공 판정을 내립니다. (즉, 1개 AZ가 완전히 전멸해도 쓰기 가동성이 무너지지 않습니다.)
* **읽기(Read) 검증:** 6개의 사본 중 **최소 3개의 승인**으로 초고속 데이터를 읽어옵니다.


* **자가 치유(Self-Healing):** 백엔드에서 P2P(Peer-to-Peer) 통신을 통해 손상된 스토리지 블록을 백그라운드에서 자동으로 지속 복구합니다. 스토리지 볼륨은 10GB 단위로 자가 증설되어 **최대 128TB까지 자동 확장**됩니다.

#### 📚 Aurora 엔드포인트(Endpoint) 분리 매커니즘

Aurora는 연결 관리(**Connection-level 로드 밸런싱**)를 위해 지 진입 주소를 분리 노출합니다.

* **Writer Endpoint:** 데이터 삽입 및 수정을 담당하며, 클러스터 내의 오직 단 1대의 마스터(Primary) DB 인스턴스를 상시 타겟팅합니다.
* **Reader Endpoint:** 데이터 조회를 전담하며, 뒤에 붙은 **최대 15개의 Aurora 읽기 복제본(Replicas)** 풀로 트래픽을 자동 부하 분산 처리합니다. (10ms 미만의 극초저 복제 지연을 달성합니다.)
* **Custom Endpoint:** 대형 분석용 쿼리나 고사양 연산 처리를 위해 특정 고성능 복제본 인스턴스들만 별도의 묶음 주소로 커스텀 정의하여 격리 배포하는 방식입니다.

#### ✨ Aurora의 핵심 고급 기능 세트

* **Aurora Serverless:** 트래픽이 아예 없을 때는 DB를 완전히 잠재우고, 예측 불가능한 스파이크 트래픽이 몰아칠 때만 인프라 사양을 완전 자동 확장/축소하는 초당 과금형 경제적 모델입니다.
* **Aurora Multi-Master:** 클러스터 안의 **모든 가상 DB 노드가 동시에 읽기/쓰기 권한**을 행사합니다. 특정 노드가 무너져도 30초 대기 없이 즉각적인 무중단 쓰기 장애 조치(Failover)가 연속 유지되어야 할 때 채택합니다.
* **Global Aurora:** 기본 리전(1개) 외에 전 세계에 최대 5개의 보조 읽기 리전(Global Database)을 구성합니다. 리전 간 복제 지연은 **평균 1초 미만**이며, 메인 리전 대형 재해 시 다른 리전을 메인으로 승격시키는 **RTO(복구 시간 목표)를 1분 미만**으로 달성할 수 있습니다.
* **Aurora Database Cloning:** 스냅샷 복원 방식과 달리, 기존 라이브 데이터 볼륨의 포인터를 그대로 공유하여 **프로덕션 가동에 1프로의 영향도 주지 않고 스테이징/테스트용 카피 DB 클러스터를 초고속으로 복제 생성**하는 기술입니다. 데이터 변경 분이 발생한 블록만 독자 기록하므로 비용 효율성이 극대화됩니다.
* **Aurora Machine Learning:** SQL 구문 인터페이스 내부에서 **Amazon SageMaker**나 **Amazon Comprehend** 머신러닝 서비스를 API 연동 방식으로 직접 호출하여, 복잡한 ML 코드 작성 없이 데이터베이스 레벨에서 사기 감지, 감정 분석, 추천 시스템 쿼리를 수행하는 인공지능 융합 기술입니다.

### 🚧 3. 데이터베이스 커넥션 관리자: Amazon RDS Proxy

애플리케이션 가상 서버들이 데이터베이스에 직접 수천 개의 접속 세션을 맺으면 DB의 CPU와 RAM 자원이 커넥션 유지 오버헤드로 인해 고갈됩니다. RDS 프록시는 이 문제를 원천 차단하는 완전 관리형 가상 프록시입니다.

* **커넥션 풀링(Connection Pooling):** 수많은 애플리케이션 연결을 프록시 내부 풀에 안전하게 수용하고, 실제 다운스트림 DB 인스턴스로는 압축된 소수의 정제된 연결만 공유 및 다중화(Multiplexing)하여 전송하므로 DB 효율성이 극대화됩니다.
* **초고속 장애 조치:** RDS/Aurora의 가상 페일오버 발생 시, 애플리케이션 코드를 고치거나 재부팅할 필요 없이 프록시가 내부 단에서 연결 인계를 전담하여 **장애 조치 지연 시간을 최대 66%까지 대폭 단축**시킵니다.
* **철저한 VPC 전용 보안:** **퍼블릭 인터넷 진입 주소를 절대 가질 수 없으며**, 오직 내부 VPC 공간에서만 접근 가능합니다. **AWS Secrets Manager**와 통합되어 DB 계정 자격 증명을 안전하게 은닉하고, DB 연결 시 강력한 **IAM 인증 체계를 강제**할 수 있습니다.

### ⚡ 4. Amazon ElastiCache (인메모리 캐시 서비스)

관계형 DB 전면에 배치되어 마이크로초(Microseconds) 단위의 응답 속도를 보장하고, 읽기 집약적 워크로드의 압도적인 부하 완화 및 웹 어플라이언스의 무상태(Stateless) 세션 공유 저장을 지원하는 고속 인메모리 데이터베이스 서비스입니다.

#### ℹ️ Redis vs Memcached 아키텍처 정밀 비교

| 비교 기술 표준 | 🟥 ElastiCache for Redis | 🟦 ElastiCache for Memcached |
| --- | --- | --- |
| **고가용성 체계** | **Multi-AZ 아키텍처 지원 (자동 페일오버)** | 고가용성 미지원 (단일 노드 아키텍처) |
| **데이터 영구 내구성** | **AOF(Append Only File) 지속성 및 백업/복원 제공** | 데이터 휘발성 전용 (백업 및 복원 기능 전무) |
| **수평 확장 방식** | 읽기 전용 복제본(Read Replicas) 구성 가능 | **다중 노드 데이터 샤딩(Sharding) 분산 파티셔닝** |
| **내부 아키텍처** | 싱글 스레드 기반 연산 통제 | **멀티 스레드(Multi-threaded) 코어 처리** |
| **특수 자료 구조** | Sets, **Sorted Sets(실시간 게임 리더보드용)** 지원 | 단순 Key-Value 구조 저장 표준 |

#### 🧲 개발자 필수 습득 캐싱 디자인 패턴 (DVA-C02 단골)

##### ① Lazy Loading / Cache-Aside (지연 로딩 패턴)

[Image diagram showcasing the Cache-Aside architecture step by step checking cache first before reading from the main relational database on miss]

* **동작 시나리오:** 애플리케이션이 데이터를 조회할 때 **캐시 스토어를 항상 먼저 찌릅니다.**
* **Cache Hit:** 캐시에 데이터가 존재하면 즉시 반환하고 종료합니다.
* **Cache Miss (3번의 네트워크 호출 오버헤드):** 캐시에 없으면 ➡️ 원본 관계형 DB로 넘어가 데이터를 읽어온 후 ➡️ 이를 캐시에 먼저 밀어 넣고 ➡️ 클라이언트에 최종 반환합니다.


* **장단점:** 실제 요청이 들어온 알짜배기 데이터만 캐시에 적재되므로 메모리 낭비가 없고 캐시 노드가 터져도 서비스는 유지됩니다. 다만, 원본 DB 데이터가 수정되었을 때 캐시 내 데이터는 과거의 잔재(**Stale 데이터 상태**)로 오염될 수 있으므로, 이를 방지하기 위한 **TTL(Time-To-Live) 유효 기간 설정이 필수적으로 동반**되어야 합니다.

##### ② Write Through (즉시 기록 패턴)

* **동작 시나리오:** 데이터의 삽입, 수정, 업데이트 작업이 발생할 때 **원본 데이터베이스와 캐시 스토어 양쪽에 데이터를 동시에 즉각 기록**하는 방식입니다.
* **장단점:** 캐시 데이터가 항상 최신 상태를 100% 유지하므로 데이터 오염(Stale) 리스크가 전혀 없고 조회가 매우 빠릅니다. 반면, 쓰기 작업 시 매번 두 공간에 동시 기록하므로 **쓰기 지연 패널티**가 유발되며, 한 번도 읽지 않을 쓰레기 데이터까지 캐시에 가득 차는 메모리 이탈률(Churn)이 발생할 수 있어 보통 지연 로딩 패턴과 결합해 최적화합니다.

#### 🆕 엔터프라이즈 하이브리드 캐시: Amazon MemoryDB for Redis

* **핵심 정의:** Redis의 마이크로초 단위 초고속 인메모리 스펙을 그대로 유지하면서, 분산 트래픽 트랜잭션 로그 아키텍처를 결합하여 **단 1비트의 데이터 유실도 허용하지 않는 강력한 '내구성'을 결합한 고성능 인메모리 데이터베이스**입니다.
* **차별점:** 일반 캐시는 보조 저장소이지만, MemoryDB는 그 자체로 **완벽한 메인 주 데이터베이스(Primary Database)** 역할을 완벽히 수행해 낼 수 있습니다.
