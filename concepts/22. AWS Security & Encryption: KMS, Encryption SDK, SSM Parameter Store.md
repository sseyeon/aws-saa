## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🔄 1. Encryption 101 (암호화의 3대 분류)

* **전송 중 암호화 (Encryption in flight):** 데이터가 네트워크로 전송되기 전에 암호화되고, 서버 수신 후 복호화됩니다. SSL/TLS 인증서(HTTPS)를 통해 구현하며, 중간자 공격(MITM)을 원천 방지합니다.
* **저장 시 서버 측 암호화 (Server-side encryption at rest):** 서버가 데이터를 수신한 '후' 암호화하여 저장하고, 클라이언트로 전송하기 직전에 복호화하여 보냅니다. 데이터 키를 사용하여 암호화된 형태로 저장하며, KMS와 같은 키 관리 시스템과의 통신이 필수적입니다.
* **클라이언트 측 암호화 (Client-side encryption):** 데이터가 클라이언트 측에서 이미 암호화되어 서버로 전송되므로, **서버는 절대로 데이터를 복호화할 수 없습니다.** 데이터를 수신하는 최종 클라이언트만 복호화 권한을 가지며, **봉투 암호화(Envelope Encryption)** 기법을 활용할 수 있습니다.
* **⚠️ 절대 주의:** 암호 데이터는 어떠한 경우에도 평문(Plaintext)으로 저장하면 안 되며, 특히 소스 코드에 남기는 행위는 금지됩니다.

### 🔑 2. AWS KMS (Key Management Service)

AWS 암호화 아키텍처의 핵심 서비스로, 키 권한 제어를 위해 IAM 정책 및 키 정책과 완벽하게 통합됩니다. 모든 키 사용 이력은 `CloudTrail` 로그에 실시간 기록되어 감사에 활용됩니다.

#### 🗂️ KMS Key 유형 및 비용 구조

* **대칭 키 (Symmetric Key - AES-256):** 암호화와 복호화에 **동일한 단일 키**를 사용합니다. KMS와 통합된 대칭 CMK 아키텍처에서는 사용자가 키 파일 자체에 절대 직접 접근할 수 없으며, 오직 KMS API 호출을 통해서만 사용할 수 있습니다.
* **비대칭 키 (Asymmetric Key - RSA & ECC):** 퍼블릭 키(암호화)와 **프라이빗 키(복호화/전자서명)** 쌍으로 구성됩니다. 퍼블릭 키는 외부 다운로드가 가능하나 프라이빗 키는 KMS 내부에 격리됩니다. AWS 외부의 사용자가 다운로드한 퍼블릭 키로 암호화한 데이터를, AWS 내부에서 프라이빗 키로 복호화하는 시나리오에 활용됩니다.
* **소유권별 분류 및 비용:**
* **AWS 관리형 키:** `aws/service-name` (예: `aws/rds`, `aws/ebs`) 형태이며 **기본 무료**입니다.
* **고객 관리형 키 (CMK):** 사용자가 직접 생성하며 **개당 월 $1**의 비용이 발생합니다.
* **가져온 키 구성 요소 (Imported Key):** 외부의 256-bit 대칭키를 직접 업로드하여 사용하며 **개당 월 $1**의 비용이 청구됩니다.
* *공통 비용:* KMS API 호출 10,000회당 **$0.03**의 비용이 추가됩니다.


* **자동 키 로테이션 (Key Rotation) 규칙:**
* **AWS 관리형 키:** 1년에 한 번씩 시스템이 자동으로 키를 교체합니다.
* **고객 관리형 키 (CMK):** 기본 비활성화 상태이며, **사용자가 활성화 옵션을 켜야만 1년에 한 번씩 자동으로 교체**됩니다.
* **가져온 자체 키 (Imported Key):** 자동 로테이션을 지원하지 않으며, **KMS 별칭(Alias)을 사용하여 수동으로만 교체**해야 합니다.



#### 🌐 리전/계정 간 자원 복사 및 키 공유 메커니즘

KMS 키는 기본적으로 특정 리전에 종속(Region-scoped)됩니다.

* **교차 리전 스냅샷 복사 (Copying Snapshots across regions):**
1. 원본 리전에서 암호화된 EBS 스냅샷을 생성하면, 원본 리전의 KMS 키로 암호화됩니다.
2. 이를 타 리전으로 복사할 때, **원본 키를 그대로 들고 갈 수 없으므로** 대상 리전의 KMS 키를 지정하여 재암호화해야 합니다. 이 재암호화 프로세스는 AWS 백엔드에서 자동으로 처리됩니다.
3. 대상 리전으로 넘어간 스냅샷을 해당 리전의 KMS 키로 복원하여 EBS 볼륨을 생성합니다.


* **교차 계정 스냅샷 복사 (Copying Snapshots across accounts):**
1. Source 계정에서 자체 고객 관리형 키(CMK)로 암호화된 EBS 스냅샷을 생성합니다.
2. Target 계정이 해당 키를 쓸 수 있도록 KMS 키 정책(Key Policy)에 Target 계정 ID 권한을 명시적으로 추가합니다.
3. 암호화된 스냅샷을 Target 계정으로 공유합니다.
4. **Target 계정 측에서는 공유받은 스냅샷을 그대로 사용할 수 없으므로**, 반드시 스냅샷 복사본을 한 번 더 생성하면서 **Target 계정 소유의 자체 CMK 키로 재암호화**한 뒤 볼륨을 생성해야 합니다.



#### 🛡️ KMS Key Policies (키 정책)

KMS 키의 접근 제어를 담당하는 리소스 기반 정책입니다. **S3 버킷 정책과 달리, KMS 키는 키 정책이 없으면 계정 루트 권한을 포함한 그 누구도 키에 접근할 수 없습니다.**

* **기본 키 정책:** 사용자 지정 정책을 넣지 않으면 자동 생성되며, **AWS 계정의 루트 사용자에게 키에 대한 100% 완전한 권한을 부여**하여 계정 내 IAM 정책을 통해 권한이 제어되도록 통로를 열어줍니다.
* **사용자 지정 키 정책:** 키를 관리할 권한자(Key Administrators)와 사용할 권한자(Key Users)를 명확히 쪼갤 수 있으며, 특히 **교차 계정(Cross-Account) 액세스 권한을 위임할 때 필수적으로 작성**해야 합니다.

#### 🗺️ KMS Multi-Region Keys (다중 리전 키)

서로 다른 리전 간에 완전히 동일한 키 ID와 키 구성 요소(Key Material)를 공유하는 상호 교환 가능한 키 아키텍처입니다. 하나의 리전에 기본 키(Primary Key)를 생성하면 다른 리전들로 복제본(Replica Key)이 배포되는 방식입니다.

* **특징:** 전역(Global) 서비스는 아니며 독립 관리되지만, 기본 키에서 자동 키 로테이션을 수행하면 복제본 리전들도 싱크가 맞추어집니다. 원본 리전에서 암호화한 데이터를 타 리전에서 교차 리전 API 호출이나 데이터 재암호화 단계 없이 **로컬 리전의 복제 키로 즉시 복호화할 수 있어 지연 시간이 극도로 단축**됩니다.
* **고급 클라이언트 측 암호화 결합 시나리오:**
* **with DynamoDB 전역 테이블 (Global Tables):** `Amazon DynamoDB Encryption Client`를 사용해 특정 민감 필드를 클라이언트 측 암호화한 뒤 저장하면 타 리전으로 데이터가 자동 복제됩니다. 이때 다중 리전 키를 배치해 두면, 타 리전의 로컬 클라이언트들이 원본 리전으로 API를 쏘지 않고 **본인 리전의 복제 KMS 키로 로컬 API를 호출하여 즉시 복호화**하므로 속도가 매우 빠릅니다.
* **with Global Aurora:** `AWS Encryption SDK`를 사용해 DB의 특정 속성을 클라이언트 측 암호화하여 타 리전으로 넘깁니다. 데이터베이스 최고 관리자(DBA)라 할지라도 KMS 복호화 권한 키가 없다면 민감 필드 데이터를 절대 열어볼 수 없도록 철저히 격리합니다.



#### 📦 S3 Replication Encryption (S3 복제 시 암호화 고려사항)

* **기본 동작:** S3 교차 버킷 복제를 켜면 암호화되지 않은 객체와 `SSE-S3`로 암호화된 객체는 기본 복제됩니다.
* **⚠️ 복제 불가능:** 고객 제공 키를 사용하는 **`SSE-C` 암호화 객체는 절대로 복제되지 않습니다.**
* **SSE-KMS 복제 조건:** `SSE-KMS` 객체를 복제하려면 수동으로 옵션을 활성화하고 아래 3가지 요건을 맞춰야 합니다.
1. 대상(Target) 버킷에서 객체를 다시 암호화할 KMS 키를 명시적으로 지정해야 합니다.
2. 복제를 수행하는 IAM 역할(Role)에 원본 KMS 키에 대한 `kms:Decrypt` 권한과 대상 KMS 키에 대한 `kms:Encrypt` 권한을 모두 부여해야 합니다.
3. 이 과정에서 KMS 초당 호출 한도를 초과하는 **스로틀링(Throttling) 오류**가 발생할 수 있으므로, 대량 복제 시 서비스 할당량(Service Quotas) 상향 요청이 필요합니다.


* *참고:* S3 백엔드는 현재 다중 리전 키도 독립 키로 취급하므로, 다중 리전 키를 쓰더라도 객체는 복호화된 후 대상 리전 키로 재암호화 과정을 거칩니다.



#### 📀 AMI Sharing Process Encrypted via KMS (암호화된 AMI 공유 절차)

KMS 키로 암호화된 소스 계정의 AMI를 타 계정과 공유하여 EC2를 띄우기 위한 필수 4단계 프로세스입니다.

1. 소스 계정의 AMI 내부 스냅샷이 소스 계정 고객 관리형 KMS 키로 암호화되어 있는지 확인합니다.
2. AMI 이미지 속성을 수정하여 공유하고자 하는 **대상(Target) AWS 계정 ID를 시작 권한(Launch Permissions)에 명시적으로 추가**합니다.
3. 스냅샷 암호화에 사용된 소스 계정의 KMS 키 정책을 수정하여 대상 계정(또는 대상의 IAM 역할)이 키를 사용할 수 있도록 공유 허용 권한을 추가합니다.
4. 대상 계정의 IAM 정책에는 반드시 `kms:DescribeKey`, `kms:ReEncrypt*`, `kms:CreateGrant`, `kms:Decrypt` 권한이 탑재되어 있어야 합니다. (인스턴스를 시작할 때 대상 계정은 볼륨 재암호화를 위해 자신의 로컬 KMS 키를 새롭게 지정하여 완전히 소유권을 가져올 수 있습니다.)

### 🗃️ 3. 데이터 보안 창고: SSM Parameter Store vs AWS Secrets Manager

| 비교 항목 | 📝 SSM Parameter Store | 🔐 AWS Secrets Manager |
| --- | --- | --- |
| **핵심 목적** | 일반 구성 설정값 및 암호의 계층형 스토리지 | 대주기적 암호 강제 교체(Rotation) 중심의 보안 저장소 |
| **비용 구조** | **Standard 티어 기본 무료** (고급 티어 사용 시 비용 발생) | **암호(Secret)당 월 $0.40** + 10,000회 호출당 $0.05 |
| **최대 크기** | Standard: 4KB / Advanced: 8KB | 암호당 최대 10KB |
| **암호 교체 기능** | 자체 자동 교체 기능 없음 | **Lambda 연동을 통한 자동 암호 교체(Rotation) 내장** |
| **주요 통합 대상** | CloudFormation, 전사 아키텍처 파라미터 공유 | **Amazon RDS**(MySQL, PostgreSQL, Aurora) 자격 증명 연동 |

#### 📁 SSM Parameter Store 핵심 디테일

* **계층 구조 (Hierarchy):** `/my-department/my-app/db-password` 형태로 경로 구조화가 가능합니다. 이를 통해 IAM 정책에서 `/my-department/*` 경로 패턴 하나로 하위 모든 파라미터의 접근 권한을 한 번에 제어하고 관리 정책을 대폭 간소화할 수 있습니다.
* **시크릿 매니저 레퍼런스:** 파라미터 스토어 경로 구문에 `/aws/reference/secretsmanager/secret_ID` 형식을 사용하면, 파라미터 스토어를 거쳐 Secrets Manager에 저장된 실시간 암호값에 접근할 수 있습니다.
* **공용 퍼블릭 파라미터 사용:** AWS가 전역으로 발행하는 최신 공용 매개변수를 소스 코드나 스크립트에서 조회할 수 있습니다. (예: 특정 리전의 최신 리눅스 이미지 ID를 하드코딩하지 않고 `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2` 경로를 호출하여 동적 확보)
* **고급(Advanced) 파라미터 정책 기능:** 고급 티어를 쓰면 파라미터에 자체 정책을 심을 수 있습니다. 데이터의 유효 만료 기한을 강제하는 **TTL 정책**, 만료 시 알림을 보내는 **전환 정책**, 사용자별 세부 접근을 쪼개는 **다중 정책 할당** 기능이 제공됩니다.

#### 🌍 Secrets Manager Multi-Region Secrets

* **글로벌 복제:** 여러 AWS 리전으로 암호를 자동 복제하여 동기화된 읽기 전용 복제본(Read-only Replicas)을 백엔드에서 유지해 줍니다.
* **승격 기능:** 재해 복구(DR) 시 상황에 따라 읽기 전용 복제본 암호를 독자적인 독립 실행형 암호로 승격시킬 수 있습니다. 다중 리전 활성-활성 애플리케이션이나 글로벌 분산 DB 환경의 자격 증명 관리에 필수적입니다.

### 🌐 4. AWS Certificate Manager (ACM) 및 엔드포인트 아키텍처

웹사이트 전송 중 암호화(HTTPS)를 위한 TLS/SSL 인증서를 중앙에서 프로비저닝, 관리 및 배포하는 서비스입니다. 퍼블릭 인증서 발급은 **완전 무료**이며 만료 60일 전 자동 갱신(Auto-Renewal)을 지원합니다.

* **❌ 인프라 제한:** **EC2 인스턴스 내부에는 ACM 인증서를 직접 설치할 수 없습니다.** ACM은 프라이빗 키 추출이 불가능하므로 반드시 ELB(Load Balancer), CloudFront 배포, API Gateway 등 통합 연동 서비스를 전면에 내세워야 합니다.

#### 🔏 인증서 신청 및 검증 프로세스

1. 도메인 유형 정의: 완전한 도메인 이름(FQDN, `corp.example.com`) 또는 와일드카드 형태(`*.example.com`) 목록을 작성하여 신청합니다.
2. 유효성 검증(Validation) 방식 2가지 선택:
* **DNS 유효성 검증 (강력 권한 추천):** DNS 설정(예: Route 53)에 ACM이 제공하는 전용 CNAME 레코드를 삽입하여 소유권을 증명합니다. **인증서 자동 갱신이 완벽하게 무중단 자동 처리**되므로 운영상 선호됩니다.
* **이메일 유효성 검증:** WHOIS 데이터베이스에 등록된 도메인 관리자 연락처로 인증 메일을 발송하여 승인을 받는 수동 방식입니다.



#### 📥 외부 인증서 가져오기 (Importing Public Certificates)

* **제약 사항:** 타사 외부 기관에서 발급받은 기존 인증서를 ACM으로 업로드하여 쓸 수 있으나, 이 경우 **ACM에 의한 자동 갱신이 절대 불가능**합니다. 만료 전 사용자가 직접 새 인증서를 다시 인포트해야 합니다.
* **만료 방어 메커니즘:** ACM은 가져온 인증서의 만료 45일 전부터 매일 만료 경고 이벤트를 생성하여 `Amazon EventBridge`에 던져줍니다. 또한 `AWS Config`의 관리형 규칙인 `acm-certificate-expiration-check`를 결합하면 커스텀 지정한 임계 일수 전부터 만료 예정인 불량 인증서를 실시간 추적할 수 있습니다.

#### 🛡️ API Gateway Endpoint 유형별 ACM 결합 규칙

API Gateway와 ACM 인증서를 통합하여 사용자 지정 도메인 이름(Custom Domain Name)을 매핑할 때, 엔드포인트 유형별로 리전 제약 규칙이 엄격하게 작동합니다.

* **1. 엣지 최적화 엔드포인트 (Edge-Optimized - 기본값):** 글로벌 클라이언트의 요청을 최단거리 CloudFront 엣지 로케이션으로 라우팅하여 지연 시간을 줄입니다. 백엔드 API Gateway가 어느 리전에 있든 관계없이, **ACM 인증서는 반드시 버지니아 북부(`us-east-1`) 리전에 생성되어 있어야만** CloudFront 배포 레이어에 정상 바인딩됩니다.
* **2. 리전 엔드포인트 (Regional):** 클라이언트가 API Gateway와 물리적으로 동일한 리전에 있거나 자체 커스텀 CloudFront 아키텍처를 직접 제어하고 싶을 때 씁니다. 이 경우 **ACM 인증서는 반드시 API Gateway가 구동 중인 리전과 완전히 동일한 리전**에 생성되어야 합니다.
* *3. 프라이빗 엔드포인트 (Private):* 인터페이스 VPC 엔드포인트(ENI)를 통해 외부망이 아닌 오직 VPC 내부에서만 사설 통신하는 유형으로, 접근 통제를 위한 리소스 정책이 필수적으로 요구됩니다.

### 🧱 5. 레이어별 에지 인프라 방어 세트: WAF, Shield, Firewall Manager

#### 🛡️ AWS WAF (Web Application Firewall - L7 방어)

HTTP/HTTPS 프로토콜 패킷 내부를 깊숙이 들여다보고 악성 웹 요청을 차단합니다. `ALB`, `API Gateway`, `CloudFront`, `AppSync`, `Cognito 사용자 풀` 상단에 장착 가능합니다.

* **Web ACL(액세스 제어 목록) 구성 규칙:**
* **IP Set:** 최대 10,000개의 악성 블랙리스트 IP 주소를 단일 규칙으로 관리합니다. 초과 시 여러 규칙을 레이어로 묶어 사용합니다.
* **문자열/패턴 필터링:** HTTP 헤더, 본문(Body), URI 문자열의 패턴을 분석하여 웹 보안의 최대 적인 **SQL 인젝션(SQL Injection)** 및 **교차 사이트 스크립팅(XSS)** 취약점 공격을 정밀 탐지·차단합니다.
* **지역 일치 (Geo-match):** 특정 국가 코드를 기반으로 트래픽 접근을 원천 차단하거나 허용합니다.
* **속도 기반 규칙 (Rate-based rules):** 동일 IP에서 초당 유입되는 요청 횟수 제한 임계치를 설정하여 인프라 마비를 노리는 애플리케이션 레이어 HTTP Flood 계열의 DDoS 공격을 자동으로 탐지하고 차단합니다.
* *적용 범위:* 웹 ACL은 기본 리전별로 귀속되나, **CloudFront에 부착하는 Web ACL만은 전역(Global)으로 관리**됩니다. 규칙 그룹(Rule Group) 단위로 묶어 여러 Web ACL에 재사용할 수 있습니다.


* **⚠️ 고정 IP 인프라 우회 해결책 (NLB 미지원 이슈):** WAF는 4계층인 네트워크 로드 밸런서(NLB)에 직접 붙일 수 없습니다. 반드시 ALB를 거쳐야 하지만 ALB는 고정 IP를 주지 않습니다. 클라이언트에게 고정 IP 진입점을 반드시 제공하면서 L7 WAF 방화벽까지 가동하고 싶다면, **`AWS Global Accelerator`를 전면에 내세워 고정 IP 2개를 확보한 뒤 뒤쪽의 ALB 엔드포인트에 WAF를 바인딩**하는 아키텍처를 구성해야 합니다.

#### 🛡️ AWS Shield (DDoS 특화 방어)

* **Shield Standard:** 모든 AWS 고객에게 비용 청구 없이 **기본 활성화된 자동 방어 레이어**입니다. 가장 흔하게 발생하는 3계층/4계층 인프라 대상 공격인 `SYN Flood`, `UDP Reflection Attack` 등을 에지단에서 자동으로 걷어냅니다.
* **Shield Advanced:** 월 조직당 **$3,000**의 고정 비용이 청구되는 프리미엄 구독형 서비스입니다. `EC2`, `ELB`, `CloudFront`, `Global Accelerator`, `Route 53`을 고도로 정밀한 대규모 공격으로부터 전방위 보호합니다.
* *특전 1:* AWS DDoS 대응팀(DRT / SRT)이 인프라를 상시 모니터링하며 전용 기술 지원을 보장합니다.
* *특전 2:* DDoS 공격 방어 성공 과정에서 트래픽 및 인프라 사용량이 비정상 폭등하여 발생한 **요금 폭탄(사용량 급증 비용)을 면제/보상**해 줍니다.
* *특전 3:* **자동 L7 DDoS 완화 기능**이 내장되어 있어, 공격 감시 즉시 백엔드에서 정상 패턴을 분석해 **AWS WAF 차단 규칙을 스스로 자동 생성, 평가 및 즉각 배포**하여 애플리케이션 마비를 방어합니다.



#### 🏢 AWS Firewall Manager (전사 방화벽 가드레일)

* **역할:** AWS Organizations 조직 내에 존재하는 수많은 계정과 리전 전반의 방화벽 보안 정책을 중앙에서 전사 일괄 배포 및 강제 적용하는 상위 거버넌스 관리 도구입니다.
* **통제 가능한 보안 정책 기술 종류:** 1. 여러 계정의 `AWS WAF` 웹 ACL 규칙 일괄 제어 및 배포 자동화
2. 전사 자원에 대한 `AWS Shield Advanced` 구독 배포 및 활성화
3. `EC2`, `ALB`, VPC `ENI` 인스턴스들에 대한 공통 **보안 그룹(Security Group)** 규격 강제 배포
4. VPC 수준의 `AWS Network Firewall` 규칙 중앙 통제
5. `Amazon Route 53 Resolver DNS Firewall` 규칙 제어
* **운영 특징:** 정책은 리전 단위로 수립되며, 향후 개발자들에 의해 조직 내에 **새로운 AWS 계정이나 신규 리소스가 생성되는 순간 Firewall Manager가 감지하여 중앙 보안 방화벽 규칙을 즉시 자동 강제 바인딩**시킵니다. 일관된 전사 보안 규정 준수를 달성할 수 있습니다.

## 🏗️ 3단계: 디도스(DDoS) 완화 모범 사례 (DDoS Mitigation Best Practices)

인프라 아키텍처 설계 시 외부 공격 노출면을 최소화하고 트래픽 폭주를 완화하기 위한 전천후 인프라 구축 핵심 가이드라인입니다.

### 🏢 1. 공격 노출면 최소화 및 리소스 은닉 (Attack Surface Reduction)

* **백엔드 자원 은닉 (BP1, BP4, BP6):** 인터넷 대중에 우리 아키텍처의 핵심인 EC2 인스턴스의 실제 IP 주소나 Lambda 함수의 엔드포인트를 노출해서는 안 됩니다. 전면에 `CloudFront`, `API Gateway`, 혹은 `Elastic Load Balancing(ALB)`을 배치하여 실제 백엔드 인프라의 위치와 자원 구조를 완벽하게 난독화하고 숨깁니다.
* **API 엔드포인트 요새화 (BP4):** 외부 진입로가 되는 API Gateway 레이어에 `WAF`를 필수 결합하고, 초당 요청 한도를 통제하는 **버스트(Burst) 제한**, 악성 패킷을 걸러내는 **헤더 필터링**, 인가된 사용자만 통과시키는 **API 키 검증**을 활성화하여 에지 최적화 모드로 제어력을 강화합니다.
* **초소형 방화벽 필터링 (BP5):** 서브넷 인터페이스 경계선에서 네트워크 ACL(NACL)과 보안 그룹(Security Group)을 사용하여 철저히 허용된 신뢰 IP 대역의 트래픽만 들어오도록 4계층 이하의 진입 구멍을 최소화합니다. 외부 노출이 불가피한 탄력적 IP(EIP)는 `Shield Advanced` 보호 대상 자원으로 등록합니다.

### 🌊 2. 인프라 레이어 대규모 트래픽 방어 (Infrastructure Layer Defense)

* **에지 위치 완화 아키텍처 (BP1, BP3):** 트래픽이 우리 VPC 내부망으로 치고 들어오기 전, 전 세계 수백 개의 엣지 로케이션 인프라를 활용해 충격을 분산 흡수해야 합니다.
* **`CloudFront`:** 정적 콘텐츠를 전 세계 엣지에 캐싱하여 백엔드 오리진 서버의 컴퓨팅 부하를 100%에 가깝게 방어합니다. 일반적인 인프라 공격인 SYN Flood나 UDP 반사 공격은 에지단에 연결된 Shield 아키텍처가 EC2에 닿기도 전에 분산 소멸시킵니다.
* **`AWS Global Accelerator`:** CloudFront와 오리진의 프로토콜 사명이 맞지 않는 특수 환경에서 유용합니다. 전 세계 에지 진입점을 통해 전용 애니캐스트 고정 IP를 제공하며, 유입되는 트래픽은 글로벌 에지망에서 즉시 1차 정화된 후 AWS 전용 고속 백본 네트워크를 타고 안전하게 내부 ALB로 진입합니다.
* **`Amazon Route 53`:** 도메인 이름 확인 요청(DNS 쿼리)을 글로벌 엣지에 분산 배치하여, DNS 인프라 자체를 타겟으로 삼는 대규모 디도스 공격 공격 유형으로부터 서비스 가용성을 철저히 보호합니다.


* **고탄력 확장 레이어 결합 (BP6, BP7):** `Elastic Load Balancing(ALB)`은 트래픽 폭증 추이에 맞춰 백엔드에서 스스로 대역폭을 확장하며 요청을 하위 여러 가용 영역의 EC2 인스턴스들로 유연하게 분산시킵니다. 여기에 **Auto Scaling(오토 스케일링)** 그룹을 결합하여, 디도스나 급격한 대규모 플래시 크라우드 트래픽 유입 시 컴퓨팅 자원을 수평 확장하여 인프라 다운을 방어합니다.

### 💻 3. 애플리케이션 레이어 정밀 방어 (Application Layer Defense)

* **WAF 지능형 필터링 (BP1, BP2):** ALB 및 CloudFront 위에 부착된 `AWS WAF`는 들어오는 모든 HTTP 요청의 서명(Signature)을 실시간 분석합니다. 악성 공격자의 유입은 속도 기반 규칙(Rate-based)을 통해 임계치 초과 즉시 해당 IP를 자동 차단 장부에 올립니다. IP 평판 데이터베이스(IP Reputation) 규칙이나 익명 프록시/VPN IP 차단 관리형 규칙을 결합해 사전에 유해 트래픽을 필터링합니다. CloudFront의 지리적 제한(Geo-Restriction)을 켜서 특정 국가발 대규모 유해 트래픽 유입을 물리적으로 단절시킬 수 있습니다.
* **Shield Advanced 자동 연동:** Advanced 구독을 켜두면, 시스템이 자체적으로 7계층 이상 징후를 모니터링하다가 공격 감지 즉시 백엔드 단에서 **WAF 차단 규칙을 한치의 오차도 없이 자동으로 빌드, 검증 및 전 세계 엣지에 초고속 배포**하여 애플리케이션 가용성을 실시간 수호합니다.

---

### 🕵️ 6. 지능형 위협 탐지 및 자동 보안 평가

#### 🧠 Amazon GuardDuty (지능형 위협 탐지 탑)

AWS 계정 내부 및 인프라 전반의 위협을 실시간 감시하는 지능형 탐지 서비스입니다. 별도의 소프트웨어나 에이전트를 서버에 설치할 필요 없이, 콘솔에서 **클릭 단 한 번으로 활성화**되어 인프라 성능에 1%의 영향도 주지 않고 독립 구동됩니다.

* **핵심 탐지 기술:** 내부 머신러닝 알고리즘을 사용해 정상적인 인프라 활동의 기준선을 세운 뒤, 비정상적인 행위(이상 탐지)를 잡아내며 제3자 인텔리전스 위협 평판 데이터를 통합 적용합니다. 특히 암호화폐 채굴 공격(Crypto-mining)에 대한 특화 탐지 레이어를 제공합니다.
* **⚠️ 무조건 외워야 할 5대 핵심 입력 데이터 소스 (100% 출제):** GuardDuty는 다음 5가지 로그를 백엔드에서 실시간 분석합니다. (로그 파일들을 사용자가 수동으로 수집하거나 넘겨줄 필요 없이 GuardDuty가 직접 긁어갑니다.)
1. **CloudTrail 관리 이벤트 로그:** 비정상적인 API 호출 패턴, 무단 인프라 자원 배포, VPC 서브넷 변경 및 감사 트레일 삭제 시도 감지
2. **CloudTrail S3 데이터 이벤트 로그:** 버킷 내 민감 객체에 대한 무단 `GetObject`, `ListObjects`, 대량 `DeleteObject` 행위 추적
3. **VPC 플로우 로그 (VPC Flow Logs):** 인프라 내부와 외부 인터넷 간의 비정상적인 트래픽 흐름, 알려진 악성 C&C 서버 IP 주소와의 은밀한 통신 시도 식별
4. **DNS 로그 (DNS Logs):** 해킹되어 외부로 손상된 EC2 인스턴스가 내부 데이터를 인코딩하여 DNS 쿼리 요청 속에 숨겨 외부로 유출하려는 행위(DNS Tunneling) 감지
5. **Kubernetes 감사 로그 (EKS Audit Logs):** Amazon EKS 클러스터 내부의 수상한 API 활동 및 잠재적 컨테이너 클러스터 손상 징후 탐지


* **보안 조치 자동화 파이프라인:** 위협 이벤트를 발견하는 순간 GuardDuty는 상세 결과물(Finding)을 생성합니다. 이를 실시간 낚아채기 위해 `Amazon EventBridge` 규칙을 구성하고, 타겟으로 **AWS Lambda**를 지정해 악성 EC2의 보안그룹을 즉시 격리하거나, **Amazon SNS**를 연동하여 보안 담당자 장비로 경고 메일을 즉시 발송하는 자동 대응 체계를 구축할 수 있습니다.

#### 🔍 Amazon Inspector (자원 내부 취약점 자동 스캐너)

실행 중인 인프라 자원 내부에 잠재된 소프트웨어 취약점과 의도하지 않은 네트워크 노출 경로를 정기적·지속적으로 자동 스캔하는 보안 평가 도구입니다.

* **평가 대상 자원 딱 3가지 (★오답 유도 주의):** Amazon Inspector는 오직 **1) 실행 중인 EC2 인스턴스**, **2) Amazon ECR에 푸시된 컨테이너 이미지**, **3) 배포된 AWS Lambda 함수**에만 사용할 수 있습니다. S3 버킷이나 RDS 등의 스캔에는 사용할 수 없습니다.
* **세부 작동 방식 및 스캔 항목:**
* **EC2 인스턴스 스캔:** 인스턴스 내부에 탑재된 **AWS Systems Manager (SSM) 에이전트**의 통신 기능을 활용해 내부 운영체제(OS) 환경을 샅샅이 뒤집니다. 실행 중인 소프트웨어 애플리케이션 패키지들을 수집하여 글로벌 공통 취약점 데이터베이스(**CVE**)와 대조 분석합니다. 또한, 내부가 아닌 네트워크 외부 관점에서 보안 그룹과 라우팅 테이블을 분석하여 의도하지 않은 외부 인터넷으로의 네트워크 접근 가능성(보안 구멍)이 열려 있는지 평가합니다.
* **Amazon ECR 컨테이너 스캔:** 개발자가 ECR 리포지토리로 컨테이너 이미지를 푸시하는 순간, 이미지 내 레이어 패키지들의 보안 취약점을 자동으로 정밀 평가합니다.
* **AWS Lambda 스캔:** 새로운 Lambda 함수 코드가 배포되거나 업데이트될 때, 함수 내부 소스 코드 자체의 보안 결함 및 함께 패키징된 라이브러리 종속성(Dependencies) 내의 소프트웨어 취약성을 자동 분석합니다.


* **보안 허브 통합:** 스캔 평가 작업을 완료하면 발견된 모든 이슈와 취약점 항목들을 종합 위험 점수(Risk Score)로 수치화하여 우선순위를 정한 뒤, `AWS Security Hub`로 자동 보고합니다. 동시에 발견 사항을 `Amazon EventBridge`로 쏴주어 실시간 티켓팅 및 알림 시스템과 연동할 수 있습니다.

#### 👁️ AWS Macie (S3 개인정보 파수꾼)

S3 버킷 내에 저장된 수많은 대규모 데이터 자산들을 대상으로 작동하는 완전 관리형 데이터 보안 및 개인 정보 보호 서비스입니다.

* **동작 방식:** 내부적으로 강력한 **머신러닝 알고리즘과 정밀 패턴 일치(Pattern Matching) 기술**을 구동합니다.
* **핵심 타겟:** 버킷 내부의 파일들을 자동으로 스캔하여 이름, 주민등록번호, 신용카드 번호, 주소 등 외부로 절대 유출되어서는 안 되는 **개인 식별 정보 (PII, Personally Identifiable Information)** 및 민감한 금융 자산 정보를 정밀 식별해 냅니다. 위험 수준이 높은 민감 데이터 발견 시 즉시 보안 경고(Findings)를 생성하여 관리자에게 안내하고 데이터를 안전하게 격리 보호할 수 있는 기반을 제공합니다.
