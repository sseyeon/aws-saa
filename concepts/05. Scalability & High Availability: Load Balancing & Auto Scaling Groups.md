# ⚖️ AWS 고가용성, 로드 밸런싱 및 오토 스케일링 핵심 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **단일 가상 서버의 체급 증가(t2.micro ➡️ t2.large) + 하드웨어 한계 직면 + 분산 불가능한 DB 환경** $\rightarrow$ `Vertical Scalability (수직 확장성 / Scale Up)`
* **서버 개수 자체를 증감(Scale Out/In) + 탄력성(Elasticity)의 근간 + 현대 분산 애플리케이션 표준 아키텍처** $\rightarrow$ `Horizontal Scalability (수평 확장성)`
* **최소 2개 이상의 가용 영역(Multi-AZ) 분산 배치 + 데이터 센터 전체 재해 발생 시 서비스 무중단 생존 보장** $\rightarrow$ `High Availability (고가용성)`
* **Layer 7 HTTP/HTTPS 기반 전용 라우팅 + URL 경로(/users), 호스트명, 쿼리 스트링 조건별 타겟 그룹 매핑** $\rightarrow$ `ALB (Application Load Balancer)`
* **클라이언트의 원본 실제 IP 주소를 백엔드 EC2 인스턴스가 인지하도록 전송하는 HTTP 헤더명** $\rightarrow$ `X-Forwarded-For`
* **Layer 4 TCP/UDP 초고성능 트래픽 처리 + 가용 영역별 고정 IP/탄력적 IP 할당 필수 + 마이크로초 단위 극초저지연** $\rightarrow$ `NLB (Network Load Balancer)`
* **Layer 3 IP 패킷 수준 검사 + 3rd Party 가상 어플라이언스 플릿(방화벽, IDS/IPS) 배치 + 포트 6081 GENEVE 프로토콜** $\rightarrow$ `GWLB (Gateway Load Balancer)`
* **쿠키(Cookie)를 활용하여 동일한 클라이언트의 연속된 요청을 특정 백엔드 EC2 인스턴스로 강제 고정 바인딩** $\rightarrow$ `Sticky Sessions (고정 세션)`
* **하나의 로드 밸런서 리스너 인터페이스에 다수의 도메인별 SSL/TLS 인증서를 바인딩하여 호스트명 매칭 접속 지원** $\rightarrow$ `SNI (Server Name Indication)`
* **인스턴스 교체/비정상 판정 시, 현재 처리 중인 활성 요청(In-flight Requests)이 안전하게 완료될 때까지 신규 트래픽을 차단하고 대기** $\rightarrow$ `Deregistration Delay / Connection Draining (등록 취소 지연)`
* **비정상(Unhealthy) 자원 자동 교체 + 최소/최대 인스턴스 보유 대수 보장 + CloudWatch 경보 기반 동적 인프라 스케일링** $\rightarrow$ `ASG (Auto Scaling Group)`
* **평균 CPU 사용률을 항상 40% 선으로 유지하기 위해 인프라가 알아서 계산하여 유연하게 가동 대수를 맞추는 정책** $\rightarrow$ `Target Tracking Scaling (대상 추적 조정)`
* **시작 템플릿(Launch Template) 업데이트 후, 지정한 최소 정상 백분율을 유지하면서 순차적으로 구형 인스턴스를 신형으로 전면 교체** $\rightarrow$ `Instance Refresh (인스턴스 새로 고침)`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### ⏫ 1. 확장성(Scalability) 및 고가용성(High Availability)

* **수직 확장 (Vertical Scaling):** 단일 머신의 사양(vCPU, RAM) 자체를 업그레이드하는 행위입니다. 분산 아키텍처 적용이 까다로운 전통적인 RDBMS(RDS)나 인메모리 캐시(ElastiCache) 시스템의 초기 용량 산정에 주로 쓰이나, 물리적인 하드웨어 한계(Hardware Limit)가 명확히 존재합니다.
* **수평 확장 (Horizontal Scaling = Elasticity):** 동일한 스펙의 인스턴스 대수 자체를 동적으로 늘리거나 줄이는 방식입니다. 클라우드의 핵심 가치인 **탄력성**을 대변하며, 현대적인 무상태(Stateless) 웹 서버 및 마이크로 서비스 분산 구조의 필수 조건입니다.
* **고가용성 (High Availability):** 시스템을 최소 2개 이상의 독립된 가용 영역(Multi-AZ)에 상호 이중화 분산 배치하여, 특정 데이터 센터 가동 중단(물리 재해) 사태가 터지더라도 데이터 유실 및 서비스 가동 중단 없이 트래픽을 연속적으로 생존 처리하는 능력을 뜻합니다.

---

### 🔀 2. Elastic Load Balancing (ELB) 완전 분석

ELB는 단일 진입점(제공되는 고정 DNS 주소)을 외부 공용 공간에 노출하고, 다운스트림 대상 그룹(Target Groups)으로 트래픽을 균등하게 분산 전달하는 완전 관리형 부하 분산 서비스입니다.

#### 🏥 헬스 체크 (Health Checks) 메커니즘

로드 밸런서는 지정된 프로토콜, 포트, 특정 URL 라우팅 경로(예: `HTTP / 80 / /health`)를 통해 다운스트림 자원들에 주기적으로 핑(Ping)을 날립니다. 이때 백엔드 애플리케이션이 표준 정상 응답 코드인 '200 OK'를 반환하지 못하면 해당 인스턴스를 비정상(`Unhealthy`) 자원으로 분류하고 트래픽 전송 대상에서 즉시 제외시킵니다.

#### 🌐 로드 밸런서 3대 유형 비교 마스터

##### ① Application Load Balancer (ALB - Layer 7 레이어)

* **주요 특징:** 오직 **HTTP/HTTPS 및 WebSocket, HTTP/2 트래픽** 처리에 최적화된 고급 지능형 로드 밸런서입니다. 컨테이너 기반 마이크로 서비스(Docker, Amazon ECS) 환경과 찰떡궁합을 자랑합니다.
* **지능형 라우팅 규칙:**
* **URL 경로 기반:** `example.com/users`와 `example.com/posts`를 서로 다른 타겟 그룹으로 라우팅.
* **호스트 이름 기반:** `one.example.com`과 `other.example.com`을 분리 분배.
* **기타 파라미터:** HTTP 헤더 정보, 쿠키, 쿼리 스트링(Query String) 값을 분석하여 정밀 라우팅.


* **네트워크 투명성:** ALB 뒤에 숨은 백엔드 EC2는 ALB의 프라이빗 IP가 트래픽을 전달하므로 클라이언트의 실제 IP를 다이렉트로 보지 못합니다. 원본 클라이언트의 IP 주소를 읽어오려면 반드시 **`X-Forwarded-For`** 헤더를 추출해야 합니다. (포트는 `X-Forwarded-Port`, 프로토콜은 `X-Forwarded-Proto` 참조)

##### ② Network Load Balancer (NLB - Layer 4 레이어)

* **주요 특징:** **TCP, UDP, TLS** 네트워크 전송 계층 트래픽을 초당 수백만 건 이상 다이렉트로 초고속 처리하는 고성능 로드 밸런서입니다. 약 100ms 수준의 극초저지연(ALB는 약 400ms)을 달성합니다.
* **★시험 단골 키워드 (고정 IP):** ALB와 달리, NLB는 **가용 영역(AZ)당 단 하나의 고정 퍼블릭 IP**를 가집니다. 사용자가 원할 경우 **탄력적 IP(EIP)를 가용 영역별로 직접 바인딩**할 수 있으므로, 방화벽 단에서 특정 퍼블릭 IP만 진입을 허용해야 하는 **화이트리스트(Whitelisting) 연동 시나리오의 무조건적인 정답**입니다.

##### ③ Gateway Load Balancer (GWLB - Layer 3 레이어)

* **주요 특징:** 가상 방화벽, 침입 탐지 및 방지 시스템(IDS/IPS), 딥 패킷 검사(DPI) 어플라이언스 등 **3rd Party 네트워크 가상 보안 장비 플릿을 효율적으로 스케일링하고 중앙 관리**하기 위한 특수 로드 밸런서입니다.
* **메커니즘:** IP 패킷 전체 레이어인 Layer 3에서 동작하며, 전체 네트워크 진입/출입 경로를 투명하게 장악하는 게이트웨이 역할을 수행합니다. 내부적으로 **포트 6081번의 GENEVE 프로토콜**을 사용하여 트래픽을 보안 어플라이언스로 캡슐화해 실어 나릅니다.

### 🔗 3. 로드 밸런서 고급 트래픽 통제 엔지니어링

#### 🔗 Sticky Sessions (고정 세션 / Session Affinity)

* **정의:** 클라이언트가 던지는 연속적인 비즈니스 요청이 세션 만료 전까지 **항상 백엔드의 동일한 특정 EC2 인스턴스로만 전달되도록 고정**하는 기술입니다. (ALB 및 구형 CLB 지원)
* **메커니즘:** 클라이언트와 로드 밸런서 간에 '만료 기간(Duration)'이 정의된 전용 쿠키(Cookie)를 교환하여 구현합니다.
* **⚠️ 아키텍처 부작용:** 트래픽이 특정 인스턴스로 쏠리는 현상이 발생하여 백엔드 자원 간 부하 불균형(Imbalance)이 유발될 수 있습니다.
* **쿠키 분류 유형:**
* **애플리케이션 기반 커스텀 쿠키:** 백엔드 타겟이 직접 생성한 커스텀 속성을 따릅니다. 단, `AWSALB`, `AWSALBAPP`, `AWSALBTG` 같은 예약어는 이름으로 쓸 수 없습니다.
* **애플리케이션 쿠키:** 로드 밸런서가 자체 발행하며 쿠키명은 `AWSALBAPP` 고정입니다.
* **기간 기반 쿠키 (Duration-based):** 로드 밸런서가 전적으로 관리하며 ALB인 경우 `AWSALB`, CLB인 경우 `AWSELB` 라는 이름으로 발행됩니다.



#### 🚰 교차 영역 로드 밸런싱 (Cross-Zone Load Balancing)

* **정의:** 로드 밸런서의 가용 영역별 노드가 자신과 다른 AZ에 속해 있는 백엔드 인스턴스들에게도 트래픽을 균등하게 찢어서 넘겨주는 기능입니다.

[Image diagram showing Cross-Zone Load Balancing on AWS distributing traffic evenly to all instances across multiple Availability Zones]

* **형태별 동작 비교:**
* **ALB:** **기본값으로 무조건 활성화**되어 있으며, AZ 간 데이터 전송에 따른 추가 네트워크 비용 비용 청구가 전혀 없습니다. (대상 그룹 단위 비활성화 가능)
* **NLB / GWLB:** **기본값으로 비활성화** 상태입니다. 만약 필요에 의해 명시적으로 활성화할 경우, **가용 영역(AZ) 경계를 넘어가는 데이터 전송량에 대해 추가 네트워크 요금이 청구**되므로 주의해야 합니다.



#### 📜 SSL/TLS 암호화 및 SNI (Server Name Indication)

* **SSL/TLS 종료 (SSL Termination):** 클라이언트가 로드 밸런서까지는 안전하게 HTTPS 암호화 통신으로 들어오고, 로드 밸런서가 백엔드 EC2 인스턴스로 트래픽을 넘길 때는 복호화하여 일반 HTTP로 전달함으로써 백엔드 서버의 복호화 연산 부담을 지워주는 구조입니다.
* **ACM (AWS Certificate Manager):** AWS에서 X.509 실명 인증서를 발급 및 만료 자동 갱신 관리해 주는 서비스로, 로드 밸런서의 HTTPS 리스너에 다이렉트로 바인딩됩니다.
* **★SNI (Server Name Indication) 핵심 메커니즘:**
* 과거에는 하나의 로드 밸런서 인터페이스가 단 하나의 SSL 인증서만 품을 수 있어서 도메인별로 로드 밸런서를 각각 사야 했습니다.
* SNI 기술을 쓰면 클라이언트가 초기 SSL 핸드셰이크 단계에서 "내가 접속하려는 도메인은 `app.example.com`이야"라고 **호스트 명 명시적 선언**을 보냅니다. 이를 통해 **단 하나의 로드 밸런서 가상 리스너 포트가 다수의 스마트 SSL 인증서 풀을 품고 있다가 도메인에 딱 맞는 인증서를 역동적으로 찾아서 반환**해 줄 수 있습니다.
* **⚠️ 제약 조건:** 최신 세대인 **ALB, NLB 및 CloudFront에서만 정상 지원**되며, 구형 CLB(Classic) 아키텍처에서는 SNI가 절대 작동하지 않습니다.



#### 🚰 등록 취소 지연 (Deregistration Delay / Connection Draining)

* **정의:** 배포나 오토스케일링 과정에서 특정 EC2 인스턴스가 타겟 그룹에서 빠지거나(`Deregistering`), 헬스 체크 결과 비정상(`Unhealthy`) 상태로 전락했을 때, 칼같이 연결을 끊지 않고 **일정 시간(1~3600초, 기본값 300초) 유예 기간을 주는 기능**입니다.
* **목적:** 이미 해당 서버에서 세션을 맺고 열심히 트랜잭션을 수행 중이던 **기존 요청(In-flight Requests)들이 정상적으로 응답을 완료할 때까지 안전하게 대기**시켜 주며, 해당 기간 동안 신규 유입 트래픽은 완전히 차단합니다. 0초로 세팅하면 이 안전 장치가 완전히 꺼집니다.

### 📈 4. Auto Scaling Group (ASG)

ASG는 사전에 정의한 인프라 규칙과 모니터링 지표에 기반하여 EC2 인스턴스 자원을 자동으로 스케일 아웃/인 처리하고, 정해진 최소(`Min`), 최대(`Max`), 최초 희망 용량(`Desired`) 범위 내에서 정상 서버 대수를 자가 치유(Health Check 대체) 유지하는 완전 무료 관리형 서비스입니다.

#### ✅ ASG 작동의 뼈대: 시작 템플릿 (Launch Template)

ASG가 새 인스턴스를 무인으로 찍어낼 때 참조하는 청사진 명세서입니다. **AMI ID, 인스턴스 타입 스펙, EC2 User Data 스크립트, EBS 스토리지 구조, 매핑할 보안 그룹(Security Groups), SSH 키 페어, EC2용 IAM 역할(Role)** 정보를 완벽히 내포하고 있습니다.

#### 📑 4대 크기 조정 정책 (Scaling Policies)

1. **Target Tracking Scaling (대상 추적 조정):** 가장 직관적이고 추천되는 모드입니다. "전체 ASG의 평균 CPU 사용률을 상시 40%로 고정 유지하라"고 선언하면, AWS가 알아서 계산하여 서버를 늘리거나 줄입니다.
2. **Simple / Step Scaling (단계별 조정):** CloudWatch 알람 조건과 정밀 매핑합니다. (예: CPU 사용률 70% 초과 알람 트리거 시 인스턴스 2대 추가 배치, 30% 미만 추락 알람 시 1대 삭제)
3. **Scheduled Actions (예약된 조정):** 비즈니스 트래픽 사용 패턴을 미리 완벽히 알고 있을 때 타임 스케줄링을 거는 방식입니다. (예: 매주 금요일 오후 5시 퇴근 시간 전 상시 최소 용량을 10대로 강제 확장)
4. **Predictive Scaling (예측 조정):** AWS의 머신러닝 알고리즘이 과거의 오랜 트래픽 패턴 데이터 추이를 스스로 분석하여, 다가올 피크 타임 주기 직전에 인스턴스 용량을 미리 선제적으로 확장 시켜 놓는 최첨달 스마트 정책입니다.

#### 📉 스케일링 쿨다운 (Scaling Cooldowns)

* **메커니즘:** 스케일 아웃이나 스케일 인 활동이 한 번 탕! 발생하여 인프라에 변화가 생기면, 그 직후 **지정한 일정 쿨다운 기간(기본값 300초) 동안 ASG는 추가적인 스케일링 액션을 일체 동결 차단**합니다.
* **이유:** 새로 켜진 인스턴스가 OS 부팅을 완료하고 애플리케이션 캐시를 올려 트래픽을 정상 분담할 때까지 시간이 걸리므로, CloudWatch 전체 평균 지표가 안정화될 때까지 기다려 **인프라가 과도하게 무한정 증설되는 메트릭 교란 현상을 방지**하기 위함입니다. (골든 AMI를 구축해 쓰면 부팅 시간을 단축시켜 쿨다운 효율을 높일 수 있습니다.)

#### 🆕 Instance Refresh (인스턴스 새로 고침)

* **목적:** 개발자가 소스 코드를 새로 반영하여 시작 템플릿(Launch Template) 버전을 신형 AMI로 업데이트했을 때, 기존에 구동 중이던 구버전 EC2 인스턴스 플릿 전체를 무중단으로 안전하게 신버전으로 물갈이 교체하는 ASG 전용 고유 빌트인 기능입니다.
* **통제 변수:** **최소 정상 백분율(Minimum Healthy Percentage)** 조건을 설정하여, 교체 작업 중에도 전체 인프라 용량의 최소 70~80%는 상시 정상 가동 상태(Healthy)를 강제 유지하도록 롤링 배포 흐름을 제어할 수 있으며, 서버가 트래픽을 받을 준비가 될 때까지의 **인스턴스 워밍업 시간**을 정밀하게 지정할 수 있습니다.
