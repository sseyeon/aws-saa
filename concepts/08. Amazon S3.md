# 🪣 Amazon S3 객체 스토리지 및 수명 주기 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **전 계정/리전 유일무이한 전역 네임스페이스 + 11 9s($99.999999999\%$) 내구성 보장** $\rightarrow$ `Amazon S3`
* **S3에는 실제 폴더 구조가 없으며, 디렉토리처럼 보이는 착시를 유도하는 슬래시(`/`) 포함 전체 파일 경로** $\rightarrow$ `객체 키 (Object Key)`
* **단일 파일 용량 100MB 초과 또는 최대 5TB 업로드 시 네트워크 최적화 및 전송 실패 리스크를 방지하는 분할 업로드 기술** $\rightarrow$ `Multipart Upload (멀티파트 업로드)`
* **S3 콘솔/API 단에서 교차 계정(Cross Account) 및 대규모 퍼블릭 읽기 권한을 JSON 포맷으로 일괄 제어하는 리소스 정책** $\rightarrow$ `S3 Bucket Policy (S3 버킷 정책)`
* **버킷 정책으로 실수로 Public 권한을 열어두더라도 계정/버킷 레벨에서 데이터 유출을 원천 전면 차단하는 마스터 보안 스위치** $\rightarrow$ `Block Public Access (퍼블릭 액세스 차단)`
* **동일 키(Key) 덮어쓰기 시 파일 유실 방지 + 삭제 시 실제 데이터 삭제 대신 임시 마커만 꼽아 복원을 지원하는 추적 기능** $\rightarrow$ `S3 Versioning (버전 관리)`
* **버전 관리 활성화 필수 + 서로 다른 리전(CRR) 또는 동일 리전(SRR) 간 비동기식 실시간 백라운드 객체 복사 기술** $\rightarrow$ `S3 Replication (S3 복제)`
* **기존에 이미 적재되어 있던 레거시 객체들이나 과거 복제 실패 건들을 일괄 타겟 버킷으로 강제 밀어 넣는 대규모 배치 제어** $\rightarrow$ `S3 Batch Replication`
* **자주 쓰지는 않지만 재해 복구(DR)용으로 빠른 밀리초(ms) 단위 즉시 반환이 요구되며, 검색 시 바이트당 비용이 추가되는 클래스** $\rightarrow$ `S3 Standard-IA (Infrequent Access)`
* **검색 옵션 3종(Expedited 1~5분, Standard 3~5시간, Bulk 5~12시간) 제공 및 최소 90일 보존 조건의 아카이빙 클래스** $\rightarrow$ `S3 Glacier Flexible Retrieval`
* **데이터 액세스 패턴을 전혀 예측할 수 없는 가변 워크로드 환경 + 알아서 5개 계층을 동적으로 오르내리며 비용을 자동 절감하는 클래스** $\rightarrow$ `S3 Intelligent-Tiering`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🪣 1. S3 버킷(Buckets) 및 객체(Objects) 명세 규칙

Amazon S3는 디렉토리 개념이 없는 완전한 **Key-Value 기반 플랫(Flat) 객체 스토리지**입니다.

* **글로벌 버킷 네이밍 스펙:** 전 세계 모든 AWS 고객을 통틀어 단 하나만 존재해야 하는 전역 고유명(Globally Unique)을 가집니다. 단, 실제 인프라는 생성 시 지정한 **특정 리전(Region) 내부에 바인딩**되어 구동됩니다.
* **명명 규칙 가이드:** 소문자, 숫자, 하이픈(`-`)만 허용 (대문자 ❌, 밑줄 `_` ❌, IP 주소 형식 ❌, 길이는 3~63자 제한, `xn--` 접두사 및 `s3alias` 접미사 절대 불가, 소문자나 숫자로 시작 필수)


* **폴더 구조의 실체:** S3 콘솔에서 보이는 폴더 구조는 가짜입니다. 실제로는 `s3://버킷명/접두사(Prefix)/파일명` 전체 문자열 자체가 하나의 고유한 **키(Key)** 이름일 뿐이며, 슬래시(`/`)를 포함한 아주 긴 이름의 키만 존재합니다.
* **용량 제한 및 전송 최적화:** 단일 객체의 최대 저장 용량은 **5TB**입니다. 매우 큰 파일을 업로드하는 경우 네트워크 최적화 및 복력 향상을 위해 파일을 쪼개서 병렬 전송하는 **멀티파트 업로드(Multipart Upload)** 아키텍처가 강제됩니다.
* **메타데이터 & 태그:** * **메타데이터:** 시스템 또는 사용자 정의 텍스트 키/값 쌍의 목록입니다.
* **태그:** 최대 10개까지 유니코드 키/값 쌍 지정이 가능하며 보안 정책이나 수명 주기 제어에 유용하게 쓰입니다.

### 🔒 2. S3 4중 보안 아키텍처 및 정적 웹 호스팅

S3 보안은 유저 기반(IAM) 정책과 리소스 기반(버킷 정책, ACL) 권한의 합집합으로 최종 통제됩니다.

#### ⚔️ 권한 평가 알고리즘 (IAM vs Bucket Policy)

* **평가 방식:** 유저의 IAM Policy 혹은 대상 S3의 Bucket Policy 중 최소 어느 한 곳에서 명시적 허용(ALLOW)이 떨어져야 진입할 수 있습니다. 단, 인프라 내 어느 곳이라도 단 하나의 **명시적 거부(Explicit DENY)** 조항이 적발되면 타 정책의 ALLOW 여부와 상관없이 무조건 차단됩니다.
* **S3 버킷 정책 (Bucket Policies):** JSON 기반 정책 구조로 `Resources`, `Effect`, `Actions`, `Principal` 정보를 포함합니다. 버킷에 대한 공개 액세스 부여, 업로드 시 암호화 강제화, 다른 계정에 액세스 권한 부여(Cross Account)를 지원합니다.
* **ACL (Access Control List):** 객체 및 버킷 수준에서 더 세분화되거나 덜 일반적인 권한 설정을 제공하며 필요시 비활성화할 수 있습니다.

#### ⛔ Block Public Access (퍼블릭 액세스 차단 마스터 스위치)

기업의 데이터 유출 사고를 막기 위한 전사적 안전장치입니다. 계정 또는 버킷 레벨에서 설정 가능하며, 개발자가 실수로 버킷 정책을 오픈하여 퍼블릭 공개로 만들더라도 **이 스위치가 활성화되어 있으면 모든 퍼블릭 접근권이 강제로 무력화 거부**됩니다.

#### 🌐 정적 웹사이트 호스팅 (Static Website Hosting)

S3는 인프라 서버 없이 정적 웹 사이트를 초고속 호스팅할 수 있습니다. 활성화 시 리전 규격에 따라 `http://버킷명.s3-website-리전명.amazonaws.com` 또는 `http://버킷명.s3-website.리전명.amazonaws.com` 형태의 엔드포인트 URL이 발급됩니다. 만약 접속 시 **403 Forbidden 오류**가 떨어진다면, 버킷 정책이 공개 읽기(Public Read)를 허용하는지 가장 먼저 점검해야 합니다.

### 📑 3. 버전 관리(Versioning) 및 복제(Replication) 메커니즘

#### 📑 버전 관리의 동작 특성

* 버킷 단위로 활성화되며, 동일한 파일명(Key)으로 계속 덮어쓰더라도 과거 히스토리가 고유 버전 ID를 달고 누적 저장됩니다. 의도하지 않은 삭제와 덮어쓰기로부터 데이터를 보호하고 쉽게 롤백할 수 있습니다.
* **삭제 마커 (Delete Marker):** 버전 관리 중인 객체를 단순 삭제하면 파일이 증발하지 않고 최상단에 **삭제 마커**가 추가되어 숨겨집니다. 이 마커만 제거하면 과거 데이터를 100% 무중단 복원할 수 있습니다.
* **주의 사항:** 버전 관리가 활성화되기 전에 업로드된 파일은 `null` 버전을 가지게 됩니다. 또한 버전 관리를 일시 중단(Suspend)하더라도 이미 생성된 과거 버전 파일들은 삭제되지 않고 그대로 유지됩니다.

#### 🗂️ 실시간 데이터 복제 프로세스 (CRR / SRR)

* **필수 전제 조건:** 데이터를 쏘는 소스(Source) 버킷과 데이터를 받는 타겟(Target) 버킷 **모두 반드시 버전 관리(Versioning)가 활성화**되어 있어야 합니다. 복제는 백그라운드에서 비동기(Asynchronous)로 처리되며, S3 서비스가 타겟 버킷에 파일을 쓸 수 있도록 읽기/쓰기 권한이 포함된 적절한 IAM Role 권한 바인딩이 동반되어야 합니다. 버킷은 서로 다른 AWS 계정 소유여도 상관없습니다.
* **리전 경계별 분류:** * **CRR (Cross-Region):** 법무 규정 준수, 내부 체제 관리, 다른 리전 사용자의 접근 지연 시간 축소, 계정 간 복제 목적.
* **SRR (Same-Region):** 시스템 로그 통합, 운영 환경과 개발 환경 간의 실시간 복제 목적.


* **⚠️ 시험 단골 예외 조항 (Replication Notes):**
1. 복제 기능을 켠 **그 시점 이후로 새롭게 생성되는 신규 객체만 자동 복제**됩니다. 기존 파일이나 복제에 실패한 건들을 복사하려면 **S3 배치 복제(Batch Replication)** 기능을 수동으로 구동해 주어야 합니다.
2. **삭제 작업 (Delete Operations):** 소스 버킷의 삭제 마커를 대상 버킷으로 복제하는 것은 옵션 설정으로 제어할 수 있습니다. 단, **특정 버전 ID를 명시하여 수행한 영구 삭제 행위는 타겟 버킷으로 복제 전파되지 않습니다.**
3. **연쇄 복제 불가 (No Chaining):** 버킷 1 ➡️ 버킷 2 복제 규칙이 있고, 버킷 2 ➡️ 버킷 3 복제 규칙이 있을 때, 최초 버킷 1에 들어온 객체는 버킷 2까지만 복사되며 버킷 3까지 연쇄 복제되지 않습니다.

### 🗄️ 4. S3 스토리지 클래스 (Storage Classes) 매트릭스

S3는 모든 스토리지 클래스가 99.999999999%(11 9s)의 동일한 데이터 내구성(Durability)을 자랑합니다. (가용 영역 전체가 파괴되더라도 데이터 유실 위험 제로 수준). 차이점은 오직 인프라 가동성인 가용성(Availability)과 **데이터 검색 속도, 그리고 비용 구조**입니다. 수동 변경 외에도 S3 수명 주기(Lifecycle) 구성을 통해 자동 전환이 가능합니다.

#### 🗄️ S3 Intelligent-Tiering 내부 자동 계층화 상세 구조

* **Frequent Access tier (기본값):** 데이터 업로드 시 기본 상주 계층.
* **Infrequent Access tier (자동):** 연속 **30일** 동안 액세스가 없는 객체 이동.
* **Archive Instant Access tier (자동):** 연속 **90일** 동안 액세스가 없는 객체 이동.
* **Archive Access tier (선택 사항):** **90일에서 700일+** 사이로 커스텀 설정하여 이동 가능.
* **Deep Archive Access tier (선택 사항):** **180일에서 700일+** 사이로 커스텀 설정하여 이동 가능.
