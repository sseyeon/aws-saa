# 🌐 AWS VPC 핵심 네트워킹 및 하이브리드 보안 마스터 치트키

## 🎯 1단계: 한 눈에 보는 VPC/네트워크 핵심 키워드 매칭

시험 문제에 아래 핵심 단어가 나오면 고민하지 말고 바로 해당 서비스를 고르세요!

* **사설 서브넷의 인스턴스가 아웃바운드 인터넷 통신 + AWS 관리형(자동 확장) + IPv4 전용** $\rightarrow$ `NAT Gateway`
* **사설 서브넷의 인스턴스가 아웃바운드 인터넷 통신 + 인스턴스 유형별 대역폭 종속 + 소스/목적지 확인 해제 필수** $\rightarrow$ `NAT Instance`
* **IPv6 전용 + 사설 인스턴스의 아웃바운드 허용 + 외부 인바운드 진입 원천 차단** $\rightarrow$ `Egress-only Internet Gateway`
* **서브넷 경계 방화벽 + 번호순(우선순위) 평가 + 무작위 휘발성 포트(Ephemeral Ports) 고려 + Stateless** $\rightarrow$ `NACL (Network ACL)`
* **인스턴스 레벨 방화벽 + 인바운드 허용 시 아웃바운드 자동 개방 + Stateful** $\rightarrow$ `Security Group (보안 그룹)`
* **두 VPC의 프라이빗 내부망 연결 + CIDR 중복 불가 + 비전이적(Non-transitive) 연결** $\rightarrow$ `VPC Peering`
* **인터넷 안 거치고 S3 / DynamoDB 사설 연결 + 라우팅 테이블 대상 등록 + 완전 무료** $\rightarrow$ `Gateway Endpoint`
* **인터넷 안 거치고 S3(온프레미스 연동용) / 기타 대부분의 AWS 서비스 사설 연결 + ENI 기반 + 시간/GB당 과금** $\rightarrow$ `Interface Endpoint (PrivateLink)`
* **네트워크 연결성 테스트 및 도달 가능 여부 디버깅 + 실제 패킷 전송 없이 구성 분석** $\rightarrow$ `Reachability Analyzer`
* **ENI 패킷 캡처 및 복사 + 타겟으로 ENI/NLB 지정 + 침입 탐지 및 위협 모니터링 연동** $\rightarrow$ `Traffic Mirroring`
* **수천 개 VPC + 온프레미스(VPN/DX) 중앙 집중 허업앤스포크 모델 + 전이적 연결 허용 + IP 멀티캐스트 지원** $\rightarrow$ `Transit Gateway`
* **VPC 전체 경계 통제 + L3~L7 양방향 패킷 정밀 검사 + 상태 기반 도메인 화이트리스트 규칙(Suricata 기반)** $\rightarrow$ `AWS Network Firewall`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🔢 1. CIDR (Classless Inter-Domain Routing) & IP 주소 체계

* **구성 요소:** Base IP(범위의 시작 주소)와 Subnet Mask(고정 비트 수, 기호 `/`)로 표현됩니다.
* **마스크 범위별 IP 개수 규칙:**
* `/32`: 단 1개의 고유 IP 주소 (`WW.XX.YY.ZZ/32`)
* `/24`: 마지막 옥텟 변경 가능 $\rightarrow$ $2^{32-24} = 256$개 IP 주소
* `/16`: 마지막 두 개 옥텟 변경 가능 $\rightarrow$ $2^{32-16} = 65,536$개 IP 주소
* `/8`: 마지막 세 개 옥텟 변경 가능 $\rightarrow$ 대규모 네트워크
* `/0`: 전 세계 모든 IP 주소 공간 (`0.0.0.0/0`)


* **사설 IP (Private IP) RFC 1918 규격:** 사설망 내부(LAN/VPC)에서만 통신 가능하며 인터넷 라우팅이 불가능한 주소대역입니다.
* `10.0.0.0 – 10.255.255.255 (10.0.0.0/8)`: 대규모 엔터프라이즈 환경
* `172.16.0.0 – 172.31.255.255 (172.16.0.0/12)`: **AWS 기본(Default) VPC 주소 범위**
* `192.168.0.0 – 192.168.255.255 (192.168.0.0/16)`: 일반 홈 네트워크 환경
* *참고:* 사설 IP 주소대역을 제외한 나머지 모든 주소 공간은 공용 IP(Public IP)로 분류됩니다.

### 🧱 2. AWS VPC (Virtual Private Cloud) & Subnet 핵심 기반

#### 🌐 VPC 인프라 기본 한도 및 규칙

* 단일 AWS 리전에 생성 가능하며, 리전당 기본 최대 **5개**의 VPC 생성이 가능합니다(소프트 리밋).
* 하나의 VPC당 최대 **5개**의 CIDR 블록을 연동할 수 있습니다.
* **VPC 허용 크기 범위:** 최소 `/28` (16개 IP)부터 최대 `/16` (65,536개 IP)까지 지원합니다. 사설 리소스이므로 위 RFC 1918 범위 내에서만 정의해야 하며, **타 네트워크나 온프레미스 기업망과 CIDR 주소가 절대 겹치지 않도록 설계해야 합니다.**
* **기본(Default) VPC:** 모든 AWS 계정 생성 시 리전별 자동 제공됩니다. 서브넷 미지정 상태로 EC2를 띄우면 여기에 생성됩니다. 모든 서브넷이 인터넷 게이트웨이(IGW)와 라우팅되어 있어 내부에 생성된 EC2는 자동으로 퍼블릭 IPv4 및 퍼블릭/프라이빗 DNS 이름을 부여받고 즉시 인터넷 통신이 가능합니다.

#### ⚠️ AWS 서브넷별 예약 IP 5가지 (★시험 단골 출제)

AWS는 모든 서브넷의 **처음 4개 주소와 마지막 1개 주소, 총 5개의 IP를 고정 예약**하므로 사용자가 EC2 인스턴스 등에 절대 할당할 수 없습니다.

> **예시: 서브넷 CIDR 블록이 `10.0.0.0/24`인 경우 예약 주소**
> 1. `10.0.0.0`: 네트워크 자체의 기본 주소 (Network Address)
> 2. `10.0.0.1`: VPC 내부 라우터 통신용 예약 주소
> 3. `10.0.0.2`: AWS가 제공하는 자체 기본 Route 53 DNS 매핑용 예약 주소
> 4. `10.0.0.3`: 향후 기능 확장을 위해 AWS 내부적으로 선점한 예약 주소
> 5. `10.0.0.255`: 네트워크 브로드캐스트(Broadcast) 주소. 단, **AWS VPC는 브로드캐스트 전송 메커니즘을 지원하지 않으므로** 사용은 불가능하나 규격상 예약됩니다.
>
>

* **💡 시험용 계산 팁:** 필요한 실사용 IP 주소가 **29개**인 경우:
* `/27` 서브넷은 총 32개 IP 주소를 주지만, 예약 IP 5개를 빼면 **27개**만 남으므로 할당 불가능합니다 ($32 - 5 = 27 < 29$).
* 따라서 한 단계 더 큰 `/26` 서브넷(총 64개 IP, 실사용 59개)을 반드시 채택해야 합니다.

### 🛣️ 3. 외부 연결 및 주소 변환 인프라 (IGW, Bastion, NAT)

#### 🗺️ Internet Gateway (IGW)

* VPC 내부 자원을 외부 퍼블릭 인터넷 세상과 다리 놓아주는 관문입니다. 수평 확장이 기본 보장되며 고가용성/중복성을 가집니다.
* **1:1 매핑 원칙:** 하나의 VPC에는 단 하나의 IGW만 결합(Attach)할 수 있으며, 역의 관계도 같습니다.
* **⚠️ 라우팅 필수:** IGW를 생성 및 VPC에 연결한 것만으로는 통신이 안 됩니다. 반드시 서브넷과 연동된 **라우팅 테이블을 수정하여 `0.0.0.0/0` (모든 IPv4 트래픽)의 대상(Target)을 해당 `igw-xxxxxxxx`로 지정**해야만 비로소 퍼블릭 서브넷으로서 기능합니다.

#### 🏰 Bastion Hosts (배스천 호스트)

* 사설(Private) 서브넷에 숨겨진 EC2 인스턴스에 개발자나 관리자가 안전하게 SSH 터널링 보안 접속을 수행할 수 있도록 디딤돌 역할을 해주는 특수 퍼블릭 EC2 인스턴스입니다.
* 퍼블릭 서브넷에 배치되며, 내부망을 통해 모든 사설 서브넷 인스턴스들과 연결됩니다.
* **보안 그룹 설계 규칙:** 배스천 호스트의 인바운드 22번(SSH) 포트는 무조건 `0.0.0.0/0`으로 전면 개방하면 안 되며, **기업의 공용 고정 CIDR 블록이나 작업자 본인의 특정 IP 주소로만 접근을 엄격히 한정**해야 합니다. 사설 서브넷 EC2의 보안 그룹은 오직 배스천 호스트의 보안 그룹 ID(또는 프라이빗 IP)로부터 들어오는 SSH 요청만 허용하도록 문을 잠가야 합니다.

#### 📟 NAT Instance (과거 유산, 시험 출제용)

* 사설 서브넷 인스턴스들이 패킷을 밖으로 내보내(아웃바운드) 인터넷 업데이트 등을 수행할 수 있게 변환해주는 퍼블릭 서브넷 전용 EC2 인스턴스입니다. (2020년 말 표준 지원 종료)
* **필수 조건 3가지:** 1. 인스턴스 네트워크 설정에서 '소스/목적지 확인(Source/Destination Check)' 옵션을 반드시 비활성화(False)로 꺼야 라우터 역할을 수행할 수 있습니다.
2. 고정 인터넷 통신을 위해 **탄력적 IP(Elastic IP)** 결합이 필수적입니다.
3. 사설 서브넷 라우팅 테이블의 `0.0.0.0/0` 타겟을 해당 NAT 인스턴스의 `i-xxxxxx` ID로 지정해야 합니다.
* **한계점:** 고가용성(HA)이 내장되어 있지 않아 사용자가 직접 ASG(Auto Scaling Group) 및 멀티 AZ 기반 스크립트를 짜야 하며, 네트워크 대역폭 한계치가 선택한 EC2 인스턴스 스펙(Type) 크기에 종속되므로 병목이 발생하기 쉽고, 인바운드/아웃바운드 보안 그룹 규칙을 수동 관리해야 합니다.

#### 🚀 NAT Gateway (AWS 관리형 대안)

* AWS가 100% 관리하는 고가용성, 초고대역폭 기반 주소 변환 서비스입니다. 사용자가 OS나 보안 그룹을 관리할 필요가 전혀 없습니다.
* 퍼블릭 서브넷에 배포되며 자체 탄력적 IP(EIP)를 부여받아 작동합니다. 프라이빗 서브넷 인스턴스들의 통신 경로는 다음과 같습니다: `프라이빗 서브넷 인스턴스 -> NAT Gateway -> 인터넷 게이트웨이(IGW) -> 외부 인터넷`.
* **대역폭 확장:** 기본 초당 **5 Gbps**의 처리량을 보장하며 트래픽 폭증 시 최대 **45 Gbps**까지 백엔드에서 자동 확장됩니다.
* **⚠️ 주의 공간 제한:** NAT Gateway는 자신이 속한 **동일한 서브넷 내부의 인스턴스 통신에는 주소 변환 도움을 줄 수 없으며**, 반드시 타 사설 서브넷에서 이 NAT Gateway가 위치한 퍼블릭 서브넷 경로로 트래픽을 쏠 때만 정상 작동합니다.
* **고가용성(High Availability) 설계 아키텍처:**
* NAT Gateway 자체는 단일 가용 영역(AZ) 내에서만 이중화 복원력을 가집니다. 즉, **해당 가용 영역(AZ) 전체가 물리적으로 다운되면 그 영역에 생성된 NAT Gateway도 정지**합니다.
* 따라서 진정한 결함 허용(Fault-tolerance) 시스템을 구축하려면 **여러 AZ(예: AZ-A, AZ-B)마다 각각 독립적인 NAT Gateway를 개별 배포**하고 각 사설 서브넷 라우팅 테이블이 본인 가용 영역의 NAT Gateway를 바라보게 격리 설계해야 합니다.
* 교차 리전/교차 AZ 장애 조치(Cross-AZ Failover)를 위해 라우팅 테이블을 타 가용 영역의 NAT Gateway와 복잡하게 x자로 꼬아서 연결할 필요는 없습니다. (가용 영역이 다운되면 어차피 그 안의 컴퓨팅 리소스도 죽으므로 타 지역 NAT를 빌려 쓸 이유가 없기 때문)

### 🛡️ 4. 서브넷 vs 인스턴스 이중 방화벽 체계: SGs & NACLs

AWS VPC 네트워크 보안은 2개의 방화벽 레이어가 상호 보완하며 보호합니다.

#### 📜 Network Access Control List (NACL - 서브넷 단위, Stateless)

서브넷 경계선에 위치하여 인바운드/아웃바운드 트래픽을 통제하는 거대 방화벽입니다. 하나의 서브넷은 무조건 단 하나의 NACL과만 결합할 수 있습니다.

* **규칙 평가 우선순위:** 모든 규칙은 고유의 번호(1~32,766)를 가집니다. **번호가 낮을수록 우선순위가 높으며**, 위에서부터 순차 매칭되다가 최초로 조건이 부합하는 규칙을 만나는 즉시 평가를 종료(First Match Win)하고 트래픽 통과 여부를 결정합니다.
* **기본 낙수효과 및 거부:** 규칙 장부의 가장 마지막은 항상 **별표(`*`) 번호 규칙**으로 마감되며, 위에서 일치하는 규칙이 단 하나도 없으면 모든 요청을 전면 거부(DENY)합니다. 관리 편의성을 위해 AWS는 규칙 번호를 100단위(`100, 200, 300`)씩 간격을 띄워 추가하는 것을 강력히 권장합니다.
* **기본(Default) vs 사용자 정의(Custom) NACL 규칙 체계:**
* **기본 제공 NACL:** VPC 생성 시 자동 부여되는 기본 NACL은 인바운드 및 아웃바운드 모두 모든 트래픽 허용(`ALLOW`)으로 열려 있습니다.
* **사용자 정의 NACL:** 사용자가 콘솔에서 새로 생성한 커스텀 NACL은 **인바운드/아웃바운드 불문하고 모든 트래픽 전면 거부(`DENY`)** 상태로 생성되므로 필요한 포트를 직접 한땀한땀 열어주어야 합니다. 특정 악성 블랙리스트 IP 주소를 서브넷 진입 전단에서 확실히 차단(Deny)하고자 할 때 가장 유용합니다.


* **⚠️ Stateless(상태 비저장) 특성과 휘발성 포트(Ephemeral Ports) 대란:**
* NACL은 패킷의 상태를 기억하지 못하므로, 들어오는 문(Inbound)을 열어주었어도 **나가는 문(Outbound) 규칙에 나가는 경로가 명시되어 있지 않으면 패킷이 밖으로 탈출하지 못하고 응답이 증발**합니다.
* **클라이언트의 통신 원리:** 클라이언트가 서버의 웹 포트(80/443)로 요청을 찌르면, 클라이언트 장비 운영체제(OS)는 응답을 받아먹기 위해 무작위 임시 포트인 휘발성 포트(Ephemeral Ports)를 자체 개방합니다. (리눅스: `32,768 – 60,999` / 윈도우 및 IANA 표준: `49,152 – 65,535`)
* **NACL 규칙 구성 설계 핵심:** 따라서 웹 서버 서브넷의 인바운드 포트 80을 열어주었다면, 웹 서버가 클라이언트에게 웹 페이지 답장을 정상 송신할 수 있도록 **NACL 아웃바운드 규칙에 대상 클라이언트의 휘발성 포트 범위(`32,768 - 65,535`)에 대한 트래픽 허용 규칙을 반드시 함께 명시**해 주어야 통신이 성립됩니다.



#### 🛡️ Security Group (보안 그룹 - 인스턴스 단위, Stateful)

* 가상 서버(EC2) 인스턴스 레벨에 직접 부착하는 초소형 정밀 방화벽입니다.
* **Stateful (상태 저장) 특성:** 인바운드 규칙을 통과하여 내부로 진입한 패킷은 AWS 가상화 레이어가 통신 상태를 기억하므로, **아웃바운드 규칙의 차단 여부와 일절 상관없이 무조건 나가는 응답 패킷 통과를 보장**합니다. (그 반대의 아웃바운드 시작 요청도 동일하게 귀환 인바운드가 자동 보장됩니다.) 따라서 NACL과 달리 복잡한 휘발성 포트 개방 규칙을 아웃바운드에 중복 설계할 필요가 없습니다.
* **규칙 제약:** 오직 허용(`ALLOW`) 규칙만 등록할 수 있으며, 특정 IP 차단(`DENY`) 규칙은 선언이 불가능합니다. (차단은 무조건 상위 레이어인 NACL에서 수행해야 합니다.)

#### 🔍 네트워킹 문제 트러블슈팅 매트릭스 (★시험 단골 패턴)

VPC 내 통신 오류 발생 시 원인 분석 공식입니다.

1. **인바운드 REJECT (접속 자체가 안 됨):** 상위 레이어인 NACL 인바운드 규칙에서 막혔거나, 인스턴스의 보안 그룹 인바운드 허용 장부에 해당 IP/포트가 누락된 상태입니다.
2. **인바운드 ACCEPT인데 아웃바운드 REJECT (들어왔는데 응답을 못 줌):** 보안 그룹은 Stateful 하므로 들어온 요청의 나가는 응답을 막지 않습니다. 따라서 이 현상은 **100% NACL 아웃바운드 규칙에서 나가는 패킷을 거부(`DENY`)했거나 휘발성 포트 대역 개방이 누락**된 상태입니다.

### 🤝 5. VPC 확장 및 프라이빗 서비스 연동 아키텍처

#### 🔗 VPC Peering (VPC 피어링)

* AWS의 글로벌 전용 백본 네트워크망을 활용해 두 개의 VPC 간에 사설(Private) 통신 전용 통로를 뚫어주는 기능입니다. 서로 다른 AWS 계정 간, 또는 서로 다른 리전 간(Inter-Region) VPC 매핑도 완벽 지원합니다.
* **철칙 2가지:**
1. 두 VPC 간의 **CIDR 주소 블록이 1비트라도 겹치면(Overlap) 피어링 연결 생성이 원천 불가능**합니다.
2. **비전이적 (Non-transitive) 특성:** VPC A와 VPC B가 피어링되어 있고, VPC B와 VPC C가 피어링되어 있다고 해서, **VPC A가 VPC B를 징검다리 삼아 VPC C와 통신할 수 없습니다.** A와 C가 통신하려면 반드시 A-C 간의 독자적인 피어링 연결을 직접 새로 수립해야 합니다.


* **라우팅:** 연결 성립 후, 통신을 원하는 각 VPC 하위 서브넷 라우팅 테이블에 상대방 VPC CIDR 대역의 타겟을 피어링 연결 ID인 `pcx-xxxxxx`로 지정해 주어야 통신이 흐르기 시작합니다. 보안 그룹 설정 시 동일 리전 내에 있다면 상대방 VPC의 보안 그룹 ID를 내 보안 그룹의 인바운드 소스로 상호 참조(Reference)하는 고도화된 설계가 가능합니다.

#### 🕳️ VPC Endpoints (VPC 엔드포인트 - 인터넷 없는 AWS 연동)

원래 S3, DynamoDB, CloudWatch, SSM 등 모든 AWS의 개별 서비스들은 고유의 인터넷 퍼블릭 URL 엔드포인트를 가집니다. 따라서 사설 서브넷 EC2가 이들과 통신하려면 NAT Gateway 등을 타고 퍼블릭 인터넷 망으로 나갔다 들어와야 합니다. **VPC 엔드포인트를 쓰면 외부 퍼블릭 인터넷망으로 한 발짝도 나가지 않고 AWS 사설 백본 네트워크를 통해 이 서비스들과 초고속 직통 프라이빗 통신**을 달성할 수 있습니다.

* **1. 게이트웨이 엔드포인트 (Gateway Endpoint):**
* **대상 서비스 딱 2가지:** **`Amazon S3` 및 `Amazon DynamoDB**` 전용입니다.
* **작동 원리:** 서브넷 라우팅 테이블에 접두사 목록(Prefix List, 예: `pl-xxxxxxxx` - S3의 리전 공용 IP 대역 세트) 형태로 타겟 타겟 게이트웨이 엔드포인트 ID(`vpce-xxxxxx`)가 자동 등록되는 방식입니다.
* **특징:** 네트워킹 레이어상 **100% 완전 무료**이며, 인스턴스에 가상 네트워크 카드를 박는 방식이 아니므로 **보안 그룹(Security Group)을 연결하거나 사용하지 않습니다.** 라우팅 수정만으로 전사 자원이 S3에 무료 통신을 할 수 있게 열리므로 시험 문제에서 S3 비용 최소화 사설 연결이 나오면 무조건 게이트웨이 엔드포인트가 정답입니다.


* **2. 인터페이스 엔드포인트 (Interface Endpoint - AWS PrivateLink 기반):**
* **대상 서비스:** S3(일부 특수 목적용)를 포함하여 CloudWatch, SSM, 데이터베이스 등 **게이트웨이 2종을 제외한 수백 개의 거의 모든 AWS 서비스**들이 이 방식을 씁니다.
* **작동 원리:** 서브넷 내부에 프라이빗 IP 주소를 부여받은 가상 네트워크 카드인 탄력적 네트워크 인터페이스(ENI)를 직접 프로비저닝하여 진입점(Entry Point)을 구축하는 방식입니다.
* **특징:** 인스턴스 레벨로 ENI가 박히기 때문에 **반드시 보안 그룹(Security Group)을 연결하여 접근 포트를 통제**해 주어야 합니다. 인프라 비용으로 **시간당 유지비 및 처리된 데이터 용량(GB)당 비용이 추가 청구**되는 유료 서비스입니다.


* **💡 S3 전용 엔드포인트 선택의 기로 (★시험 단골 핵심):**
* AWS 내부 VPC 안에서 돌아가는 애플리케이션이 S3를 찌를 때는 비용이 제로인 **게이트웨이 엔드포인트**를 고르는 것이 압도적으로 유리합니다.
* 하지만 온프레미스 사설 데이터 센터(Site-to-Site VPN 또는 Direct Connect로 연결된 환경)나 타 리전의 외부 네트워크 환경에서 우리 VPC를 경유해 S3로 사설 접근을 뻗어 나가고 싶을 때는 라우팅 테이블 전파가 불가능한 게이트웨이 방식을 쓸 수 없으므로, 프라이빗 IP 주소 기반으로 온프레미스 라우팅이 통하는 **인터페이스 엔드포인트**를 전면에 배치해야 합니다.



#### 📡 AWS PrivateLink (VPC Endpoint Services)

* **개념:** 내가 만든 VPC 내부의 특정 백엔드 서비스(예: 자체 빌드한 고객 관리 API 프로그램)를 퍼블릭 인터넷 노출이나 복잡한 VPC 피어링 그물망 구성 없이, 다른 수백 수천 개의 고객사(Consumer) VPC 채널로 비공개 안전하게 공유·노출하고 싶을 때 사용하는 기술입니다.
* **구현 아키텍처:** 서비스 제공자(Provider) VPC 전단에 Network Load Balancer(NLB)를 심고 이를 PrivateLink 서비스로 등록하면, 소비자(Consumer) VPC 측에서는 해당 주소를 기반으로 인터페이스 엔드포인트(ENI)를 뚫어 1:1 사설 연결을 맺습니다. 양사 간의 인터넷 게이트웨이, NAT, 라우팅 테이블 오픈 및 CIDR 주소 중복을 신경 쓸 필요가 전혀 없는 초격리 네트워크 공유 기법입니다.

#### 📊 VPC Flow Logs (VPC 흐름 로그) & 분석 에코시스템

VPC, 서브넷, 또는 개별 ENI 레이어를 통과하는 모든 IP 트래픽의 메타데이터 합격/불합격 명세를 포착하여 기록하는 감사 기능입니다.

* **구문 핵심 변수 명세:** 로그 한 줄(Record)에는 `version, account-id, interface-id, srcaddr(출발지 IP), dstaddr(목적지 IP), srcport(출발 포트), dstport(목적 포트), protocol, packets, bytes, action, log-status` 등이 남습니다.
* **문제 해결 접근법:** `action` 필드에 기록된 값이 `ACCEPT`인지 `REJECT`인지를 분석하여 보안 그룹이나 NACL 중 어느 방화벽 장부에서 악성 포트 스캔 공격이나 통신 패킷 드롭이 터졌는지 실시간 추적합니다. 수집된 흐름 로그는 `Amazon S3` 또는 `CloudWatch Logs`로 전송할 수 있습니다.
* **고급 분석 결합 아키텍처:**
* **CloudWatch Logs 연동:** 흐름 로그가 쌓이면 `CloudWatch Contributor Insights` 기능을 돌려 우리 네트워크 대역폭을 가장 많이 갉아먹고 있는 상위 10대 헤비 업로더 IP 주소(`srcaddr`)를 즉각 식별해 냅니다. 또는 SSH(22)/RDP(3389) 특정 악성 접속 포트에 매칭되는 지표 필터(Metric Filter)를 생성해두고 평소보다 요청 빈도가 폭증하면 `CloudWatch Alarm`을 가동하여 `SNS` 경보를 운영자 폰으로 쏠 수 있습니다.
* **S3 버킷 연동:** 대용량 로그 파일이 S3에 쌓이면 표준 SQL 구문으로 즉시 페타바이트급 데이터 검색이 가능한 `Amazon Athena`를 연결해 악성 IP 주소의 행동 패턴을 쿼리하고, 이를 비즈니스 대시보드 도구인 `Amazon QuickSight`와 이어 붙여 트래픽 통계 시각화 화면을 구축합니다.

### 🏢 6. 하이브리드 온프레미스 연동망 (VPN, DX, Transit Gateway)

#### 🗺️ AWS Site-to-Site VPN (가성비 공용 인터넷망 기반 크립토 터널)

온프레미스 기업 데이터 센터와 AWS VPC 간을 퍼블릭 인터넷 선로를 활용해 프라이빗하게 이어주는 암호화 연결 솔루션입니다. 인터넷망을 밟고 가지만 **IPsec 암호화** 프로토콜 터널링이 걸리므로 안전합니다.

* **핵심 구성 장비 2가지:**
1. **가상 프라이빗 게이트웨이 (VGW - 가상 사설 게이트웨이):** VPN 연결 시 AWS VPC 네트워크의 맨 앞단 경계선에 부착되는 전용 VPN 종단 수신기(Concentrator)입니다. 고유의 자율 시스템 번호(ASN)를 부여할 수 있습니다.
2. **고객 게이트웨이 (CGW):** 온프레미스 데이터 센터 쪽에 위치한 물리 방화벽 장비 또는 소프트웨어 엔드포인트입니다. 만약 CGW 장비가 사설 NAT 벽 뒤에 숨어 있다면 `NAT 탐색(NAT-T)` 옵션을 켜고 외부 NAT 장비의 퍼블릭 IP를 진입점으로 뚫어주어야 합니다.


* **⚠️ 주의 운영 필수 단계:** VPN 터널이 뚫린 후, VPC 서브넷과 연관된 라우팅 테이블 설정으로 들어가 **가상 프라이빗 게이트웨이(VGW)의 '라우트 전파(Route Propagation)' 옵션을 반드시 활성화(Enable)** 해주어야 온프레미스 데이터 센터의 사설 라우팅 주소 장부가 VPC 내부로 자동 전파되어 상호 통신이 가능해집니다. 온프레미스에서 VPC 내부 EC2를 향해 네트워크 선로 점검용 핑(`ping`)을 날려보고 싶다면, EC2 보안 그룹 인바운드 규칙에 **ICMP 프로토콜** 허용을 명시해야 합니다.

#### 🕸️ AWS VPN CloudHub (다점포 허브앤스포크 VPN 모델)

* **개념:** 본사, 지사 A, 공장 B, 연구소 C 등 전 세계 물리적으로 흩어진 다수의 온프레미스 거점들을 하나의 대형 가상 사설 게이트웨이(VGW)에 여러 개의 Site-to-Site VPN 회선으로 줄줄이 엮어주는 저비용 네트워크 허브 아키텍처입니다.
* **특징:** **동적 라우팅** 설정을 기반으로 구동되며, 지사 A에서 출발한 트래픽이 AWS VGW 허브를 징검다리 삼아 본사나 공장 B의 사설망까지 안전하게 공용 인터넷 VPN 선로를 타고 종단 간 통신을 할 수 있게 허용해 줍니다.

#### 🚊 AWS Direct Connect (DX - 초고속 전용 오프라인 사설 전용선)

공용 인터넷망을 0.1%도 거치지 않고, 온프레미스 데이터 센터와 AWS 리전 간을 물리적인 전용 광케이블 선로로 1:1 다이렉트 직결하는 엔터프라이즈 하이브리드 네트워킹의 끝판왕 서비스입니다.

* **장점:** 대용량 데이터 전송 시 인터넷 망의 혼잡에 영향을 받지 않으므로 네트워크 지연 시간(Latency)이 극도로 일정하게 유지되며, 대량 이그레스 트래픽 발생 시 인터넷 회선 과금 방식보다 데이터 전송 비용이 획기적으로 절감됩니다. IPv4와 IPv6 통신을 동시에 지원합니다.
* **연결 공급 유형 2가지:**
* **Dedicated Connections (전용 연결):** 초당 `1Gbps, 10Gbps, 100Gbps` 규격의 물리 포트 자체를 고객이 통째로 독점 선점하는 방식입니다. AWS에 공식 요청을 넣으면 파트너 통신사가 라우팅 구성을 마감합니다.
* **Hosted Connections (호스팅 연결):** 초당 `50Mbps` 소용량부터 `500Mbps, 1Gbps, 10Gbps`까지 AWS Direct Connect 파트너 통신사가 미리 뚫어놓은 거대 물리 파이프라인의 대역폭 일부를 슬라이싱하여 임대 분할 사용하는 방식입니다. 필요에 따라 유연하게 용량을 떼고 붙일 수 있습니다.


* **⚠️ 암호화 및 구축 기간 반전 디테일:** **Direct Connect는 전송 중인 패킷 데이터에 대한 자체 암호화 처리를 기본적으로 수행하지 않습니다.** (공용 인터넷이 아닌 완전 격리된 물리 전용선이므로 안전하다고 판단하기 때문). 만약 기업 보안 규정상 전용선 내부에서도 무조건 암호화가 돌아가야 한다면, **Direct Connect 회선 위단에 Site-to-Site VPN을 얹어서 IPsec으로 이중 래핑 암호화 프라이빗 연결을 가동**해야 합니다. 또한, 물리 선로 공사가 동반되므로 최초 직결망 최종 개통 완료까지 최소 **1달 이상**의 긴 시간이 소요되므로 일주일 내의 초고속 데이터 마이그레이션 요구 조건에는 정답이 될 수 없습니다.
* **Direct Connect Gateway (DXGW):** 단 하나의 전용 DX 회선만 개통해두면, 동일한 AWS 계정 내에서 전 세계 서로 다른 리전(예: 서울, 버지니아, 아일랜드)에 흩어져 있는 수많은 멀티 리전 VPC들을 단 한 번에 프라이빗 연결로 묶어낼 수 있는 중앙 집중형 연결 허브 게이트웨이입니다.
* **💥 인프라 고가용성(Resiliency) 및 백업 다중화 가이드라인:**
* 기업의 핵심 미션 크리티컬 워크로드를 돌릴 때 DX 회선 하나만 쓰면 물리 선로 단선 시 전체 시스템이 마비됩니다.
* **최대 가용성 달성:** 서로 다른 물리적 위치를 가지는 독립된 Direct Connect 로케이션 2곳을 소싱하고, 각 로케이션마다 회선을 2개씩 이중화로 총 4개 회선을 뚫으면 중단 확률이 0%에 수렴하는 완벽한 내결함성이 완성됩니다.
* **가성비 백업 다중화 (★시험 빈출 패턴):** 보조 DX 회선을 하나 더 파는 것은 비용 낭비가 심하므로, **평소에는 초고속 전용선인 Direct Connect를 메인(기본) 통신로로 쓰다가, 비상사태 단선 시 공용 인터넷 기반의 `Site-to-Site VPN` 회선으로 트래픽이 자동 우회(Failover)되도록 백업 선로를 구성**해 두는 아키텍처가 비용 효율성 부문 최선의 정답입니다.



#### 🔀 AWS Transit Gateway (전사 네트워크 중앙 통제 허브)

VPC 개수가 수십 수백 개로 늘어나고 온프레미스 본사/지사망이 얽히면 VPC 피어링 그물망 구조(`Mesh 토폴로지`)가 꼬여 관리가 불가능한 카오스가 찾아옵니다. Transit Gateway는 이 복잡한 선로들을 단 하나의 중앙 집중식 교차로로 한데 모아주는 클라우드 엔터프라이즈 라우터(스타 토폴로지 모델)입니다.

* **전이적 라우팅 (Transitive Routing) 해방:** 피어링과 달리 전이적 통신을 기본 지원하므로, Transit Gateway에 연결된 모든 VPC와 온프레미스 망들은 별도의 개별 피어링 연결을 파지 않아도 중앙 라우터를 거쳐 서로 자유롭게 통신할 수 있습니다. `Resource Access Manager (RAM)`를 연동하면 멀티 계정(Organizations) 간에도 라우터를 완벽 공유할 수 있으며 리전 간 Transit Gateway끼리의 피어링도 지원합니다. AWS 인프라 중 유일하게 **IP 멀티캐스트(Multicast)** 통신 프로토콜 전송을 지원합니다.
* **🏎️ VPN 대역폭 대폭발: ECMP (Equal-Cost Multi-Path Routing) 기술 결합**
* 원래 일반 가상 사설 게이트웨이(VGW)에 Site-to-Site VPN을 직접 꽂으면 단일 터널당 최대 대역폭 한계치는 무조건 **1.25 Gbps**로 고정 제한됩니다. VPN이 기본 터널 2개로 이중화 구성되어 있어도 평소엔 1개 터널만 활성화되기 때문입니다.
* **Transit Gateway에 VPN을 꽂는 순간:** **등가 다중 경로 라우팅(ECMP)** 기술이 활성화되면서, 백엔드에 구성된 **2개의 이중화 VPN 터널을 동시에 활성화하여 트래픽을 양분 분산 전송**하기 시작합니다. 즉, VPN 회선 하나당 대역폭이 2.5 Gbps ($1.25 \times 2$)로 즉시 2배 폭증합니다.
* 여기에 만족하지 않고 Site-to-Site VPN 연결 개수를 2개, 3개 더 추가하여 Transit Gateway에 묶어버리면 대역폭이 5 Gbps, 7.5 Gbps 단위로 선형 배수 증가하는 초고속 암호화 하이브리드 통로를 커스텀 빌드할 수 있습니다. (단, Transit Gateway를 거쳐 가는 모든 트래픽은 데이터 처리 용량 1GB당 이그레스 비용이 부과되는 점을 비용 트레이드오프로 계산해야 합니다.)

### 🛡️ 7. VPC 전체 전천후 정밀 방화벽: AWS Network Firewall

기존의 NACL과 보안 그룹은 오직 IP와 포트 번호 기반으로만 패킷을 통제하는 L4 수준의 한계를 가집니다. **VPC 내부로 유입되거나 나가는 모든 트래픽의 패킷 데이터 본문(Body)까지 딥 패킷 인스펙션(DPI) 기법으로 샅샅이 뒤져 악성코드 감염 및 해킹 징후를 통제하고 싶을 때 사용하는 L3~L7 통합 완전 관리형 방화벽 서비스**입니다.

* **아키텍처 및 검사 범위:** 백엔드에서 AWS의 Gateway Load Balancer(GWLB) 기술을 기반으로 구동되며, `VPC 간 교차 트래픽, 외부 인터넷으로 나가는 아웃바운드, 외부에서 들어오는 인바운드, 온프레미스 연동망(DX/VPN)` 등 인프라의 모든 사방향 경계선 패킷을 정밀 검사합니다. 규칙 일치 이력 로그는 분석을 위해 `S3, CloudWatch Logs, Kinesis Data Firehose`로 실시간 떨궈줄 수 있습니다.
* **🔥 초정밀 제어 기능 명세 (Fine-Grained Controls):**
* 수만 개의 IP 및 특정 포트 차단은 기본 지원합니다.
* 특정 프로토콜 차단 기술: 아웃바운드 통신망에서 윈도우 파일 공유 프로토콜인 **SMB 프로토콜**이 감지되면 패킷을 즉각 차단하도록 선언 가능합니다.
* **상태 기반 도메인 목록 화이트리스트 규칙:** 소스 코드 내 오픈소스 라이브러리 업데이트를 위해 외부 협력사망이나 오픈소스 저장소 대역만 열어주고 싶다면, `*.mycorp.com` 이나 `*.github.com` 처럼 **정규 표현식을 가미한 일반 패턴 매칭 기술 및 도메인 이름 기반의 아웃바운드 통신 화이트리스트**를 강제 구현할 수 있습니다.
  *오픈소스 네트워크 침입 방지 엔진 표준인 **Suricata 호환 규칙 규격**을 내장하여, 실시간 흐름(Active Flow) 속에서 바이러스 침입 패턴이 감지되면 즉각 트래픽을 드롭(`DROP`), 허용(`ALLOW`), 혹은 경고(`ALERT`) 조치하는 고차원 네트워크 위협 방어 인프라를 제공합니다.

## 💰 4단계: 클라우드 네트워크 비용 최적화 핵심 요약 (Simplified Costs)

AWS 인프라 아키텍처 설계 시 네트워크 통신 비용 청구 폭탄을 방어하기 위한 최우선 정형 공식입니다.

1. **공용(Public) IP 대신 무조건 사설(Private) IP를 사용하여 내부 통신하라:** 동일 VPC 내부라 할지라도 EC2 간에 퍼블릭 IP나 탄력적 IP 주소를 목적지로 두고 패킷을 쏘면, 트래픽이 외부 인터넷망 경계선을 밟고 유턴해 들어오므로 불필요한 송신(Egress) 네트워크 비용이 청구됩니다. 프라이빗 사설 IP를 쓰면 비용이 완전 무료이거나 극소화되며 내부 백본 통신이므로 속도와 대역폭 성능도 훨씬 뛰어납니다.
2. **비용과 가용성의 트레이드오프 - 동일 가용 영역(AZ) 통신 법칙:**
* 트래픽이 서로 다른 가용 영역(예: AZ-A의 EC2와 AZ-B의 RDS) 경계선을 넘나들 때마다 기가바이트(GB)당 데이터 전송 비용이 양방향으로 청구됩니다.
* 비용을 완벽히 아끼려면 최대한 모든 컴퓨팅 자원을 **단일 가용 영역(Same AZ)** 내에 결집시켜 통신하게 짜야 합니다. 단, 이 경우 해당 AZ가 물리 화재 등으로 다운되면 백업 장애 조치(Failover)가 안 되므로 시스템 전체가 정지하는 아키텍처 가용성 희생이 동반됨을 명심해야 합니다.


3. **이그레스 트래픽(Egress - 밖으로 나가는 트래픽)을 내부에 가두어라:** 외부 인터넷 세상으로 데이터를 내보내는 송신 트래픽 비용은 AWS 과금 체계 중 가장 비쌉니다. (반대로 외부에서 AWS 내부로 밀어 넣는 인바운드 인그레스(Ingress) 트래픽은 100% 언제나 무료입니다.) 따라서 최대한 인터넷 외부망으로 데이터가 나가지 않도록 아키텍처를 가두어야 합니다.

#### 📦 S3 / CloudFront 데이터 전송 비용 구조 정밀 해부 (미국 리전 기준)

* **S3 내부로 데이터 업로드(Ingress):** 완전 무료
* **S3에서 외부 일반 인터넷으로 직접 데이터 다운로드:** **$0.09 / GB** (가장 비쌈)
* **S3 Transfer Acceleration 가동 시:** 전 세계 엣지를 타고 유입되므로 업로드/다운로드 속도가 50%~500% 급상승하는 대신, 기본 데이터 전송료 외에 **1GB당 $0.04~$0.08의 추가 프리미엄 할증 비용**이 붙습니다.
* **S3에서 바로 옆의 CloudFront(CDN)로 데이터를 넘길 때:** **100% 완전 무료**
* **CloudFront 엣지에서 최종 인터넷 사용자에게 데이터 다운로드:** **$0.085 / GB** (S3에서 직접 인터넷으로 던지는 $0.09보다 기본 단가 자체가 저렴함)
* **🚀 비용 절감 아키텍처 결합 결론:** 대량의 정적 파일 다운로드가 일어나는 서비스라면 S3 링크를 유저에게 직접 뿌리지 말고, 전단에 **CloudFront를 연동하여 캐싱 레이어를 구축**해야 합니다. S3-CloudFront 간 전송료는 무료이며, 전 세계 엣지에 파일이 캐싱되므로 S3를 직접 찌르는 API 호출 횟수 비용 자체가 소멸하고, 인터넷 송신 단가도 더 저렴하므로 **최대 7배 이상의 네트워킹 운영 비용을 극적으로 절감**할 수 있습니다.
* **S3 교차 리전 복제 (CRR) 비용:** 타 리전 S3 버킷으로 백업 복제본 객체가 넘어갈 때 **$0.02 / GB**의 리전 간 전송 비용이 청구됩니다.
