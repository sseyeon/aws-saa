# 1. Docker

- 앱을 배포하기 위한 소프트웨어 개발 플랫폼
- 앱은 **컨테이너**에 패키지화되어 어떤 운영 체제에서든 실행할 수 있게 됨
- 앱은 실행되는 위치와 관계없이 동일하게 작동됨
    - 어떤 기기(머신)에서든 실행 가능
    - 호환성 문제 없음
    - 예측 가능한 동작
    - 작업량 감소
    - 유지 및 배포가 쉬움
    - 모든 언어, 운영 체제, 기술과 호환됨
- 사용 사례: 마이크로서비스 아키텍처, 온프레미스에서 AWS 클라우드로의 앱 이동 등

## 1-1. 🐬 Docker on an OS

![](https://velog.velcdn.com/images/gagaeun/post/875a585a-b92a-45d3-a91b-b0dfd4a41bf9/image.png)

- 첫 번째 도커 컨테이너는 Java 애플리케이션을 포함, 두 번째 도커 컨테이너에는 Node.js 애플리케이션이 있음.
- 다수의 도커 컨테이너가 동시에 실행될 수 있어서 Java 혹은 Node.js 애플리케이션을 가진 여러 도커 컨테이너가 있을 수 있음.
- 도커 내에서 MySQL 등의 데이터베이스도 실행 가능. 다용도로 활용 가능.
- 서버 관점에서는 모두 도커 컨테이너로 보임!

## 1-2. 🖼️ Where are Docker images stored?

> 도커에서 **이미지**는 애플리케이션과 그 실행 환경을 포함하는 패키지이다. 이미지는 컨테이너화된 애플리케이션을 구축하고 배포하기 위해 필요한 모든 파일, 라이브러리, 종속성 및 설정을 포함한다. 이 이미지는 컨테이너의 기반이 되며, 컨테이너를 실행하기 위한 실행 환경을 제공한다.
> 
- 도커 이미지는 도커 리포지토리에 저장됨
- **Docker Hub ([https://hub.docker.com](https://hub.docker.com/))**
    - **퍼블릭** 리포지토리
    - 다양한 기술 또는 운영 체제에 맞는 기본 이미지를 찾을 수 있음 (ex. Ubuntu, MySQL)
- **Amazon ECR (Amazon Elastic Container Registry)**
    - **프라이빗** 리포지토리
    - **퍼블릭** 리포지토리로도 제공됨 (**Amazon ECR Public Gallery** [https://gallery.ecr.aws](https://gallery.ecr.aws/))

## 1-3. 🧐 Docker vs. Virtual Machines

- 도커는 "어떤 의미로는" 가상화 기술이지만 정확히는 아님
- 리소스는 호스트와 공유됨 → 따라서 한 서버에서 다수의 컨테이너를 공유할 수 있음
    
    ![](https://velog.velcdn.com/images/gagaeun/post/0d2e0a0d-4904-47a0-b2d8-450c102b9dce/image.png)
    

(왼) 가상 머신 아키텍처

- 인프라, 호스트 운영체제, 하이퍼바이저, 앱, Guest 운영체제가 있음
- 즉 EC2의 원리임. 다시 말해 EC2 머신은 하이퍼바이저에 실행되는 가상 머신과도 같음
- 그래서 Amazon이 EC2 인스턴스를 다양한 소비자에게 제공할 수 있으며 가상 머신의 EC2 인스턴스는 각자 분리되어 있음
- 즉, 리소스를 공유하지 않음

(오) 도커 컨테이너

- 인프라, EC2 인스턴스 같은 호스트 OS, 도커 Daemon 위에 많은 컨테이너가 있음
- 도커 Daemon에서 가볍게 실행되는 컨테이너라 공존할 수 있음
- 네트워킹이나 데이터 등을 공유할 수 있음
- 가상 머신보다 덜 안전하지만 하나의 서버에 많은 컨테이너를 실행할 수 있기 때문에 도커 컨테이너를 많이 사용함

## 1-4. ✔️ Getting Started with Docker

![](https://velog.velcdn.com/images/gagaeun/post/cb524a32-40a9-48f7-ae2e-b0544b282970/image.png)

1. **Dockerfile 작성** - 도커 컨테이너를 구성하는 파일 작성
2. **도커 이미지 구축** - 작성한 Dockerfile을 사용하여 도커 이미지를 구축. 도커 이미지는 베이스 이미지에 추가 파일 및 설정을 포함하여 생성
3. **도커 이미지 Push** - 구축한 도커 이미지를 도커 리포지토리(Docker Hub, Amazon ECR 등)에 저장하기 위해 Push
4. **도커 이미지 Pull** - 도커 리포지토리에서 필요한 이미지를 Pull하여 가져옴
5. **도커 컨테이너 실행** - Pull한 도커 이미지를 실행하여 도커 컨테이너로 실행. 이때, 도커 이미지를 구축할 때 사용한 코드와 설정이 실행됨

## 1-5. ❔ Docker Containers Management on AWS

- **Amazon Elastic Container Service (Amazon ECS)**: 도커 관리를 위한 Amazon의 전용 플랫폼
- **Amazon Elastic Kubernetes Service (Amazon EKS)**: 쿠버네티스의 관리형 버전 (오픈 소스)
- **AWS Fargate**: Amazon의 서버리스 컨테이너 플랫폼. ECS, EKS 둘 다 함께 작동할 수 있음
- **Amazon ECR**: 컨테이너 이미지를 저장하는데 사용


# 2. Amazon ECS
## 2-1. 💡 두 가지 실행 유형 (Launch Types)

Docker 컨테이너(ECS에서는 이를 **태스크, Task**라고 부릅니다)를 띄울 때, 기반이 되는 하드웨어 인프라를 내가 관리할지 AWS에 맡길지 선택해야 합니다.

### ① EC2 Launch Type (수동 인프라 관리형)

* **개념:** 내가 직접 EC2 인스턴스들을 생성하고 관리하는 방식입니다.
* **메커니즘:** 이 EC2 인스턴스들 내부에는 ECS 에이전트(Agent)라는 소프트웨어가 실행되어 내 ECS 클러스터에 "저 준비됐어요!" 하고 등록됩니다. 컨테이너의 배치와 라이프사이클은 AWS가 제어하지만, EC2 자체의 OS 패치나 모니터링은 내 몫입니다.

### ② Fargate Launch Type (완전 관리형 서버리스)

* **개념:** **"우리는 EC2 인스턴스 신경 쓰기 싫고, 딱 Docker 컨테이너만 띄우고 싶어!"** 할 때 쓰는 서버리스 옵션입니다.
* **메커니즘:** 내가 가상 서버를 프로비저닝할 필요가 전혀 없습니다. 컨테이너 명세서인 태스크 정의(Task Definitions)를 만들고 "이 컨테이너는 CPU X만큼, RAM Y만큼 줘"라고 설정만 하면, AWS가 알아서 인프라를 바닥에 깔고 태스크를 실행합니다. 확장할 때도 단순히 태스크 개수만 늘리면 되므로 관리가 극도로 단순해집니다.

## 2-2. 📄 두 가지 IAM Role의 명확한 구분 (★시험 단골)

ECS 환경에서는 권한이 두 갈래로 나뉩니다. 누가 쓰는 권한인지를 구별하는 것이 핵심입니다.

1. **EC2 Instance Profile (EC2 인스턴스 역할):**
* **대상:** 컨테이너가 아니라, 컨테이너를 등에 업고 있는 배경 인프라인 'EC2 인스턴스(ECS 에이전트)'가 사용하는 권한입니다.
* **용도:** ECS 에이전트가 AWS API를 호출하거나, 컨테이너 로그를 **CloudWatch Logs**로 보내거나, ECR(컨테이너 이미지 저장소)에서 Docker 이미지를 다운로드하거나, SSM Parameter Store/Secrets Manager에서 비밀번호를 읽어올 때 이 권한을 씁니다. (Fargate는 인스턴스를 안 쓰니 필요 없습니다.)


2. **ECS Task Role (태스크 역할):**
* **대상:** 실제 실행 중인 **Docker 컨테이너(태스크) 내부의 애플리케이션 코드**가 사용하는 권한입니다.
* **용도:** 컨테이너 안에서 돌아가는 프로그램이 비즈니스 로직을 수행하기 위해 **SQS 대기열에 메시지를 넣거나, S3 버킷에서 파일을 읽어올 때** 사용합니다. **태스크 정의(Task Definition)** 수준에서 꼼꼼하게 부여합니다.

## 2-3. ℹ️ 로드 밸런서 연동 (Load Balancer Integrations)

사용자들이 내 컨테이너 웹서비스에 접속할 수 있도록 앞에 대문을 달아주는 작업입니다.

* **Application Load Balancer (ALB):** **거의 모든 일반적인 상황에서 정답**입니다. 동적 포트 매핑을 지원하여, 하나의 EC2 인스턴스 안에 동일한 웹 컨테이너가 여러 개 떠 있어도 트래픽을 알아서 척척 분산해 줍니다.
* **Network Load Balancer (NLB):** 극도의 고성능, 초당 수백만 개의 초고속 처리가 필요하거나, 고정 IP(Static IP)를 써야 할 때, 혹은 다른 VPC와 안전하게 연결하는 **AWS PrivateLink**와 연동할 때 선택합니다.
* **Classic Load Balancer (CLB):** 구형 장비입니다. Fargate와 호환도 안 되고 고급 기능이 전혀 없으므로 **절대 권장하지 않습니다.**

## 2-4. 📁 데이터 볼륨 연동 (EFS로 영구 저장하기)

기본적으로 컨테이너는 껐다 켜지면 내부 데이터가 사라지는 휘발성 개체입니다. 데이터를 영구 보관하려면 외부 저장소를 마운트해야 합니다.

* **최고의 짝꿍 (Amazon EFS):** ECS 태스크에 **EFS 파일 시스템을 네트워크 드라이브처럼 마운트**할 수 있습니다.
* **특징:** EC2 방식과 Fargate 방식 모두 완벽히 지원합니다. EFS는 다중 가용 영역(Multi-AZ)을 지원하므로, 내 컨테이너가 1a 영역에 있든 1b 영역에 있든 상관없이 **동일한 영구 스토리지 데이터에 동시 접근 및 공유**가 가능합니다.
* **서버리스 콤보:** `Fargate + EFS` 조합을 쓰면 컴퓨팅도 서버리스, 저장소도 서버리스인 완벽한 **Serverless 아키텍처**가 완성됩니다.
* **⚠️ 주의 (오답 함정):** **Amazon S3는 ECS 태스크에 직접 파일 시스템(Volume) 형태로 마운트할 수 없습니다.** (S3는 API나 SDK로 통신해야 합니다.)

## 2-5. ⏫️ ECS Service Auto Scaling (수평 확장 법칙)

수요에 따라 컨테이너 개수를 유연하게 늘리고 줄이는 기법입니다. 내부 인프라에 따라 계층이 분리됩니다.

### 1) Task 레벨 스케일링 (ECS Service Auto Scaling)

실제 컨테이너(태스크)의 개수 자체를 늘리는 스케일링입니다. **AWS Application Auto Scaling** 메커니즘을 기반으로 작동합니다.

* **추적 지표 (Metrics):**
  * 컨테이너들의 **평균 CPU 사용률**
  * 컨테이너들의 **평균 메모리 사용률 (RAM 수치)**
  * ALB에서 유입되는 **타깃당 요청 수 (Request Count Per Target)**


* **스케일링 정책 옵션:**
  * **Target Tracking (대상 추적):** "우리 컨테이너 평균 CPU 60% 유지해 줘"라고 타깃을 주면 알아서 늘리고 줄입니다.
  * **Step Scaling (단계별 스케일링):** CloudWatch 경보가 울리면 그 수치 단계에 따라 "태스크 3개 추가", "더 심해지면 5개 추가" 식으로 대응합니다.
  * **Scheduled Scaling (예약 스케일링):** 블랙 프라이데이나 출근 시간처럼 예측 가능한 변화(Predictable changes)에 맞춰 특정 날짜/시간에 미리 늘려둡니다.



### 2) 인프라 레벨 스케일링 (EC2 Launch Type 한정)

만약 **EC2 Launch Type**을 쓰고 있다면, 컨테이너만 늘린다고 해결되지 않습니다. 방을 채울 컨테이너는 늘어나는데, 방을 놓을 땅(EC2 인스턴스)이 부족해질 수 있으니까요.

* **해결책 ➔ ECS Cluster Capacity Provider (용량 공급자):** ECS 태스크들의 요구량(CPU, RAM)을 모니터링하다가 인프라 공간이 부족해지면, 연동된 **EC2 Auto Scaling Group(ASG)을 트리거하여 기반 가상 서버를 자동으로 프로비저닝하고 확장**해 줍니다.
* 💡 **Fargate가 편한 이유:** Fargate는 애초에 바닥에 EC2 인스턴스가 없는 서버리스 모델이기 때문에 인프라 레벨 스케일링을 고민할 필요 없이 **Task 레벨 스케일링만 신경 쓰면 되므로 확장이 극도로 간단**해집니다.

## 2-6. 아키텍처 연동 3대장 예시 패턴

노트 마지막에 언급된 아키텍처 연동 사례들은 실무 및 시험에서 가장 세련되게 쓰이는 패턴들입니다.

### ① EventBridge를 통한 이벤트 트리거 방식 (ECS tasks invoked by EventBridge)

S3 버킷에 대용량 원본 영상 파일이 업로드되는 등의 특정 이벤트가 발생하면, 이를 **Amazon EventBridge**가 캐치하여 "야! ECS, 이 이벤트 처리용 컨테이너 태스크 하나 실행해!" 하고 다이렉트로 태스크를 실행(Invoke)시키는 구조입니다.

### ② EventBridge 크론탭 스케줄 방식 (ECS tasks invoked by EventBridge Schedule)

"매일 밤 12시 정각에 정산 컨테이너를 띄워서 배치 작업을 돌리고 배치 종료 시 컨테이너를 꺼라" 하는 요구사항이 있을 때 씁니다. EventBridge 스케줄러(Cron 식 활용)가 정해진 시간에만 ECS 태스크를 딱 실행시키는 똑똑한 구조입니다.

### ③ SQS 대기열 버퍼 방식 (ECS – SQS Queue Example)

앞 섹션에서 배운 디커플링의 정석입니다. 대규모 주문이나 작업 요청이 들어오면 우선 **SQS 대기열**에 저장합니다. ECS 서비스 내부의 태스크들은 SQS를 계속 폴링(Polling)하면서 메시지를 처리합니다. 이때 CloudWatch가 SQS의 대기열 길이(`ApproximateNumberOfMessages`)를 모니터링하다가 쌓인 편지가 너무 많으면 **ECS Service Auto Scaling을 트리거하여 컨테이너 태스크 개수를 쫙 늘려서** 광속으로 처리해 버리는 환상의 콤보 아키텍처입니다.

### 🎯 한눈에 외우는 SAA 시험 ECS 정답 공식

1. **"가상 서버 관리 없이 컨테이너만 독립적으로 배포 및 단순 수평 확장을 원함"** ➔ **ECS Fargate Launch Type**
2. **"컨테이너 내부 애플리케이션 코드가 S3나 DynamoDB 등 다른 AWS API를 호출해야 함"** ➔ **ECS Task Role 설정**
3. **"ECS 에이전트가 ECR에서 이미지를 땡겨오거나 CloudWatch로 로그를 보내지 못함"** ➔ **EC2 Instance Profile 권한 확인**
4. **"여러 가용 영역의 ECS 컨테이너들이 파일 데이터를 유실 없이 영구적으로 공유해야 함"** ➔ **Amazon EFS 마운트 (`S3 마운트는 오답`)**
5. **"컨테이너 수평 확장 시 추적할 수 있는 메特릭 지표"** ➔ **CPU 사용률, 메모리 사용률, ALB 타깃당 요청 수**
6. **"대규모 요청 유실 방지를 위해 SQS의 대기열 길이에 맞춰 컨테이너 개수를 조절하고 싶다"** ➔ **SQS + CloudWatch + ECS Service Auto Scaling 조합**

# 3. Amazon ECR (도커 이미지 금고)

컨테이너를 실행하려면 먼저 빵틀에 해당하는 'Docker 이미지'가 필요합니다. ECR은 이 도커 이미지를 안전하게 보관하고 관리하는 AWS 전용 저장소(Registry)입니다.

* **저장 위치와 통합:** 우리가 ECR에 이미지를 업로드(Push)하면, 내부 백그라운드에서는 **Amazon S3 버킷에 안전하게 저장**됩니다. 당연히 앞서 배운 Amazon ECS나 뒤에 배운 EKS와 완벽하게 찰떡궁합으로 통합됩니다.
* **공개 범위:** 우리 회사 사람만 쓰는 **프라이빗(Private)** 저장소와, 전 세계 누구나 다운받을 수 있는 **퍼블릭(Public)** 저장소(Amazon ECR Public Gallery)를 모두 지원합니다.
* **🔒 권한 에러 해결사 (IAM):** 앞 섹션에서 배운 내용과 이어집니다! ECS나 EKS가 ECR에서 이미지를 땡겨오지 못하고 권한 에러(`ImagePullBackOff` 등)를 뱉는다면? **100% IAM 정책(Policy) 문제**이므로 권한을 살펴봐야 합니다.
* **부가 기능들:** 이미지에 구멍이 없는지 확인하는 **취약점 스캐닝**, 똑같은 태그로 덮어쓰지 못하게 막는 **버전 관리 및 태그 설정**, 오래된 이미지는 자동으로 지워주는 **수명 주기(Lifecycle) 확인** 기능을 지원합니다.

# 4. Amazon EKS (클라우드에 구애받지 않는 오픈소스 대장)

목적은 ECS와 똑같습니다. 컨테이너를 대규모로 자동 배포, 확장, 관리(오케스트레이션)하는 서비스입니다. 하지만 뼈대가 다릅니다. ECS는 AWS 전용 기술이고, **EKS는 전 세계 표준 오픈소스인 'Kubernetes(쿠버네티스)'를 AWS 환경에서 알아서 관리해 주는 서비스**입니다.

* **최고의 장점 (Cloud-agnostic):** 쿠버네티스는 특정 클라우드에 종속되지 않습니다. 즉, EKS에서 돌리던 컨테이너 설정 그대로 Azure나 GCP, 온프레미스 서버실로 **언제든 쉽게 마이그레이션**할 수 있습니다.
* **설계 특징:** 여러 리전에 걸쳐 인프라를 짤 때, **리전 당 하나의 EKS 클러스터를 배포**하는 것이 기본 원칙입니다.
* **모니터링:** 시스템이 잘 돌아가는지 로그와 메트릭을 수집할 때는 **CloudWatch Container Insights**를 사용합니다.

### ✅ Amazon EKS – Node Types (컨테이너가 들어갈 컴퓨터 유형)

쿠버네티스에서 컨테이너가 실제로 실행되는 가상 서버를 '노드(Node)'라고 부릅니다. 노드를 다루는 방식은 3가지가 있습니다.

1. **관리형 노드 그룹 (Managed Node Groups):**
* **특징:** AWS가 노드(EC2 인스턴스)의 생성, 업데이트, 종료를 알아서 관리해 줍니다.
* **메커니즘:** 쿠버네티스가 다루기 좋게 EKS 전용 오토 스케일링 그룹(ASG)으로 묶여서 작동합니다. 비용 절감을 위해 온디맨드와 스팟 인스턴스를 모두 지원합니다. (**가장 추천되는 기본 방식**)


2. **자체 관리형 노드 (Self-Managed Nodes):**
* **특징:** 내가 직접 EC2 인스턴스를 만들어서 EKS 클러스터에 수동으로 등록하고 ASG로 관리하는 방식입니다.
* **메커니즘:** 내 입맛대로 OS나 설정을 커스텀해야 할 때 쓰며, EKS에 최적화된 미리 빌드된 AMI 등을 활용해 구성합니다. 당연히 온디맨드/스팟 인스턴스 설정이 가능합니다.


3. **AWS Fargate:**
* **특징:** 서버리스 모드입니다. 가상 서버(노드)라는 개념 자체를 내가 관리할 필요가 전혀 없으며, 패치나 유지 보수도 제로입니다.



### 🗂️ Data Volumes (EKS의 영구 저장 주머니)

EKS 컨테이너에 외장 하드(볼륨)를 연결할 때는 표준 규격인 **CSI (Container Storage Interface) 호환 드라이버**를 사용해 스토리지 클래스 매니페스트(설정 파일)를 지정해야 합니다. 다음 저장소들을 꽂을 수 있습니다.

* **Amazon EBS:** 단일 노드에 빠르게 붙는 고성능 블록 스토리지
* **Amazon EFS:** **Fargate 모드에서 필수적으로 쓰이며**, 여러 노드가 동시에 공유할 수 있는 다중 AZ 파일 시스템
* **Amazon FSx 시리즈 (Lustre / NetApp ONTAP):** 초고성능 연산이나 특정 엔터프라이즈 파일 시스템이 필요할 때 연동

# 5. AWS App Runner (인프라 몰라도 되는 고속 도로)

ECS나 EKS는 너무 훌륭하지만, 개발자 입장에서는 클러스터니, 노드니, 태스크 정의니 공부할 게 너무 많고 세팅이 번거롭습니다. **"난 인프라 경험 없고 알고 싶지도 않아! 그냥 내가 짠 웹 앱 소스코드나 Docker 이미지 줄 테니까 알아서 인터넷에 띄워줘!"** 할 때 쓰는 완전 관리형 서비스입니다.

* **작동 방식:** 내 소스코드(GitHub 등)나 ECR에 있는 Docker 이미지를 App Runner에 툭 던져줍니다. 그러면 지가 알아서 코드를 빌드하고, 로드 밸런서 달고, 암호화 적용하고, 고가용성 세팅을 마친 뒤 **인터넷 URL을 뿅 하고 뽑아줍니다.**
* **인프라 제로:** 오토 스케일링, 로드 밸런싱을 뒤에서 100% 자동으로 처리하므로 서버 설정을 아예 안 건드립니다.
* **연결성:** 인프라를 숨겨주지만 엄연히 **VPC 액세스를 지원**하기 때문에, 내 사설 네트워크 안에 숨겨둔 RDS 데이터베이스, ElastiCache, SQS 대기열과 안전하게 통신할 수 있습니다.
* **사용 사례:** 빠른 시간 안에 프로덕션에 배포해야 하는 웹 애플리케이션, REST API, 마이크로서비스에 최적입니다.

1. **"이미 온프레미스나 타사 클라우드(GCP, Azure)에서 쿠버네티스를 쓰고 있고, 이 아키텍처 그대로 AWS로 이사 오고 싶다"** ➔ **Amazon EKS**
2. **"EKS 환경인데 가상 서버 노드를 패치하거나 관리하는 번거로움 없이 서버리스로 컨테이너를 띄우고 싶다"** ➔ **EKS + AWS Fargate 조합**
3. **"EKS 컨테이너들이 영구적으로 데이터를 공유해야 하고, 특히 Fargate 모드를 쓰고 있다"** ➔ **CSI 드라이버를 통해 Amazon EFS 연동**
4. **"인프라 지식이나 설정 경험이 전혀 없는 개발 팀이, 코드나 Docker 이미지 하나만 가지고 오토 스케일링과 로드 밸런싱이 지원되는 웹 대고객 서비스를 광속으로 배포하고 싶다"** ➔ **AWS App Runner**
5. **"ECS/EKS가 이미지를 땡겨오지 못해 컨테이너가 안 뜬다"** ➔ **IAM의 ECR 접근 권한(Policy) 확인**
