# 🔥 Container 마스터 치트키
## Docker, ECS, Fargate, ECR, EKS, App Runner

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **가상 서버 관리 없이 컨테이너만 독립 배포 + 단순 수평 확장** $\rightarrow$ `ECS Fargate Launch Type`
* **컨테이너 내부 앱 코드가 S3, DynamoDB 등 다른 AWS API를 호출** $\rightarrow$ `ECS Task Role 설정`
* **ECS 에이전트가 ECR 이미지를 못 가져오거나 CloudWatch로 로그를 못 보냄** $\rightarrow$ `EC2 Instance Profile 권한 확인`
* **여러 AZ의 ECS 컨테이너들이 파일 데이터를 영구적으로 공유** $\rightarrow$ `Amazon EFS 마운트 (S3 마운트는 오답)`
* **완전 서버리스 컨테이너 + 영구 파일 공유** $\rightarrow$ `Fargate + EFS 조합`
* **컨테이너 스케일링 추적 지표** $\rightarrow$ `CPU 사용률 / 메모리 사용률 / ALB 타깃당 요청 수`
* **SQS 대기열 길이에 맞춰 컨테이너 개수 자동 조절** $\rightarrow$ `SQS + CloudWatch + ECS Service Auto Scaling`
* **EC2 Launch Type에서 인프라 공간 부족 시 자동 서버 확장** $\rightarrow$ `ECS Cluster Capacity Provider → ASG 트리거`
* **S3 업로드 이벤트로 컨테이너 태스크 자동 실행** $\rightarrow$ `S3 → EventBridge → ECS Task 실행`
* **정해진 시간에만 배치 컨테이너 실행 (크론 방식)** $\rightarrow$ `EventBridge 스케줄 → ECS Task`
* **ECS/EKS가 이미지를 못 가져와 컨테이너가 안 뜸 (ImagePullBackOff)** $\rightarrow$ `IAM의 ECR 접근 권한(Policy) 확인`
* **온프레미스 or GCP/Azure의 쿠버네티스 그대로 AWS로 이전** $\rightarrow$ `Amazon EKS`
* **EKS 환경에서 노드 관리 없는 서버리스 컨테이너** $\rightarrow$ `EKS + AWS Fargate`
* **EKS Fargate 모드에서 컨테이너 영구 저장소** $\rightarrow$ `CSI 드라이버 + Amazon EFS`
* **인프라 지식 없는 팀이 코드/이미지만 주면 자동 배포 + LB + 오토스케일링** $\rightarrow$ `AWS App Runner`
* **거의 모든 컨테이너 앞 로드 밸런서 정답** $\rightarrow$ `ALB (동적 포트 매핑 지원)`
* **초당 수백만 요청 / 고정 IP / PrivateLink가 필요한 컨테이너 LB** $\rightarrow$ `NLB`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📌 1. Docker 기초

* **정의:** 앱과 실행 환경을 하나의 컨테이너로 패키지화하는 플랫폼. OS에 무관하게 어디서든 동일하게 실행됩니다.
* **이미지 저장 위치:** Docker Hub (퍼블릭) / Amazon ECR (프라이빗, 퍼블릭 갤러리도 제공).

#### Docker vs Virtual Machine

| 구분 | VM (가상 머신) | Docker 컨테이너 |
| --- | --- | --- |
| 격리 방식 | 하이퍼바이저 위에서 Guest OS 포함 완전 격리 | Docker Daemon 위에서 OS 공유 |
| 리소스 공유 | 공유 안 함 (독립 할당) | 공유 (더 많은 컨테이너 실행 가능) |
| 보안 | 더 안전 | 상대적으로 덜 안전 |
| 밀도 | 낮음 | 높음 (하나의 서버에 많은 컨테이너 구동 가능) |

#### AWS 컨테이너 관리 서비스 4종

| 서비스 | 역할 |
| --- | --- |
| Amazon ECS | AWS 전용 컨테이너 오케스트레이션 플랫폼 |
| Amazon EKS | 쿠버네티스(오픈소스) 관리형 서비스 |
| AWS Fargate | ECS/EKS 모두와 연동되는 서버리스 컨테이너 |
| Amazon ECR | Docker 이미지 저장소 |

### 📌 2. Amazon ECS - 두 가지 Launch Type

#### ① EC2 Launch Type (수동 인프라 관리형)

* 내가 직접 EC2 인스턴스를 생성하고 관리합니다.
* EC2 내부에 ECS 에이전트(Agent)가 실행되어 ECS 클러스터에 등록됩니다.
* 컨테이너 배치/라이프사이클은 AWS가 제어하지만 EC2 OS 패치와 모니터링은 사용자 몫입니다.

#### ② Fargate Launch Type (완전 관리형 서버리스)

* 가상 서버를 프로비저닝할 필요가 전혀 없습니다.
* 태스크 정의(Task Definitions)에서 CPU/RAM만 지정하면 AWS가 알아서 인프라를 깔고 태스크를 실행합니다.
* 확장 시 태스크 개수만 늘리면 되어 관리가 극도로 단순합니다.

---

### 📌 3. ECS IAM Role 두 가지 구분 (★시험 단골)

| Role | 사용 주체 | 주요 용도 |
| --- | --- | --- |
| EC2 Instance Profile | EC2 인스턴스(ECS 에이전트) | ECR 이미지 pull, CloudWatch 로그 전송, SSM/Secrets Manager 비밀번호 읽기 |
| ECS Task Role | 컨테이너 내부 앱 코드 | S3 파일 읽기, SQS 메시지 전송 등 비즈니스 로직 AWS API 호출 |

> **⚠️ 오답 함정:** Task Role은 태스크 정의(Task Definition) 단위로 부여합니다. Fargate는 EC2 인스턴스가 없으므로 EC2 Instance Profile이 필요 없습니다.

### 📌 4. 로드 밸런서 연동

| LB 유형 | 사용 시나리오 |
| --- | --- |
| ALB | 거의 모든 일반적인 상황의 정답. 동적 포트 매핑으로 동일 EC2 내 다수 컨테이너 분산 |
| NLB | 초당 수백만 요청의 극한 고성능 필요, 고정 IP, AWS PrivateLink 연동 |
| CLB | 구형 장비. Fargate 호환 안 됨. 절대 권장하지 않음 |

---

### 📌 5. 데이터 볼륨 연동

* **최고의 조합 = Amazon EFS:** EC2 방식과 Fargate 방식 모두 지원. Multi-AZ 공유 가능.
* **Fargate + EFS = 완전 서버리스 아키텍처:** 컴퓨팅도 서버리스, 저장도 서버리스.
* **⚠️ 오답 함정:** Amazon S3는 ECS 태스크에 파일 시스템(Volume) 형태로 직접 마운트할 수 없습니다. S3는 반드시 API/SDK로 통신해야 합니다.

### 📌 6. ECS Service Auto Scaling

#### 1단계: Task 레벨 스케일링 (컨테이너 개수 조절)

AWS Application Auto Scaling 메커니즘 기반으로 실제 태스크 수를 늘리고 줄입니다.

| 추적 지표 | 내용 |
| --- | --- |
| CPU 사용률 | 컨테이너 평균 CPU |
| 메모리 사용률(RAM) | 컨테이너 평균 RAM |
| ALB 타깃당 요청 수 | Request Count Per Target |

| 스케일링 정책 | 내용 |
| --- | --- |
| Target Tracking | 지표 목표값 유지 (예: 평균 CPU 60%) |
| Step Scaling | CloudWatch 경보 수치 단계에 따라 추가 수량 조정 |
| Scheduled Scaling | 블랙 프라이데이 등 예측 가능한 시간에 미리 확장 |

#### 2단계: 인프라 레벨 스케일링 (EC2 Launch Type 한정)

* **문제:** 태스크는 늘어나는데 띄울 EC2(서버) 공간이 부족할 수 있습니다.
* **해결:** ECS Cluster Capacity Provider가 CPU/RAM 부족을 감지하면 연동된 ASG를 트리거하여 EC2를 자동 프로비저닝합니다.
* **Fargate 장점:** 서버리스라 인프라 레벨 스케일링 고민이 없습니다. Task 레벨만 관리하면 됩니다.

### 📌 7. ECS 아키텍처 연동 3대 패턴

#### ① EventBridge 이벤트 트리거

S3에 파일이 업로드되는 등의 이벤트 → EventBridge가 캐치 → ECS 태스크를 다이렉트로 실행(Invoke). 일회성 작업 처리에 적합합니다.

#### ② EventBridge 크론 스케줄

"매일 밤 12시에 정산 컨테이너를 실행하라"처럼 크론 식으로 정해진 시간에만 ECS 태스크를 실행합니다. 배치 작업에 최적입니다.

#### ③ SQS 대기열 버퍼 방식

대규모 요청 → SQS 대기열 저장 → ECS 태스크가 SQS 폴링하며 처리 → CloudWatch가 `ApproximateNumberOfMessages`를 감시하다 임계치 초과 시 ECS Service Auto Scaling 트리거 → 태스크 수 자동 증가.

### 📌 8. Amazon ECR (Docker 이미지 금고)

| 항목 | 내용 |
| --- | --- |
| 저장 백엔드 | 내부적으로 Amazon S3에 저장 |
| 공개 범위 | 프라이빗 저장소 + 퍼블릭 저장소(ECR Public Gallery) 모두 지원 |
| IAM 권한 에러 | 이미지 pull 실패(ImagePullBackOff) = 100% IAM 정책 문제 |
| 부가 기능 | 취약점 스캐닝, 버전 관리/태그 설정, 수명 주기(Lifecycle) 자동 정리 |

### 📌 9. Amazon EKS (클라우드 비종속 쿠버네티스)

ECS와 목적은 동일(컨테이너 오케스트레이션)하지만, 오픈소스 쿠버네티스(Kubernetes)를 AWS에서 관리해 주는 서비스입니다.

* **최대 장점 (Cloud-agnostic):** Azure, GCP, 온프레미스 등 어떤 환경으로도 동일한 쿠버네티스 설정 그대로 마이그레이션 가능.
* **설계 특징:** 리전당 1개 EKS 클러스터 배포가 기본 원칙.
* **모니터링:** CloudWatch Container Insights 사용.

#### EKS Node Types (컨테이너가 실행되는 노드 유형)

| 노드 유형 | 설명 |
| --- | --- |
| Managed Node Groups | AWS가 EC2 생성/업데이트/종료를 관리. EKS 전용 ASG로 묶임. 온디맨드/스팟 지원. 가장 추천 |
| Self-Managed Nodes | 직접 EC2를 만들어 EKS 클러스터에 수동 등록. 커스텀 OS/설정 필요 시 사용 |
| Fargate | 서버리스. 노드 관리/패치 완전 불필요 |

#### EKS Data Volumes (영구 저장소)

CSI(Container Storage Interface) 호환 드라이버 + 스토리지 클래스 매니페스트 지정이 필수입니다.

| 스토리지 | 특징 |
| --- | --- |
| Amazon EBS | 단일 노드 고성능 블록 스토리지 |
| Amazon EFS | Fargate 모드에서 필수. 다중 노드 동시 공유, Multi-AZ |
| Amazon FSx for Lustre | 초고성능 HPC 연산용 |
| Amazon FSx for NetApp ONTAP | 엔터프라이즈 파일 시스템 필요 시 |

> **⚠️ 핵심:** EKS + Fargate 모드에서 영구 저장소가 필요하면 반드시 EFS + CSI 드라이버 조합입니다.

### 📌 10. AWS App Runner (인프라 몰라도 되는 고속 배포 도로)

"인프라 경험 없고 알고 싶지도 않다. 소스코드나 Docker 이미지만 줄 테니 알아서 인터넷에 띄워달라"는 요구에 최적화된 완전 관리형 서비스입니다.

* **작동 방식:** 소스코드(GitHub 등) 또는 ECR의 Docker 이미지를 전달 → App Runner가 자동 빌드 + 로드 밸런서 + 암호화 + 오토 스케일링 + URL 제공.
* **인프라 제로:** 오토 스케일링과 로드 밸런싱을 100% 자동으로 처리합니다.
* **연결성:** VPC 액세스를 지원하여 프라이빗 네트워크 내 RDS, ElastiCache, SQS와 안전하게 통신 가능.
* **사용 사례:** 빠른 시간 안에 프로덕션에 배포해야 하는 웹 앱, REST API, 마이크로서비스.
