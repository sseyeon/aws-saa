# 🛡️ AWS 재해 복구(DR) 및 엔터프라이즈 마이그레이션 마스터 치트키

## 🎯 1단계: 한 눈에 보는 DR/마이그레이션 핵심 키워드 매칭

시험 문제에 아래 핵심 단어가 나오면 고민하지 말고 바로 해당 서비스를 고르세요!

* **비용 최저 + 가동 인프라 없음 + RPO/RTO 수 시간~수 일 단위 최고치** $\rightarrow$ `Backup and Restore (백업 및 복구)`
* **핵심 DB만 실시간 복제 + 웹/앱 서버는 재해 시 CloudFormation 등으로 즉시 자동 프로비저닝 (최소화된 핵심 코어 상시 가동)** $\rightarrow$ `Pilot Light (파일럿 라이트)`
* **전체 시스템을 최소 스케일(ASG 최소 용량)로 상시 가동 + 재해 시 Route 53 전환 후 즉시 Auto Scaling 확장** $\rightarrow$ `Warm Standby (웜 대기)`
* **두 개 이상의 리전에 동일 스케일의 인프라 Active-Active 가동 + RTO/RPO 몇 초 단위 최저치 + 비용 최고** $\rightarrow$ `Multi-Site / Hot Site (다중 사이트)`
* **실시간 원본 DB 가동 유지 + 이기종/동종 DB 간 가동 중단 없는 마이그레이션 + EC2 복제 인스턴스 필수 생성** $\rightarrow$ `DMS (Database Migration Service)`
* **이기종 DB 간 마이그레이션 + 테이블 구조 및 스키마/저장 프로시저 자동 변환** $\rightarrow$ `SCT (Schema Conversion Tool)`
* **온프레미스 인프라 리호스팅(Lift-and-Shift) + 에이전트 기반 지속적 블록 레벨 복제 + 다운타임 최소화** $\rightarrow$ `MGN (Application Migration Service)`
* **온프레미스 서버 사전 조사 + VM 환경 소프트웨어 종속성 및 리소스 사용량 맵핑** $\rightarrow$ `Application Discovery Service`
* **중앙 집중형 백업 관리 + 크로스 계정/리전 백업 일원화 + PITR(지정 시간 복구) 지원** $\rightarrow$ `AWS Backup`
* **랜섬웨어 방어 + 루트 사용자도 백업 삭제 불가 + WORM(Write Once Read Many) 강제화** $\rightarrow$ `AWS Backup Vault Lock`
* **기존 온프레미스 vSphere 툴셋 및 인프라 그대로 유지 + AWS 베어메탈 인프라 확장 연동** $\rightarrow$ `VMware Cloud on AWS`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### ⏱️ 1. 재해 복구(DR) 핵심 지표 및 4대 전략

재해 복구의 모든 의사결정은 비즈니스 연속성을 담보하는 두 가지 핵심 시간 지표를 기준으로 트레이드오프(비용 vs 속도) 관계를 가집니다.

```
[ 과거 데이터 손실 범위 ] <--- (RPO) --- [ 재해 발생 시점 ] --- (RTO) ---> [ 서비스 복구 완료 시점 ]
                                        └─────── 다운타임(Downtime) ───────┘

```

* **RPO (Recovery Point Objective, 복구 시점 목표):** 재해 발생 시 "얼마만큼의 데이터 손실을 감수할 수 있는가"의 기준입니다. 백업 주기(매일, 매시간 등)를 결정하며, RPO가 낮을수록 실시간 복제 기술이 필요합니다.
* **RTO (Recovery Time Objective, 복구 시간 목표):** 재해 발생 순간부터 "얼마나 빨리 서비스를 정상화(애플리케이션 가동)할 수 있는가"의 기준입니다. 즉, 서비스의 총 다운타임 한계치를 의미합니다.

#### 📊 DR 4대 전략 비교 매트릭스

| 전략명 | 가동 인프라 상태 (상시) | RPO / RTO 수준 | 비용 | 재해 발생 시 대응 메커니즘 |
| --- | --- | --- | --- | --- |
| **Backup & Restore** (백업 및 복구) | 없음 (스냅샷/백업본만 S3/Glacier 적재) | **최고치** (수 시간 ~ 수 일) | **최저** | AMI로 EC2 빌드, RDS 스냅샷 기반 신규 DB 생성 등 **전체 인프라 새로 재생산** |
| **Pilot Light** (파일럿 라이트) | **핵심 코어(DB)만 상시 가동/복제**, 웹·앱 서버는 정지 상태 | **높음~중간** | **저렴** | Route 53 장애 조치 야기 $\rightarrow$ 복제 중이던 RDS 정상 승격, EC2 인스턴스 즉시 대량 생성 |
| **Warm Standby** (웜 대기) | **전체 인프라가 최소 규모로 항시 구동** (ASG 최소화, ELB 대기) | **낮음~중간** | **중간** | Route 53 트래픽 우회 $\rightarrow$ ASG 가동 정책에 의해 **프로덕션 로드에 맞게 서버 즉시 확장(Scale-out)** |
| **Multi-Site / Hot Site** (다중 사이트) | **동일 스케일의 프로덕션 환경 다중화** (Active-Active 가동) | **최저치** (몇 초 ~ 몇 분) | **최고** | Route 53 가중치/지리적 라우팅 등으로 상시 분산. 한쪽 붕괴 시 **지연 시간 없이 즉각 수용** |

#### 💡 All-AWS 다중 리전 및 DR 구축 아키텍처 팁

* 온프레미스를 배제하고 100% AWS 클라우드 내에서 멀티 리전 DR을 구성하는 경우, **Aurora Global Database**를 채택하면 기본 리전의 데이터가 타 리전의 Slave(읽기 전용 복제본)로 초 단위 미만 밀리세컨드 단위로 실시간 복제되므로 Multi-Site 급 RPO를 달성할 수 있습니다.
* **백업 핵심:** EBS 스냅샷 및 RDS 자동 백업을 상시 수행하고, S3 교차 리전 복제(CRR)와 수명 주기 정책(Lifecycle Policy)을 연동하여 백업본을 S3 IA 또는 Glacier로 정기 이관합니다. 온프레미스 연동 시에는 Snowball이나 Storage Gateway를 데이터 전송 통로로 잡습니다.
* **고가용성 & 복제:** Route 53 장애 조치(Failover) 라우팅을 사용해 기본 리전 장애 시 DR 리전으로 DNS를 자동 마이그레이션합니다. 다중 AZ 인프라(RDS, ElastiCache Multi-AZ) 및 EFS, S3의 자체 가용성을 기본으로 깔고 갑니다. Direct Connect 장애에 대비해 백업용 Site-to-Site VPN을 교차 구성합니다.
* **자동화 및 카오스 엔지니어링:** CloudFormation이나 Elastic Beanstalk 템플릿을 사전 정의하여 재해 선언 즉시 전체 인프라 환경을 완전 자동 코드로 재생성합니다. 평소 인프라 복원력을 검증하기 위해 Netflix의 "simian-army(Chaos Monkey)"처럼 가동 중인 가상 서버를 무작위로 강제 종료시키는 카오스 테스트 환경을 구축하기도 합니다.

### 🔄 2. 데이터베이스 마이그레이션 기술 (DMS & SCT)

#### 🔄 AWS DMS (Database Migration Service)

* 원본(Source) 데이터베이스를 서비스 중단 없이(Live 상태 유지) 안전하고 빠르게 AWS 클라우드로 이관하는 동기화 서비스입니다.
* **복제 엔진:** DMS 구동을 위해서는 백엔드에서 복제 작업을 대행하고 데이터를 중계할 **EC2 기반의 복제 인스턴스(Replication Instance)를 무조건 별도 생성/할당**해야 합니다.
* **CDC (Continuous Data Replication, 지속적 데이터 복제):** 최초 1회 전체 데이터를 마이그레이션한 이후에도 원본 DB의 변경 사항(트랜잭션 로그)을 실시간으로 캐치하여 대상 DB에 계속 밀어 넣어주는 연속 복제 기능을 지원합니다. 마이그레이션 중에도 소스 DB는 완벽하게 Active 상태를 유지합니다.

#### 🛠️ AWS SCT (Schema Conversion Tool)

* **이기종 간(Heterogeneous) 마이그레이션 필수품:** 소스 DB와 대상 DB의 엔진 종류가 다를 때(예: 상용 오라클 $\rightarrow$ 오픈소스 기반 Aurora PostgreSQL / SQL Server $\rightarrow$ Amazon Redshift 등), 데이터 테이블 구조, 스키마, 뷰, 저장 프로시저(Stored Procedure)를 대상 데이터베이스 엔진용 언어로 자동 변환해 주는 전용 도구입니다.
* **💡 시험용 매칭 공식:**
* **동종 마이그레이션 (Oracle $\rightarrow$ Oracle / On-premise Postgres $\rightarrow$ RDS Postgres):** 엔진 구조가 완벽히 같으므로 플랫폼이 클라우드로 바뀔 뿐입니다. 따라서 **SCT가 전혀 필요 없으며** DMS만 단독 사용합니다.
* **이기종 마이그레이션 (SQL Server $\rightarrow$ Aurora / Teradata $\rightarrow$ Redshift):** 무조건 **[SCT로 스키마 변환] $\rightarrow$ [DMS로 실시간 데이터 복제]** 두 단계 연동 프로세스를 골라야 정답입니다.



#### 🗄️ Source 및 Target 연동 가능 리스트 (노트 내용 100% 반영)

DMS는 이기종 플랫폼을 가리지 않고 광범위한 소스와 타겟을 매핑합니다.

```
[ SOURCE ]                                     [ TARGET ]
- 온프레미스/EC2 DB (Oracle, MS SQL,           - 온프레미스/EC2 DB (위 엔진들과 동일)
  MySQL, MariaDB, PostgreSQL, MongoDB,        - Amazon RDS & Aurora 전체
  SAP, DB2 등)                                 - Amazon Redshift / DynamoDB / S3
- Azure SQL Database                           - OpenSearch Service
- Amazon RDS & Aurora 전체                     - Kinesis Data Streams / Apache Kafka
- Amazon S3                                    - DocumentDB / Amazon Neptune
- DocumentDB                                   - Redis / Babelfish

```

### 🐬 3. RDS & Aurora 전용 마이그레이션 상세 옵션

#### 🐬 1. RDS & Aurora MySQL 마이그레이션

* **RDS MySQL $\rightarrow$ Aurora MySQL (AWS 내부 이관)**
* **옵션 1 (스냅샷 기반):** 원본 RDS MySQL의 DB 스냅샷을 수동 생성한 뒤, 이를 소스로 삼아 Aurora MySQL DB 클러스터로 직접 복원합니다. 스냅샷 촬영 및 복원 시간 동안 소스 DB 쓰기를 정지해야 하므로 **다운타임(Downtime)이 무조건 발생**합니다.
* **옵션 2 (읽기 전용 복제본 승격):** 원본 RDS MySQL 아래에 'Aurora 읽기 전용 복제본(Read Replica)'을 바로 생성하여 동기화를 돌립니다. 이후 복제 지연(Replication Lag) 지표가 '0'으로 수렴하는 시점에 복제본 관계를 끊고 **독립적인 DB 클러스터로 승격**시킵니다. 인프라 비용과 승격 대기 시간이 길어지지만 다운타임을 최소화할 수 있습니다.


* **외부 독립 MySQL $\rightarrow$ Aurora MySQL (외부망에서 클라우드로 이관)**
* **옵션 1 (S3 백업 복원법):** 대용량 DB에 유리한 기법으로, 오픈소스 툴인 **Percona XtraBackup**을 사용하여 온프레미스 MySQL의 물리 백업 파일을 통째로 추출하여 Amazon S3 버킷에 먼저 업로드합니다. 이후 해당 S3 경로를 소스로 지정하여 Aurora MySQL DB 클러스터를 다이렉트로 복원 생성합니다.
* **옵션 2 (mysqldump 방식):** 소형 DB에 유용하며, `mysqldump` 명령어로 데이터 스크립트를 추출해 수동 마이그레이션합니다. 데이터 파일이 클 경우 S3 백업 방식보다 속도가 현저히 느립니다.
* **실시간 상시 가동 필수 요건:** 위 백업 복원 과정 중에도 원본 외부 MySQL이 중단 없이 돌아가야 한다면 무조건 **DMS**를 붙여 실시간 복제 작업을 동반해야 합니다.



#### 🐘 2. RDS & Aurora PostgreSQL 마이그레이션

* **RDS PostgreSQL $\rightarrow$ Aurora PostgreSQL (AWS 내부 이관)**
* MySQL과 메커니즘이 완벽히 동일합니다. **[옵션 1: DB 스냅샷 생성 후 Aurora 복원 (다운타임 발생)]** 또는 **[옵션 2: Aurora 읽기 전용 복제본 생성 후 복제 지연 0 확인 및 독립 클러스터 승격]** 중 아키텍처 요구사항에 맞춰 선택합니다.


* **외부 독립 PostgreSQL $\rightarrow$ Aurora PostgreSQL (외부망에서 클라우드로 이관)**
* 온프레미스 PostgreSQL 백업 파일을 정형 생성하여 Amazon S3 버킷에 우선 적재합니다.
* 이후 Aurora PostgreSQL 타겟 DB 클러스터를 먼저 생성한 뒤, 내부 전용 확장 기능인 **`aws_s3` 플러그인 확장(Extension) 명령**을 호출하여 S3에 저장된 백업 데이터 셋을 백엔드로 가질러 와서 일괄 로드(Import)합니다.
* 원본 외부 PostgreSQL이 라이브 상태로 계속 가동되어야 하는 요구조건이라면 예외 없이 **DMS**를 구성하여 마이그레이션합니다.

### 🏢 4. 온프레미스 하이브리드 이관 솔루션 어레이

* **Amazon Linux 2 가상 머신(VM) 다운로드:** AWS 온프레미스 연동 전략의 일환으로, Amazon Linux 2 가상 머신 이미지를 가상 디스크 포맷(`.iso` 등)으로 다운로드하여 로컬 VMWare, KVM, VirtualBox, MS Hyper-V 하이퍼바이저 환경에 직접 로드해 로컬 테스트 환경을 구축할 수 있습니다.
* **EC2 VM 가져오기/내보내기 (VM Import/Export):** 온프레미스에서 기존에 운영 중이던 가상 머신 이미지 자체를 통째로 AWS EC2 인스턴스(AMI)로 마이그레이션하거나, 반대로 AWS EC2 인스턴스를 로컬 온프레미스 가상 환경용 파일로 내보낼 수 있는 양방향 이관 기술입니다. 온프레미스 가상 머신의 DR 리포지토리 보관 전략으로 활용됩니다.
* **AWS Server Migration Service (SMS):** 온프레미스에서 구동 중인 라이브 가상 머신(VM)들을 AWS EC2의 AMI로 단계적/증분 복제해 주는 전통 마이그레이션 서비스입니다. (현재는 후술할 MGN으로 통합 대체되는 추세입니다.)
* **VMware Cloud on AWS:** 온프레미스 데이터 센터 전체를 VMWare vSphere 기반으로 운영하던 엔터프라이즈 기업이, 관리 도구·명령어·스킬셋을 하나도 바꾸지 않고 AWS의 유연한 베어메탈 클라우드 인프라 하드웨어를 확장 용량(컴퓨팅/스토리지) 및 하이브리드 재해 복구(DR) 사이트로 그대로 상속받아 연동할 수 있게 해주는 강력한 파트너십 솔루션입니다.

### 📋 5. 엔터프라이즈 백업 총괄 거거넌스: AWS Backup

개별 서비스별로 파편화되어 수동 관리되던 스냅샷 및 백업 스크립트 업무를 단 하나의 대시보드에서 전사 정책 기반으로 통합 제어·자동화하는 완전 관리형 백업 중앙 통제 플랫폼입니다.

* **지원 통합 대상 리소스 풀 (노트 내용 100% 매칭):**
* `Amazon EC2 / Amazon EBS`
* `Amazon S3` (객체 스토리지 포함)
* `Amazon RDS 전체 엔진 / Amazon Aurora / Amazon DynamoDB`
* `Amazon DocumentDB / Amazon Neptune`
* `Amazon EFS / Amazon FSx (Lustre & Windows File Server)`
* `AWS Storage Gateway (Volume Gateway 백업 연동)`


* **핵심 거버넌스 기능:** 단 하나의 백업 플랜(Backup Plan) 정책을 선언하면 **리전 간 백업본 교차 전송(Cross-Region)** 및 조직 내 계정 간 백업본 복제(Cross-Account)를 자동 수행합니다. 지원 리소스에 한해 특정 시점으로 완벽히 되돌리는 **지정 시간 복구(PITR)** 기술을 명시적으로 지원하며, 정기 예약은 물론 즉시 실행하는 온디맨드 백업을 지원합니다. 자원에 태그(`Tag`)를 붙여두면 해당 태그 기반으로 백업 정책이 자동 타겟팅되도록 자동화 규칙을 짤 수 있습니다.
* **백업 정책 세부 라이프사이클 윈도우 구성 요소:**
* **백업 빈도(Frequency):** 12시간 주기, 매일, 매주, 매월 단위 정기 스케줄러 세팅 및 리눅스 표준 `cron` 표현식 스케줄링 완벽 지원
* **백업 윈도우(Backup Window):** 백업이 시작되고 돌아가야 하는 특정 타임존 시간 슬롯 한정 설정
* **Cold Storage 이관 주기:** 비용 절감을 위해 생성된 백업본을 언제 콜드 스토리지(Glacier)로 강제 이관할 것인지 일단위, 주단위, 월단위, 연단위로 일괄 지정 (절대 이전하지 않음 옵션 선택 가능)
* **보존 기간(Retention):** 백업 데이터를 며칠, 몇 주, 몇 달, 몇 년간 살려두고 자동 폐기할 것인지 설정 (항상 보존 옵션 선택 가능)



#### 🔒 AWS Backup Vault Lock (볼트 락 - 보안 절대 방어선)

* **WORM (Write Once Read Many) 규정 강제:** 백업 저장소인 Vault 자체를 통째로 잠가버려, 한 번 기록된 백업 파일은 절대 변조·수정·삭제할 수 없도록 물리적 가드레일을 치는 보안 기능입니다.
* **악성 행위 원천 차단:** 랜섬웨어 감염이나 내부 탈옥 사용자의 악의적인 백업본 강제 삭제 시도를 원천 차단합니다. **볼트 락이 발동되면, 최고 절대 권한자인 루트 사용자(Root User)라 할지라도 설정된 보존 기간 만료 전에는 백업 데이터를 절대로 임의 삭제하거나 보존 기간을 임의로 단축 변경할 수 없습니다.**

### 🛸 6. 현대적 대규모 마이그레이션 전용 서비스 (Discovery & MGN)

#### 🔍 AWS Application Discovery Service

온프레미스 인프라를 클라우드로 이관하기 전, 도대체 로컬 데이터 센터에 서버가 몇 대가 있고 사양은 어떻고, 프로그램끼리 어떻게 얽혀있는지 사전에 실시간 정밀 자동 스캔을 떠서 마이그레이션 계획 수립을 도와주는 사전 조사 도구입니다. 데이터 분석 결과는 **AWS Migration Hub** 대시보드로 전송되어 한눈에 매핑 상태를 추적 관리할 수 있습니다.

* **1. 에이전트리스 방식 (Agentless Discovery Collector):** 가상화 환경 인벤토리를 가볍게 스캔하는 방식입니다. 가상 머신 서버에 내부 프로그램을 깔 필요 없이 하이퍼바이저 단에서 VM 구성 정보 및 CPU, 메모리, 디스크 사용량 등 **성능 히스토리 요약본**만 빠르게 수집합니다.
* **2. 에이전트 기반 방식 (Agent-based Discovery Agent):** 스캔 대상 온프레미스 가상/물리 서버 OS 내부에 직접 조사 프로그램(Agent)을 설치 구동하는 정밀 방식입니다. 시스템 내부 구성 정보와 세부 성능은 물론, 현재 가동 중인 프로세스 명세와 **서버와 서버 간에 패킷을 주고받는 세부 네트워크 연결 종속성(Dependency) 매핑 정보**까지 한땀한땀 완벽하게 긁어 모아 옵니다.

#### 🛸 AWS MGN (Application Migration Service)

* **현세대 표준 Lift-and-Shift 탑티어 솔루션:** 과거의 Server Migration Service(SMS) 및 CloudEndure Migration 기술을 완벽하게 계승·대체하는 최신 마이그레이션 전용 서비스입니다. 온프레미스의 가상, 물리, 타 클라우드 환경 서버를 아키텍처 변경 없이 그대로 AWS EC2로 1:1 리호스팅(Lift-and-shift)하여 네이티브 구동되도록 원클릭 변환해 줍니다.
* **작동 메커니즘:** 소스 서버의 OS 단에서 지속적인 블록 레벨 복제(Continuous Block-Level Replication)를 수행하므로, 마이그레이션이 진행되는 동안에도 실제 서비스 구동 마이그레이션 다운타임과 비용을 제로에 가깝게 최소화하여 깔끔하게 클라우드로 서버를 이주 시켜 줍니다. 다양한 플랫폼, OS, DB 종류를 가리지 않고 광범위하게 지원합니다.

### 🚚 7. 대용량 데이터 클라우드 전송 비용 및 시간 계산 전략

> **시나리오 예시:** 총 용량 **200 TB**의 거대 데이터 셋을 온프레미스에서 AWS 클라우드로 전송해야 합니다. 현재 우리 회사의 인터넷 전용선 대역폭 한계치는 **100 Mbps**입니다. 리전 간 지속 복제 요구사항 시에는 Site-to-Site VPN이나 DX 선로 위에 DMS 또는 **AWS DataSync** 툴을 얹어 자동 연속 전송 파이프라인을 가동해야 합니다. 그렇다면 1회성 마이그레이션 시 최적의 수단은 무엇일까요?

#### ⏰ 1회성 대용량 전송 옵션별 전송 시간 산출 공식

* **옵션 1: 일반 공용 인터넷 또는 Site-to-Site VPN (`100 Mbps`) 활용**
* 네트워킹 장비 추가 설치가 필요 없어 비용이 안 들고 즉시 전송을 누를 수 있어 간편해 보입니다.
* **소요 시간 수학적 계산:**

$$200\text{ TB} \times 1,000\text{ GB/TB} \times 1,000\text{ MB/GB} \times 8\text{ Mb/MB} = 1,600,000,000\text{ Mb}$$


$$1,600,000,000\text{ Mb} / 100\text{ Mbps} = 16,000,000\text{초} \approx \mathbf{185일}\text{ (반년 이상 소요)}$$


* **결론:** 네트워크 대역폭 병목으로 인해 절대 정답이 될 수 없습니다.


* **옵션 2: Direct Connect 전용선 (`1 Gbps`) 신규 개통 활용**
* 회선 속도를 1Gbps로 10배 업그레이드하여 전용선 다이렉트 전송을 때립니다.
* **소요 시간 수학적 계산:**

$$200\text{ TB} \times 1,000\text{ GB/TB} \times 8\text{ Gb/GB} = 1,600,000\text{ Gb}$$


$$1,600,000\text{ Gb} / 1\text{ Gbps} = 1,600,000\text{초} \approx \mathbf{18.5일}$$


* **결론:** 순수 전송 시간은 18.5일로 줄었으나, **Direct Connect 선로 개통을 위한 물리적 공사 및 파트너 통신사 연동 등 초기 설치 행정 절차에만 최소 1달 이상**이 먼저 소요되므로 급한 마이그레이션 조건에는 탈락입니다.


* **옵션 3: AWS Snowball 하드웨어 가방 동원 (★비즈니스 최적 정답)**
* 물리적 대용량 스토리지 장비인 Snowball 기기 2~3대를 병렬로 동시에 신청하여 데이터 센터에 입고시킵니다. 로컬 내부망 로컬 LAN 전송(초고속)으로 데이터를 장비에 다 때려 넣은 뒤, DHL/우체국 택배로 AWS 데이터 센터에 발송하여 다이렉트 이관합니다.
* **소요 시간:** 장비 배송 및 로컬 전송, 클라우드 업로드 전체 프로세스를 합쳐 단 **일주일(1 Week)** 내에 200TB 이관이 완료됩니다. 대용량 이관 후 변경 분 동기화 시에는 DMS를 결합 가동하여 마무리를 짓습니다. 아키텍처 시험에서 수십~수백 TB급 1회성 급진 마이그레이션 문제가 나오면 무조건 Snowball이 탑티어 정답입니다.
