# 🔥 Serverless 마스터 치트키
## Lambda, Edge Functions, CloudFront Functions vs Lambda@Edge

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **서버 관리 없이 코드/함수만 배포, 호출 시에만 과금** $\rightarrow$ `AWS Lambda`
* **호출당 비용, 프리티어 100만 건/월, RAM 증가 시 CPU도 자동 향상** $\rightarrow$ `AWS Lambda 기본 특성`
* **15분(900초)을 넘기는 무거운 배치 작업** $\rightarrow$ `EC2 또는 ECS가 정답 (Lambda 오답)`
* **Lambda가 VPC 내부의 프라이빗 RDS, ElastiCache에 접근해야 함** $\rightarrow$ `Lambda VPC 배포 + 서브넷 + 보안그룹 + ENI 생성`
* **Lambda 동시 접속 폭주로 RDS 커넥션이 고갈/타임아웃 발생** $\rightarrow$ `Lambda VPC 배포 + RDS Proxy`
* **일반 Docker 이미지를 그대로 실행** $\rightarrow$ `ECS 또는 Fargate (Lambda 오답)`
* **1ms 미만의 극초고속 HTTP 헤더 조작, 캐시 키 정규화, URL 재작성** $\rightarrow$ `CloudFront Functions`
* **엣지에서 외부 라이브러리/파일 시스템 접근, 무거운 연산(수ms 이상)** $\rightarrow$ `Lambda@Edge`
* **Origin 요청/응답까지 엣지에서 수정 필요** $\rightarrow$ `Lambda@Edge (4개 구간 전부 가능)`
* **단순 JWT 토큰 검사, 쿠키/쿼리스트링 변환** $\rightarrow$ `CloudFront Functions`
* **타사 API 통합, AWS SDK 사용, 대규모 데이터 통합 (엣지에서)** $\rightarrow$ `Lambda@Edge`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📌 1. Serverless 개념 정의

* 서버가 없다는 뜻이 아닙니다. **서버가 보이지 않거나, 관리하지 않거나, 프로비저닝하지 않는 것**을 의미합니다.
* 개발자는 코드/함수 배포만 하면 됩니다.

**AWS 서버리스 서비스 목록:**

| 카테고리 | 서비스 |
| --- | --- |
| 컴퓨팅 | Lambda, Fargate |
| 데이터베이스 | DynamoDB, Aurora Serverless |
| 메시징 | SNS, SQS, Kinesis Data Firehose |
| API/통합 | API Gateway, Step Functions |
| 인증 | Cognito |
| 스토리지 | S3 |

---

### 📌 2. AWS Lambda 핵심 장점 5가지

* **관리 서버 없음:** 인프라 프로비저닝 불필요.
* **호출 시에만 과금:** 함수가 실행된 시간(ms 단위) + 호출 횟수만 청구.
* **프리 티어:** 매달 요청 **100만 건** + **40만 GB-초** 컴퓨팅 시간 무료 제공.
* **AWS 전체와 통합:** API Gateway, Kinesis, DynamoDB, S3, CloudFront, EventBridge, SNS, SQS, Cognito 등과 유기적으로 연동.
* **최대 10GB RAM:** 함수당 최소 128MB ~ 최대 10GB (1MB 단위 조정). **RAM을 늘리면 CPU와 네트워크 성능도 자동 비례 향상.**

### 📌 3. Lambda 지원 언어

| 유형 | 언어 |
| --- | --- |
| 기본 지원 런타임 | Node.js, Python, Java, C#(.NET Core), Go, PowerShell, Ruby |
| Custom Runtime API | Rust 등 커뮤니티 지원 언어 |
| 컨테이너 이미지 | Lambda Runtime API를 구현한 이미지만 실행 가능 |

> **⚠️ 오답 함정:** 임의의 일반 Docker 이미지 실행 → Lambda 불가. ECS 또는 Fargate 사용해야 합니다.

### 📌 5. Lambda 주요 통합 서비스

| 서비스 | 통합 방식 |
| --- | --- |
| API Gateway | HTTP 요청 수신 → Lambda 함수 호출 |
| Kinesis | 실시간 스트리밍 데이터를 Lambda가 실시간 변환/가공 |
| DynamoDB Streams | DynamoDB 데이터 변경 이벤트 → Lambda 트리거 |
| S3 | 객체 업로드 이벤트 → Lambda 트리거 (예: 썸네일 생성) |
| CloudFront (Lambda@Edge) | 엣지 로케이션에서 로직 실행 |
| EventBridge | 크론 배치 실행, 인프라 이벤트 대응 |
| SNS | 주제 메시지 도착 시 Lambda 트리거 |
| SQS | 대기열 메시지 폴링하며 비동기 처리 |
| Cognito | 회원가입/로그인 이벤트 시 커스텀 로직 실행 |

### 📌 6. Lambda VPC 연동 & RDS Proxy

#### Lambda + VPC (프라이빗 리소스 접근)

* 기본적으로 Lambda는 AWS 공용 네트워크에서 실행되어 프라이빗 서브넷 내의 RDS, ElastiCache 등에 접근 불가.
* 접근하려면 **Lambda에 VPC ID + 서브넷 + 보안그룹을 지정** → Lambda가 서브넷 안에 ENI(가상 랜카드)를 생성하여 통신.

#### Lambda + RDS Proxy (커넥션 폭발 방지)

* **문제:** Lambda가 무수히 동시 실행되어 RDS에 직접 연결하면 DB 커넥션이 고갈되고 타임아웃 발생.
* **해결:** Lambda와 RDS 사이에 **RDS Proxy**를 배치. RDS Proxy가 커넥션 풀링으로 다수의 Lambda 요청을 묶어 관리합니다.

| RDS Proxy 효과 | 내용 |
| --- | --- |
| 확장성 향상 | 커넥션 풀링으로 Lambda 동시성 폭증 대응 |
| 장애 조치 개선 | 장애 조치 시간 66% 감소 |
| 보안 강화 | Secrets Manager 연동으로 자격 증명 관리 |

> **⚠️ 핵심 제약:** RDS Proxy는 퍼블릭 액세스 불가 → Lambda도 반드시 VPC 내에 배포 필수.

### 📌 7. Edge Functions: CloudFront에서의 로직 실행

보통 함수는 특정 리전에 배포하지만, 최신 애플리케이션은 사용자와 가장 가까운 **엣지 로케이션**에서 로직이 실행되길 요구합니다. 100% 서버리스로 전 세계에 배포됩니다.

#### 엣지 함수 10대 유즈케이스

1. 웹사이트 보안 및 개인정보 보호
2. 엣지에서 동적 웹 애플리케이션 실행
3. SEO(검색 엔진 최적화) 처리
4. 다양한 오리진 간 지능적인 라우팅
5. 엣지 단에서 악성 봇 완화
6. 실시간 이미지 변환 및 리사이징
7. A/B 테스팅
8. 사용자 인증 및 권한 부여
9. 트래픽 몰릴 때 우선순위 지정
10. 사용자 행위 추적 및 분석

### 📌 8. CloudFront Functions vs Lambda@Edge 완벽 비교 (★시험 단골)

| 특징 | CloudFront Functions | Lambda@Edge |
| --- | --- | --- |
| 언어 | JavaScript | Node.js 또는 Python |
| 시작 지연 | 1ms 미만 | 수ms ~ |
| 초당 처리량 | 수백만 요청 | 수천 요청 |
| 수정 가능 구간 | Viewer 요청, Viewer 응답 (2개) | Viewer 요청, Origin 요청, Origin 응답, Viewer 응답 (4개) |
| 외부 네트워크 접속 | 불가 | 가능 |
| 파일 시스템 접근 | 불가 | 가능 |
| AWS SDK/라이브러리 | 불가 | 가능 |
| 배포 방식 | CloudFront 내에서 관리 | us-east-1에서 작성 후 전 세계 자동 복제 |

#### CloudFront Functions가 정답인 상황

* 캐시 키 정규화 (헤더/쿠키/쿼리스트링 변환으로 캐시 히트율 향상)
* HTTP 요청/응답 헤더 삽입, 수정, 삭제
* URL 재작성 및 리디렉션
* 단순 JWT 토큰 등 가벼운 인증 검사

#### Lambda@Edge가 정답인 상황

* 수ms 이상의 무거운 연산이나 높은 CPU/RAM이 필요한 처리
* AWS SDK 등 외부 라이브러리 사용
* 외부 API나 타사 서비스와 네트워크 통신이 필요한 데이터 통합
* Origin 요청/응답 단계에서의 로직 처리
