# 1. Serverless

- 서버리스 서비스를 사용하는 개발자는 **더이상 서버를 관리할 필요가 없다**. 서버가 없다는 뜻이 아닌, 관리할 필요가 없다는 것이다. 개발자는 단지 코드를 배포하기만, 함수를 배치하기만 하면 된다.
- 초기의 서버리스는 Faas (Function as a Service)를 의미했지만, 지금의 서버리스는 더 많은 것을 의미한다.
- AWS Lambda로부터 서버리스는 개발되었고, 이제는 데이터베이스, 메시징, 스토리지 등을 포함한 모든 관리형 서비스를 의미한다.
- **서버리스는 서버가 없다는 것을 의미하지 않고**, 서버가 보이지 않거나, 관리하지 않고, 프로비저닝하지 않는 것을 의미한다.

# 2. AWS Lambda
## 1. 🔎 AWS Lambda의 존재 이유와 5대 장점 (Benefits)

전통적인 **EC2 가상 서버**는 프로비저닝이 필요하고 RAM/CPU 크기가 고정되며 24시간 켜져 있어 비용이 계속 나갑니다. 반면 **AWS Lambda**는 관리할 서버가 없고 온디맨드로 호출 시에만 작동하며 자동으로 스케일링되는 가상 함수입니다.

* **쉬운 가격 정책 (호출 및 시간제 청구):** 함수가 실행된 시간(밀리초 단위)과 호출 횟수만큼만 돈을 냅니다. 특히 **매달 요청 1백만 건과 40만 GB-초의 컴퓨팅 시간**이 상시 무료 프리 티어로 제공되어 매우 저렴합니다.
* **AWS 전체 서비스와 완벽 통합:** API Gateway부터 S3, SQS까지 AWS 생태계 전체와 유기적으로 엮입니다.
* **다양한 프로그래밍 언어 지원:** 특정 언어에 종속되지 않고 폭넓은 선택지를 제공합니다.
* **CloudWatch를 통한 손쉬운 모니터링:** 실행 로그와 메트릭이 자동으로 기록되어 관리가 편합니다.
* **쉬운 추가 리소스 확보 (최대 10GB RAM):** 함수당 최소 128MB에서 최대 10GB까지 1MB 단위로 메모리를 할당할 수 있으며, **RAM을 증가시키면 CPU와 네트워크 성능도 비례해서 자동 향상**됩니다.

---

## 💻 AWS Lambda 언어 지원 (Language Support) 및 실행 규칙

* **기본 지원 런타임:** Node.js (JavaScript), Python, Java (Java 8 호환), C# (.NET Core), Golang, PowerShell, Ruby
* **Custom Runtime API:** 커뮤니티에서 지원하는 커스텀 API를 활용해 **Rust** 같은 언어도 실행할 수 있습니다.
* **컨테이너 이미지 지원:** Lambda 규격(Runtime API)을 구현한 가벼운 컨테이너 이미지를 실행할 수 있습니다. 단, **아무 제약이 없는 임의의 일반 Docker 이미지를 실행할 때는 ECS 또는 Fargate를 사용하는 것이 권장**됩니다.

---

## 🛠️ AWS Lambda 핵심 통합 서비스 (Integrations Main Ones)

현업에서 Lambda가 실무적으로 어떤 이벤트들과 엮여서 자동화를 수행하는지 보여주는 명단입니다.

* **API Gateway:** REST API를 생성하고 유저의 HTTP 요청을 받아 뒤에 있는 람다 함수를 호출합니다.
* **Kinesis:** 실시간으로 쏟아지는 스트리밍 데이터를 람다가 실시간으로 전달받아 즉시 변환하고 가공합니다.
* **DynamoDB:** NoSQL 데이터베이스에 새로운 데이터가 추가되거나 수정되는 트리거(Streams)가 생기면 람다가 깨어나 후속 작업을 합니다.
* **S3:** 사용자가 버킷에 파일을 업로드하는 등 이벤트가 발생하면 언제든 람다 함수를 작동시킵니다. (예: **실시간 썸네일 생성**)
* **CloudFront:** 전 세계 엣지 단에서 로직을 실행하는 **Lambda@Edge** 형태로 통합됩니다.
* **CloudWatch Events & EventBridge:** AWS 인프라 상태 변화에 대응하거나, 정해진 시간마다 자동화를 돌릴 때 씁니다. (예: **서버리스 크론탭 배치 작업**)
* **CloudWatch Logs:** 람다가 뱉은 로그를 수집하거나, 다른 곳으로 스트리밍하여 전달합니다.
* **SNS:** 푸시 알림이나 SNS 주제(Topic)에 메시지가 들어왔을 때 람다가 이를 받아 대처합니다.
* **SQS:** 비동기 결합의 정석으로, SQS 대기열에 쌓인 메시지들을 람다가 폴링하여 차례대로 처리합니다.
* **Cognito:** 사용자가 회원가입을 하거나 데이터베이스에 로그인할 때마다 람다가 응답하여 커스텀 로직을 수행합니다.

## 📍 AWS Lambda 스펙 제약 조건 (VPC 및 RDS Proxy 포함)

인프라 설계 시 타임아웃이나 용량 초과로 인한 장애를 막기 위해 반드시 외워야 하는 리전당 한도 수치입니다.

* **최대 실행 시간:** **900초 (15분)** ➔ 15분을 넘기는 무거운 배치는 EC2/ECS가 정답입니다.
* **환경 변수 용량:** 최대 **4 KB**까지 설정 가능합니다.
* **임시 디스크 용량 (`/tmp` 폴더):** 최소 512 MB에서 최대 **10 GB**까지 사용 가능하며, 시작 시 다른 대용량 파일을 로드하고 압축을 풀 때 유용합니다.
* **동시 실행 횟수:** 리전당 기본 **1,000회** (요청 시 증가 가능)
* **배포 파일 크기:** 압축된 `.zip` 형태일 때는 최대 **50 MB**, 압축을 풀었을 때 코드와 라이브러리 전체 총합은 **250 MB**를 넘을 수 없습니다.

### 🚧 Lambda 인프라 구성 패턴 (VPC & RDS Proxy)

* **VPC 내부 리소스 접근:** 기본적으로 람다는 AWS 공용 네트워크에서 돌기 때문에, 프라이빗 서브넷 안의 RDS나 ElastiCache에 접근하려면 **Lambda에 VPC ID와 서브넷을 지정하고 보안 그룹을 추가**해야 합니다. 이때 람다는 서브넷 안에 가상 랜카드인 **ENI**를 생성해 통신합니다.
* **RDS Proxy 활용:** 람다가 무수히 켜지며 RDS 데이터베이스에 다이렉트로 붙으면 DB 커넥션이 폭발해 타임아웃이 납니다. 이를 막기 위해 중간에 연결을 묶어 관리해 주는 **RDS Proxy**를 둡니다. (커넥션 풀링으로 확장성 향상, 장애 조치 시간 66% 감소, Secret Manager 연동으로 자격 증명 보안 강화). 단, RDS Proxy는 퍼블릭 액세스가 안 되므로 **Lambda 함수도 반드시 사용자의 VPC 내에 배포**되어야 합니다.

## 💡 Edge Function: CloudFront를 통한 엣지 단의 커스터마이징

보통은 함수를 특정 리전(서울 등)에 배포하지만, 모던 애플리케이션은 지연 시간을 최소화하기 위해 사용자와 가장 가까운 **엣지 로케이션**에서 로직이 실행되기를 요구합니다. 이를 엣지 함수(Edge Function)라고 하며, 서버 관리 없이 전역 배포되는 100% 서버리스 구조입니다.

### 🎯 엣지 함수의 10대 유즈케이스 (Use Cases)

1. **웹사이트 보안 및 개인정보 보호**
2. **엣지에서 동적 웹 애플리케이션 실행**
3. **검색 엔진 최적화 (SEO) 처리**
4. **다양한 오리진과 데이터 센터 간 지능적인 라우팅**
5. **엣지 단에서의 악성 봇 완화**
6. **실시간 이미지 변환 및 리사이징**
7. **A/B 테스팅 수행**
8. **사용자 인증 및 권한 부여**
9. **트래픽 몰릴 때 사용자 우선순위 지정**
10. **사용자 행위 추적 및 분석**

---

### 🥊 두 가지 엣지 함수 유형 완벽 비교 (CloudFront Functions vs Lambda@Edge)

AWS는 성능과 스펙에 따라 두 가지 선택지를 제공합니다.

| 특징 | ⚡ CloudFront Functions | 🧠 Lambda@Edge |
| --- | --- | --- |
| **언어 및 성능** | **JavaScript**로 작성된 초경량 함수<br>

<br>시작 시간 **1ms 미만**, 초당 수백만 요청 처리 | **Node.js** 또는 **Python**으로 작성<br>

<br>초당 수천 개 요청 처리 |
| **변경 가능한 영역** | **Viewer(뷰어) 요청** 및 **Viewer 응답**만 변경 가능<br>

<br>(클라이언트와 CDN 사이만 관여) | **Viewer 요청, Origin 요청, Origin 응답, Viewer 응답**<br>

<br>4개 구간 전부 수정 가능 |
| **네트워크 / 파일** | 외부 인터넷 접속 **불가**, 파일 시스템 접근 **불가** | 외부 서비스 네트워크 접속 **가능**, 파일 시스템 접근 **가능** |
| **스펙 특징** | CloudFront의 네이티브 기능으로 고성능/고확장성 요구 시 적합 | 조정 가능한 CPU/RAM, 3rd party 라이브러리(AWS SDK 등) 의존성 지원, HTTP 본문(Body) 접근 가능 |
| **배포 방식** | 모든 코드가 CloudFront 내에서 직접 관리됨 | 한 리전(`us-east-1`)에서 작성하면 전 세계 엣지로 자동 복제 |

### 🛠️ 세부 유즈케이스 매칭 가이드

* **CloudFront Functions가 정답인 상황:**
* **캐시 키 정규화:** 헤더, 쿠키, 쿼리 문자열, URL 속성을 변환하여 CDN 캐시 히트율을 최적화할 때
* **헤더 조작:** HTTP 요청/응답 헤더를 삽입, 수정, 삭제할 때
* **URL 재작성 및 리디렉션:** 주소를 바꾸거나 다른 페이지로 팅겨줄 때
* **단순 토큰 인증:** 사용자가 보낸 단순 JWT 토큰 등을 가볍게 검사하여 요청을 허용/거부할 때


* **Lambda@Edge가 정답인 상황:**
* **무거운 연산:** 수 밀리초 이상의 실행 시간이나 높은 CPU/RAM이 필요할 때
* **외부 라이브러리 필요:** AWS SDK 등을 써서 다른 AWS 서비스에 접근해야 할 때
* **데이터 통합:** 외부 API나 타사 서비스에 네트워크 통신을 날려 대규모 데이터 통합을 해야 할 때

### 🎯 최종 요약 정답 공식

1. **"호출당 비용, 프리티어 1백만 건, RAM에 비례하는 CPU 성능"** ➔ **AWS Lambda 기본 특성**
2. **"1ms 미만의 극도로 빠른 지연 시간으로 HTTP 헤더를 조작하거나 캐시 키 정규화"** ➔ **CloudFront Functions**
3. **"엣지에서 외부 타사 라이브러리를 쓰거나 파일 시스템에 접근하여 무거운 처리(이미지 변환 등) 수행"** ➔ **Lambda@Edge**
4. **"람다 함수를 내 내부 DB(RDS)와 안전하게 엮고 동시성 부하를 덜어주고 싶다"** ➔ **VPC 내에 Lambda 배포 + RDS Proxy 조합**
