# 💻 AWS EC2 코어 인프라 및 핵심 구매 옵션 마스터 치트키

## 🎯 1단계: 한 눈에 보는 EC2 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **인스턴스 첫 부팅 시 딱 한 번 실행 + 초기 설정/패키지 설치 자동화 + 루트 계정 구동** $\rightarrow$ `EC2 User Data (부트스트래핑)`
* **인바운드 기본 차단 / 아웃바운드 기본 허용 + 허용(Allow) 규칙만 정의 + 상태 저장(Stateful) 방화벽** $\rightarrow$ `Security Group (보안 그룹)`
* **웹 서버 접속 불가 시 "타임아웃(Timeout)" 발생** $\rightarrow$ `보안 그룹(Security Group) 규칙 미설정 (네트워크 차단)`
* **웹 서버 접속 불가 시 "Connection Refused" 에러 발생** $\rightarrow$ `보안 그룹은 통과했으나 웹 서버 애플리케이션 미구동/포트 미개방`
* **다운로드 받은 비밀키(개인키) 없이 + 브라우저 기반으로 + 22번 포트를 경유하여 즉시 Linux 인스턴스 원격 접속** $\rightarrow$ `EC2 Instance Connect`
* **장기 워크로드(1~3년 약정) + 인스턴스 패밀리/사이즈/OS/테넌시 관계없이 유연하게 최대 72% 할인 적용** $\rightarrow$ `EC2 Savings Plans`
* **On-Demand 대비 최대 90% 할인 + 언제든 중단 가능한 배치/분산 데이터 분석 워크로드 + 비용 최적화** $\rightarrow$ `Spot Instance (스팟 인스턴스)`
* **특정 AZ의 가상 서버 용량을 무조건 확보(할인 목적 X, 단기/중단 없는 필수 구동 자원 확보)** $\rightarrow$ `Capacity Reservation (용량 예약)`
* **법적 규제 및 컴플라이언스 만족 + 기존 보유 서버 라이선스(BYOL) 그대로 사용 + 물리적 서버 통째로 대여** $\rightarrow$ `Dedicated Host (전용 호스트)`
* **스팟 자원 회수 시 새로 생성되는 것을 막기 위한 완전 삭제 절차** $\rightarrow$ `1) Spot Request(스팟 요청) 먼저 취소 ➡️ 2) 구동 중인 Spot Instance 종료`
* **다양한 인스턴스 타입/AZ 풀 중에서 가격 및 가용성 전략(priceCapacityOptimized 등)에 따라 스팟 및 온디맨드 복합 자동 할당** $\rightarrow$ `Spot Fleet (스팟 플릿)`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 💻 1. Amazon EC2 & 인스턴스 유형(Type) 마스터

Amazon EC2는 클라우드에서 가상 머신을 대여하는 IaaS(Infrastructure as a Service)의 핵심이며 가상 서버, 스토리지(EBS/EFS), 부하 분산(ELB), 자동 확장(ASG)을 아우르는 근간 서비스입니다.

#### 📝 EC2 User Data (부트스트래핑)

* **정의:** 인스턴스가 생성되어 **최초로 부팅될 때 딱 1번만 자동 실행**되는 스크립트(셸 스크립트 등)입니다. 소프트웨어 설치, OS 업데이트, 인터넷 파일 다운로드 등 초기 환경 구축을 자동화합니다.
* **보안 가이드레일:** User Data는 **루트(root) 권한**으로 실행됩니다. 실행이 완료된 후에는 내부 코드를 수정할 수 없으며, AWS 콘솔 및 API 상에서 암호화되지 않은 텍스트(Plain Text)로 노출되므로 패스워드, API Secret Key 등 **민감한 정보를 절대 포함해서는 안 됩니다.**
* **주의점:** 스크립트에 수행할 작업을 많이 넣을수록 첫 부팅 완료(인스턴스 사용 가능 상태)까지 걸리는 시간이 길어집니다.

#### 🏷️ EC2 인스턴스 명명 규칙 (Naming Convention)

> 예: `m5.2xlarge`

* **`m` (Instance Class):** 인스턴스의 패밀리(클래스) 목적을 나타냅니다.
* **`5` (Generation):** 세대 번호이며, 숫자가 높을수록 최신 하드웨어 기반의 성능 향상을 의미합니다.
* **`2xlarge` (Size):** 해당 패밀리 내에서 할당된 CPU, RAM, 대역폭 등의 자원 규모를 나타냅니다.

#### 🗂️ 5대 인스턴스 목적별 클래스 정리

1. **General Purpose (범용 - M 또는 T 시리즈):** CPU, 메모리, 네트워크 자원이 균형 있게 배분된 타입입니다. 웹 서버, 코드 레포지토리, 테스트/개발 환경 등 가장 무난하게 쓰입니다. (`M`edium / `M`ost 사용으로 암기)
2. **Compute Optimized (컴퓨팅 최적화 - C 시리즈):** 고성능 프로세서(CPU) 중심의 연산 집약적 작업에 쓰입니다. **배치 처리(Batch), 미디어 트랜스코딩, 고성능 웹 서버, 고성능 컴퓨팅(HPC), 전용 게임 서버**에 적합합니다. (`C`PU 최적화로 암기)
3. **Memory Optimized (메모리 최적화 - R 또는 X 시리즈):** 대규모 데이터 세트를 RAM 위에 올려 초고속 가공하는 작업에 최적화되어 있습니다. **관계형(RDBMS)/비관계형(NoSQL) 대형 DB, 분산 인메모리 캐시 스토어(Redis, Memcached), 실시간 빅데이터 처리**에 필수적입니다. (`R`AM 최적화로 암기)
4. **Storage Optimized (스토리지 최적화 - I, D, H 시리즈):** 로컬 NVMe SSD/HDD에 매우 대량의 고빈도 I/O(읽기/쓰기) 및 높은 순차 처리가 필요한 워크로드용입니다. **고빈도 OLTP 시스템, 분산 NoSQL DB, 데이터 웨어하우스(Data Warehousing), 분산 파일 시스템**에 매핑됩니다. (`I`/O 성능 중심으로 암기)
5. **Accelerated Computing (가속화된 컴퓨팅):** 그래픽 처리, 데이터 패턴 분석 등을 위해 별도의 하드웨어 가속기(GPU, FPGA 등)를 탑재한 인스턴스 군입니다.

### 🛡️ 2. 보안 그룹 (Security Groups) 핵심 가인드라인

보안 그룹은 EC2 인스턴스 바로 앞단에 위치하여 인바운드/아웃바운드 가상 네트워크 트래픽을 제어하는 "상태 저장형(Stateful) 가상 방화벽"입니다.

#### ⚙️ 심층 분석 및 운영 특징

* **허용(Allow) 규칙 전용:** 특정 트래픽을 거부(Deny)하는 규칙은 만들 수 없으며, **오직 허용할 IP 대역, 포트, 프로토콜만 지정**할 수 있습니다.
* **보안 그룹 간 상호 참조:** IP 주소뿐만 아니라 **"다른 보안 그룹의 ID"를 소스(Source)로 지정**할 수 있습니다. (예: WAS 보안 그룹 설정 시, ALB 보안 그룹 ID만 인바운드로 허용하여 전면 보안 강화)
* **다대다(M:N) 매핑:** 하나의 보안 그룹을 여러 EC2 인스턴스에 동시 적용할 수 있으며, 반대로 하나의 EC2 인스턴스에도 여러 개의 보안 그룹을 중첩 연결할 수 있습니다.
* **네트워크 경계성:** 보안 그룹은 리전(Region) 및 VPC 단위로 종속됩니다. 다른 리전으로 전환하면 완전히 새로 만들어야 합니다.
* **위치적 특성:** EC2 인스턴스 "외부"의 AWS 네트워크 인프라 레이어에서 동작합니다. 따라서 인바운드가 차단된 트래픽은 EC2 인스턴스 내부 OS까지 도달하지 못하므로, EC2 내부 로그에서는 차단 기록을 확인할 수 없습니다.
* **기본 규칙:** 생성 직후 기본적으로 모든 인바운드는 차단(Deny)되어 있으며, **모든 아웃바운드는 허용(Allow)** 상태입니다.

#### 🚨 네트워크 에러 디버깅 치트키 (시험 단골 문항)

* **타임아웃(Connection Timeout) 발생:** 트래픽 요청을 보냈으나 묵묵부답으로 세션이 끊기는 현상입니다. 이는 100% **"보안 그룹(Security Group) 규칙"에서 해당 인바운드 포트를 열어주지 않아** 외부에서 아예 접근하지 못하고 패킷이 드롭되는 상태입니다.
* **연결 거부(Connection Refused) 발생:** 트래픽이 보안 그룹 방화벽을 무사히 통과하여 EC2 인스턴스 내부까지는 들어갔으나, 정작 내부에서 응답해야 할 **웹 서버 애플리케이션(Nginx, Apache 등)이 꺼져 있거나 해당 포트를 리스닝(Listening)하고 있지 않을 때** 발생하는 OS단의 에러입니다. (보안 그룹은 정상입니다.)

#### 🔌 반드시 암기해야 할 표준 클래식 포트 번호

* **`22`:** SSH (Linux 원격 접속) / SFTP (SSH 기반 안전한 파일 전송)
* **`21`:** FTP (전통 파일 전송 프로토콜)
* **`80`:** HTTP (암호화되지 않은 일반 웹 트래픽)
* **`443`:** HTTPS (SSL/TLS 암호화 웹 트래픽)
* **`3389`:** RDP (Windows 가상 머신 원격 데스크톱 접속)

#### 🔗 EC2 Instance Connect

* **특징:** 로컬 PC에 SSH 개인키(`.pem`)를 다운로드하거나 복잡한 SSH 클라이언트 설정을 관리할 필요 없이, AWS 웹 콘솔 화면에서 즉시 대상 Linux 서버(Amazon Linux 2 기본 지원)에 안전하게 원격 로그인하는 기능입니다.
* **핵심 가드레일:** 편의성이 극대화된 웹 기반 접속 방식이지만, 백엔드에서는 여전히 표준 SSH 프로토콜을 사용하므로 **해당 EC2 인스턴스의 보안 그룹 규칙에서 22번 인바운드 포트는 반드시 열려 있어야** 정상 작동합니다.

### 💰 3. EC2 인스턴스 구매 옵션 (Purchasing Options)

| 구매 옵션 | 핵심 메커니즘 | 요금 방식 및 할인율 | 추천 워크로드 환경 |
| --- | --- | --- | --- |
| **On-Demand** | 약정 없이 필요한 만큼 즉시 대여 | 초 단위(Linux/Windows)<br>

<br>또는 시간 단위 청구 (**기준가**) | 단기적, 중단 없음, 예측 불가능한 스파이크성 트래픽 애플리케이션 |
| **Reserved Instances (RI)** | 1년 또는 3년 기간 동안 특정 인스턴스 스펙(타입, 리전 등)을 약정 | **최대 72% 할인**<br>

<br>(선결제 많을수록 할인율 ⬆️) | 사용량이 24시간 항시 일정하고 고정된 시스템 (예: 상시 가동 프로덕션 DB) |
| **Convertible RI** | RI 기간 도중 인스턴스 패밀리, OS, 타입을 자유롭게 유연 변경 가능 | **최대 66% 할인**<br>

<br>(일반 RI보다 낮은 할인율) | 장기 가동이 확실하지만 중간에 아키텍처 스펙 변경 가능성이 있을 때 |
| **Savings Plans** | 1년 또는 3년 동안 **"시간당 $X 지불"** 형태의 총 자원 금액을 약정 | **최대 72% 할인**<br>

<br>(인스턴스 사이즈/OS 변경 유연) | 인스턴스 패밀리(M5 등) 내에서 유연하게 사이즈 및 OS를 바꾸며 비용을 절감할 때 |
| **Spot Instances** | AWS 여유 용량 인프라를 경매 방식으로 저렴하게 대여 | **최대 90% 할인** (**비용 최저**) | **중단되어도 무방한(Resilient)** 배치 작업, 빅데이터 분석, 이미지 렌더링, 분산 워크로드 |
| **Dedicated Hosts** | 사용자에게 물리적 서버 전체 가시성과 배치 제어권을 통제 할당 | 물리 서버 기준 과금 (**비용 최고**) | **복잡한 라이선스 모델(BYOL, 코어/소켓당 과금)** 요구 기업, 강력한 대기업 법적 규제 준수(Compliance) |
| **Dedicated Instances** | 타인과 하드웨어를 공유하지 않는 격리 환경에서 인스턴스 실행 | 인스턴스별 별도 과금 | 물리적 서버 수준의 제어는 필요 없으나 하드웨어 물리 격리만 원할 때 |
| **Capacity Reservation** | 특정 AZ에 가상 서버 자원을 지정 기간 동안 무조건 강제 선점 | **할인 혜택 없음** (실행 안 해도 On-Demand 요금 상시 부과) | 특정 AZ에서 반드시 구동되어야 하는 단기 이벤트성 핵심 워크로드 용량 확보 |

* **💡 Convertible RI vs Savings Plans 차이:** Convertible RI는 인스턴스 속성 변경 시 교환 절차가 필요하지만, Savings Plans는 계약 금액 내에서 인스턴스 크기, OS, 테넌시가 바뀌면 별도 조작 없이 자동으로 할인 혜택이 적용되므로 현대 아키텍처에서 훨씬 선호됩니다.

### 🛑 4. 스팟 인스턴스 (Spot Instance) 및 스팟 플릿 (Spot Fleet)

#### 🛑 중단 및 종료 메커니즘 (Termination)

* 스팟 인스턴스는 사용자가 설정한 최대 입찰가(Max Spot Price)보다 실시간 AWS 시장 스팟 가격(Current Spot Price)이 높아지면 AWS에 의해 강제 회수(종료/중지)됩니다. 이때 인스턴스가 날아가기 딱 **2분 전 유예 시간 알림**을 줍니다. (장애 복구 능력이 있는 애플리케이션 빌드가 필수적입니다.)
* **스팟 요청 유형:**
* **One-time Request (일회성 요청):** 요청이 수락되어 인스턴스가 실행되는 즉시 요청(Request) 객체는 스스로 폐기됩니다. 이후 인스턴스를 수동 종료해도 재실행되지 않습니다.
* **Persistent Request (지속성 요청):** 지정한 유효기간(Valid From ~ Until) 동안 요청이 상시 살아있습니다. 시장 가격 변동으로 인해 인스턴스가 도중에 중단/종료되더라도, 시장 가격이 다시 사용자의 입찰가보다 낮아지면 **지속성 요청이 활성화되어 스팟 인스턴스를 자동으로 재시작(Restart)** 시킵니다.



#### ⚠️ 스팟 인스턴스 완전 영구 종료 프로세스 (시험 필수)

* 만약 지속성 요청(Persistent Request)으로 켜진 스팟 인스턴스를 그냥 일반 EC2 끄듯이 인스턴스만 종료(`Terminate`)시키면, 살아있는 스팟 요청 객체가 "어? 인스턴스가 부족하네?" 하고 **새로운 스팟 인스턴스를 무한대로 재차 띄우는 대참사**가 발생합니다.
* **올바른 삭제 순서:**
```
[1단계: Spot Request(스팟 요청) 취소 (Cancel)] ➡️ [2단계: 연결된 구동 중인 Spot Instance 종료 (Terminate)]

```


* 요청 상태가 Open, Active, Disabled인 상태에서 취소해야 하며, 요청을 먼저 지워야만 인스턴스를 종료했을 때 좀비처럼 다시 살아나지 않습니다.



#### 🚢 스팟 플릿 (Spot Fleet)

스팟 인스턴스 단일 요청과 달리, 사용자가 사전에 정의한 **다양한 인스턴스 타입 풀(Launch Pools - 예: m5.large, c5.large)과 다중 AZ 배치 구성 조건**을 기반으로, 예산 및 대상 용량(Target Capacity)에 맞춰 온디맨드와 스팟 인스턴스 군을 복합적으로 조합하여 자동 관리해 주는 스마트 통합 관리 레이어입니다.

#### 🎯 스팟 플릿 4대 할당 전략 (Allocation Strategy)

* **lowestPrice:** 가장 비용이 저렴한 인프라 풀에서만 인스턴스를 가져옵니다. **비용 최적화 중심**의 매우 짧은 단기 워크로드에 추천됩니다.
* **diversified:** 사전에 정의해 둔 모든 인스턴스 런치 풀에 인스턴스를 균등하게 분산 배치합니다. 하나의 풀이 폭등하여 회수되어도 타격이 적으므로 **가용성이 우수**한 장기 워크로드에 쓰입니다.
* **capacityOptimized:** 인스턴스 공급 부족 위험이 없는, 현재 시점 가장 **예비 용량이 풍부한 최적의 풀**을 찾아 할당합니다. 중단 가능성을 최소화합니다.
* **priceCapacityOptimized (★AWS 추천 기본값):** 가용 용량이 가장 넉넉하여 중단 리스크가 가장 적은 풀들을 1차 선별한 후, 그 안에서 **가장 요금이 저렴한 풀을 2차 선택**합니다. 대다수의 일반적인 분산 배치 워크로드에 가장 완벽한 밸런스를 제공합니다.
