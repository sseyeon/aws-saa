# 1. S3 데이터 암호화 (Encryption)


이번 파트는 S3에 저장되는 소중한 내 데이터를 도둑맞지 않게 잠그는 **"암호화(Encryption)"** 장치들입니다.
크게 "누가 열쇠(Key)를 쥐고 흔드느냐"를 기준으로 4가지 방식을 깔끔하게 쪼개 드릴게요!

---

## 🧊 1-1. 서버 측 암호화 (Server-Side Encryption, SSE)

내가 파일을 그냥 날것(평문)으로 S3에 던지면, **S3 서버가 받자마자 알아서 암호화해서 저장**해 두는 방식입니다. 꺼내 갈 때도 알아서 복호화(암호 해독)해서 줍니다.

누가 열쇠를 관리하느냐에 따라 3가지로 나뉩니다.

### ① SSE-S3 : "가장 편한 자동 자물쇠"

* **특징:** 열쇠 관리, 생성, 교체까지 **모두 AWS가 알아서 소유하고 처리**합니다.
* **실무 맥락:** "보안 규정상 암호화는 무조건 켜야 하는데, 귀찮은 설정이나 열쇠 관리는 신경 쓰고 싶지 않다" 할 때 씁니다. 현재 S3 버킷을 새로 만들면 기본적으로 켜져 있는 기본값입니다.
* **시험 키워드:** 헤더에 `"x-amz-server-side-encryption": "AES256"`이 붙어 날아갑니다.

### ② SSE-KMS : "세밀한 통제가 가능한 고급 자물쇠" (★시험 단골)

* **특징:** AWS의 열쇠 전문 관리 서비스인 **KMS**를 이용합니다. 내가 직접 열쇠를 폐기하거나, 특정 직원만 열쇠를 쓸 수 있게 권한을 쪼갤 수 있습니다. 누가 이 열쇠로 파일 열어봤는지 CloudTrail 로그로 감시(감사)도 가능합니다.
* **시험 키워드:** 헤더에 `"aws:kms"`가 붙습니다.

#### ⚠️ 무조건 외워야 하는 SSE-KMS의 치명적인 한계 (Limitation)

실무에서 이거 모르고 트래픽 몰리는 대형 서비스에 적용했다가 서비스 터집니다.

* 파일을 S3에 업로드할 때마다 `GenerateDataKey`라는 KMS API를 호출하고, 다운로드할 때마다 `Decrypt`라는 KMS API를 호출합니다.
* 문제는 KMS 서비스가 처리할 수 있는 초당 요청 수(리전별로 5,500 ~ 30,000 req/s)에 제한이 있다는 점입니다.
* **시험 단골 문제:** "KMS로 암호화된 S3 버킷에서 대규모 빅데이터 분석을 돌렸더니 **스로틀링(초과 오류, HTTP 503) 에러**가 난다. 어떻게 해결해야 하는가?" ➡️ "KMS API 할당량(Quota) 증가를 요청하거나 S3 버킷 키를 활용해 KMS 호출을 줄인다"가 정답으로 나옵니다.

### ③ SSE-C : "열쇠는 절대 AWS에 못 준다"

* **특징:** 회사의 아주 엄격한 보안 규정 때문에 암호화 열쇠만큼은 AWS 클라우드에 올리지 못하고 **우리 회사 내부(온프레미스)에 보관**해야 할 때 씁니다.
* **작동 원리:** S3에 파일을 보낼 때, **우리 열쇠를 HTTP 헤더에 같이 실어서** 보냅니다. S3는 그 열쇠로 파일을 잠그자마자 **우리 열쇠를 메모리에서 즉시 지워버립니다.** (저장 안 함) 즉, 꺼내 올 때도 우리가 그 똑같은 열쇠를 헤더에 쥐여주지 않으면 S3는 파일을 열어주지 못합니다.
* **필수 조건:** 열쇠가 인터넷 세상에 노출되면 안 되므로, 반드시 **HTTPS(보안 통신)가 강제**됩니다.

> ⭐ **KMS 스로틀링 제한(Limitation) - 시험 단골**
> 
> 
> SSE-KMS로 암호화된 버킷은 업로드할 때 GenerateDataKey, 다운로드할 때 Decrypt API를 매번 호출합니다.
> 
> 초당 요청 수가 매우 높은 환경(리전별 초당 5,500 ~ 30,000 req/s 제한)에서는 **KMS Throttling(503 오류)**이 발생할 수 있습니다.
> 
> - **해결책:** Service Quotas를 통해 KMS 한도 증설을 요청하거나, S3 버킷 키(Bucket Keys) 기능을 활성화하여 KMS 호출 횟수를 줄여야 합니다.

---

## 💻 1-2. 클라이언트 측 암호화 (Client-Side Encryption)

* **특징:** S3 서버를 전혀 믿지 않는 방식입니다. **내 컴퓨터(웹 서버나 앱)에서 아예 파일을 암호화로 꽁꽁 묶은 뒤에** S3에 던집니다.
* S3 입장에서는 이게 이미지인지 텍스트인지 알 길도 없고, 그냥 깨진 외계어 파일로 보관만 해줍니다. 열쇠와 암호화 주기는 100% 고객이 관리합니다.

---

## 🔒 1-3. 전송 중 암호화 (Encryption in transit)

인터넷 망을 타고 데이터가 이동할 때 해커가 중간에 패킷을 훔쳐보는 걸 막는 보안입니다.

* S3는 일반 `HTTP(80포트)` 주소와 암호화된 `HTTPS(443포트)` 주소 두 가지를 다 제공합니다.
* 당연히 **HTTPS**를 쓰는 것이 강력하게 권장되며, 앞서 배운 **SSE-C는 무조건 HTTPS가 필수**입니다.

---

## 🛠️ 1-4. 기본 암호화(Default) vs 버킷 정책(Bucket Policy)

"우리 회사 S3 양동이에는 무조건 암호화된 파일만 들어와야 해!"라고 강제 규칙을 정하는 두 가지 방법의 순서 문제입니다.

* **버킷 정책 (거부권):** "헤더에 암호화 표시(`AES256`) 안 달고 들어오는 무암호화 나쁜 파일들은 입구 컷(Deny) 시켜라!" 하고 문지기를 세우는 것입니다.
* **기본 암호화 (보정):** "유저가 암호화 지정을 깜빡하고 그냥 올렸네? 그럼 내가 대신 기본 자물쇠(SSE-S3) 채워서 저장해 줄게." 하고 받아주는 정비소입니다.

#### ⚠️ 시험 출제 포인트 (평가 순서)

AWS는 항상 **버킷 정책을 '기본 암호화'보다 먼저 검사**합니다.
만약 버킷 정책에 "암호화 헤더 없는 놈은 다 차단해!"라고 적어두고, 동시에 "기본 암호화"도 켜놨다고 해볼게요. 유저가 암호화 헤더 없이 생짜 파일을 올리면 어떻게 될까요?

* 기본 암호화 기능이 작동해서 자동으로 자물쇠를 채워주기도 전에, **버킷 정책 문지기가 먼저 발견하고 "헤더 없네? 저리 가!" 하고 에러를 내며 차단**해 버립니다. 두 기능의 우선순위를 묻는 지문이 나오면 이 순서를 기억하셔야 합니다.

---

### 💡 1초 완성 요약표

* **SSE-S3:** AWS가 알아서 다 해주는 기본 자동 자물쇠.
* **SSE-KMS:** 내가 열쇠 권한을 통제하고 기록을 감시할 수 있는 고급 자물쇠 (단, **트래픽 폭탄 시 스로틀링 제한** 조심!).
* **SSE-C:** 열쇠는 내 주머니에 두고, 일할 때만 잠시 AWS에 빌려주는 방식 (**HTTPS 필수**).
* **Client-Side:** 내 컴퓨터에서 미리 다 잠가서 보내는 방식.
* **순서:** 파일이 들어올 땐 **버킷 정책(문지기) ➡️ 기본 암호화(정비소)** 순으로 거친다.

# 2. 권한 및 접근 제어 (CORS & Access)
이 파트는 백엔드와 프론트엔드가 협업할 때, 그리고 S3에 저장된 이미지나 폰트를 가져올 때 무조건 한 번은 마주치는 "웹 브라우저의 보안 자물쇠"에 대한 이야기입니다. SAA 시험에서도 정말 단골로 출제되는 핵심 개념이에요.

딱딱한 명칭 대신, 웹 브라우저가 왜 이런 깐깐한 규칙을 만들었는지 **"신분증 검사"** 비유로 아주 쉽게 풀어드릴게요!

---
## CORS
### 2-1. Origin(출처)이 정확히 뭐야?

쉽게 말해서 웹사이트의 "정확한 풀 주소(문패)"를 뜻합니다. 노트에 적힌 대로 아래 3가지가 단 하나라도 다르면 브라우저는 "서로 다른 동네(Cross-Origin)"라고 인식합니다.

1. **Protocol (프로토콜):** `http`인지 `https`인지
2. **Host (도메인):** `naver.com`인지 `daum.net`인지
3. **Port (포트):** 웹 서버 구멍 번호 (`:80`, `:443` 등)

* **Same Origin (같은 동네):** `https://example.com/movie`와 `https://example.com/music` (앞부분이 완벽히 일치!)
* **Different Origin (다른 동네):** `https://example.com`과 `http://example.com` (프로토콜이 다름!), `https://api.example.com` (서브 도메인이 다름!)

---

### 2-2. CORS는 대체 왜 필요한 거야? (보안의 이유)

만약 브라우저에 아무런 제약이 없다면 엄청난 해킹 사건이 터집니다.
내가 해커가 만든 피싱 사이트(`http://bad-hacker.com`)에 접속했는데, 그 사이트 안에 심겨 있는 자바스크립트 코드가 브라우저 몰래 내 진짜 은행 사이트(`https://my-bank.com`)로 "내 계좌에서 돈 이체해!"라는 요청을 강제로 보낼 수 있거든요.

이를 막기 위해 웹 브라우저(크롬, 사파리 등)는 "기본적으로 내가 접속한 사이트랑 다른 동네(Origin)로 요청을 보내는 건 전부 차단한다!"라는 철칙(SOP, Same-Origin Policy)을 세웠습니다.

---

### 2-3. 그럼 CORS Headers는 뭐야? (합법적인 문 열어주기)

그런데 실무에서는 무조건 다른 동네끼리 데이터를 주고받아야 하는 합법적인 상황이 생깁니다.
예를 들어 유저가 내 웹사이트(`https://my-service.com`)에 접속했는데, 이 사이트가 쓰는 예쁜 글꼴 폰트나 고화질 이미지는 AWS S3 버킷 주소(`https://my-s3-bucket.s3.amazonaws.com`)에 저장되어 있는 거죠.

1. 유저의 브라우저가 `my-service.com` 화면을 그리다가, "어? 이미지랑 폰트는 `my-s3-bucket`이라는 다른 동네에서 가져와야 하네?" 하고 S3에 요청을 보냅니다.
2. S3가 데이터를 보내주는데, 브라우저가 중간에 가로채서 검사를 합니다. **"잠깐, 너 다른 동네잖아! 보안상 못 보여줘!"** 하고 에러를 뿜으며 화면을 깨버립니다. 이 에러가 바로 악명 높은 **CORS 에러**입니다.
3. 이 문제를 해결하려면, S3 바구니에 자물쇠를 풀어주는 설정(**CORS Headers**)을 심어야 합니다.

---

### 2-4. Amazon S3 - CORS 설정 (시험에 자주 나오는 이유)

S3 관리자 화면에 가서 아래와 같이 "CORS 정책"을 JSON이나 XML 텍스트로 적어두면 브라우저가 통과시켜 줍니다.

```json
[
  {
    "AllowedOrigins": ["https://my-service.com"],
    "AllowedMethods": ["GET"]
  }
]

```

> **해석:** "우리 S3 버킷은 `https://my-service.com`이라는 동네에서 들어오는 `GET(읽기)` 요청은 안전하니까 허용해 줄게!"

이렇게 설정해 두면 S3가 브라우저에게 파일을 넘겨줄 때 `Access-Control-Allow-Origin: https://my-service.com`이라는 허가증(헤더)을 같이 쥐여서 보냅니다. 브라우저가 이 허가증을 읽고 "아, S3 주인이 얘네 동네는 안전하다고 허락해 줬구나!" 하고 안심하며 이미지를 정상적으로 화면에 띄워줍니다. (만약 모든 동네를 허용하고 싶다면 `*`을 적으면 됩니다.)


## MFA Delete (멀티팩터 인증 삭제 보호)
이번 파트는 S3의 '최종 방어선(MFA)', '행동 기록 장치(Access Logs)', 그리고 '임시 출입증(Pre-Signed URL)'에 대한 내용입니다.

실무에서 해킹이나 실수로 데이터가 날아가는 걸 막고, 누가 뭘 뒤져봤는지 감시하며, 특정 유저에게만 잠깐 안전하게 문을 열어줄 때 쓰는 필수 기술들입니다. 머리에 쏙 박히게 정리해 드릴게요!

---

### 2-5. MFA Delete (실수와 해킹을 막는 '2차 비밀번호')

> **"버전 관리 중단이나 영구 삭제는 내 스마트폰 OTP 코드를 대라!"**

회사 직원의 AWS 계정이 해킹당했거나, 신입 사원이 실수로 S3의 중요 데이터를 통째로 날려버리는 대참사를 막기 위한 최종 브레이크 장치입니다. (이 기능을 쓰려면 지난번에 배운 버전 관리(Versioning)가 필수로 켜져 있어야 합니다.)

* **언제 MFA(OTP 코드)를 요구할까?**
* 단순히 파일을 지우는 게 아니라, 복구조차 불가능하게 "객체 버전을 영구적으로 완전히 삭제"하려고 할 때.
* 버킷의 안전장치인 **"버전 관리 기능을 꺼버리려고(중단)"** 할 때.


* **언제는 안 요구할까? (시험 단골 말장난)**
* 버전 관리를 처음 켤 때, 혹은 그냥 삭제된 파일 리스트를 구경할 때는 OTP 코드가 필요 없습니다.


* **루트 계정(버킷 소유자)만 제어 가능:** 이 MFA Delete 기능을 켜고 끄는 막강한 권한은 오직 회사의 대장 계정(Root 계정)만 AWS CLI를 통해서 처리할 수 있습니다. 일반 IAM 직원은 절대 손댈 수 없습니다.

# 3. 모니터링 & 임시 접근 제어
## S3 Access Logs (CCTV 블랙박스 기록)

> **"누가 내 양동이를 뒤져봤는지 다 기록해라"**

금융권이나 IT 회사에서는 보안 감사(Audit)를 나옵니다. "누가 2026년 5월에 고객 개인정보 파일을 다운로드해 갔는지 기록 내놓으세요"라고 할 때 제출하는 CCTV 블랙박스입니다.

* **철저한 기록:** 로그인을 한 사람이든, 거부당한 사람이든, 다른 AWS 계정이든 상관없이 **내 S3 양동이에 찌른 모든 요청(성공/실패)이 100% 텍스트 로그 파일로 기록**됩니다. 이 로그들을 나중에 Athena 같은 도구로 SQL 쿼리를 날려 분석합니다.
* **핵심 제약 조건:** 로그가 저장될 '로그 방(버킷)'은 반드시 원본 S3 양동이와 동일한 AWS 리전(예: 둘 다 서울 리전)에 있어야 합니다.

### 🚨 경고: 무한 루프 헬게이트 (집에서는 절대 금지!)

노트에 무시무시하게 적혀 있던 내용이죠! 인프라 초보들이 실무에서 정말 많이 치는 대형 사고입니다.

* **원리:** S3에 파일이 들어오면 ➡️ S3는 "파일 들어옴!" 하고 로그를 한 줄 씁니다.
* **대참사:** 만약 "내 S3 양동이의 로그를 자기 자신에게 저장해라"라고 바보같이 세팅해 버리면 어떻게 될까요?
1. 유저가 파일을 1개 올림.
2. S3가 자기 자신에 "로그 파일"을 씀.
3. S3 입장에서는 "어? 방금 새 로그 파일이 들어왔네?" 하고 **그 로그 파일이 들어온 것에 대한 또 다른 로그**를 씀.
4. "어? 또 새 로그가 들어왔네?" ➡️ 무한 반복 🔄


* **결과:** 1초에 수만 개의 로그 파일이 폭발적으로 생겨나며 **양동이 크기가 기하급수적으로 커지고, 비용 폭탄**을 맞게 됩니다. 로그는 반드시 '별도의 분리된 S3 양동이'에 담아야 합니다.

## Pre-Signed URLs (3분 짜리 임시 프리패스 출입증)

> **"로그인한 유저에게만 딱 10분 동안 다운로드 권한 주기"**

넷플릭스나 유료 인터넷 강의 사이트를 만든다고 해볼게요. 동영상 파일들을 보관하는 S3 양동이는 해커들이 훔쳐가지 못하게 **Public(공개)을 다 막아두고 꽁꽁 숨겨놔야** 합니다.
그럼 돈을 낸 유저들에게는 어떻게 영상을 보여줄까요? 유저한테 우리 회사 AWS 마스터 비밀번호를 줄 순 없잖아요?

이때 쓰는 기술이 바로 Pre-Signed URL(미리 서명된 URL)입니다.

* **작동 원리:** 백엔드 서버가 유저의 로그인 여부를 확인한 뒤, AWS SDK 코드를 써서 "내(개발자) 권한을 딱 5분 동안만 빌려주는 특수 가짜 주소"를 동적으로 발행해서 유저 브라우저에 던져줍니다.
* **권한 상속:** 이 주소를 받아 든 유저는 AWS 계정이 없어도, 주소가 만료되기 전까지는 **그 주소를 만들어준 개발자의 권한(GET으로 읽거나 PUT으로 업로드하는 권한)을 그대로 상속받아** 일시적으로 S3와 다이렉트 거래를 할 수 있습니다.
* **만료 기한:** 콘솔에서는 최대 12시간, CLI나 코딩(SDK)으로는 최대 7일(168시간)까지 유통기한 타이머를 빠듯하게 걸어둘 수 있습니다. 기한이 지나면 주소는 그냥 터진 링크가 됩니다.


# 4. 데이터 보존 및 규정 준수 (WORM 모델)
이번 파트는 S3의 '강력한 절대 보안(Lock)'과 대규모 데이터를 다룰 때 쓰는 '지능형 접근 기술(Access Point & Lambda)'에 대한 내용입니다.

실무에서 금융권 규제를 맞추거나, 수천 명의 권한을 관리하거나, 데이터를 가공해서 전달할 때 쓰는 고급 기술들이에요. 어려운 용어를 싹 걷어내고 직관적인 비유로 하나씩 쪼개 드릴게요!

---

## 4-1. S3 Glacier Vault Lock (지하 냉동고 통째로 용접하기, 볼트 잠금)

> **"한 번 쓰면 끝, 누구도 절대 삭제 변경 불가 (WORM)"**

회사에 금융 감독원이나 국세청 같은 국가 기관이 들이닥쳤습니다. "당신들 5년 치 회계 장부 가져와 봐. 혹시 중간에 조작하거나 삭제한 거 아니야?" 라고 의심합니다. 이때 우리 회사의 결백함을 증명할 치트키가 바로 **Vault Lock**입니다.

* **WORM (Write Once Read Many):** "한 번 쓰고, 읽기는 여러 번 한다." 즉, 수정과 삭제가 절대 불가능한 읽기 전용 상태를 뜻합니다.
* **동작 원리:** Glacier(장기 보관소)에 백업 파일을 넣고 Vault Lock 자물쇠를 채우는 순간, **그 규칙은 영구적으로 수정·삭제가 불가능**해집니다.
* **얼마나 강력한가:** 회사의 사장님은 물론이고, AWS의 최고 권한을 가진 **루트(Root) 사용자나 AWS 본사 직원 할아버지가 와도 이 잠금을 풀거나 데이터를 지울 수 없습니다.** 데이터 조작이 원천 차단되므로 법적 규정(Compliance)을 완벽하게 만족하게 됩니다.

---

## 4-2. S3 Object Lock (파일 하나하나에 타이머 자물쇠 채우기, 객체 잠금)

위의 Vault Lock이 보관소 전체를 용접하는 거라면, **Object Lock**은 양동이(Bucket) 안의 '특정 파일(Object) 딱 한 개'에 자물쇠를 채우는 기술입니다. (단, 파일의 과거 이력을 추적해야 하므로 **버전 관리(Versioning)** 기능이 꼭 켜져 있어야 합니다.)

자물쇠 모드는 크게 두 가지로 나뉩니다.

### ① Compliance (규정 준수 모드) - 최고 철통 보안

* **원리:** 루트 사용자를 포함해 **그 어떤 유저도 절대 지우거나 설정을 바꿀 수 없습니다.**
* 만약 보관 기간을 3년으로 설정했다면, 3년 동안은 그 누구도 데이터에 손을 댈 수 없습니다. 중간에 보관 기간을 줄이는 꼼수도 안 통합니다.

### ② Governance (거버넌스 모드) - 관리자 예외 허용

* **원리:** 일반 직원들은 절대 못 지우게 막아두지만, **'특별한 대장 권한(IAM)'을 가진 인프라 팀장님 같은 관리자는 예외적으로 파일을 지우거나 잠금을 풀 수 있게** 문을 열어두는 모드입니다.

### 🔒 특수 기능들

* **Retention Period (보관 기간):** "이 파일은 앞으로 100일 동안 보호한다"처럼 타이머를 맞춰두는 것입니다. 필요하면 기간을 더 늘릴 수는 있습니다.
* **Legal Hold (법적 보관):** 타이머(기간)가 없습니다. 법적 소송이나 재판이 끝날 때까지 **무기한으로 삭제를 차단**하는 자물쇠입니다. 권한을 가진 사람이 "이제 해제해도 돼" 하고 자물쇠를 직접 열어주기 전까지는 평생 보호됩니다.

---

## 4-3. S3 Access Points (폴더별 전용 출입문 만들기)

> **"하나의 거대한 버킷을 다루는 스마트한 출입구"**

회사의 하나의 거대한 S3 양동이 안에 `재무부`, `마케팅부`, `개발부` 폴더가 다 들어있다고 해볼게요. 수백 명의 직원에게 권한을 주려면 S3 버킷 정책(Bucket Policy) 텍스트 파일이 수천 줄이 넘어가면서 관리가 불가능해지고 꼬이기 시작합니다.

이 난장판을 해결하기 위해, S3 양동이로 들어오는 '부서별 전용 출입문(Access Point)'을 따로 파주는 것입니다.

* **재무부 전용 문:** 주소는 `finance-access-point` 이고, 오직 재무부 방화벽과 직원만 통과하는 정책을 심어둡니다.
* **마케팅부 전용 문:** 주소는 `marketing-access-point` 이고, 마케팅 파일만 보게 만듭니다.
* **효과:** 복잡한 메인 버킷 정책을 건드릴 필요 없이, 각자 부서 문 앞에만 간단한 규칙을 적어두면 되므로 대규모 권한 관리가 엄청나게 쉬워집니다.

---

## 4-4. S3 Object Lambda (파일을 꺼내기 직전의 마법 필터)

SAA 시험에서 가장 핫하고 세련된 기술입니다. 유저가 S3에서 파일을 다운로드해서 가져가기 **'직전 찰나의 순간'에 Lambda(코드)를 실행시켜 데이터를 실시간으로 변형**해서 던져주는 기술입니다.

### 💡 기막힌 실무 사용 사례 3가지 (시험 단골 주제)

1. **개인정보 마스킹 (PII 삭제):** * 회사의 원본 엑셀 파일에는 회원들의 이름과 주민번호가 그대로 적혀있습니다.

* 일반 분석가 직원이 이 파일을 다운로드 요청하면, **Object Lambda가 주민번호 뒷자리를 `******`로 실시간으로 변환**해서 분석가에게 전달합니다. 원본 파일은 건드리지 않고 보안을 지키는 거죠.

2. **데이터 포맷 변환:** * 구형 시스템은 XML 파일만 알아듣고, 신형 시스템은 JSON 파일만 알아듣는 상황입니다.

* S3에 XML로 저장해 두고, 신형 시스템이 다운로드할 때 **Lambda가 0.1초 만에 JSON으로 싹 번역**해서 건네줍니다.

3. **동적 워터마크 및 이미지 크기 조절:** * 유저가 S3에 있는 고화질 원본 이미지를 요청하면, **그 유저의 아이디를 이미지 구석에 워터마크로 쾅 박거나**, 스마트폰 기종에 맞게 이미지 해상도를 줄여서 다운로드되게 만듭니다.
