# 🔐 Amazon S3 Security 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **열쇠 관리 귀찮고 그냥 AWS가 다 알아서 기본 암호화 해줬으면, AES256 헤더** $\rightarrow$ `SSE-S3`
* **KMS 키로 통제, CloudTrail 감사 로그, 누가 열어봤는지 추적** $\rightarrow$ `SSE-KMS`
* **SSE-KMS 환경에서 대규모 트래픽 시 HTTP 503 스로틀링 발생** $\rightarrow$ `KMS 할당량 증설 또는 S3 버킷 키 활성화`
* **열쇠는 절대 AWS에 못 줌, 고객이 직접 키 관리, HTTPS 필수** $\rightarrow$ `SSE-C`
* **S3 서버를 전혀 믿지 않음, 업로드 전 클라이언트가 직접 암호화** $\rightarrow$ `클라이언트 측 암호화 (Client-Side Encryption)`
* **버킷 정책과 기본 암호화 중 무엇이 먼저 평가되는가** $\rightarrow$ `버킷 정책(문지기) 먼저, 그 다음 기본 암호화(보정)`
* **프론트에서 S3 이미지 가져올 때 브라우저가 차단, 다른 Origin 허용 설정** $\rightarrow$ `CORS (AllowedOrigins 설정)`
* **루트 계정 전용, 영구 버전 삭제나 버전 관리 중단 시 OTP 요구** $\rightarrow$ `MFA Delete`
* **S3 모든 요청을 텍스트 로그로 기록, 감사(Audit), Athena 분석** $\rightarrow$ `S3 Access Logs`
* **Access Logs를 같은 버킷에 저장 시** $\rightarrow$ `무한 루프 발생 (절대 금지), 별도 버킷 사용 필수`
* **로그인 없는 유저에게 S3 파일을 임시로 다운로드/업로드 허용, 유효기간 설정** $\rightarrow$ `Pre-Signed URL`
* **Pre-Signed URL 최대 유효기간 - 콘솔** $\rightarrow$ `최대 12시간`
* **Pre-Signed URL 최대 유효기간 - CLI/SDK** $\rightarrow$ `최대 7일 (168시간)`
* **WORM, 루트 포함 누구도 절대 삭제·변경 불가, 금융 법적 규제 컴플라이언스 완벽 만족** $\rightarrow$ `S3 Glacier Vault Lock`
* **파일 하나에 기간 타이머 자물쇠, 루트 포함 절대 불가** $\rightarrow$ `S3 Object Lock - Compliance 모드`
* **파일 자물쇠, 특수 IAM 권한 관리자만 예외적으로 해제 가능** $\rightarrow$ `S3 Object Lock - Governance 모드`
* **법적 소송 끝날 때까지 무기한 보호, 기간 없음** $\rightarrow$ `S3 Object Lock - Legal Hold`
* **단일 버킷에 부서별 전용 엔드포인트·정책 분리, 복잡한 버킷 정책 단순화** $\rightarrow$ `S3 Access Points`
* **S3 다운로드 직전 실시간 개인정보 마스킹, XML→JSON 변환, 동적 워터마크** $\rightarrow$ `S3 Object Lambda`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 🔑 1. S3 데이터 암호화 (Encryption)

S3에 저장되는 데이터를 보호하는 암호화 방식입니다. "누가 열쇠(Key)를 쥐고 관리하느냐"를 기준으로 4가지 방식으로 나뉩니다.

#### 서버 측 암호화 (Server-Side Encryption, SSE)

파일을 그냥 S3에 전송하면, S3 서버가 받자마자 알아서 암호화해서 저장합니다. 꺼내 갈 때도 자동으로 복호화해서 줍니다.

**① SSE-S3: "가장 편한 자동 자물쇠"**

* AWS가 열쇠 관리, 생성, 교체까지 모두 소유하고 처리합니다.
* 현재 S3 버킷을 새로 만들면 기본적으로 켜져 있는 기본값입니다.
* 시험 키워드: 헤더에 `"x-amz-server-side-encryption": "AES256"`이 붙습니다.

**② SSE-KMS: "세밀한 통제가 가능한 고급 자물쇠" (★시험 단골)**

* AWS의 열쇠 전문 관리 서비스인 KMS를 이용합니다.
* 직접 열쇠를 폐기하거나, 특정 사용자만 열쇠를 쓸 수 있게 권한을 쪼갤 수 있습니다.
* 누가 이 열쇠로 파일을 열어봤는지 CloudTrail 로그로 감사(감사)가 가능합니다.
* 시험 키워드: 헤더에 `"aws:kms"`가 붙습니다.

**⚠️ SSE-KMS의 치명적인 한계 (Limitation) - 무조건 암기**

* 파일을 S3에 업로드할 때마다 `GenerateDataKey` KMS API를 호출하고, 다운로드할 때마다 `Decrypt` KMS API를 호출합니다.
* KMS 서비스가 처리할 수 있는 초당 요청 수(리전별로 5,500 ~ 30,000 req/s)에 제한이 있습니다.
* 시험 단골 문제: "KMS로 암호화된 S3 버킷에서 대규모 빅데이터 분석을 돌렸더니 스로틀링(HTTP 503 에러)이 난다. 어떻게 해결하는가?" $\rightarrow$ "KMS API 할당량(Quota) 증가를 요청하거나 S3 버킷 키(Bucket Keys) 기능을 활성화하여 KMS 호출 횟수를 줄인다"가 정답입니다.

**③ SSE-C: "열쇠는 절대 AWS에 못 준다"**

* 암호화 열쇠만큼은 AWS 클라우드에 올리지 못하고 회사 내부(온프레미스)에 보관해야 할 때 씁니다.
* S3에 파일을 보낼 때, 열쇠를 HTTP 헤더에 같이 실어서 보냅니다.
* S3는 그 열쇠로 파일을 잠그자마자 열쇠를 메모리에서 즉시 지워버립니다. (저장 안 함)
* 꺼내 올 때도 동일한 열쇠를 헤더에 넣어주지 않으면 S3는 파일을 열어주지 못합니다.
* 필수 조건: 반드시 **HTTPS(보안 통신)가 강제**됩니다.

#### 클라이언트 측 암호화 (Client-Side Encryption)

* S3 서버를 전혀 믿지 않는 방식입니다.
* 내 컴퓨터(웹 서버나 앱)에서 아예 파일을 암호화로 묶은 뒤에 S3에 업로드합니다.
* S3 입장에서는 깨진 외계어 파일로 보관만 해주며, 열쇠와 암호화 주기는 100% 고객이 관리합니다.

#### 전송 중 암호화 (Encryption in Transit)

* 인터넷 망을 타고 데이터가 이동할 때 중간에 패킷을 훔쳐보는 걸 막는 보안입니다.
* S3는 일반 `HTTP(80포트)` 주소와 암호화된 `HTTPS(443포트)` 주소 두 가지를 모두 제공합니다.
* HTTPS를 쓰는 것이 강력하게 권장되며, **SSE-C는 무조건 HTTPS가 필수**입니다.

### 🛡️ 2. 기본 암호화(Default) vs 버킷 정책(Bucket Policy) 평가 순서

버킷에 무조건 암호화된 파일만 들어오게 강제하는 두 가지 방법과 그 순서입니다.

* **버킷 정책 (거부권):** "헤더에 암호화 표시(`AES256`) 안 달고 들어오는 파일들은 입구에서 차단(Deny)해라!" 하고 문지기를 세우는 것입니다.
* **기본 암호화 (보정):** "유저가 암호화 지정을 깜빡하고 그냥 올렸네? 그럼 내가 대신 기본 자물쇠(SSE-S3) 채워서 저장해 줄게." 하고 받아주는 정비소입니다.

**⚠️ 시험 출제 포인트 (평가 순서)**

AWS는 항상 **버킷 정책을 '기본 암호화'보다 먼저 검사**합니다. 버킷 정책에 "암호화 헤더 없는 요청은 다 차단해!"라고 적어두고, 동시에 "기본 암호화"도 켜놓은 상태에서 유저가 암호화 헤더 없이 파일을 올리면? 기본 암호화 기능이 자동으로 자물쇠를 채워주기도 전에, **버킷 정책 문지기가 먼저 발견하고 차단**해 버립니다.

### 🌐 3. CORS (Cross-Origin Resource Sharing)

브라우저의 보안 자물쇠인 CORS는 SAA 시험 단골 개념입니다.

#### Origin(출처)의 정의

웹사이트의 "정확한 풀 주소(문패)"를 뜻합니다. 아래 3가지 중 단 하나라도 다르면 브라우저는 "서로 다른 동네(Cross-Origin)"로 인식합니다.

1. **Protocol (프로토콜):** `http`인지 `https`인지
2. **Host (도메인):** `naver.com`인지 `daum.net`인지
3. **Port (포트):** 웹 서버 포트 번호 (`:80`, `:443` 등)

* **Same Origin (같은 Origin):** `https://example.com/movie`와 `https://example.com/music` (프로토콜+호스트+포트가 완벽히 일치)
* **Different Origin (다른 Origin):** `https://example.com`과 `http://example.com` (프로토콜이 다름), `https://api.example.com` (서브 도메인이 다름)

#### CORS가 필요한 이유 (Same-Origin Policy)

브라우저는 기본적으로 "내가 접속한 사이트랑 다른 Origin으로 요청을 보내는 건 전부 차단한다!"는 철칙(SOP, Same-Origin Policy)을 가집니다. 이는 악성 스크립트가 다른 사이트에 몰래 요청을 보내는 해킹을 막기 위한 보안 장치입니다.

#### Amazon S3 CORS 설정

웹사이트(`https://my-service.com`)가 S3 버킷(`https://my-s3-bucket.s3.amazonaws.com`)의 이미지나 폰트를 가져올 때 CORS 에러가 발생합니다. 이를 해결하려면 S3 버킷에 CORS 정책을 설정해야 합니다.

```json
[
  {
    "AllowedOrigins": ["https://my-service.com"],
    "AllowedMethods": ["GET"]
  }
]
```

* 설정 효과: S3가 파일을 넘겨줄 때 `Access-Control-Allow-Origin: https://my-service.com` 허가증(헤더)을 같이 보내주어 브라우저가 정상적으로 허용합니다.
* 모든 Origin을 허용하려면 `"AllowedOrigins": ["*"]`로 설정합니다.

### 🔐 4. MFA Delete (멀티팩터 인증 삭제 보호)

버전 관리 중단이나 영구 삭제 시 OTP 코드를 요구하는 최종 방어선 장치입니다.

* **전제 조건:** S3 버전 관리(Versioning) 기능이 반드시 켜져 있어야 합니다.
* **MFA(OTP 코드)를 요구하는 경우:**
  * 객체 버전을 영구적으로 완전히 삭제하려고 할 때
  * 버킷의 버전 관리 기능을 꺼버리려고(중단) 할 때
* **MFA를 요구하지 않는 경우 (시험 단골 말장난):**
  * 버전 관리를 처음 켤 때
  * 삭제된 파일 리스트를 조회만 할 때
* **루트 계정(버킷 소유자)만 제어 가능:** MFA Delete 기능을 켜고 끄는 권한은 오직 루트 계정만 AWS CLI를 통해 처리할 수 있습니다. 일반 IAM 사용자는 절대 손댈 수 없습니다.

### 📹 5. S3 Access Logs (CCTV 블랙박스 기록)

"누가 내 S3를 뒤져봤는지 다 기록해라" — 보안 감사(Audit) 시 제출하는 CCTV 블랙박스입니다.

* **철저한 기록:** 로그인한 사람이든, 거부당한 사람이든, 다른 AWS 계정이든 상관없이 S3에 찌른 모든 요청(성공/실패)이 100% 텍스트 로그 파일로 기록됩니다.
* **Athena 분석:** 기록된 로그를 나중에 Athena로 SQL 쿼리를 날려 분석합니다.
* **핵심 제약 조건:** 로그가 저장될 버킷은 반드시 원본 S3 버킷과 **동일한 AWS 리전**에 있어야 합니다.

**🚨 무한 루프 헬게이트 경고 (절대 금지!)**

로그를 저장하는 버킷을 원본 버킷과 동일하게 설정하면 다음과 같은 대참사가 발생합니다.

1. 유저가 파일을 1개 올림 → S3가 "파일 들어옴!" 로그를 씀
2. "어? 새 로그 파일이 들어왔네?" → 그 로그 파일이 들어온 것에 대한 또 다른 로그를 씀
3. "어? 또 새 로그가 들어왔네?" → 무한 반복

1초에 수만 개의 로그 파일이 폭발적으로 생겨나며 **비용 폭탄**을 맞게 됩니다. 로그는 반드시 별도의 분리된 S3 버킷에 저장해야 합니다.

### 🔗 6. Pre-Signed URLs (임시 프리패스 출입증)

"로그인한 유저에게만 딱 10분 동안 다운로드 권한 주기" — 프라이빗 S3 파일에 임시 접근을 허용하는 기술입니다.

* **작동 원리:** 백엔드 서버가 유저의 로그인 여부를 확인한 뒤, AWS SDK 코드를 써서 "발급자의 권한을 딱 N분 동안만 빌려주는 특수 가짜 주소"를 동적으로 발행해서 유저에게 전달합니다.
* **권한 상속:** 이 URL을 받은 유저는 AWS 계정이 없어도, 주소가 만료되기 전까지는 **그 주소를 만들어준 발급자의 권한(GET으로 읽거나 PUT으로 업로드하는 권한)을 그대로 상속받아** 일시적으로 S3와 직접 거래할 수 있습니다.
* **만료 기한 (반드시 암기):**
  * **콘솔:** 최대 **12시간**
  * **CLI / SDK:** 최대 **7일 (168시간)**

### 🔒 7. 데이터 보존 및 규정 준수 (WORM 모델)

#### S3 Glacier Vault Lock (지하 냉동고 통째로 용접하기)

* **WORM (Write Once Read Many):** 한 번 쓰고, 수정과 삭제가 절대 불가능한 읽기 전용 상태.
* Glacier에 파일을 넣고 Vault Lock 자물쇠를 채우는 순간, 그 규칙은 영구적으로 수정·삭제가 불가능해집니다.
* **최강의 방어:** 루트(Root) 사용자나 AWS 본사 직원 할아버지가 와도 잠금을 풀거나 데이터를 지울 수 없습니다.
* 법적 규정(Compliance)을 완벽하게 만족하여 금융권 감사, 국가 기관 규제 대응에 사용됩니다.

#### S3 Object Lock (파일 하나하나에 타이머 자물쇠 채우기)

* Vault Lock이 보관소 전체를 용접하는 거라면, Object Lock은 특정 파일(Object) 하나에 자물쇠를 채우는 기술입니다.
* **전제 조건:** 버전 관리(Versioning) 기능이 반드시 켜져 있어야 합니다.

**자물쇠 모드 비교:**

| 모드 | 보호 대상 | 예외 |
| --- | --- | --- |
| **Compliance (규정 준수 모드)** | 루트 사용자 포함 그 어떤 유저도 절대 삭제·변경 불가 | 없음 (절대 철통) |
| **Governance (거버넌스 모드)** | 일반 사용자는 절대 못 지움 | 특별한 IAM 권한을 가진 관리자는 예외적으로 해제 가능 |

**특수 기능:**

* **Retention Period (보관 기간):** "이 파일은 앞으로 100일 동안 보호한다"처럼 타이머를 설정합니다. 필요하면 기간을 더 늘릴 수 있습니다.
* **Legal Hold (법적 보관):** 타이머(기간)가 없습니다. 법적 소송이나 재판이 끝날 때까지 **무기한으로 삭제를 차단**하는 자물쇠입니다. 권한을 가진 사람이 직접 해제하기 전까지는 평생 보호됩니다.

### 🚪 8. S3 Access Points (폴더별 전용 출입문 만들기)

하나의 거대한 S3 버킷 안에 `재무부`, `마케팅부`, `개발부` 폴더가 모두 들어있을 때, 수백 명의 직원 권한을 S3 버킷 정책(Bucket Policy) 단 하나로 관리하면 수천 줄이 넘어 관리가 불가능해집니다.

* **해결책:** S3 버킷으로 들어오는 '부서별 전용 출입문(Access Point)'을 따로 생성합니다.
* **재무부 전용 문:** `finance-access-point` — 재무부 직원만 통과하는 정책을 설정합니다.
* **마케팅부 전용 문:** `marketing-access-point` — 마케팅 파일만 접근 가능하게 설정합니다.
* **효과:** 복잡한 메인 버킷 정책을 건드릴 필요 없이, 각 부서 문 앞에만 간단한 규칙을 적어두면 되므로 대규모 권한 관리가 단순화됩니다.

### ⚡ 9. S3 Object Lambda (파일 꺼내기 직전의 마법 필터)

유저가 S3에서 파일을 다운로드해서 가져가기 **직전 찰나의 순간에 Lambda(코드)를 실행시켜 데이터를 실시간으로 변형**해서 전달하는 기술입니다. SAA 시험에서 가장 핫한 기술 중 하나입니다.

#### 기막힌 실무 사용 사례 3가지 (시험 단골 주제)

1. **개인정보 마스킹 (PII 삭제):** 원본 파일에는 회원들의 이름과 주민번호가 있습니다. 일반 분석가 직원이 이 파일을 다운로드 요청하면, Object Lambda가 주민번호 뒷자리를 `******`로 실시간으로 변환해서 전달합니다. 원본 파일은 건드리지 않습니다.
2. **데이터 포맷 변환:** S3에 XML로 저장해 두고, 신형 시스템이 다운로드할 때 Lambda가 즉시 JSON으로 번역해서 건네줍니다.
3. **동적 워터마크 및 이미지 크기 조절:** 유저가 고화질 원본 이미지를 요청하면, 그 유저의 아이디를 이미지 구석에 워터마크로 삽입하거나 스마트폰 기종에 맞게 해상도를 줄여서 다운로드되게 만듭니다.

### 💡 1초 완성 요약표

| 암호화 방식 | 열쇠 관리자 | 핵심 특징 | HTTPS |
| --- | --- | --- | --- |
| SSE-S3 | AWS 자동 관리 | 기본값, AES256 헤더 | 선택 |
| SSE-KMS | 고객 (KMS) | CloudTrail 감사, 스로틀링 주의 | 선택 |
| SSE-C | 고객 직접 관리 | 키를 HTTP 헤더로 전달, AWS는 메모리에서 즉시 삭제 | 필수 |
| Client-Side | 고객 100% | S3는 내용 모름, 업로드 전 암호화 | 선택 |

* **평가 순서:** 파일이 들어올 땐 **버킷 정책(문지기) $\rightarrow$ 기본 암호화(정비소)** 순으로 거칩니다.
* **"법적 규제, 루트 유저도 절대 삭제 불가, WORM"** $\rightarrow$ **Glacier Vault Lock** 또는 **S3 Object Lock (Compliance mode)**
* **"단일 버킷 정책이 너무 커지고 복잡해져서 사용자 그룹별로 전용 진입 주소와 정책을 분리하고 싶다"** $\rightarrow$ **S3 Access Points**
* **"S3에서 데이터를 검색(Retrieve)할 때 개인정보를 숨기거나 포맷을 변환하는 가공을 거치고 싶다"** $\rightarrow$ **S3 Object Lambda**
