# 🛡️ AWS 보안, 자격 증명 및 다중 계정 관리 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 서비스/기능 키워드 매칭

시험 문제에 아래 핵심 단어가 나오면 고민하지 말고 바로 해당 서비스를 고르세요!

* **Multi-Account 통합 결제 + OU(조직 단위) 계정 그룹화** $\rightarrow$ `AWS Organizations`
* **조직 내 전체 계정에 강제 적용 + 관리 계정 예외 + IAM 사용자/역할 최고 권한 제한** $\rightarrow$ `SCP (서비스 제어 정책)`
* **로그인한 클라이언트의 IP 대역 제한** $\rightarrow$ `aws:SourceIP (IAM 조건)`
* **특정 AWS 리전에서만 API 호출 허용/제한** $\rightarrow$ `aws:RequestedRegion (IAM 조건)`
* **모바일/웹 앱 사용자 + 소셜 로그인(구글/페이스북) + 사용자 데이터베이스 풀** $\rightarrow$ `Cognito 사용자 풀 (CUP)`
* **임시 AWS 자격 증명 발급 + AWS 리소스 직접 액세스 + DynamoDB 행 수준 보안(RLS)** $\rightarrow$ `Cognito 자격 증명 풀`
* **모든 조직 계정 단일 로그인(SSO) + 3rd Party 앱(Salesforce, Okta) 통합 + 권한 세트** $\rightarrow$ `AWS IAM Identity Center`
* **온프레미스 AD와 "신뢰 관계(Trust)" 구축 + 클라우드에 복제 배포** $\rightarrow$ `AWS Managed Microsoft AD`
* **온프레미스 AD에 자격 증명 요청 우회/전송 (프록시 역할) + 로컬 관리 전용** $\rightarrow$ `AD Connector`
* **Microsoft 소프트웨어 불필요 + 저렴한 독립형 AD 호환 관리 디렉터리** $\rightarrow$ `Simple AD`
* **개발자 권한 위임 + 자체 권한 상승(Admin 획득) 방지 + 최대 허용 한도 설정** $\rightarrow$ `IAM Permission Boundary (권한 경계)`
* **AWS 모범 사례 멀티 계정 랜딩 존 + SCP로 예방 + Config로 탐지적 가드레일** $\rightarrow$ `AWS Control Tower`

## 📑 2단계: 서비스 및 아키텍처 개념 100% 디테일 통합

### 🏢 1. AWS Organizations & SCP

여러 개의 AWS 계정을 중앙에서 통합 관리하는 **글로벌 서비스**입니다.

* **계정 아키텍처:** 조직을 생성한 메인 계정이 관리 계정(Management Account)이 되며, 귀속된 다른 계정들은 **멤버 계정**이 됩니다. 멤버 계정은 오직 **단 하나의 조직**에만 속할 수 있습니다.
* **비용 최적화:** 모든 계정의 비용을 관리 계정의 지불 방법 하나로 묶는 통합 결제(Consolidated Billing)를 지원합니다. 조직 내 모든 계정의 누적 사용량을 집계하여 볼륨 할인을 받으며, 계정 간 예약 인스턴스(RI)와 **Savings Plans** 할인이 자동으로 공유됩니다.
* **자동화 및 운영 장점:** API를 통해 AWS 계정 생성을 자동화할 수 있습니다. 단일 계정 내에 수많은 VPC를 쪼개어 쓰는 것보다, 계정 자체를 분리하는 것이 보안적 독립성이 훨씬 뛰어납니다. 청구 목적의 태그 기준을 전사 적용할 수 있으며, 한 번의 설정으로 모든 계정의 `CloudTrail` 및 `CloudWatch Logs`를 중앙 S3/로깅 계정으로 강제 전송할 수 있습니다. 관리 목적으로 계정 간 교차 계정(Cross-Account) 역할을 손쉽게 설정합니다.

#### 🛡️ 서비스 제어 정책 (Service Control Policies, SCP)

* **작동 원리:** 조직 단위(OU) 또는 개별 계정에 적용되어 해당 환경 내의 모든 IAM 사용자, 그룹, 역할이 수행할 수 있는 **최대 권한 한도**를 제한하는 정책입니다.
* **⚠️ 예외 사항:** **SCP는 관리 계정(Management Account)에는 절대로 적용되지 않습니다.** 관리 계정은 언제나 모든 작업을 수행할 수 있는 무소불위의 권한을 유지합니다.
* **평가 로직:** IAM과 동일하게 **기본적으로 모두 거부(Default Deny)** 상태이므로, 사용하고자 하는 서비스에 대한 명시적 허용(Allow)이 반드시 존재해야 통과됩니다.
* **전략 2가지:**
  ![](https://velog.velcdn.com/images/gagaeun/post/e15c7edd-e3ab-432c-bd14-0eaad0a09ac7/image.png)
1. **Blocklist (차단 목록):** 기본적으로 모든 것을 허용(`*`)해 두고, 특정 유해한 액션(예: 루트 권한 변경, 특정 리전 외 자원 생성)만 명시적으로 거부(`Deny`)하는 전략입니다.
2. **Allowlist (허용 목록):** 허용할 핵심 서비스만 명시적으로 지정하고 나머지는 철저히 차단하는 폐쇄적인 전략입니다.

### 🔑 2. IAM 고급 기능 및 정책 평가 로직

#### 🔍 IAM Conditions (조건부 제어)

IAM 정책 문장(Statement) 내부에서 특정 상황일 때만 정책이 발동하도록 제어하는 핵심 속성입니다. 사용자 정책, S3 버킷 리소스 정책, VPC 엔드포인트 정책 등 모든 정책에 삽입할 수 있습니다.

* **`aws:SourceIP`:** API 호출을 시도하는 클라이언트의 소스 IP 주소를 검증합니다. 지정된 특정 IP 주소 대역(CIDR) 범위 외에서 들어오는 모든 API 요청을 명시적으로 거부(`Deny`)할 때 주로 사용합니다.
  ![](https://velog.velcdn.com/images/gagaeun/post/af9f202f-b587-4c3a-82a3-c34f03887e75/image.png)
* **`aws:RequestedRegion`:** AWS의 글로벌 조건 키로, API 호출이 발생하는 **대상 AWS 리전**을 제한합니다. (예: `eu-central-1`, `eu-west-1` 리전 이외에서 발생하는 EC2, RDS, DynamoDB 생성 호출을 원천 차단)
  ![](https://velog.velcdn.com/images/gagaeun/post/53c2ded8-4818-4427-acd2-3ee5ac7abcd0/image.png)
* **`ec2:ResourceTag` 및 `aws:PrincipalTag`:** 속성 기반 액세스 제어(ABAC)의 핵심입니다.
  ![](https://velog.velcdn.com/images/gagaeun/post/b9a8bea6-3a9c-425a-ab60-d7c7675de459/image.png)
* `ec2:ResourceTag/Project`: 조작하려는 대상 EC2 인스턴스의 태그 키 `Project` 값이 `DataAnalytics`와 일치할 때만 인스턴스 시작/종료를 허용합니다.
* `aws:PrincipalTag/Department`: 요청을 보내는 주체(IAM 사용자/역할)의 태그 키 `Department` 값이 `Data`인 경우에만 해당 작업을 수행할 수 있도록 상호 매칭 검증을 수행합니다.


* **`aws:MultiFactorAuthPresent`:** 멀티팩터 인증(MFA) 여부를 검증합니다. 이 조건이 `false`로 평가되면 EC2 인스턴스 중지 및 종료와 같은 고위험 작업을 명시적으로 거부하여 보안을 강화합니다.
  ![](https://velog.velcdn.com/images/gagaeun/post/5c9c3718-8794-481b-97b0-c37387cd88e5/image.png)


#### 🪣 S3 전용 IAM 권한 타겟 지정 규칙

S3 보안 정책 작성 시 ARN(Amazon Resource Name)을 지정하는 수준을 명확히 분리해야 합니다. 형식은 `arn:partition:service:region:account-id:resource`를 따릅니다.

* **버킷 수준 권한:** `s3:ListBucket`, `s3:GetBucketLocation` 등 버킷 자체를 조회하는 액션은 버킷 고유 ARN인 `arn:aws:s3:::test` 형식에 매핑해야 합니다.
* **객체 수준 권한:** `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject` 등 버킷 내부의 파일(객체)을 조작하는 액션은 반드시 뒤에 와일드카드를 붙여 `arn:aws:s3:::test/*` 형태로 객체 수준 경로를 지정해야 정상 동작합니다.

#### 🤝 교차 계정 액세스: IAM 역할(Role) vs 리소스 기반 정책 (★시험 단골 분석)

다른 AWS 계정(Account A)의 사용자가 내 계정(Account B)의 S3 버킷에 접근하도록 구성하는 방법은 크게 두 가지로 나뉩니다.

| 비교 항목 | 1. IAM 역할(Role) 수임 방식 | 2. 리소스 기반 정책(Resource-based) 방식 |
| --- | --- | --- |
| **작동 아키텍처** | 사용자가 Account B의 IAM 역할을 **수임(AssumeRole)**하여 권한을 획득함 | Account B의 S3 버킷에 **버킷 정책**을 붙여 Account A 사용자를 직접 허용함 |
| **자신의 권한 유지 여부** | 역할을 맡는 순간 **기존 자신의 권한을 완전히 포기**하고 수임한 역할 권한만 행사함 | 역할을 전환하지 않으므로 **자신의 기존 권한을 그대로 유지**한 채 대상 리소스에 접근함 |
| **추천 사용 사례** | 타 계정 자원을 장시간 조작하거나 대단위 관리 작업을 대행할 때 | 계정 A의 유저가 본인 계정 DynamoDB를 스캔하면서 동시에 계정 B S3 버킷에 파일을 업로드할 때 |

* **리소스 기반 정책 지원 자원:** Amazon S3 버킷, Amazon SNS 토픽, Amazon SQS 큐 등 주체가 권한을 유지하며 통제할 수 있도록 다양한 자원에서 이를 지원합니다.
* **`aws:PrincipalOrgID` 조건 키:** 리소스 기반 정책 내부에 이 조건을 삽입하면, 특정 계정 ID를 일일이 적지 않고도 **우리 AWS Organizations 조직 내에 소속된 멤버 계정들만** 안전하게 리소스에 접근하도록 글로벌 통제가 가능해집니다.

#### 🔀 Amazon EventBridge 대상 권한 매커니즘

이벤트 규칙이 트리거되어 대상(Target)으로 이벤트를 보낼 때, 대상 서비스 종류에 따라 권한 부여 방식이 철저히 양분됩니다.

* **리소스 기반 정책 사용 대상:** `Lambda`, `SNS`, `SQS`, `CloudWatch Logs`, `API Gateway` 등으로 이벤트를 보낼 때는 해당 대상 리소스 측 정책에서 EventBridge의 접근을 허용해 주어야 합니다.
* **IAM 역할(Role) 사용 대상:** `Kinesis Data Streams`, `Systems Manager Run Command`, `ECS Task` 등으로 이벤트를 보낼 때는 EventBridge 서비스가 해당 역할을 수임하여 자원을 실행할 수 있도록 IAM 세팅이 필요합니다.
* **✨ 한 줄 요약:** SNS/SQS/Lambda는 **리소스 기반 정책**, Kinesis는 **IAM 역할**을 사용합니다!

#### 🚧 IAM Permission Boundaries (권한 경계)

사용자와 역할(Role)에만 적용할 수 있으며 **그룹(Group)에는 적용할 수 없는** 고급 권한 통제 기능입니다. 관리형 정책을 사용하여 해당 주체가 가질 수 있는 '최대 허용 한도(Max Privileges)'를 강제로 설정합니다.

* **권한 평가 메커니즘:** 예를 들어 권한 경계 정책에는 `S3, CloudWatch, EC2`만 허용되어 있습니다. 그런데 이 사용자에게 일반 IAM 정책으로 `iam:CreateUser` 권한을 부여했습니다. 이 경우 사용자는 다른 사용자를 생성할 수 **없습니다.** 실제 권한은 **[IAM 정책]과 [권한 경계 정책]의 교집합(공통 분모)** 영역만 최종 승인되기 때문입니다. 경계선 밖에 있는 `iam:CreateUser`는 무력화됩니다.
* **주요 사용 사례 (권한 위임):** 루트 관리자가 아닌 일반 개발자 팀장에게 새 IAM 사용자를 생성할 수 있는 권한을 위임할 때 사용합니다. 개발자가 자신이나 하위 사용자의 정책을 직접 커스텀하게 할당할 수 있게 허용하면서도, 권한 경계(Boundary)를 미리 쳐두면 스스로에게 Administrator(관리자) 권한을 부여하는 **"권한 상승(Privilege Escalation)"을 완벽하게 예방**할 수 있습니다.

#### 📊 IAM 최종 권한 평가 로직 (Evaluation Logic)

AWS의 모든 정책 평가 프로세스는 아래의 철저한 계층 구조를 따르며, 단 한 곳에서라도 명시적 거부(Explicit Deny)가 발생하면 다른 모든 허용을 무시하고 최종 거부 처리됩니다.

$$\text{기본 거부 (Default Deny)} \longrightarrow \text{SCP 평가} \longrightarrow \text{리소스 기반 정책 평가} \longrightarrow \text{IAM 권한 경계 평가} \longrightarrow \text{일반 IAM 정책 평가} \longrightarrow \text{최종 허용 결정}$$

* **분석 예시 (SQS 정책 검증):**
* *질문 1:* `sqs:CreateQueue` 액션을 실행할 수 있는가? $\rightarrow$ **No!** 정책 하단에 `sqs:*`에 대한 명시적 거부(`Deny`) 블록이 존재하므로 해당 와일드카드 내부에 포함되는 Create 액션은 차단됩니다.
* *질문 2:* 다른 문장에 `sqs:DeleteQueue`에 대한 명시적 허용(`Allow`)이 있고, 동시에 `sqs:*`에 대한 명시적 거부(`Deny`)가 있다면 실행 가능한가? $\rightarrow$ **No!** 명시적 거부가 허용보다 무조건 최우선 순위로 작동하여 자동 거부됩니다.
* *질문 3:* 정책 내부에 EC2 관련 언급이 전혀 없을 때 `ec2:DescribeInstances`를 실행할 수 있는가? $\rightarrow$ **No!** 명시적 거부는 없지만, 명시적 허용(`Allow`) 또한 존재하지 않으므로 기본 거부(Default Deny) 원칙에 의해 실행이 불가능합니다.

### 👤 3. Amazon Cognito (고객용 인증 인프라)

웹 및 모바일 애플리케이션의 일반 외부 사용자(고객)들의 가입, 로그인 및 자격 증명 제어를 담당하는 완전 관리형 서버리스 서비스입니다. 수백만 명의 모바일 사용자, 외부 소셜 로그인 연동 등의 키워드가 나오면 내부 임직원용인 IAM이 아니라 무조건 Cognito가 정답입니다.

#### 🔐 1. Cognito 사용자 풀 (Cognito User Pools, CUP)

* **개념:** 애플리케이션 사용자를 위한 서버리스 신원 데이터베이스(장부)를 생성하는 기능입니다.
* **기능:** 사용자 이름/이메일/비밀번호 기반 로그인, 비밀번호 재설정 프로세스 구축, 이메일 및 전화번호 소유권 검증, 사용자 다중인증(MFA) 활성화를 제공합니다. Facebook, Google, Apple 등 소셜 로그인 및 기업 연동용 **SAML 2.0 / OIDC** 신원 공급자 연동을 전면 지원합니다.
* **인프라 통합 패턴:**
* **with API Gateway:** 사용자가 먼저 CUP에 로그인하여 토큰(JWT)을 발급받은 뒤, API Gateway로 요청을 보낼 때 이 토큰을 제출합니다. API Gateway가 토큰 유효성을 자체 검증한 후 자격 증명 정보로 변환하여 백엔드 Lambda 함수로 안전하게 넘겨줍니다.
* **with Application Load Balancer (ALB):** ALB 리스너 규칙 상단에 Cognito 사용자 풀 인증 레이어를 배치합니다. 사용자가 접근하면 ALB가 CUP와 통신하여 유효 로그인 여부를 판단하고, 로그인 성공 시 백엔드 타겟 그룹 인스턴스로 요청을 리다이렉트하며 사용자의 자격 증명 메타데이터를 **HTTP 추가 헤더**에 담아 전송합니다.
* *효과:* 백엔드 어플리케이션 소스코드 내부에서 복잡한 로그인 검증 로직을 직접 코딩할 필요 없이, 인프라 진입점(API Gateway, ALB)에서 검증 책임을 100% 전담 처리합니다.



#### 🔑 2. Cognito 자격 증명 풀 (Cognito Identity Pools / Federated Identities)

* **개념:** 로그인에 성공한 사용자들에게 실제 AWS 임시 자격 증명(IAM Key)을 발급하여, 사용자가 중간 서버를 거치지 않고 S3 버킷이나 DynamoDB 같은 AWS 리소스에 **직접(Direct) 안전하게 액세스**할 수 있도록 통로를 열어주는 기능입니다.
* **자격 증명 소스:** 앞서 설명한 `Cognito 사용자 풀(CUP)`을 자격 증명 풀의 신원 공급자로 매핑하여 결합해 쓰거나, 외부 타사 소셜 로그인을 직접 소스로 삼을 수 있습니다.
* **권한 통제:** 자격 증명 풀에 임시 연결되는 IAM 정책은 Cognito 내부에서 사전에 정의됩니다. 로그인에 성공한 '인증된 사용자'와 로그인하지 않은 '게스트(비인증) 사용자'별로 각각 기본 IAM 역할을 다르게 상속하도록 설계할 수 있습니다. `user_id` 컨텍스트를 활용해 개별 사용자별로 극도로 세분화된 리소스 제어가 가능합니다.
* **⚡ DynamoDB 행 수준 보안 (Row-Level Security, RLS):** * 자격 증명 풀과 결합하여 DynamoDB 정책 내부에 `조회 조건(Condition)`을 삽입할 수 있습니다.
* 조건절 내부의 **`dynamodb:LeadingKey` 값이 현재 로그인한 사용자의 `Cognito 자격 증명 사용자 ID`와 일치해야만 데이터 Read/Write를 허용**하도록 제한하는 고도의 보안 기법입니다.
* 이를 통해 사용자는 DynamoDB 테이블의 전체 데이터가 아니라, 오직 **자기 자신에게 귀속된 행(Row) 데이터에만 철저히 격리되어 접근**할 수 있게 됩니다.

### 🆔 4. AWS IAM Identity Center (임직원용 SSO)

구 AWS Single Sign-On(SSO)의 차세대 후속 제품으로, 기업 내부 임직원 및 개발자들의 다중 계정 인프라 접근을 단 한 번의 로그인으로 일괄 통합 관리하는 서비스입니다.

* **단일 로그인(SSO) 대상 리소스:**
1. AWS Organizations 구조 내에 존재하는 모든 개별 AWS 계정 콘솔 및 CLI 접근
2. 외부 비즈니스 사스(SaaS) 클라우드 애플리케이션 (Salesforce, Box, Microsoft 365 등)
3. SAML 2.0 표준 팩터를 지원하는 모든 커스텀 사내 어플리케이션
4. EC2 Windows 인스턴스 원격 로그인


* **신원 데이터베이스 연동 (Identity Providers):** Identity Center 자체 내장 저장소를 쓸 수도 있고, 기존 기업들이 온프레미스나 타 클라우드에서 운영 중이던 서드파티 ID 공급자(Active Directory, OneLogin, Okta 등)와 네이티브하게 동기화 연동할 수 있습니다.
* **중앙 집중형 권한 제어 (★최다 출제 개념):**
* 임직원이 통합 포털에 한 번 로그인했다고 해서 모든 계정의 모든 자원에 마음대로 들어갈 수 있는 것은 아닙니다.
* 관리자는 자격 증명 센터 내부에 하나 이상의 IAM 정책들을 묶어둔 '권한 세트 (Permission Sets)'를 정의한 후, "인사팀 그룹은 A 계정의 ReadOnly 권한 세트를 매핑", "DevOps팀 그룹은 전체 계정의 Administrator 권한 세트를 매핑"하는 방식으로 다중 계정 권한(Multi-Account Permissions)과 애플리케이션 할당을 정교하게 제어합니다.


* **🎨 속성 기반 액세스 제어 (ABAC):** 사용자의 프로필 속성 데이터(예: 비용 센터 `cost center`, 직책 `title`, 지역 `locale`)를 권한 세트 정책과 결합합니다. 권한 세트 양식을 단 한 번만 정의해 두면, 사용자의 직책이나 부서 속성이 변경됨에 따라 AWS 내 자원 접근 권한이 유연하고 정밀하게 자동 수정 및 격리 조치됩니다.

### 🌲 5. AWS Directory Services (디렉터리 인프라 엔지니어링)

기업의 자원 관리 장부인 Microsoft Active Directory(AD) 환경을 AWS와 융합하는 서비스 라인업입니다.

* **🏛️ AWS Managed Microsoft AD:**
* AWS 클라우드 내부에 실제 Windows 서버 기반의 자체 관리형 Microsoft AD를 자동으로 생성하고 가용성을 관리해 주는 서비스입니다. 멀티팩터 인증(MFA)을 기본 지원합니다.
* **신뢰 관계 (Trust Relationship):** 온프레미스 인프라에 이미 존재하는 기존 AD와 클라우드의 Managed AD 간에 **양방향 또는 단방향 신뢰 관계**를 설정할 수 있습니다. 이를 통해 클라우드로 자원을 확장하면서도 사용자의 계정 신원 정보를 유기적으로 공유 및 통합 관리할 수 있게 됩니다.


* **🔌 AD Connector (디렉터리 게이트웨이 / 프록시):**
* 클라우드 내부에 별도의 계정 데이터베이스나 Windows 서버를 일절 생성하지 않습니다.
* 오직 인증 요청이 들어오면 이를 온프레미스 Active Directory로 그대로 토스(전송)해 주는 **실시간 프록시(우회 통로) 역할**만 수행합니다. 멀티팩터 인증(MFA)을 지원합니다.
* 모든 사용자 데이터 관리와 자격 증명 처리는 철저히 온프레미스 AD 본진에서만 이루어지기를 원하는 아키텍처에 완벽히 부합합니다. (단, 매번 온프레미스로 네트워크 전송이 일어나므로 지연 시간이 다소 늘어날 수 있습니다.)


* **🍃 Simple AD:**
* Microsoft AD 실제 소프트웨어를 전혀 사용하지 않는, **Samba 4 기반의 AD 호환 관리형 디렉터리**입니다.
* 온프레미스 인프라가 전혀 존재하지 않는 상태에서, 가볍고 저렴하게 중앙 집중식 Windows 자원 관리 및 Linux 사용자 계정 관리를 클라우드 단독형으로 구현하고 싶을 때 최적의 가성비 선택지입니다. 온프레미스 AD와의 신뢰 관계 결합은 지원하지 않습니다.

### 🏰 6. AWS Control Tower (거버넌스 자동화 탑)

AWS가 제안하는 다중 계정 모범 사례 환경인 '랜딩 존(Landing Zone)'을 단 몇 번의 마우스 클릭만으로 안전하고 규정에 준수하게 자동 구축·유지 관리해 주는 최상위 거버넌스 오케스트레이션 서비스입니다. 내부적으로 **AWS Organizations** 엔진을 활용하여 신규 계정 생성 및 관리를 수행합니다.

* **주요 핵심 이점:** 계정 생성 및 보안 기준 인프라 배포 환경 설정을 완전히 자동화하며, 강력한 **가드레일(Guardrails)** 기능을 배치하여 조직 전반의 보안 정책 관리 자동화, 비정상 정책 위반 실시간 감지, 대화형 중앙 대시보드를 통한 시각적 규정 준수 모니터링을 상시 수행합니다.

#### 🚧 거버넌스를 통제하는 가드레일 2대장 메커니즘 (★100% 출제)

Control Tower 내에 소속된 모든 하위 계정들의 탈선을 막기 위해 배치되는 두 가지 성격의 제어 장치입니다.

1. **예방적 가드레일 (Preventive Guardrail):**
* **구현 기술:** AWS Organizations의 SCP(서비스 제어 정책)를 백엔드에서 생성하여 적용합니다.
* **동작 방식:** 규정을 위반하는 행위 자체를 물리적으로 사전 차단합니다. (예: "우리 조직 내 모든 계정은 서울 리전 외에 다른 리전에는 어떠한 자원도 생성할 수 없다"는 리전 제한 SCP를 박아두어 정책 위반을 사전에 예방)


2. **탐지적 가드레일 (Detective Guardrail):**
* **구현 기술:** **AWS Config Rules** 감사를 백엔드에서 구동하여 구현합니다.
* **동작 방식:** 사용자가 자원을 생성하는 것을 막지는 않지만, 생성된 자원의 내부 스펙을 실시간 감시하여 규정에 어긋나면 즉시 경고를 날리고 대시보드에 마킹합니다. (예: "조직 내 배포된 모든 EBS 디스크나 S3 버킷 중 필수 태그가 누락된 불량 리소스를 실시간 추적하여 식별"하는 감사 수행)
