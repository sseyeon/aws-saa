# 1. Amazon CloudFront (CDN)
## 1-1. CloudFront 기본 개념: 넷플릭스 서울 지사 이야기

만약 넷플릭스 본사 서버가 미국에만 있다면, 한국 유저가 불금에 영상을 클릭할 때 태평양 바다 건너 미국까지 데이터가 왔다 와야 하므로 버퍼링 지옥이 펼쳐집니다.

이를 해결하기 위해 전 세계 주요 도시마다 미니 보관소인 엣지 로케이션(Edge Location)을 차려두고, 인기 있는 영상들을 미리 복사(**캐싱, Caching**)해 두는 서비스가 바로 **CloudFront**입니다.

* **창고에 물건이 있다면 (Cache Hit):** 미국 본사 안 거치고 서울 엣지에서 0.001초 만에 바로 영상을 꺼내 줍니다.
* **창고가 비어있다면 (Cache Miss):** 서울 엣지가 미국 원본(오리진)으로 쏜살같이 달려가서 파일을 긁어와 자기 창고에 저장(로컬 캐시)한 뒤 유저에게 줍니다. 다음 유저부터는 광속으로 받아보겠죠?
* **덤으로 얻는 방패 효과:** 트래픽 요청이 전 세계 엣지로 분산되니 대규모 디도스(DDoS) 공격을 받아도 원본 서버가 터지지 않습니다. 입구에 AWS Shield나 WAF(방화벽)를 붙여 악성 요청을 입구 컷하기 딱 좋은 구조가 됩니다.

---

## 1-2. Origins (원본 소스): "창고가 비었을 때 어디서 떼어올래?"

엣지 창고에 물건이 없을 때, 진짜 오리지널 데이터를 공급해 주는 본사 역할을 지정하는 단계입니다.

### ① S3 버킷 원본 (양동이 창고)

* 사진, 영상, 정적 파일(HTML/CSS)을 전 세계에 뿌릴 때 씁니다.
* **OAC (Origin Access Control):** 유저들이 S3 주소로 직접 들어가서 파일을 훔쳐 가면 안 되기 때문에, S3 문을 완전히 닫아걸고 "오직 CloudFront가 발행한 출입증(OAC)"을 들고 오는 요청만 S3가 열어주도록 철통 보안을 칩니다. (과거 기술인 OAI는 이제 폐기되고 OAC가 대세!)
* **Ingress (역주행 업로드):** 맨날 다운로드만 하는 게 아니라, CloudFront의 전 세계 고속도로망을 거꾸로 타고 들어가서 S3 버킷으로 파일을 빠르게 업로드할 때도 쓰입니다.

### ② Custom Origin (내가 만든 HTTP 서버 원본)

S3가 아니라, 동적인 연산을 하는 진짜 웹 서버를 원본으로 지정하는 방식입니다.

* **ALB(로드 밸런서)를 원본으로 둘 때:** 외부 창고(CloudFront)가 인터넷을 타고 노크해야 하므로 **ALB 자체는 퍼블릭(Public)** 영역에 노출되어야 합니다. 하지만 그 뒤에서 진짜 일하는 EC2 컴퓨터들은 안전하게 **프라이빗(Private)** 서브넷에 꼭꼭 숨겨둘 수 있어 실무에서 가장 애용합니다.
* **EC2 직접 원본일 때:** 중간에 중계해 줄 로드 밸런서가 없기 때문에 CloudFront 엣지가 EC2를 직접 찔러야 합니다. 따라서 **EC2도 퍼블릭** 상태여야만 CloudFront가 주소를 찾아갈 수 있습니다.
* 이때 아무나 내 EC2에 직접 들어오면 안 되므로, EC2 보안그룹(SG)에 "오직 AWS가 공식 제공하는 CloudFront IP 목록"만 들어올 수 있게 화이트리스트 문을 잠가야 합니다.
* **S3 웹사이트:** S3의 '정적 웹사이트 호스팅' 기능을 켠 버킷은 일반 S3가 아니라 HTTP 주소가 나오는 웹 서버 취급을 받기 때문에 커스텀 오리진으로 분류됩니다.

---

## 1-3. 라이벌 비교: CloudFront vs S3 교차 리전 복제(CRR)

둘 다 "데이터를 다른 지역으로 보낸다"는 개념이라 시험에서 함정으로 자주 나옵니다.

* **CloudFront (대여소 캐싱):** 전 세계 200개가 넘는 모든 엣지에 파일을 '잠깐 복사'해 둡니다. 지정한 유효기간(TTL, 보통 하루)이 지나면 파일은 알아서 사라집니다. **전 세계 대상을 상대로 한 정적 콘텐츠 전송 최적화**에 완벽합니다.
* **S3 CRR (진짜 이삿짐):** 미국 S3 양동이에 있는 진짜 원본 파일을 한국 S3 양동이로 '실시간 영구 복제'를 하는 것입니다. 캐싱이 아니라 진짜 파일이 이사 가는 개념이라, 특정 리전 유저들에게 낮은 지연 시간으로 동적 데이터를 주거나 재해 복구(백업)를 할 때 씁니다.

---

## 1-4. Geo Restriction (국가별 차단 자물쇠)

* **원리:** 유저가 접속한 IP 주소를 기반으로 어느 국가인지 판별하는 서드파티 Geo-IP DB를 활용합니다.
* **Allowlist:** "한국, 미국 유저만 들어와!" (지정한 국가만 허용)
* **Blocklist:** "특정 국가 유저들은 차단해!" (지정한 국가만 막음)
* **실무 예시:** 방송 저작권 법률 때문에 **"이 영상은 저작권 계약상 특정 국가 안에서만 재생되게 제한해 주세요"** 하는 요구사항이 나올 때 무조건 이 기능을 켭니다.

---

## 1-5. Pricing & Price Classes (돈에 맞춰 창고 줄이기)

인프라 회선 비용은 전 세계가 다 다릅니다. **북미나 유럽은 회선 가격이 저렴하지만, 남미나 아시아 쪽 엣지는 운영비가 훨씬 비쌉니다.** 그래서 AWS는 예산에 맞춰 엣지 범위를 줄일 수 있는 가성비 옵션을 줍니다.

* **요금 계층 전체 (Price Class All):** "돈은 상관없으니 전 세계 모든 엣지 다 켜줘! 전 세계 유저 모두에게 최상의 속도를 주겠다!" (비용 최고)
* **요금 계층 200:** "대부분 지역은 다 켜되, 남미처럼 터무니없이 제일 비싼 리전 몇 개만 쏙 빼줘." (가성비 세팅)
* **요금 계층 100:** "예산이 진짜 타이트해. 전 세계에서 **가장 저렴한 북미와 유럽 엣지**만 켜줘." (비용 최저, 아시아 유저들은 약간의 지연 감수)

---

## 1-6. Cache Invalidations (강제로 캐시 불태우기)

실무에서 배포 직후 "어? 왜 반영이 안 되지?" 할 때 100% 이 문제 때문입니다.

* **문제 상황:** 웹사이트 메인 로고 이미지(`logo.png`)를 새로 수정해서 S3 원본에 올렸습니다. 그런데 유저들이 자꾸 옛날 디자인이 보인다고 항의합니다.
* **이유:** 서울 엣지 로케이션 창고가 **"나 이 파일 유효기간(TTL) 하루로 설정돼서 내일까지는 본사 안 가고 내 창고에 있는 옛날 파일 그냥 줄 거야"** 하고 버티고 있기 때문입니다.
* **해결책 (Invalidation):** TTL 만료까지 마냥 기다릴 수 없으니, CloudFront에 가서 **"야! `/images/*` 경로에 있는 캐시 지금 당장 강제로 싹 다 불태워 버려!(Invalidation)"** 하고 명령을 내리는 것입니다. 그럼 창고가 깨끗이 비워지고, 다음 유저가 올 때 S3 원본에서 따끈따끈한 새 로고를 새로 받아와서 보여주게 됩니다.

---

### 🎯 한눈에 외우는 SAA 시험 정답 매칭 공식

1. **S3 보안 우회 차단** ➔ OAC
2. **ALB 뒤로 숨기기** ➔ Public ALB + Private EC2
3. **특정 국가 저작권 제어** ➔ Geo Restriction
4. **가장 저렴한 북미/유럽만 사용** ➔ Price Class 100
5. **수정사항 즉시 반영** ➔ Cache Invalidation

전체 인프라 흐름과 실무적인 유기성이 한 눈에 들어오시나요? 이 흐름을 머릿속에 얹고 기출문제를 보시면 지문이 훨씬 직관적으로 읽히실 겁니다!

# 2. AWS Global Accelerator
이번엔 **AWS Global Accelerator**군요! CloudFront와 찰떡궁합 라이벌이자, SAA 시험에서 네트워크 속도와 가용성(안정성)을 다룰 때 무조건 정답으로 튀어나오는 서비스입니다.

이것도 노트에 적어두신 뼈대 그대로, 아주 직관적이고 생생한 비유를 얹어서 머리에 콕 박히게 정리해 드릴게요!

---

## 2-1. Global Accelerator가 필요한 이유: 일반 인터넷 고속도로의 비극

만약 한국 리전에 게임 서버를 열었는데, 미국이나 유럽에 사는 유저가 접속하려고 합니다.

* **일반 공용 인터넷(Public Internet):** 유저의 요청이 수많은 국가의 통신사 라우터(Hops)를 거치며 핑퐁을 칩니다. 이 과정에서 병목현상이 생기고 핑(Ping)이 튀며 지연 시간(Latency)이 폭발합니다.
* **해결책:** 유저 집 근처에 있는 AWS 엣지 로케이션까지만 공용 인터넷을 타게 하고, 그 뒤부터 우리 서버(한국)까지는 전용선인 'AWS 사설 글로벌 네트워크'를 타고 초고속으로 달리게 만드는 것입니다. 이 우주선 전용 차선을 깔아주는 서비스가 바로 **Global Accelerator**입니다.

---

## 2-2. Unicast IP vs Anycast IP (주소 찾기 혁명)

Global Accelerator의 마법은 **애니캐스트(Anycast)** 주소에서 시작됩니다.

* **유니캐스트(Unicast) - "주소는 단 하나, 찾아오려면 고생해":** 서버 하나당 고유 IP 하나를 가집니다. 미국 유저도, 한국 유저도 무조건 한국에 있는 그 IP 주소 하나만 바라보고 전 세계 인터넷 망을 헤매며 찾아와야 합니다.
* **애니캐스트(Anycast) - "어디서나 부르면 가장 가까운 분신이 대답해":** **전 세계 모든 엣지 로케이션이 완전히 동일한 IP 주소**를 가집니다.
* 예컨대 AWS가 딱 2개의 고정 애니캐스트 IP(`1.2.3.4`, `5.6.7.8`)를 주면, 미국 유저가 `1.2.3.4`를 쳤을 때 미국 엣지로 가고, 한국 유저가 치면 서울 엣지로 자동으로 꽂힙니다. 최단 거리 라우팅의 끝판왕이죠.



---

## 2-3. 작동 원리와 핵심 장점: 1분 만에 끝나는 재해 복구

Global Accelerator를 켜면 딱 2개의 고정 애니캐스트 IP가 발급됩니다.

1. **지름길 라우팅:** 전 세계 유저가 이 IP로 접속하면 가장 가까운 엣지 로케이션으로 순간 이동합니다. 그 후, 엣지에서 진짜 서버(ALB, NLB, EC2 등)까지는 낙후된 공용 인터넷이 아니라 **안정적이고 광속인 AWS 사설망**을 타고 이동합니다.
2. **캐싱 트러블 제로:** DNS 기반 라우팅(Route 53 등)은 서버 주소가 바뀌면 전 세계 통신사 캐시에 옛날 주소가 남아있어 반영되는 데 시간이 걸립니다. 하지만 Global Accelerator는 **내 IP 주소가 평생 고정**이므로 클라이언트 캐시 문제로 먹통이 될 일이 전혀 없습니다.
3. **지능형 헬스 체크 & 1분 장애 조치(Failover):** * 한국 리전과 미국 리전 두 곳에 서버를 띄워놨다고 해봅시다. Global Accelerator가 이 서버들을 감시(Health Check)하다가, 한국 서버가 폭발하면 **1분 안에 전 세계 트래픽을 살아있는 미국 서버로 자동 우회**시킵니다. 재해 복구(DR) 시나리오의 핵심 치트키입니다.
4. **보안성 향상:** 방화벽을 칠 때 수많은 IP를 등록할 필요 없이, 외부에는 딱 이 **2개의 고정 외부 IP만 화이트리스트**로 열어두면 되니 인프라 문단속이 아주 깔끔해집니다. 당연히 초입에서 AWS Shield(디도스 방어)의 보호를 받습니다.

---

## 2-4. 라이벌 매치: CloudFront vs Global Accelerator

둘 다 "전 세계 AWS 엣지 로케이션을 쓴다", "디도스를 막아준다"는 공통점이 있어서 SAA 시험 단골 함정 카드로 나옵니다. 확실하게 갈라쳐 드릴게요.

[Image comparing AWS CloudFront for caching static content at the Edge and AWS Global Accelerator for proxying TCP/UDP traffic through the AWS network]

### 🍿 CloudFront (임시 창고형 가속)

* **핵심:** 캐싱(Caching)이 목적입니다.
* 이미지, 동영상, 정적 HTML처럼 엣지 창고에 복사해 두고 유저한테 바로 던져줄 수 있는 콘텐츠에 최적화되어 있습니다.
* 물론 API 같은 동적 콘텐츠도 가속해 주지만, 기본 본질은 "엣지에서 콘텐츠를 직접 꺼내 준다"입니다. (HTTP/HTTPS 프로토콜 위주)

### 🏎️ Global Accelerator (고속도로 패스형 가속)

* **핵심:** 캐싱 그딴 거 모릅니다. "순간 이동 프록시(Proxy)"가 목적입니다.
* 모든 요청을 캐싱 없이 **100% 실시간으로 원본 서버에 토스**합니다. 대신 통로를 사설망으로 초고속화해 주는 것입니다.
* **HTTP가 아닌 사용 사례(Non-HTTP)에 무조건 정답:** UDP 기반의 실시간 배틀로얄 게임, MQTT 프로토콜을 쓰는 사물인터넷(IoT) 장비, 실시간 VoIP 음성 통화 등 TCP/UDP 기반 애플리케이션 전체를 가속합니다.
* 주소가 절대로 바뀌면 안 되는 **고정 IP 요구사항**이 있거나, 리전이 터졌을 때 눈 깜짝할 새 장애 조치(Failover)가 필요할 때 선택합니다.

---

### 🎯 한눈에 외우는 SAA 시험 정답 매칭 공식

1. **"해외 유저가 한국 서버 접속하는데 공용 인터넷 홉(Hops) 때문에 너무 느리다"** ➔ **Global Accelerator**
2. **"HTTP가 아닌 UDP 기반 실시간 게임 / IoT 가속이 필요하다"** ➔ **Global Accelerator**
3. **"리전 장애 시 1분 미만의 신속한 재해 복구(Failover)가 필요하다"** ➔ **Global Accelerator**
4. **"보안상 전 세계에 딱 2개의 고정 IP(화이트리스트용)만 노출하고 싶다"** ➔ **Global Accelerator**
5. **"이미지나 동영상을 전 세계에 빠르게 뿌리고 싶다(캐싱)"** ➔ **CloudFront**
