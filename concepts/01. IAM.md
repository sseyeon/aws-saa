# 1. IAM
## 1-1. IAM 개요 및 Users & Groups (사용자와 그룹)

* **IAM의 정의:** AWS 리소스에 대한 인증(Authentication)과 권한 부여(Authorization)를 관리하는 **글로벌(Global) 서비스**입니다. 리전별로 따로 설정하지 않고 전 세계 모든 리전에 공통으로 적용됩니다.
* **루트 계정 (Root Account):** 계정을 처음 만들 때 사용한 이메일 계정입니다. 모든 권한을 가진 절대 권한자이므로, **생성 직후 결제나 특수 설정을 제외하고는 절대로 사용하거나 공유해서는 안 됩니다.** * **사용자 (Users):** 실제 조직 내의 '한 사람(물리적 사용자)'과 1:1로 매핑되는 개체입니다.
* **그룹 (Groups):** 사용자들의 묶음입니다. 핵심은 그룹 안에 '사용자'만 넣을 수 있고, '다른 그룹'은 포함할 수 없다(그룹 중첩 불가)는 점입니다.

### ✅ Permissions & 🏛️ Policies Structure (JSON 정책 구조)

사용자나 그룹이 무엇을 할 수 있는지 정의하는 JSON 문서입니다. AWS는 필요한 만큼의 권한만 주는 최소 권한의 원칙(Least Privilege Principle)을 고수합니다.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "EC2AndELBReadOnly",
            "Effect": "Allow",
            "Principal": "*",
            "Action": [
                "ec2:Describe*",
                "elasticloadbalancing:Describe*"
            ],
            "Resource": "*"
        }
    ]
}

```

* **Version:** 정책 언어의 버전으로, 보통 `"2012-10-17"`을 고정으로 씁니다.
* **Id:** 정책의 고유 식별자 (선택 사항).
* **Statement:** 권한을 서술하는 핵심 조항(배열 구조로 여러 개 작성 가능)이며 내부 요소는 다음과 같습니다:
* **Sid (Statement ID):** 각 문장을 구별하기 위한 ID (선택 사항).
* **Effect:** 권한을 줄지 말지 결정 (`Allow` 또는 `Deny`). **AWS는 명시적 Deny가 최우선**합니다.
* **Principal:** 이 정책을 적용받을 대상(사용자, 계정, 역할). 단, 사용자나 그룹에 *직접* 부착하는 정책(Identity-based)에서는 생략되는 경우가 많습니다.
* **Action:** 허용하거나 거부할 API 호출 목록 (예: `ec2:Describe*` -> EC2 조회 관련 모든 API).
* **Resource:** 조치가 적용될 AWS 리소스 목록 (`*`는 모든 리소스 의미).
* **Condition:** 이 정책이 발동할 조건 (예: 특정 IP 대역에서 요청할 때만 허용 등, 선택 사항).

## 1-2. 계정 잠금 장치: 비밀번호 정책 & MFA

### 🔐 비밀번호 정책 (Password Policy)

관리자는 IAM 사용자들의 비밀번호 규정을 강제하여 보안을 높일 수 있습니다.

* 최소 길이 설정, 복잡성 요구(대문자/소문자/숫자/특수문자 섞기).
* 사용자가 직접 비밀번호를 바꿀 수 있게 허용하거나 금지.
* 일정 기간(예: 90일) 후 만료시켜 새 비밀번호 설정 요구 및 과거 비밀번호 재사용 금지.

### 🔐🔐🔐 MFA (Multi-Factor Authentication, 다중 인증)

"내가 아는 것(비밀번호) + 내가 가진 것(보안 장치)"의 결합입니다. 루트 계정은 무조건 필수이며, 일반 IAM 사용자도 설정이 권장됩니다. 비밀번호가 해킹당해도 물리 장치가 없으면 로그인을 못 하므로 계정을 안전하게 보호합니다.

* **Virtual MFA device (가상 MFA 장치):** 하나의 스마트폰 앱에서 여러 개의 토큰을 지원하므로, 여러 AWS 계정이나 사용자를 등록할 수 있어 편리합니다. (**Google Authenticator, Authy** 등).
* **U2F (Universal 2nd Factor) Security Key:** 하나의 물리 보안 키로 여러 루트 계정과 IAM 사용자를 지원하는 하드웨어 키입니다. (**Yubikey** 등).
* **Hardware Key Fob MFA Device:** 일정한 시간마다 번호가 바뀌는 전통적인 OTP 기계 형태입니다. (**Gemalto** 제공).
* **AWS GovCloud 전용 하드웨어 키:** 미국 정부 전용 클라우드 환경을 위한 특수 하드웨어 키입니다. (**SurePassID** 제공).

## 1-3. 🖱️ AWS에 접근하는 3가지 방법 (Console, CLI, SDK)

| 접근 방식 | 보호 수단 | 특징 |
| --- | --- | --- |
| **Management Console** | 비밀번호 + MFA | 웹 브라우저를 통해 마우스로 시각적으로 조작하는 콘솔 |
| **AWS CLI** | Access Keys | 터미널(Command-line shell) 창에서 명령어로 조작 |
| **AWS SDK** | Access Keys | 개발자가 작성하는 소스코드(애플리케이션) 내부에 심어서 조작 |

### 🔑 액세스 키 (Access Keys)

CLI와 SDK로 AWS API에 다이렉트 인증을 요청할 때 사용하는 자격 증명입니다. 비밀번호와 똑같으므로 절대 깃허브 등에 유출하면 안 됩니다.

* **Access Key ID:** 아이디 역할 ($username$)
* **Secret Access Key:** 비밀번호 역할 ($password$)

### 📋 AWS CLI와 📦 AWS SDK의 차이

* **AWS CLI:** 터미널에서 `aws ec2 describe-instances` 같은 명령어를 치면 즉시 AWS API를 호출하여 리소스를 관리합니다. 스크립트(Shell 등)를 짜서 작업을 자동화할 때 유용합니다. (실제로 내부적으로는 파이썬용 SDK인 Boto 위에 구현되어 있습니다.)
* **AWS SDK:** 개발자가 코딩할 때 라이브러리 형태로 애플리케이션 소스코드에 포함시키는 것입니다. 프로그래밍 언어별로 따로 존재합니다. (JavaScript, Python, Java, Go, C++ 등 지원).

## 1-4. 💡 IAM 역할 (IAM Role) - 서비스용 신분증

* **정의:** 사람이 아니라 'AWS 서비스(EC2, Lambda 등)'에 권한을 주어 다른 AWS 리소스에 안전하게 접근하도록 만드는 가상 신분증입니다.
* **작동 방식:** EC2 내부 코드에 자격 증명(Access Key)을 절대 하드코딩하지 않습니다. 대신 'EC2 Instance Role'을 만들어서 EC2 인스턴스에 툭 부착해 주면, 인스턴스가 AWS로부터 아주 안전한 **단기 임시 자격 증명**을 발급받아 S3나 DynamoDB 같은 다른 서비스와 통신하게 됩니다.
* **대표 예시:** EC2 인스턴스 역할, Lambda 함수 역할, CloudFormation 실행 역할 등.

## 1-5. 🔒 IAM 보안 감사 도구 (SAA 시험 단골 비교 문제)

시험에서 계정 보안 상태를 점검하거나 불필요한 권한을 쳐내야 할 때, 아래 두 도구의 스코프(범위)를 정확히 비교해서 물어봅니다.

1. **IAM 자격 증명 보고서 (Credentials Report) ➔ 계정 전체(Account-level) 통계**
* **형태:** CSV 파일로 다운로드 가능.
* **내용:** 우리 AWS 계정에 존재하는 **모든 사용자**의 자격 증명 명단이 나옵니다. "누가 MFA를 안 켰는지", "누가 비밀번호를 90일 넘게 안 바꿨는지", "누가 액세스 키를 언제 마지막으로 썼는지" 일괄 확인하는 대시보드 역할을 합니다.


2. **IAM 액세스 관리자 (Access Advisor) ➔ 개별 사용자/그룹별(User-level) 정밀 분석**
* **형태:** 콘솔 화면에서 확인 가능.
* **내용:** 특정 사용자나 그룹에게 부여된 권한 중 "실제로 그 권한을 마지막으로 사용한 시간이 언제인지"를 보여줍니다. 예를 들어 어떤 직원에게 S3 접근 권한을 줬는데 최근 6개월 동안 한 번도 안 썼다면, 그 권한을 안전하게 쳐내서 '최소 권한의 원칙'을 실현하는 데 도움을 줍니다.

## 🧾 IAM Guidelines & Best Practices (설계 모범 사례)

* 루트 계정은 처음에 사용자 만들 때 빼고 봉인하세요.
* **1인 = 1개의 IAM 사용자**를 가집니다. 절대 하나의 계정을 공유하지 마세요.
* 개개인에게 권한을 주지 말고 **사용자를 그룹에 넣고, 그룹에 정책을 부여**하여 관리하세요.
* 강력한 비밀번호 정책을 걸고, 무조건 **MFA**를 활성화하세요.
* EC2나 Lambda 등 AWS 서비스가 움직일 때는 무조건 IAM Role(역할)을 쥐여주세요.
* CLI/SDK 자격 증명인 **Access Key는 비밀번호와 같으므로** 철저히 격리하고 절대 공유 금지입니다.
* 정기적으로 **Credential Report**와 **Access Advisor**로 권한을 감사하세요.

### 🎯 한눈에 외우는 SAA 시험 IAM 정답 공식

1. **"EC2 인스턴스 내의 애플리케이션이 안전하게 S3 버킷에 접근해야 한다"** ➔ **Access Key 하드코딩 금지, IAM Role을 EC2에 부여**
2. **"우리 계정 내 모든 유저의 MFA 설정 상태와 비밀번호 노후도를 한눈에 보고서로 뽑고 싶다"** ➔ **IAM Credential Report 생성**
3. **"특정 개발자에게 너무 많은 과도한 권한이 부여되어 있는지 실제 사용 이력을 보고 쳐내고 싶다"** ➔ **IAM Access Advisor 확인**
4. **"정책(Policy)을 만드는데 특정 조직이나 외부 서비스가 접근할 수 있도록 문을 열어준다"** ➔ **JSON 내부의 Principal 구문 활용**
5. **"그룹 내에 또 다른 팀 그룹을 중첩해서 권한을 상속하고 싶다"** ➔ **불가능 (IAM 그룹은 사용자만 담을 수 있음)**
