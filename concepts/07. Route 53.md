# 🌐 AWS Route 53 및 DNS 아키텍처 핵심 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **호스트명을 특정 IP로 치환 + 전 세계 100% 가용성 SLA 보장 + 53번 포트 완전 관리형 Authoritative DNS** $\rightarrow$ `Amazon Route 53`
* **인터넷 공용 도메인 제어** $\rightarrow$ `Public Hosted Zone (공용 호스팅 영역)`
* **사내 망 또는 특정 가상 가상 네트워크 내부 전용 도메인 처리 (`*.internal`)** $\rightarrow$ `Private Hosted Zone (프라이빗 호스팅 영역)`
* **Zone Apex (최상위 루트 도메인, 예: `example.com`)에는 절대 바인딩 불가 + 서브 도메인 한정 네임 서버 우회** $\rightarrow$ `CNAME 레코드`
* **Zone Apex(루트 도메인) 직접 매핑 가능 + AWS 리소스 변경 자동 추적 + 무료 및 자체 헬스 체크 연동** $\rightarrow$ `Alias 레코드 (별칭 레코드)`
* **단일 엔드포인트 수동 분배 + 헬스 체크 연동 불가 + 여러 IP 설정 시 클라이언트가 무작위 무작위 선택** $\rightarrow$ `Simple Routing (단순 라우팅)`
* **신규 소프트웨어 버전 카나리(Canary) 테스트 + 각 엔드포인트별 백분율(가중치 % ) 정밀 할당 분배** $\rightarrow$ `Weighted Routing (가중치 기반 라우팅)`
* **사용자와 전 세계 AWS 리전 간의 실제 네트워크 패킷 지연을 상시 측정하여 가장 쾌적한 리전으로 유도** $\rightarrow$ `Latency-based Routing (지연 시간 라우팅)`
* **주 가상 서버(Active) 장애 시 예비 가상 서버(Passive)로 트래픽을 자동 무인 절체(Primary-Secondary)** $\rightarrow$ `Failover Routing (장애 조치 라우팅)`
* **사용자의 접속 국가/대륙/미국 주(State) 위치 정보를 식별하여 맞춤형 현지화(Localization) 웹 콘텐츠 제공** $\rightarrow$ `Geolocation Routing (지리 위치 라우팅)`
* **인프라 지리적 거리 기준 처리 + 편향값(Bias, -99~99) 조절을 통한 특정 리전 영토 트래픽 인위적 확장** $\rightarrow$ `Geoproximity Routing (지리 근접 라우팅)`
* **클라이언트의 특정 CIDR 블록(ISP 네트워크망) 접두사를 식별하여 약속된 특정 백엔드로 직결 매핑** $\rightarrow$ `IP-based Routing (IP 기반 라우팅)`
* **최대 8개의 정상(Healthy) IP 백업본을 클라이언트에 동시 반환하여 가벼운 간이 DNS 로드 밸런싱 구현** $\rightarrow$ `Multi-Value Answer Routing (다중 응답 라우팅)`
* **VPC 내부 프라이빗 IP 또는 온프레미스 장비의 가용성을 연동하기 위해 경보를 우회 감시하는 헬스 체크 기술** $\rightarrow$ `CloudWatch Alarm 모니터링 기반 헬스 체크`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 📛 1. DNS 기본 용어 및 인프라 매커니즘

* **Domain Registrar (도메인 등록대행자):** 도메인 이름(`example.com`)을 공인 기관으로부터 임대/구입 대행해 주는 주체입니다. (예: Amazon Route 53, GoDaddy 등)
* **TLD (Top Level Domain):** 도메인의 가장 우측 꼬리표인 `.com`, `.net`, `.org`, `.gov` 등을 뜻합니다.
* **SLD (Second Level Domain):** 사용자가 직접 네이밍한 `amazon`, `google` 등의 고유 명칭 영역입니다.
* **Zone File:** DNS 네임서버 내부에서 관리하는 도메인 매핑 규칙 파일셋입니다.

---

### 🌐 2. Route 53 호스팅 영역 (Hosted Zones)

* **Public Hosted Zone:** 인터넷망 전체에 도메인을 공개 선언합니다. 외부 사용자가 `www.example.com`을 쳤을 때 AWS 상의 퍼블릭 ELB나 CloudFront 등으로 트래픽을 유도할 수 있는 공용 컨테이너입니다.
* **Private Hosted Zone:** 하나 이상의 **지정된 특정 VPC 내부 공간에서만 전용**으로 활성화되는 가상 내부 도메인 영역입니다. (예: `db.company.internal`). 인터넷망에는 절대 노출되지 않으므로 사내 백엔드 인프라 보호에 필수적입니다.

---

### ⏳ 3. 레코드 통제 기술: TTL (Time To Live)

* **High TTL (예: 24시간):** 클라이언트 및 로널 DNS 리졸버 캐시에 매핑 정보가 하루 종일 상주합니다. Route 53에 쿼리를 묻는 횟수가 격감하므로 비용 비용이 절감되지만, 백엔드 IP가 변경되었을 때 전 세계에 반영되는 속도가 매우 느립니다.
* **Low TTL (예: 60초):** 변경 사항이 거의 실시간 반영되므로 유연한 인프라 제어가 가능하지만, 캐시가 자주 지워지므로 Route 53으로 들어오는 질의 트래픽이 폭증하여 **비용 과금 오버헤드**가 유발됩니다. (★단, 아래 별칭 레코드는 예외입니다.)

---

### ⚖️ 4. 면접 및 시험 필수: CNAME vs Alias (별칭) 레코드

AWS 환경에서 로드 밸런서(ELB)나 CloudFront는 고정 IP가 아닌 호스트 네임(`lb-123.amazonaws.com`)을 최종 엔드포인트로 노출합니다. 이를 내 도메인과 연동할 때의 두 가지 분기점입니다.

| 비교 아키텍처 축 | 🔗 CNAME 레코드 | 🎯 Alias (별칭) 레코드 |
| --- | --- | --- |
| **Zone Apex 지원** | **루트 도메인(`example.com`) 지정 절대 불가**<br>

<br>(서브 도메인 `www.example.com`만 가능) | **루트 도메인(`example.com`)에 직접 사용 가능 (★최대 장점)** |
| **AWS 자원 연동 성능** | 외부의 일반 호스트 네임 문자열 단순 우회 | **AWS 리소스의 IP 변경 사항을 실시간 자동 감지 및 갱신** |
| **DNS 쿼리 비용** | 외부 조회 단계를 한 번 더 거치므로 추가 요금 발생 | **AWS 핵심 리전 자원 연결 시 전면 무료(Free) 제공** |
| **TTL 제어권** | 사용자가 레코드 단위로 초 단위를 강제 필수 지정 | **사용자 지정 불가** (Route 53 내부에서 완전 자동 최적화) |
| **반환 레코드 타입** | CNAME 타입으로 반환 | **최종 대상에 맞춰 A 또는 AAAA (IP형)로 다이렉트 변환 반환** |
| **자체 헬스 체크** | 불가능 | **자체 Native Health Check 연동 지원** |
| **⚠️ 매핑 불가능 자원** | 없음 | **EC2 인스턴스의 퍼블릭 DNS 네임은 ALIAS 타겟팅 불가** |

### 📑 5. Route 53 핵심 8대 라우팅 정책 (Routing Policies)

DNS는 트래픽 패킷을 물리적으로 찢어서 보내는 장치가 아닙니다. "사용자 위치나 인프라 건강 상태에 맞춰 가장 최적의 IP/호스트명 정보를 말로 답변해 주는 리퍼 대리인"임을 인지하고 매칭해야 합니다.

[Image diagram summarizing AWS Route 53 routing policies: Simple, Weighted, Latency, Failover, Geolocation, Geoproximity, and Multi-Value]

#### ① Simple (단순 라우팅)

* 하나의 레코드 이름에 여러 개의 IP 주소를 단순 나열할 수 있습니다.
* DNS 쿼리가 들어오면 등록된 모든 주소를 던져주며, 클라이언트는 그중 임의로 하나를 임의 선택해 접속합니다.
* **⚠️ 주의점:** Route 53의 핵심 무기인 **헬스 체크(Health Check) 기능과 결합할 수 없습니다.**

#### ② Weighted (가중치 기반 라우팅)

* 레코드별로 상대적 가중치 숫자 비중을 부여합니다. (예: 레코드 A에 가중치 `70`, 레코드 B에 `30` 부여 시 트래픽은 7:3 분배). 가중치의 합이 꼭 100일 필요는 없습니다.
* **가중치 `0` 할당 메커니즘:** 특정 엔드포인트의 가중치를 0으로 세팅하면 해당 서버로의 DNS 답변을 완전히 중단시킵니다. 단, 클러스터 내부의 **모든 레코드가 전부 가중치 0이 되면, 예외적으로 모든 레코드를 다시 동일한 확률로 똑같이 반환**합니다.

#### ③ Latency-based (지연 시간 기반 라우팅)

* 사용자의 네트워크 출발지와 전 세계 AWS 리전 간의 실시간 인터넷 통신 지연 속도(Latency) 메트릭 테이블을 기반으로 작동합니다.
* 사용자와 지리적으로 가장 가깝더라도 네트워크 정체가 심하다면, **지리적으로 멀지만 지연 시간이 가장 짧은 타 리전의 인프라 엔드포인트 주소를 최우선 응답**합니다.

#### ④ Failover (장애 조치 라우팅)

* 철저한 **Active-Passive(주-예비) 이중화 구조**를 만듭니다.
* 평소에는 헬스 체크가 연동된 Primary(Active) 주소만 반환하다가, 해당 서버가 Unhealthy 상태로 전락하는 순간 Route 53이 이를 즉시 감지하여 Secondary(Passive/스탠바이 대기조) 엔드포인트 IP로 답변을 자동 전환(Failover)합니다.

#### ⑤ Geolocation (지리 위치 라우팅)

* 사용자의 **실제 IP 주소가 속한 대륙, 국가, 혹은 미국 영토 내의 특정 주(State)** 정보를 조회하여 라우팅합니다.
* 지연 시간과 무관하게 한국 사용자에게는 한국어 페이지 서버 IP를, 유럽 사용자에게는 유로화 결제가 탑재된 현지 서버 IP를 강제 매핑해야 하는 **웹 사이트 지역화(Localization) 및 콘텐츠 유통 규제 통제** 시나리오의 정답입니다. 일치하는 국가가 없을 때를 대비해 `Default` 레코드를 반드시 생성해야 합니다.

#### ⑥ Geoproximity (지리 근접 라우팅)

* 사용자의 위도/경도 위치와 AWS 리전의 지리적 위치 거리를 계산하여 가까운 곳으로 매핑합니다.
* 가장 핵심 차별점은 **편향값(Bias, -99 ~ 99) 조절 기능**입니다. 특정 리전의 편향값 수치를 양수(1~99)로 높여주면, 해당 리전이 관할하는 지리적 영토 범위가 가상으로 거대하게 확장되므로 **더 먼 거리에 있는 사용자 트래픽까지 인위적으로 끌어와 흡수**할 수 있게 됩니다. (Route 53 **Traffic Flow** 기능 활성화 필수)

#### ⑦ IP-based (IP 기반 라우팅)

* 사전에 사용자가 정의한 클라이언트의 **공인 CIDR IP 블록 목록**을 Route 53에 통째로 등록해 두고 작동합니다.
* 특정 거대 기업 사내 망 IP 대역이나 특정 ISP(인터넷 서비스 제공업체)망을 타고 진입하는 최종 사용자 트래픽만 콕 집어서 전용 엔드포인트로 직결 유도함으로써 네트워크 비용을 절감하거나 성능 최적화를 도모합니다.

#### ⑧ Multi-Value Answer (다중 응답 라우팅)

* 단순 라우팅과 유사하게 다수의 엔드포인트를 등록하지만, **반드시 헬스 체크 기능과 1:1 결합**되어 작동합니다.
* 쿼리 요청 시 Route 53은 헬스 체크를 통과한 **"정상 상태(Healthy)의 레코드 중 최대 8개만 솎아내어"** 클라이언트에 동시 반환합니다. 클라이언트는 이 8개 중 하나를 골라 접속하므로 간이 DNS 로드 밸런싱이 완성됩니다. (★단, 헬스 체크 기능이 추가된 것일 뿐 하드웨어 수준의 전문 ELB 로드 밸런서를 완전히 대체할 수는 없습니다.)

### 🩺 6. Route 53 헬스 체크 (Health Checks) 아키텍처 정밀 제어

[Image diagram of Route 53 global health checkers probing a public endpoint versus calculating private metrics via CloudWatch alarms]

#### 🌐 ① 공용 엔드포인트 모니터링 (Monitor an Endpoint)

* 전 세계에 분산 배치된 약 15개의 글로벌 AWS 헬스 체커(Health Checkers)가 공용 인터넷망을 타고 대상 URL을 주기적으로 찌릅니다.
* **합의 판정:** 전체 글로벌 체커 중 **최소 18% 이상**이 대상 엔드포인트를 "정상"으로 응답해야 Route 53 최종 가용성 판정이 `Healthy`로 유지됩니다.
* **통과 기준:** 대상 서버가 프로토콜 규격(HTTP, HTTPS, TCP)에 맞춰 **표준 정상 상태 코드인 2xx 또는 3xx**를 뱉어야 통과합니다. 추가로 응답 데이터의 **초반 5120바이트 내부 문자열 문맥 검사(String Match)** 규칙을 걸어 정밀 검증할 수도 있습니다.
* **방화벽 주의점:** 이 글로벌 조사가 원활히 통과하려면, 대상 서버 전면의 인프라 방화벽/보안 그룹(Security Groups) 단에서 Route 53 헬스 체커 표준 수신 IP 대역의 인바운드 진입을 무조건 허용(Allow)해 두어야 합니다.

#### 🧮 ② 계산된 상태 확인 (Calculated Health Checks)

* 여러 하위 헬스 체크 항목들을 논리 연산자(**AND, OR, NOT**) 조합으로 묶어 하나의 마스터 상위 상태 레코드로 통합 관리하는 기능입니다.
* **최대 256개의 하위 상태 확인을 동시 모니터링**할 수 있으며, "하위 항목 중 최소 3개 이상이 통과하면 상위 마스터 웹사이트는 정상 유지 상태로 판정한다" 형태의 유연한 복합 예외 예외 처리가 가능합니다.

#### 🔒 ③ 프라이빗/온프레미스 연동 헬스 체크 (Calculated via CloudWatch Alarms)

* **⚠️ 아키텍처 제약 및 극복 (시험 단골):** Route 53 헬스 체커는 기본적으로 VPC 외부 퍼블릭 인터넷 공간에 상주하므로, **VPC 내부의 내부 프라이빗 IP 자원이나 온프레미스 전용선 내부 서버에는 직접 핑(Ping)을 날릴 수 없습니다.**
* **해결 시나리오:** 대상 프라이빗 자원의 메트릭(예: 내부 인스턴스 CPU, DynamoDB 쓰로틀링 상태, 커스텀 인프라 알람 지표)을 기반으로 **CloudWatch 경보(Alarm)를 먼저 생성**합니다. 그 후 Route 53 헬스 체크 옵션을 **"CloudWatch 경보 상태 모니터링"** 모드로 생성하여 바인딩하면, 프라이빗 공간에 단 1바이트의 외부 인터넷 포트를 개방하지 않고도 완벽한 내부 자원 DNS 자동 장애 조치(Failover) 아키텍처를 구현해 낼 수 있습니다.
