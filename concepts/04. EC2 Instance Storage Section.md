# 💾 AWS 블록 스토리지(EBS) 및 분산 파일 시스템(EFS) 핵심 마스터 치트키

## 🎯 1단계: 한 눈에 보는 핵심 키워드 매칭

시험 문제 시나리오에 아래 핵심 단어가 등장하면 고민하지 말고 바로 정답을 매칭하세요!

* **특정 AZ 국한 가상 가상 드라이브 + 단일 인스턴스 1:1 매핑 + 크기 기반 정액 과금** $\rightarrow$ `EBS 볼륨 (Elastic Block Store)`
* **비용 절감(최대 75%)을 위해 백업본을 보관하나 복원에 24~72시간이 소요되는 장기 아카이브 백업** $\rightarrow$ `EBS Snapshot Archive`
* **첫 마운트 시 데이터 초기화 지연(I/O Penalty)을 원천 차단하기 위해 추가 비용을 지불하고 즉시 고성능 복원** $\rightarrow$ `Fast Snapshot Restore (FSR)`
* **OS, 패키지 구성, 모니터링 세팅을 완전 동결 빌드하여 부팅 및 오토스케일링 확장 속도 극대화** $\rightarrow$ `AMI (Amazon Machine Image)`
* **물리적 호스트 서버 직결 디스크 + 임시 스크래치 데이터/버퍼 캐시 + 인스턴스 중지 시 데이터 완전 영구 증발** $\rightarrow$ `EC2 Instance Store`
* **미션 크리티컬 DB 워크로드 + 16,000 IOPS 이상의 대규모 연산 스펙 독립 구성** $\rightarrow$ `io1 / io2 Provisioned IOPS SSD`
* **동일 AZ 내 최대 16대 EC2 동시 R/W 마운트 + 공유 클러스터 인식 파일 시스템(Oracle RAC, Teradata)** $\rightarrow$ `EBS Multi-Attach (다중 연결)`
* **EBS 볼륨 데이터를 암호화하기 위해 거치는 우회 프로세스** $\rightarrow$ `1) 스냅샷 생성 ➡️ 2) 스냅샷 암호화 복사 ➡️ 3) 복사본으로 새 볼륨 생성`
* **Multi-AZ 다중 가상 가상 서버 동시 마운트 + Linux 전용(POSIX/NFSv4) + 사용한 용량만큼만 후불 청구** $\rightarrow$ `Amazon EFS (Elastic File System)`
* **예측 불가능한 워크로드의 파일 입출력 + 읽기 최대 3GiB/s 및 쓰기 최대 1GiB/s의 자동 처리량 조정** $\rightarrow$ `EFS Elastic Throughput Mode`
* **자주 보지 않는 공유 파일 데이터의 보관 비용을 최대 90% 이상 획기적으로 낮추는 인프라 조합** $\rightarrow$ `EFS One Zone-IA (단일 AZ + 간헐적 액세스 계층)`

## 📑 2단계: 서비스별 100% 디테일 분석 및 핵심 개념

### 💾 1. Amazon EBS (Elastic Block Store) 아키텍처

EBS는 가상 가상 서버(EC2)에 장착하여 인스턴스가 파괴되어도 데이터를 영구히 유지해 주는 "네트워크 기반 가상 USB 스틱(블록 스토리지)"입니다.

#### ⚙️ 인프라 제약 조건 및 속성

* **AZ 바인딩 가이드라인:** EBS는 생성 시 특정 가용 영역(AZ)에 완전히 잠금(Lock)됩니다. `us-east-1a`에 존재하는 EBS 볼륨은 `us-east-1b`에 있는 EC2 인스턴스에 다이렉트로 장착할 수 없습니다. 타 AZ로 드라이브를 옮기려면 **반드시 스냅샷 생성 후 해당 타 AZ에서 복원**하는 단계를 밟아야 합니다.
* **과금 구조의 차이:** 드라이브 안에 데이터를 1GB만 채웠든 100GB를 채웠든 관계없이, 초기에 **프로비저닝(할당) 설정한 총 드라이브 공간 크기를 기준으로 정액 요금이 매 시간 청구**됩니다.
* **인스턴스 종료 시 삭제(Delete on Termination) 속성:**
* **루트(Root) 볼륨:** 기본적으로 인스턴스 해체 시 함께 완전 자동 삭제되도록 기본 설정(`Enabled`)되어 있습니다. (설정 해제 가능)
* **추가 장착 볼륨:** 인스턴스를 파괴해도 데이터 유실을 막기 위해 기본적으로 자동 삭제 기능이 꺼져(`Disabled`) 있습니다.



#### 📸 EBS 스냅샷(Snapshot)의 고급 기능

* **EBS Snapshot Archive:** 스냅샷 데이터 중 장기 보관이 필요하나 자주 열어보지 않는 백업본을 **최대 75% 저렴한 아카이브 티어**로 격리 이전하는 기능입니다. 다만, 급하게 데이터 복원이 필요할 때 **최소 24시간에서 최대 72시간의 대기 지연 시간**이 강제됩니다.
* **Recycle Bin (휴지통):** 관리자의 실수로 스냅샷이 영구 삭제되는 대참사를 방지하기 위해 최소 1일에서 최대 1년까지 가상 보존 기간 규칙을 지정하여 안전하게 되살릴 수 있는 완충 구역입니다.
* **Fast Snapshot Restore (FSR):** 대용량 스냅샷을 기반으로 새 EBS 볼륨을 복원하면, 네트워크 드라이브 특성상 첫 블록 읽기 시 지연(I/O Penalty)이 발생합니다. FSR은 추가 비용을 지불하고 **스냅샷의 전체 초기화를 강제로 선행 처리하여 마운트 즉시 백프로 성능을 보장**하는 초고속 복원 솔루션입니다.

### 💽 2. 임시 하드코어 스토리지: EC2 Instance Store

네트워크를 경유하는 EBS와 달리, EC2 인스턴스가 가동 중인 **실제 물리 호스트 블레이드 서버 본체에 박혀 있는 고성능 물리 디스크**를 직접 할당받아 쓰는 블록 스토리지입니다.

* **초고성능 I/O:** 가상 가상 네트워크 대역을 통하지 않으므로 디스크 입출력(IOPS) 지연이 거의 없는 압도적인 하드웨어 원본 속도를 제공합니다.
* **⚠️ 휘발성(Ephemeral)의 치명적 한계:** 가상 서버를 **중지(Stop), 최대 절전 모드(Hibernate), 또는 종료(Terminate)하는 순간 인스턴스 스토어 안의 모든 블록 데이터는 즉시 완전 초기화(Reset)되어 복구가 불가능**합니다. (단, OS 내부 소프트웨어 재부팅 시에는 유지됩니다.)
* **관리의 주체:** 물리 호스트 컴퓨터의 하드웨어 결함 발생 시 데이터가 즉각 유실되므로, 중요 데이터의 백업 및 분산 복제 아키텍처 구성 책임은 전적으로 사용자(개발자)에게 있습니다. **임시 버퍼, 캐시 스토어, 스크래치 데이터(임시 가공 데이터)** 용도로만 매핑해야 합니다.

### 💿 3. EBS 볼륨 유형(Volume Types) 정밀 비교

EBS 볼륨은 크게 SSD 계열(부팅 가능)과 HDD 계열(부팅 불가능)의 2대 진영, 총 6개 세부 유형으로 쪼개집니다.

| 볼륨 유형 | 부팅 여부 | 핵심 메커니즘 및 스펙 기준 | 최적 추천 비즈니스 워크로드 환경 |
| --- | --- | --- | --- |
| **gp3** (General Purpose) | **가능** | 비용과 성능의 완벽한 균형.<br>

<br>**기본 3,000 IOPS / 125 MiB/s 무료 제공** | 시스템 부팅 디스크, 가상 데스크톱(VDI), 일반적인 개발 및 테스트 환경 |
| **gp2** (General Purpose) | **가능** | **볼륨 크기와 IOPS 성능이 강제 연동됨.**<br>

<br>(GiB당 3 IOPS 비율, 최대 16,000 제한) | 구형 아키텍처 범용 스토리지<br>

<br>(크기가 작으면 IOPS 성능도 처참함) |
| **io1 / io2** (Provisioned IOPS) | **가능** | **스토리지 크기와 완벽히 독립적으로 IOPS 지정.**<br>

<br>Nitro 기종 최대 64,000 IOPS 지원 | 미션 크리티컬 대형 RDBMS / NoSQL,<br>

<br>일관된 저지연 고성능이 필수인 핵심 DB |
| **io2 Block Express** | **가능** | 밀리초 미만(Sub-millisecond)의 극초저지연.<br>

<br>**최대 256,000 IOPS 폭발적 성능** | 엔터프라이즈급 초대형 핵심 메인프라 DB, EBS 다중 연결 인프라 선도 |
| **st1** (Throughput Optimized) | 불가 | 저비용 대용량 HDD.<br>

<br>**최대 처리량 500 MiB/s 중심 구조** | **연속적인 대량 데이터 입출력 필수 워크로드**<br>

<br>(빅데이터 대량 연산, 데이터 웨어하우스, 대용량 로그 분석) |
| **sc1** (Cold HDD) | 불가 | **AWS 블록 스토리지 중 비용 최저.**<br>

<br>최대 처리량 250 MiB/s 제약 | **접근 빈도가 극도로 낮은 데이터 보관**<br>

<br>(규정 규제용 장기 아카이브 파일 데이터 보관용) |

#### 🧲 EBS Multi-Attach (다중 연결 스펙)

* **대상 모델:** 오직 최고 성능 라인업인 **io1 / io2 패밀리**에서만 활성화할 수 있는 특수 기능입니다.
* **동작 원리:** 원래 EBS는 1:1 연결이 원칙이지만, 이 기능을 켜면 **동일 AZ 내에 존재하는 최대 16대의 EC2 인스턴스가 단 하나의 고성능 EBS 볼륨에 동시 마운트**되어 동시 읽기/쓰기 권한을 행사합니다.
* **⚠️ 아키텍처 주의점:** 데이터 오염을 막기 위해 가상 인스턴스들이 동시 쓰기 경합을 제어할 수 있는 클러스터 인식 파일 시스템(Cluster-aware File System - 예: Oracle RAC, Teradata 전용 파일 아키텍처)을 OS 내부단에서 필수로 구축해 써야 합니다. 일반적인 XFS, EXT4 파일 시스템은 다중 연결 환경에서 데이터가 완전히 깨지므로 사용이 불가능합니다.

### 🔒 4. EBS 암호화(Encryption) 프로세스 전환 기법

EBS 암호화를 활성화하면 AWS KMS(Key Management Service)의 AES-256 키 표준을 기반으로 **볼륨 내부 저장 데이터(Data at rest), EC2와 볼륨 간의 네트워크 전송 데이터(Data in transit), 파생된 모든 스냅샷 및 스냅샷 복원 볼륨**까지 동시 다발적으로 무중단 암호화 자동 처리가 수행됩니다. 성능 지연(Latency) 오버헤드는 사실상 제로에 가깝습니다.

#### 🔒 암호화되지 않은 기존 라이브 EBS 볼륨을 안전하게 암호화하는 4단계 공정 (시험 단골)

1. **`스냅샷 촬영`:** 암호화되지 않은 원본 EBS 볼륨의 특정 시점 백업 **스냅샷을 먼저 생성**합니다. (이때 생성된 스냅샷은 여전히 평문(암호화 X) 상태입니다.)
2. **`암호화 복사 (★핵심 단계)`:** AWS 스냅샷 복사(`Copy Snapshot`) 기능을 실행하면서, **"암호화 활성화" 체크박스를 선택하고 매핑할 KMS 암호화 키를 지정**합니다. 이 과정을 거치면 완벽하게 암호화 정렬이 완료된 "새로운 암호화 스냅샷 복사본"이 탄결됩니다.
3. **`볼륨 복원`:** 해당 암호화 스냅샷 복사본을 기반으로 **새로운 EBS 볼륨을 생성**합니다. 이 볼륨은 태생부터 암호화 속성을 상속받아 생성됩니다.
4. **`인스턴스 재장착`:** 기존 평문 볼륨을 EC2에서 탈착(`Detach`)해 버리고, 새로 빌드한 암호화 완료 EBS 볼륨을 원본 인스턴스에 최종 바인딩(`Attach`)하여 안전하게 서비스를 재개합니다.

### 📁 5. Amazon EFS (Elastic File System) 정밀 아키텍처

EFS는 여러 가용 영역(Multi-AZ)에 흩어져 있는 수백, 수천 개의 EC2 가상 서버들이 네트워크 파일 시스템 표준 프로토콜(**NFSv4.1**)을 통해 동시에 달라붙어 데이터를 공유할 수 있게 해주는 "완전 관리형 대용량 Linux 분산 파일 시스템(POSIX 표준 준수)"입니다.

#### ⚙️ 성능 모드 및 처리량 제어 엔지니어링

* **Performance Mode (성능 모드):**
* **General Purpose (범용 모드 - 기본값):** 파일 입출력당 반응 속도(지연 시간)가 가장 짧게 유지됩니다. 일반적인 웹 서버 호스팅, 콘텐츠 관리 시스템(CMS), 워드프레스 공유 스토리지에 무조건 선택합니다.
* **Max I/O:** 대형 병렬 처리가 중심입니다. 지연 시간은 다소 늘어나지만, 초당 처리량 대역폭과 극도의 병렬 쓰기 스펙을 극대화합니다. 대규모 빅데이터 연산이나 미디어 인코딩 파이프라인에 매핑합니다.


* **Throughput Mode (처리량 모드):**
* **Bursting Mode:** 파일 저장 용량 크기에 비례하여 기본 대역폭이 주어지며, 필요시 순간 버스트 대역폭을 소모하는 기본형 모드입니다.
* **Provisioned Mode:** 파일 저장 용량이 단 1GB에 불과하더라도, 스토리지 크기와 상관없이 내가 원하는 초당 처리 대역폭(MiB/s)을 돈을 더 내고 독립적으로 고정 확보하는 모드입니다.
* **Elastic Mode (★자동화 추천):** 예측 불가능한 스파이크성 비즈니스 트래픽에 맞춰 **읽기 최대 3GiB/s, 쓰기 최대 1GiB/s 범위 내에서 처리 대역폭을 완전 자동 확장/축소**해 주는 비용 효율적 모드입니다.



#### 🗂️ 비용 최적화를 위한 2대 축: 스토리지 계층 및 가용성 옵션

* **수명 주기 관리(Lifecycle Management):** 지정한 일정 기간(예: 30일) 동안 한 번도 열어보지 않은 파일 데이터를 **Standard 계층**에서 비용이 훨씬 저렴한 **Infrequent Access (EFS-IA) 계층**으로 완전 자동 이동시켜 스토리지 유지 비용을 극적으로 낮춥니다.
* **가용성 설계 전략:**
* **Standard (Multi-AZ):** 데이터를 여러 가용 영역에 분산 복제하여 프로덕션 환경의 고가용성을 완벽 보장합니다.
* **One Zone (단일 AZ):** 오직 단 하나의 가용 영역 내부에서만 데이터를 유지합니다. **개발/테스트 환경에 최적화되어 있으며 가용성 스펙을 타협하는 대신 전체 스토리지 비용을 무려 90% 이상 초대형 절감**할 수 있습니다. 기본적으로 파일 유실을 막기 위한 백업 기능이 강제 활성화됩니다. (One Zone-IA 계층과 결합하여 비용 절감 극대화 가능)
